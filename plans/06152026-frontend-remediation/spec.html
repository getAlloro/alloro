Why

The backend just completed a 7-spec remediation (security, transactions, test harness, validation, logging, db-into-models, god-file decomposition) and now passes a mechanized convention gate. The frontend (frontend/, ~164K LOC across 624 files) has never been held to any codified standard. Measured against the backend's own bars it carries the same debt the backend just cleared: 38 files over the 800-line hard ceiling (worst: IdentityModal.tsx at 3,410), 35 files bypassing the shared api/index.ts client with hand-rolled fetch/auth, an inconsistent error contract (swallow vs {successful:false} vs throw), 155 any usages despite strict mode, and ~301 console.* calls. The code-constitution skill now defines the target standard for the frontend; this plan executes the codebase to meet it.

What

A phased, behavior-preserving refactor that brings frontend/src to the constitution's frontend standard, ordered by risk. Done = the mechanized frontend tally in npm run check:conventions is driven to zero (files>800, console files, fetch-bypass, any), tsc -b and eslint are clean, a new frontend test harness is green, and every touched surface passes manual QA with no behavioral regression. This is a refactor, not a redesign — pixels and flows stay identical; only the code structure changes.

Context

Reference analog

This mirrors the completed backend remediation playbook: plans/06142026-* and the execution memory project_backend_remediation_execution. Same discipline — test harness first, mechanical/safe themes before stateful surgery, behavior-preserving extraction, commit-per-unit, gated by tsc + tests + the convention checker.

Relevant files / surfaces

  • frontend/src/api/index.ts — the shared HTTP client (apiGet/apiPost/..., getCommonHeaders). The consolidation target; the one place that reads the JWT.
  • frontend/src/api/*.ts (43 files) — domain API modules; 16 still hand-roll fetch.
  • frontend/src/hooks/queries/ — React Query hooks (95 useQuery + 74 useMutation call sites). The server-state standard.
  • frontend/src/lib/toast — the shared toast; the error-surfacing target.
  • frontend/src/components/Admin/ — flat dump of ~99 files; the folder-nesting target.
  • The 38 god-files (>800), worst-first: IdentityModal.tsx 3410, pages/admin/PracticeRanking.tsx 2532, api/websites.ts 2481, pages/admin/WebsiteDetail.tsx 2444, PMS/PMSManualEntryModal.tsx 2208, pages/admin/PageEditor.tsx 2126, …

Patterns to follow

  • Already-good, codify as-is: centralized axios client + getCommonHeaders auth; React Query for server state; Context for UI state (the .ts/.tsx split is intentional); lib/toast; strict TS + eslint flat config.
  • Decompose like the backend: data-fetching → use<Feature>() hook; pure helpers → sibling *.utils.ts; sub-views → child components; keep import surfaces stable via re-export where consumers exist.

Constraints

Must

  • Be behavior-preserving — identical UI, identical flows.
  • Land the test harness (Phase 0) before any stateful god-file decomposition (Phase 3).
  • Commit per unit; gate each with tsc -b + eslint + harness + the frontend convention tally.
  • Do high-blast-radius work (F4 error-contract) per-domain, not all at once.

Must Not

  • Mix visual redesign into this refactor (that is alloro-frontend/impeccable territory — separate plan).
  • Introduce a new state-management library or new runtime deps (test-only deps for Phase 0 excepted).
  • Change backend code or API shapes.
  • Big-bang all 38 god-files in one pass.

Out of scope

  • Visual/design changes, new features, accessibility overhaul, performance tuning beyond what falls out of decomposition.
  • Backend changes.

Risk

Level: 4 — cross-cutting, the entire frontend is the blast radius, and there is currently no behavioral safety net.

Risks identified

  • No test harness today. tsc/eslint catch type/lint errors, not behavioral regressions in stateful components. Mitigation: Phase 0 builds Vitest + React Testing Library smoke/interaction tests on the top god-files and critical flows before any decomposition.
  • Error-contract inversion (F4) is the highest-blast-radius change. Every call site assumes today's swallow/return behavior; flipping helpers to throw can surface unhandled errors UI-wide. Mitigation: do it per-domain, with React Query error surfaces + lib/toast in place first; QA each domain.
  • Stateful extraction bugs (stale closures, effect ordering, render loops) in the 38 god-files. Mitigation: worst-first, one unit per commit, harness + manual QA each; lean on F1/F3 to shrink files first.
  • Import-surface breakage from folder moves (F5) and splits. Mitigation: re-export shims where consumers exist; rely on tsc -b to prove resolution.

Blast radius: all admin pages, all client surfaces (Dashboard, Referrals Hub, Local Rankings, Reviews & Posts, Website). Every theme touches code on a live screen.

Pushback

  • Sequencing is the whole game. Doing god-file decomposition before a harness is the wrong order — it maximizes regression risk with no net. Recommended: Phase 0 (harness) → Phase 1 (mechanical, tsc/lint-safe) → Phase 2 (medium) → Phase 3 (stateful surgery).
  • Phase 1 first also shrinks the hard problem. Consolidating fetch (F1), eliminating any (F3), and stripping console (F2) pull volume out of the god-files before F6 starts.
  • Recommend executing one phase at a time with a checkpoint between phases rather than approving the whole program up front — the god-file work especially should be re-scoped after Phase 1 lands.
