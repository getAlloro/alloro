A3 — AI-Answer Visibility (AEO) Observation

Get-found, in-lane. A multi-engine observation log: run a practice's category+geo prompt set against AI answer engines on a cadence and record whether the practice is mentioned/cited — per engine, over time. Observe-only. Never a rank, never a guarantee. Built as a system (engine-adapter interface), Gemini live first.

Plan: 07152026-aeo-visibility-observation  ·  Funnel: Stage A (get found), item A3  ·  Science (do not re-derive): alloro-artifacts/research/aeo-measurement-spec.md + specs/aeo-build-spec.md
Lane: Corey+Claude spec → sandbox build + adversary → PR to dev/dave for Dave
Evidence: 🔴 EMERGING · MEASUREMENT-ONLY — rank inside AI answers is <1/100 repeatable; local AI ~1–7% of query volume. Observe, never promise.

Why

A growing share of "who's a good [x] near me" now happens inside AI answers. Alloro is totally blind to that surface today (verified: connect-lever-audit.md — grep AEO/answer-engine/perplexity/chatgpt = 0 code). A3 ends that blindness the only honest way: observe whether the practice is in the set of businesses the AI names, per engine, over time. It never claims a rank or a placement.

Feasibility de-risked live (2026-07-15): a grounded Gemini query ("good endodontists in Austin") named real, specific practices (McGuire Endodontics, Edge Endodontics), returned groundingMetadata with the live search queries + citation sources. So grounded Gemini is a real, representative-enough observation surface — labeled a proxy, not "what your customers see."

What — in scope (the SYSTEM, not one engine)

Reuse first (verified against code + the existing science spec).

  • The method is already specced — research/aeo-measurement-spec.md §4 (prompt-set runner → observation log, engine adapters, honesty flags). Build to it; do not re-derive.
  • Audit-service shape — mirror src/services/ai-seo-audit/auditTargetExecutionService.ts + auditPersistenceService.ts ("build targets → execute against an external source → persist"; run/target/result models, tenant-scoped QueryContext). Same shape.
  • Gemini client pattern — src/controllers/admin-os/feature-services/service.os-llm.ts (the @google/genai seam + the injected fake-provider test pattern). The Gemini adapter reuses this so it is testable without a live key.
  • Practice identity — PracticeFactModel / existing location data for the practice name/domain/GBP the detector matches on. No new source.
  • Category + geo — the schema-type allowlist (util.schema-business-type.ts) + location data already give category + city.

The system

  • Engine-adapter interface — AiVisibilityEngineAdapter.query(prompt, identity) → { mentioned, cited, cited_source, position, capture_method, raw_excerpt }. One interface; each engine is a pluggable adapter carrying a mandatory capture_method flag (api_grounded | api_proxy | serp_scrape). Adding an engine = write one adapter, never a rebuild.
  • Prompt-set builder — from category+geo, a small fixed set: generic ("best [category] in [city]") + one contextualized variant (per the Clovion finding). Curated + stored, not "everything".
  • Appearance detector (deterministic) — does the practice name/domain/GBP appear in the answer text OR the grounding-source titles? Record mentioned, cited, cited_source, position (stored raw, never surfaced as "rank"), raw_excerpt. Keep the raw answer (anti-fabrication).
  • Observation log — net-new table ai_visibility_observation (org/location, prompt, engine, capture_method, mentioned, cited, cited_source, position, raw_excerpt, observed_at). Idempotent insert per (location, prompt, engine, run-date). A log, never a score.
  • Runner — for a location, run the prompt set against every configured adapter, detect, persist. Callable now; the weekly scheduled cadence rides the existing scheduled-job pattern (thin follow-on).

The engines

  • Gemini (grounded) — LIVE + tested. Uses the googleSearch tool (confirmed in @google/genai 1.43; de-risk run named real practices + returned citations). capture_method: api_grounded. Citations come back as Google redirect URLs → the detector reads groundingChunks[].web.title for the source; citation-source capture is therefore partial (flagged honestly).
  • Perplexity Sonar + Google AI Overview (SerpApi) — WRITTEN, KEY-GATED, NOT live-verified. Real adapters conforming to the interface, but each activates only when its API key env var is present and is flagged "needs a live smoke-test on first key." The runner skips an adapter with no key, so nothing unverified fires. This is the "wire both, one key away" option (Corey, 2026-07-15) — the system supports them now; procurement + a smoke-test turn them on, on their own cadence.

What — explicitly NOT in A3

  • No rank number, ever. Rank inside AI answers is <1/100 repeatable. No "your AI rank #3".
  • No derived score yet. Ship the raw observation log first; a score is derived only once real per-engine time-series exists (measurement spec §3).
  • No customer-facing surface in this slice — operator/admin-first.
  • No guarantee / placement promise (Value #6). No premium AEO tier. Perplexity/SerpApi are not claimed as working until smoke-tested with a key.

Constraints

Honesty caps (binding, from the AEO specs + Value #6). Observe mention/citation, never rank · per-engine, never one collapsed number · every reading carries the churn caveat (40–60% of AI-cited domains vanish month-to-month) · every reading carries a mandatory capture_method and an API reading is labeled a proxy ("an API sample, not the exact answer a customer sees") · never a KPI Alloro moves, never a guarantee. The output is an observation, framed as a weather report.

Anti-fabrication. Record only what the engine actually returned (keep raw_excerpt). Never infer a mention that isn't in the text/citations. position is stored raw for analysis but NEVER rendered as a rank.

Sandbox-safety (this build writes). The migration is additive (new table) + reversible (tested down). The Gemini call is a read-only external query (no human/wallet side effect beyond trivial API cost); no Stripe/email/webhook. Perplexity/SerpApi adapters are dormant without keys. Prove DB paths with a rolled-back transaction in the sandbox.

  • Untested external adapters are key-gated + flagged — Perplexity/SerpApi ship only as code, run only with a key, and carry a "needs live smoke-test" marker; the runner skips them when unconfigured.
  • Not a twin of the LLM audit — the GBPAnalysis.md pillar scores the practice's own signals; A3 observes external AI answers. Different job.
  • Edge cases: engine returns no answer / times out → record nothing (or a null observation), never a fabricated miss; practice name ambiguous (common word) → require domain/GBP corroboration before marking cited; grounded Gemini returns redirect URLs → read web.title, flag citation-source capture partial; no location category/geo → skip (can't build a prompt).
  • Lane: Corey+Claude author; build+adversary in the sandbox; PR to dev/dave; Dave reviews+merges. Never push to dev/dave/main directly.

Risk

  • API ≠ UI. A grounded-Gemini reading is not the exact Google AI Overview a user sees. Mitigation: label every reading capture_method + proxy caption; never claim "what your customers see."
  • Low surface / emerging. Local AI is ~1–7% of queries. Mitigation: frame as watch-the-wave observation, not a headline lever; no over-claim in any copy.
  • Untested Perplexity/SerpApi adapters. Mitigation: key-gated + skip-when-unconfigured + "needs smoke-test" flag; only Gemini is verified end-to-end in this PR.
  • Directional hardening into a promise. Mitigation: the honesty caps are enforced in the output shape (no rank field surfaced; churn caveat attached); tests assert no rank/guarantee language.

Rev 2 — 2026-07-15

Change: Fixed after the build's fresh-adversary pass. (1) Anti-fabrication (adversary CRITICAL/HIGH): the detector used substring matching, so a competitor's lookalike name ("Smile Dental" inside "Smile Dental Group") or lookalike domain ("smiledental.com" inside "bestsmiledental.com") produced a fabricated mentioned/cited — the exact failure the honesty cap forbids. Replaced with boundary-aware matching: a name match is rejected when followed by another capitalized token (a longer, different business name); cited now requires the practice's own hostname (equality or real-subdomain via a "." label boundary) in a CITATION SOURCE only — never a substring, never a domain in prose. Conservative by design (prefers a missed positive over a fabricated one). Added the adversary's inputs as regression tests. (2) Honest reporting: the runner now separates enginesRun (recorded ≥1) from failedEngines (configured but every reading failed) from skippedEngines (no key) — a totally-failed engine is no longer reported as a real run.

Reason: Adversary (alloro-proof) BROKE-IT on lookalike-name/domain fabrication + reporting integrity.

Updated Done criteria: detector matching is boundary-aware — no competitor lookalike produces a positive (test-proven); runner reports ran / failed / skipped distinctly.

Rev 3 — 2026-07-16

Change: Fixed the four findings in Dave's second-round review of PR #165 (pinned to f01358d1). (1) Error boundaries (§3.2): observationRunner.ts wrapped the provider call and the database write in ONE try/catch, so a rejected AiVisibilityObservationModel.record() was logged as "engine query failed", counted as an engine failure, and the run RESOLVED — reporting success while having stored nothing. The two boundaries are now separate, in an extracted recordOneReading() helper: a provider failure is isolated (logged warn, reading skipped, run continues, since one flaky third party must not cost the others); a persistence failure is logged error with full context and RE-THROWN, so the caller sees a real failure. (2) Citation link loss: the SerpApi adapter mapped each reference to title || link, but the detector needs a HOSTNAME — and a reference title is a headline that usually carries no domain, so a normal {title, link} citation of the practice's own site recorded cited: false and under-reported visibility. Replaced the lossy string[] with a structured EngineCitation[] ({url, title}) across all three adapters; the detector searches both fields (engines disagree about which carries the real domain — Gemini's URI is a redirect, so only its title does) and stores the field that actually proved the match, so cited_source always contains its own evidence. The adapter no longer decides which field is useful. (3) Overstated metric: the model ignored idempotent conflicts but the runner incremented observationsRecorded regardless, so a same-day re-run restated the full set as freshly captured. record() now returns whether an insert actually occurred (.returning("id") over the ignore-conflict chain), and the summary separates observationsRecorded (new inserts) from duplicateObservations from observationsProcessed. Fixing the count alone would have introduced the opposite lie — an idempotent re-run reporting every engine as FAILED — so ran-vs-failed is now decided by completed READINGS, not by inserts. (4) §20.5 artifacts: added test.html + test-results.json (21 items; 16 pass, 5 honestly pending). Every fix is mutation-verified: reverting it fails the new test and nothing else. A self-caught gap is recorded — the first cut of (3) had no model-level test, so hardcoding return true passed the entire suite; T14 was added to close it.

Reason: User QA: Dave second-round review of PR #165, items 1–4. The round-1 §11.7 read-scoping and its §20.2 cross-org test were confirmed resolved and are untouched.
