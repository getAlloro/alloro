Why

Alloro cannot currently run -tw acceptance safely because the repository has no test:worktree adapter. Feature worktrees therefore cannot prove that the browser, API, database, queues, workers, email, and external-service boundaries belong to that worktree without risking shared dev or production systems.

This is repository infrastructure, not a per-feature implementation. Once it lands, every secondary Alloro worktree can reuse the adapter with feature-specific fixture and worker selections.

What

Add a fail-closed npm run test:worktree adapter that starts a disposable local Alloro runtime tied to the verified secondary worktree and its exact HEAD. It will emit a machine-readable manifest with a unique browser origin, authenticated bootstrap URL, health URL, dependency ports, safety modes, logs, and an exact teardown command.

The first fixture profiles will be baseline and gbp-posts. The latter provides synthetic One Endodontics-like organization, location, review, Google connection, and Google property rows without real credentials or PII, plus a local Anthropic-compatible response fixture so Google-post draft acceptance can run without an external AI call.

Completion contract: two concurrent linked worktrees can start independent runtimes, authenticate locally, exercise the named browser/API/worker surfaces, capture email locally, reject non-local server requests, and tear down without touching shared services.

Design Decisions

Database

Use a disposable pgvector Postgres container. Restore a committed schema-only baseline derived from the dev database with --no-owner, --no-privileges, and no table data. Record the baseline’s applied migration names separately, then run branch-pending Knex migrations.

Fixtures

Seed only synthetic rows in one transaction. Fixture profiles are additive and explicit. The default profile is small and feature-neutral; gbp-posts adds only the rows required for Google-post acceptance.

Authentication

Generate a fresh runtime JWT after seeding. Store it in a mode-0600 runtime file. A test-only same-origin Vite middleware endpoint consumes the file and returns a one-time bootstrap page that writes the JWT to localStorage on the runtime's unique origin, deletes the bootstrap material, and redirects to /admin. This matches the existing frontend contract that reads the JWT to build the Authorization header.

Email

Start a local HTTP capture server and route the existing n8n transport to it. Worktree mode bypasses the public-IP discovery in emailInterceptor.ts but never bypasses the local-only transport restriction.

AI And External Calls

Start a local Anthropic-compatible fixture server and point the existing SDK to it through environment configuration. Apply a Node preload outbound guard that permits loopback and the runtime’s local hosts only; unknown external server requests fail visibly.

Workers

Refactor declarations into a registry. Production keeps the complete current worker and schedule set. Worktree mode starts no workers by default and accepts an explicit named allowlist; recurring schedules remain disabled.

Ports And Origin

Node services and Vite bind port 0. Docker publishes random host ports. The browser origin uses a unique *.localhost hostname based on worktree path plus HEAD, avoiding cookie collisions across runtimes.

Lifecycle

A supervisor process owns child PIDs and the Compose project. The start command exits only after writing the manifest. The manifest’s stop command terminates recorded PIDs and that Compose project only; it never kills by port.

Constraints

Must

  • Prove the current checkout is a secondary linked Git worktree before starting anything.
  • Derive runtime identity from the absolute worktree path and exact HEAD, not branch name alone.
  • Use an allowlisted generated environment; validate that no checkout .env value is copied implicitly.
  • Require Docker and Compose explicitly; fail closed with clear guidance if unavailable.
  • Use a disposable writable Postgres database and isolated Redis instance for every runtime.
  • Use the pgvector image because current migrations and schema require the vector extension.
  • Scan the committed schema baseline and fixture artifacts for data rows, credentials, owner/grant statements, and likely PII.
  • Keep all tokens out of URLs, cookies, command output, manifests, and logs.
  • Start no worker unless explicitly named, and never start recurring schedules in worktree mode.
  • Keep normal production worker behavior unchanged when worktree mode is absent.
  • Emit a versioned JSON manifest and bounded health-check results before declaring the runtime ready.
  • Tear down after success or failure unless the caller explicitly requests --keep.
  • Support the documented admin bootstrap and seeded Pilot flow into the synthetic One Endodontics fixture.

Must Not

  • Do not connect to or write to dev, production, or any shared local database/Redis service.
  • Do not fall back to fixed ports, random-number port guesses, manual increments, or killing unrelated processes by port.
  • Do not copy, source, print, or activate the repository’s .env.
  • Do not clone production data or include real user/client PII in the baseline or fixtures.
  • Do not send email, publish Google posts, call real AI providers, start Stripe writes, or enable any third-party write path.
  • Do not add a second application auth implementation, a long-lived shared test token, or a cross-origin bootstrap token.
  • Do not put test-only routes into the Express production router.
  • Do not refactor worker processor business logic while extracting the registry.
  • Do not change dashboard UI or Alloro Docs as part of this infrastructure plan.

Out Of Scope

  • Mocking every Alloro external integration in the first release.
  • Automated Google Business Profile deployment or any sandbox/provider write.
  • Production deployment of the current Google-post feature branch.
  • Replacing Vitest unit tests or the normal dev environment.
  • Building a cross-repository universal adapter; each repository owns its application-specific adapter.

Risk

Level: 4 — Major Impact

Risks Identified

  • Production worker startup can regress during registry extraction. Mitigation: preserve a complete default registry, add a parity test that names every current worker and recurring schedule, and make selective mode reachable only through an explicit test-runtime flag.
  • The Git migration chain cannot build the original core schema from an empty database. Mitigation: commit a schema-only dev baseline and applied-migration metadata, prove it contains no table data or secrets, then run only branch-pending migrations in the disposable database.
  • A broad test harness can become an unmaintainable mock of the whole platform. Mitigation: keep a small core adapter and explicit fixture profiles. Unknown integrations remain blocked until a feature adds a deterministic local fixture.
  • Bootstrap tokens can leak through URLs or logs. Mitigation: use a same-origin tokenless, one-time bootstrap URL backed by a mode-0600 file; write the token only into unique-origin localStorage required by the existing frontend auth client, then delete the file and keep all token material out of logs/manifests.
  • Docker availability can make acceptance unavailable on some machines. Mitigation: treat Docker/Compose as an explicit prerequisite and fail closed. There is no shared database or Redis fallback.
  • The outbound guard can block legitimate local SDK behavior or miss a transport path. Mitigation: centralize the local allowlist, test HTTP/HTTPS/fetch/DNS refusal paths, and record blocked destinations without request bodies, credentials, or tokens.
  • A schema snapshot can drift behind migrations. Mitigation: store provenance and applied migration names, run branch-pending migrations every startup, and add a controlled refresh command plus schema-baseline verification tests.

Pushback

  • A sanitized production-data clone is not the right first implementation. It adds PII handling, stale data, larger artifacts, and unclear consent. The recommended path is schema-only plus synthetic fixtures.
  • Reusing normal npm run dev is unsafe. It assumes fixed ports, starts the full worker fleet, uses the checkout environment, and cannot prove dependency isolation. A dedicated runtime launcher is required.
  • Editing queue prefixes alone is not enough. Separate disposable Redis instances provide the actual isolation; named worker startup controls behavior and blast radius.

Blast radius: local developer tooling, API startup configuration, Vite dev-server configuration, email interception in non-production test mode, all BullMQ worker startup declarations, queue shutdown behavior, database bootstrap fixtures, and the -tw acceptance workflow. Existing request controllers, processor business logic, production database schema, and user-facing dashboard behavior should remain unchanged.

Revision Log

Rev 1 - 2026-07-17

Change: Replaced the non-functional httpOnly-cookie-only bootstrap with a one-time same-origin bootstrap page that stores the JWT in the unique runtime origin's localStorage and deletes the private token file.

Reason: Execution deviation: the existing frontend must read the JWT to create the Authorization header and pass protected-route session checks; an httpOnly-only token would cause a login loop unless production auth architecture was expanded beyond this plan.

Updated Done criteria: Token remains absent from URLs, cookies, manifests, and logs; the token file is mode 0600, single-use, deleted after consumption, and the resulting session is isolated by the unique runtime origin.

Rev 2 - 2026-07-17

Change: Aligned disposable pgvector Postgres to the schema baseline's PostgreSQL 17 version, replaced Vite's unsupported port-0 behavior with an OS reservation-release-bind retry using strictPort, and changed fallback teardown to remove only resources carrying the exact Compose project label.

Reason: Execution deviation: real contained startup showed PostgreSQL 16 rejected a PostgreSQL 17 dump setting, Vite mapped port 0 back to 5173 and collided under concurrency, and a redundant Compose down could not re-parse startup-only environment requirements after the supervisor had already cleaned its resources.

Updated Done criteria: None; these changes make the existing PostgreSQL compatibility, OS-assigned port, concurrent-isolation, and teardown criteria executable.
