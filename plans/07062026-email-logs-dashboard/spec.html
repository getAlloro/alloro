Why

Once every email funnels through the single sendEmail choke-point, that function is the one place to record every message the app sends. The owner needs an internal-admin view to (1) confirm what was actually sent, in full rendered HTML, so email design stays consistent, (2) see each email categorized, and (3) see delivery/open status. There is no email visibility today beyond a flat email.log file that no dashboard reads.

What

A new email_logs Postgres table written at the sendEmail choke-point; a Mailgun event webhook that flips rows to delivered/opened/bounced; and an internal-admin dashboard page (/admin/email-logs) that lists logs with category/status/date filters and renders the full sent HTML in a sandboxed viewer. Viewer only — no template editing (templates stay in code). Done when: a real email sent on dev appears in the dashboard with its category, status, and exact rendered HTML, and a Mailgun delivered/opened event updates its row.

Owner decisions recorded (grill, 2026-07-06)

  • Tracking: all-in — sent/failed (free) + delivered + opened (new Mailgun event webhook; open-tracking enabled). "Opened" is labeled best-effort/unreliable in the UI.
  • Retention: indefinite full HTML. Owner explicitly owns the compliance/BAA call — see Risk (Level 4).
  • Access: internal admin only via the existing superAdminMiddleware domain gate (@getalloro.com).

Constraints

Must

  • Log at the sendEmail choke-point only — never scatter inserts across callers.
  • Logging is best-effort and non-blocking: a logging failure must never fail or delay the actual send (wrap in try/catch, do not throw).
  • Render stored HTML in a sandboxed <iframe srcdoc> (no allow-scripts, no allow-same-origin) — stored HTML is untrusted (§17.4).
  • Mailgun event webhook verifies the HMAC signature before trusting any event (§11); rate-limited; idempotent (§21.1).
  • All DB access through a new EmailLogModel (§7.4); response contract { success, data, error } (§8.1).
  • Dashboard gated by superAdminMiddleware (backend) and it lives under /admin/* behind AdminGuard (frontend).

Must Not

  • No email template editor — viewer only.
  • No new send path — this rides the existing single sendEmail; do not add a second logger.
  • Do not block the request path on the Mailgun webhook or on logging.
  • Do not expose email bodies on any non-internal route.
  • No any; no raw fetch in components; no dangerouslySetInnerHTML for the body.

Out Of Scope

  • Editing/authoring templates from the UI.
  • Per-tenant (client-facing) email logs — this is internal-admin only.
  • Retention/purge automation (owner chose indefinite; no purge job built).
  • Click-tracking analytics beyond the raw delivered/opened status.

Risk

Level: 4 (Major Impact) — a new permanent, queryable store of cross-tenant PHI, plus a new public webhook endpoint, plus a change to the shared send choke-point.

Risks Identified

  • PHI retained indefinitely (ACCEPTED, owner-owned). Email bodies to dental/ortho practices can contain PHI (patient names in referral emails, patient messages in contact forms). Storing full HTML forever creates a standing HIPAA/BAA liability. Owner explicitly accepted and owns the compliance/BAA decision (grill 2026-07-06). Mitigation / execution gate: internal-only access; stored HTML never leaves the internal admin surface; execution requires the owner's written compliance sign-off recorded here before T1 runs. Recommended (not selected): metadata-forever + purge-HTML-after-90d.
  • New public webhook (Mailgun events). Unauthenticated inbound endpoint. Mitigation: HMAC signature verification against MAILGUN_WEBHOOK_SIGNING_KEY (validated at startup §5.6), rate-limited (§11.3), idempotent updates keyed on provider_message_id (§21.1), no reflection of input.
  • Choke-point change on every email. A bug in the insert could slow or break all sending. Mitigation: logging is post-send, try/catch, never throws, never awaited in a way that blocks failure handling; a logging error is logged via Pino and swallowed.
  • Stored-HTML XSS in the admin viewer. Rendering attacker-influenced HTML (e.g. contact-form content) could run script in the admin origin. Mitigation: sandboxed iframe srcdoc with no allow-scripts/allow-same-origin (§17.4).
  • Open-tracking is domain-wide + unreliable. Enabling opens on Mailgun rewrites links and injects a pixel into ALL email.getalloro.com mail, and reads ~50% low. Mitigation: label "opened" as best-effort in the UI; accept domain-wide effect (owner chose all-in).
  • Correlation dependency. Delivered/opened only work if Mailgun's message-id is stored at send time. Mitigation: make n8n's Respond to Webhook return Mailgun's id explicitly; verify with one live dev send before shipping tracking (pre-execution check).

Blast radius: sendEmail (every email path — auth, billing, support, notifications, leadgen, website forms), the app router (app.ts mount), the admin frontend router (Admin.tsx), Mailgun account config + the two n8n workflows (Respond node change). New env var on dev + prod (MAILGUN_WEBHOOK_SIGNING_KEY).

Pushback

  • Retention: I recommended metadata-forever + purge-HTML-after-90d; owner chose indefinite and owns the compliance call. Documented, not blocked. Execution still gated on written sign-off.
  • "Opened" value: unreliable enough to mislead. Recommend shipping it clearly labeled and not driving any decision off it.
  • Sequencing: ship logging + viewer + delivered first (backend-provable), then opens (domain config). Opens should not hold up the visibility goal.
