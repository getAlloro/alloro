Why

The Referrals Hub needs a location-specific PMS file manager so managers can see what PMS data exists, upload corrected monthly files, inspect original parsed data, edit current parsed data, download original files, and understand which months will become active before they save.

The current PMS upload path creates approved pms_jobs and the aggregator already applies the right non-destructive rule: latest approved data wins per month, capped to the latest 12 months for agent analysis. This feature should expose and manage that model instead of replacing it.

What

Add a customer-facing PMS File Manager inside /pmsStatistics, scoped to the selected location. Admins and managers can upload, edit, soft-delete, inspect, and download PMS files. Viewers can see data if the page already allows it, but cannot mutate it.

Uploads support .csv, .xls, and .xlsx. .txt support is removed from PMS upload paths. Upload preview must show incoming months and the active months that will be superseded by the new upload. Edits and deletes must append history and trigger a controlled monthly-agent rerun after blocking any active run for the location.

Scope Ledger

Objective
Location-specific PMS file manager with original-file downloads, original parsed snapshots, editable current parsed data, edit history, conflict preview, soft delete, and rerun-on-change.
Intended user
Customer admin and manager users on the Referrals Hub.
Surface
/pmsStatistics in the client dashboard, plus authenticated PMS APIs.
System of record
pms_jobs remains the PMS data record. New metadata and event rows extend it.
Out of scope
Queueing PMS mutations during active runs, changing the agent 12-month data cap, destructive hard delete, replacing the aggregator, changing ranking agents, or exposing unauthenticated admin PMS endpoints to customers.

Patterns To Follow

  • Use authenticated customer endpoints with authenticateToken, rbacMiddleware, locationScopeMiddleware, and requireRole("admin", "manager") for writes.
  • Use S3 object storage for original files. Store metadata in Postgres and return short-lived signed download URLs.
  • Keep pms_jobs.response_log as the active parsed data used by aggregation.
  • Keep pms_jobs.raw_input_data as parsed raw rows and add an immutable original parsed snapshot for the initial monthly rollup.
  • Record edits and deletes in an append-only events table with actor user id, event type, metadata, and timestamp.
  • Use React Query hooks for new server state. Do not add more direct API calls inside PMSVisualPillars.

Constraints

Must

  • Preserve non-destructive aggregation: latest approved, non-deleted job wins per month.
  • Preserve the latest 12-month cap before agent analysis.
  • Store original uploaded file bytes in S3 under a PMS-specific upload folder.
  • Expose downloads through signed URLs only.
  • Store original parsed monthly rollup separately from the current edited rollup.
  • Track edit history with actor name/id, timestamp, touched months, fields, and before/after values.
  • Soft-delete PMS jobs so they are excluded from aggregation but remain auditable and downloadable.
  • Block upload/edit/delete while an active PMS/monthly-agent run exists for the selected org/location.
  • Show active missing-month slots inside the contiguous active window from first available active month through latest active month, capped to 12 months.
  • Show incoming months and active months that will be superseded before upload submit.
  • Update Alloro Docs and the Referrals Hub replica before finalization.

Must Not

  • Do not hard-delete PMS jobs or original files from the user-facing delete flow.
  • Do not queue PMS mutations in v1.
  • Do not reuse unauthenticated admin PMS job endpoints for customer file-manager actions.
  • Do not trust organization_id from the frontend for customer endpoints.
  • Do not change ranking behavior, agent prompts, or the core 12-month cap.
  • Do not store permanent public URLs for original PMS files.
  • Do not keep supporting .txt PMS uploads.

Out Of Scope

  • Mutation queueing, multi-file batch upload, hard purge tooling, admin PMS endpoint hardening beyond avoiding those endpoints, and production data cleanup for historical PMS jobs.

Risk

Level: 4 - schema changes, original-file storage, customer write APIs, and monthly-agent reruns can affect production data and downstream tasks.

Risks Identified

  • Production migration and storage risk. New nullable columns and event table affect pms_jobs. Mitigation: additive-only migration, no destructive backfill, indexes on filter paths, reversible down migration that drops only new objects.
  • Auth and location-scope risk. Existing admin PMS job endpoints are unauthenticated. Mitigation: create separate client file-manager endpoints under authenticated RBAC and location scope; writes require admin/manager.
  • Stale agent output risk. Edits/deletes can make existing Summary/Referral Engine output stale. Mitigation: centralize PMS mutations in one service that records history, blocks active runs, and uses the existing cleanup/restart trigger path.
  • Soft-delete aggregation drift. Aggregator currently reads all approved jobs. Mitigation: update model query to exclude deleted_at IS NOT NULL and ensure key-data/list views label deleted rows correctly.
  • File privacy risk. PMS files can contain patient-adjacent data. Mitigation: S3 keys only in DB, signed URLs, auth check before URL generation, no permanent public URLs, no file contents in logs.
  • Editor duplication risk. PMS edit modals are already overlapping. Mitigation: create or refactor to a single reusable PmsJobDataEditorModal used by file manager and latest-job review paths.

Blast radius: Referrals Hub customer page, PMS upload/mapping paths, PMS aggregation, PMS automation status, monthly-agent cleanup/restart, S3 storage helpers, Alloro Docs Referrals Hub docs and replica, and any admin PMS job list that reads pms_jobs.

Pushback

  • This does not belong on the existing unauthenticated admin PMS job endpoints. Future-us will hate that. Customer file management needs its own authenticated, location-scoped API surface.
  • Hard delete is the wrong backend primitive. The UI can say delete, but the backend should soft-delete and preserve audit/file history.
  • Do not add a separate PMS-files source of truth. pms_jobs already is the upload/run record; extend it with metadata and events.

Production Migration Safety

The migration will run first on dev/dave against the dev database, then on main against production after merge. It is additive but still production-sensitive because pms_jobs is used by live aggregation. The migration must not rewrite existing PMS payloads or delete any rows. Existing jobs should remain active with null file metadata and no event history.
