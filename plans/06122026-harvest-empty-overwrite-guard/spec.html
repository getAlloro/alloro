Why

The data harvest can silently overwrite good data with nothing. The daily worker re-harvests a rolling window (GSC = 4 days) every run; the data upsert fires on any result.ok regardless of row count (dataHarvest.processor.ts:112-116); the GSC and Clarity adapters return ok: true, rowCount: 0 on an empty fetch (gscHarvestAdapter.ts:101, clarityHarvestAdapter.ts:52); and GscDataModel.upsert unconditionally replaces data on conflict (GscDataModel.ts:41-44). So if a date that already stored real data is re-harvested in-window and GSC returns an empty response that one time — transient hiccup, throttle, momentary inconsistency — the empty payload clobbers the good day. Latent (finalized GSC data is normally stable) but entirely unguarded, and the consequence is a permanently wrong day on the client and admin dashboards once it falls out of the re-harvest window.

What

Two parts:

  • Guard (prevents future clobbers): in the shared harvestSingle, only write data when result.rowCount > 0. Empty fetches still log the attempt (audit trail unchanged) but never write or overwrite stored data. Lives in the platform-agnostic path: fixes GSC and Clarity, and is a no-op for Rybbit (which hardcodes rowCount: 1 on success).
  • Backfill repair (fixes past clobbers): a one-time maintenance script that finds dates where the harvest log shows a prior success with rows_fetched > 0 but the stored gsc_data row is now empty or missing, and re-enqueues a single-date harvest for each. Dry-run by default; --apply to enqueue. Idempotent and safe to re-run.

Decisions locked: "skip empty writes" (not a placeholder-preserving conditional merge); guard + backfill only (no admin rollup panel, no log retention this pass).

Done when: a rowCount: 0 harvest no longer writes/overwrites gsc_data (while still logging the attempt), a rowCount > 0 harvest still upserts, the AI-audit and dashboard read paths are verified to treat absent days as zero, and the repair script reports/repairs clobbered dates. npx tsc --noEmit clean.

Patterns To Follow

  • Enqueue: reuse getHarvestQueue("daily").add(...) with a deterministic jobId (dedupes re-runs), exactly as the rerun endpoint does.
  • Script shape: standalone runnable under src/scripts/ (analog: worker-watchdog.ts for location/bootstrap), dry-run default, explicit --apply, structured console summary per integration.
  • Audit trail stays intact: the guard changes only the data write, never the IntegrationHarvestLogModel.create call.

Constraints

Must

  • Gate the data upsert in the shared harvestSingle on result.rowCount > 0 (all platforms).
  • Keep logging every attempt, including success / 0 rows.
  • Reuse getHarvestQueue("daily") + deterministic jobId for re-enqueue.
  • Repair script: dry-run by default, idempotent, per-integration summary, total count reported before any enqueue.

Must Not

  • Change adapter return shapes or the integration_harvest_logs table.
  • Delete gsc_data rows (unrecoverable clobbered rows stay as zeros; no destructive cleanup).
  • Add a DB migration (no schema change needed).
  • Flood the queue — report totals; rely on jobId dedupe; cap if counts are large.
  • Touch the 4-day window, the cron cadence, or the dashboard read services.

Out Of Scope

  • Admin rollup-by-date activity panel; log retention/prune.
  • Rybbit clobber protection — its rowCount is always 1, so this guard does not cover it; a Rybbit-specific emptiness check is a separate concern (noted, not done).
  • Changing harvest frequency or window size.

Risk

Level: 2 (Concern)

Risks Identified

  • Behavior change — empty days stop getting rows. For GSC/Clarity, dates with no data no longer get a (zero) gsc_data row, so findLatestReportDate anchors on the last day with data and dataDays counts only real days. Mitigation: verify service.gsc-performance.getDashboard and organizationAuditContextService treat absent days as zero (they aggregate present rows; the client sparkline already renders gaps). This is arguably more correct, but it must be confirmed, not assumed.
  • Backfill enqueue flooding. A fleet-wide repair could enqueue many jobs. Mitigation: dry-run default; print the total candidate count; deterministic jobId dedupes; cap/batch on --apply.
  • Re-harvest may still come back empty (GSC purged data >16 months, or still within lag). Mitigation: with the guard, an empty re-harvest simply skips the write — no re-clobber — and the script reports unrecoverable dates.
  • Rybbit remains unprotected by this guard. Mitigation: documented as a known gap; Rybbit's failure mode (zero-stats object, not absent rows) differs and is out of scope.

Blast radius: harvestSingle is the write path for all integrations (gsc/rybbit/clarity) across every customer. The guard only ever skips a genuine rowCount === 0 — which for Rybbit never occurs and for GSC/Clarity means "no data" — so no platform loses real data. The repair script touches the harvest queue only.

Pushback

  • The clobber may have already happened and you can't see it. The guard is necessary but not sufficient — that's exactly why the backfill is bundled. Run the script in dry-run first: if it reports zero candidates, no past damage occurred and the guard is pure prevention; if it reports candidates, those dashboards are currently understated and the repair restores them. Don't ship the guard alone and assume the past is clean.
