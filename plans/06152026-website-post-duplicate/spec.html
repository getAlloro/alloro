Why

Recreating a near-identical website post (a service page, a doctor bio, a blog entry) today means re-entering title, content, excerpt, featured image, custom fields, SEO data, categories, tags, and attachments by hand. A one-click Duplicate turns an existing post into a pre-filled draft so the editor only changes what differs.

What

A Duplicate action on each post in the Posts tab that creates a new post faithfully cloned from the source: it copies content, excerpt, featured image, custom fields, SEO data, category & tag assignments, and attachments. The new post is titled {title} [copy], forced to draft, and never inherits the source's source_url.

The action is available on both surfaces that render PostsTab: the staff admin Website Detail tab and the client DFY editor (/dfy/website). The clone logic is one shared service function; each surface adds only a thin handler + route + frontend wiring.

Done when: clicking Duplicate on either surface produces a new [copy] draft in the list with SEO, attachments, categories, and tags carried over, source_url null, and no console/tsc/convention regressions.

Patterns To Follow

  • Transaction boundary via PostModel.transaction(async (trx) => { … }) (BaseModel helper), threading trx into every model write — analog: src/controllers/pm/PmTasksController.ts:87. Honors §10.5 without inline db() in the service (§7.4).
  • Response shape: match the local sibling in each file (admin createPost → 201 { success, data }; user createUserPost → sendManagerResult(res, result, { successStatus: 201, data })). Do not introduce the ok()/fail() builders into files that don't use them.
  • Slug logic: reuse createPost's exact approach — slugify(title) then append -${Date.now().toString(36).slice(-4)} on collision via findBySlug.
  • JSONB serialization: custom_fields and seo_data come back from findByIdAndProject as parsed objects (node-pg) — JSON.stringify them on insert, mirroring createPost.
  • Frontend client: route through adminFetch (admin) / apiPost (user) — never raw fetch (§14.2). Surface errors via the existing react-hot-toast usage in PostsTab (§16.3).

Reference file: src/controllers/admin-websites/feature-services/service.post-manager.ts (the createPost function) — closest existing analog for the new duplicatePost structure, naming, logging, and error-tuple shape.

Constraints

Must

  • Write the clone logic once in service.post-manager.duplicatePost; both surfaces call it. §4.3 DRY
  • Wrap the multi-table write (post + category + tag + attachment rows) in a transaction. §10.5
  • Keep all DB access in models; thread trx through. §7.4
  • Force status: "draft" and published_at: null on the clone.
  • Set source_url: null on the clone — never inherit the import dedup key.
  • Title the clone {source.title} [copy]; regenerate the slug with collision handling.
  • Copy seo_data, custom_fields, featured_image, excerpt, content, category & tag assignments, and attachment rows.
  • Return 201 with the enriched new post; invalidate the posts cache. §8.4
  • Add duplicateUserPost to the UserWebsiteController re-export list (no phantom export). Read-before-write / No Phantom Files

Must Not

  • No new dependencies, no DB migration (existing tables/columns only).
  • Do not decompose or refactor PostsTab.tsx as part of this work (see Risk / Refactoring Rule).
  • Do not retrofit createPost to be transactional or change its response shape.
  • Do not introduce a new response/error contract into files that use the legacy shape.
  • Do not re-upload or copy S3 objects — clone attachment rows by reference (same URL).
  • No raw fetch/axios or hand-rolled auth header on the frontend. §14.2
  • No console.*, no any in new frontend code. §17.1 §17.2

Out Of Scope

  • Decomposing the 1,298-line PostsTab.tsx (tracked by the frontend-remediation plan, T6 god-files).
  • Bulk-duplicate / duplicate-to-another-project.
  • A confirmation modal or rename-on-duplicate dialog (clone is immediate; the editor renames after).
  • Deep-cloning GBP local posts (gbp-automation) — different domain, not this feature.

Risk

Level: 2 — Concern. Additive feature, no migration, no new deps, no change to existing exports.

Risks Identified

  • §13.1 file-size ceiling. PostsTab.tsx is 1,298 lines — ~500 over the ~800 hard ceiling; §13.1 says never add to a file already over it. Mitigation: add only the minimal button + handler (~15 lines); defer decomposition to the existing frontend-remediation backlog. Frontend checks are advisory until that lands (§18). Mixing a god-component refactor into feature work violates the Refactoring Rule.
  • §8.1/§8.3 legacy response contract. Admin PostsController + api/posts.ts use {success,data} + string error code, not the ok()/fail() builders. Mitigation: new handlers match the local sibling; a third shape in one handler is worse drift. Pre-existing, not fixed here.
  • source_url leakage (correctness). Copying source_url would collide with the import dedup key (findByImportDedupKey on project_id+post_type_id+source_url) and corrupt re-import matching. Mitigation: explicit source_url: null in the insert (covered by a Done check).
  • Partial-failure orphan. Without a transaction, an attachment-insert failure after the post row commits leaves a half-cloned post. Mitigation: transaction wraps all writes (§10.5) — the clone is more correct than the non-transactional createPost.
  • Shared S3 attachments. Clone rows point at the same S3 URLs. Mitigation: acceptable by design — deletePost removes only rows (no S3 cascade), so neither post orphans the other's media; replacing an image uploads a new URL.
  • Double-duplicate naming. Two clones both titled "… [copy]" (slugs differ). Mitigation: acceptable; counter suffix is a later nicety.

Blast radius:

  • service.post-manager.duplicatePost — new export; consumers = admin PostsController + user postHandlers. No existing export changes.
  • PostsTab gains an optional duplicatePostFn prop defaulting to the admin endpoint → WebsiteDetail.tsx (admin) gets it free, no change needed there; DFYWebsite.tsx overrides with userDuplicatePost on both PostsTab instances.
  • New routes are deeper/distinct paths (/:id/posts/:postId/duplicate, /posts/:postId/duplicate) — no shadowing of existing matchers. Auth inherited (admin) / explicit RBAC (user).

Pushback

  • Recommended: accept the small additive change to PostsTab.tsx and flag it for the frontend-remediation backlog rather than extracting now. Alternative (only if you want it): pull the post-card row into a PostListRow child as a separate, pre-feature refactor task — keeps §13.1 honest but widens scope. Recommendation is to not do that in this plan.
