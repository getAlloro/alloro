Why

Practice owners (Jo's request) can see individual months in Manage Data but cannot compare them. They want side-by-side month-over-month referral comparison — production, referral mix, and which referral sources moved — plus a one-paragraph AI read on what changed. This is the "Month toggling + side-by-side comparison (referrals)" feasibility item; feasibility is confirmed and this spec is the greenlit Tier B build (includes per-month source comparison).

What

A Compare button in the Referrals Hub header opens a modal where the user picks two months and sees a side-by-side dashboard: production, total/doctor/self referrals (each with a delta badge), and a ranked per-source comparison (referrals in A vs B with up/down/new/gone indicators). A Generate comparison insights button in the modal header calls a new backend endpoint that uses Claude Haiku (claude-haiku-4-5-20251001) to write a short paragraph summarizing the comparison.

Done when: the modal opens from the live hub (PmsHubSurface), both months and their sources render correctly, deltas are accurate, the insight paragraph generates from real data, and the build/lint gates pass with manual QA in dev.

Context

Key finding (verified): per-month referral sources already ship to the client. aggregatePmsData() builds each month with its own sources[] array (src/utils/pms/pmsAggregator.ts:375) and aggregateKeyData() returns months verbatim (src/controllers/pms/pms-services/pms-data.service.ts:106). The data is on the wire in /pms/keyData today; the frontend PmsKeyDataMonth type simply does not declare sources, and PMSVisualPillars strips it when remapping to monthlyData. No new data endpoint and no aggregator change are needed for the comparison.

Constraints

Must

  • Server re-derives both months from aggregatePmsData(org, location); client sends only the two month keys + location_id (org from JWT).
  • Use claude-haiku-4-5-20251001, max_tokens ~400, plain-text output.
  • New endpoint reuses [authenticateToken, rbacMiddleware, locationScopeMiddleware].
  • Type/prop additions are optional/additive only.
  • Compare button renders only when ≥2 months exist; pickers block identical A/B.
  • Disable Generate button while a request is in flight; log cost via safeLogAiCostEvent.

Must Not

  • No change to the /pms/keyData response shape or aggregatePmsData logic.
  • No new dependencies.
  • No new DB table / migration (no v1 persistence of insights).
  • Do not refactor PMSVisualPillars beyond adding state + modal mount + props.
  • Do not trust client-sent referral numbers for the insight.

Out Of Scope

  • Comparing 3+ months; cross-location comparison.
  • Persisting/caching generated insights (future: agent_results).
  • Exporting/printing the comparison; QTD/YTD comparison (months only).

Risk

Level: 2

Risks Identified

  • User-triggered LLM cost. Mitigation: explicit click only, disable-while-loading guard, cost logged via safeLogAiCostEvent (Haiku ≈ $0.001/call at ~400 tokens out).
  • Shared-type edits (PmsDashboardSurfaceProps, PmsKeyDataMonth). Mitigation: additions are optional; PmsDashboardSurface ignores onOpenCompare (only PmsHubSurface renders the button); backend payload untouched.
  • Sparse / mismatched per-month sources. Raw source values may be strings and unranked. Mitigation: normalize with Number() and sort client-side; guard ≥2 months.

Blast radius: consumers of PmsDashboardSurfaceProps = PmsHubSurface + PmsDashboardSurface (both safe with optional prop). Consumers of PmsKeyDataMonth = the monthlyData map in PMSVisualPillars + any keyData reader (safe with optional fields). New endpoint/service/modal have no existing consumers. aggregatePmsData is read-only reuse — unchanged.

Pushback

  • Server-side re-derivation (chosen) vs client-sends-numbers. Recommended: server re-derives — avoids trusting client input and keeps prompt/model server-side; no drift since both paths read the same monthly_rollup. The client-numbers shortcut was rejected despite being marginally less code.
  • No v1 caching. Recommended: generate fresh per explicit click; persistence to agent_results deferred until there's evidence users re-open the same comparison often.
