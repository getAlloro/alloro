Why

There is no automated code review in CI today — all three workflows (main.yml, dev.yml, deploy-sandbox.yml) trigger on push and only build/deploy. We want a senior-engineer-level second pair of eyes on every PR into main, checking changes against the Alloro Code Constitution, without blocking the merge. It catches architecture/convention drift and real bugs before prod, while staying advisory so it never stalls a release.

What

A new pull_request-triggered GitHub Actions workflow that, on PRs into main, computes the changed files plus their dependency-graph neighbors, sends them with the Code Constitution to Gemini (Flash tier) via the official run-gemini-cli action, and posts the model's senior-level recommendations as inline review comments on the changed lines only. The review event is COMMENT (never REQUEST_CHANGES), so it warns and never blocks.

Done when: opening a PR dev/dave → main produces inline Gemini comments scoped to the changed files, each warning written as a senior-engineer recommendation (citing §N.M when a Constitution article applies), the check shows green/neutral regardless of findings, and no comment lands on an unchanged file.

Context

How it fits the deploy flow

Work happens on dev/dave (push deploys dev). The gate sits on the PR dev/dave → main only — it does not touch dev.yml, so dev deploys keep working unchanged. Because it is warn-only, no branch protection is required; the comments are informational.

Relevant Files

  • .github/workflows/main.yml / .github/workflows/dev.yml - existing workflow structure, secrets usage, Node setup patterns to match.
  • scripts/check-conventions.sh - the existing deterministic convention checker (repo tooling analog; lives outside src//frontend/). The Gemini reviewer complements it: grep covers mechanical rules, Gemini covers judgment-level review.
  • .dependency-cruiser.cjs - existing depcruise config (currently scans src only). Source of the "related files" graph for backend changes.
  • code-constitution.html + ~/.claude/skills/code-constitution/SKILL.md - the convention source fed into the prompt (~550 lines; fits easily in Flash context).

Patterns To Follow

  • Workflow shape and secrets handling mirror main.yml/dev.yml (Node 22, actions/checkout@v6, actions/setup-node@v6).
  • Orchestration logic lives in small scripts/pr-review/*.mjs helpers, not sprawling inline YAML — same spirit as keeping logic in scripts/check-conventions.sh rather than in the deploy YAML.
  • Use pull_request (NOT pull_request_target) — same-repo PRs get secrets safely; avoids the injection surface of running untrusted head code in the base context.

Reference analog: .github/workflows/dev.yml for the workflow scaffold; scripts/check-conventions.sh for the repo-tooling-script conventions.

Constitution note

This task creates CI config and build tooling under .github/ and scripts/ — it does not modify governed runtime code in src/ or frontend/. So the §N.M Articles govern what the reviewer checks, not these files themselves. The prompt embeds the Constitution as the review rubric.

Constraints

Must

  • Warn-only: review event COMMENT; workflow never fails the PR or is marked a required check.
  • Comment ONLY on changed files; related (dependency) files are read-only context the model may reason over but must not comment on.
  • Inline comments on specific changed lines, validated against the diff before posting; drop any finding whose file/line is not in the diff.
  • Senior-engineer tone: architecture, correctness, security, performance, maintainability — cite §N.M when a Constitution article applies.
  • Model is a single configurable variable (GEMINI_MODEL), Flash tier, defaulting to gemini-3.5-flash (confirmed GA, 2026-05-19).
  • Use run-gemini-cli as the inference engine and GEMINI_API_KEY from repo secrets.
  • On synchronize, review ONLY the new commits (before..head); review the full PR only on the first run, reopen, or force-push fallback. (Rev 3)
  • No-comment is a valid outcome — return [] for clean code; never manufacture low-value findings. (Rev 3)

Must Not

  • No REQUEST_CHANGES, no failing exit code that blocks merge.
  • No edits to main.yml, dev.yml, or deploy-sandbox.yml.
  • No pull_request_target; no secrets exposed to untrusted code.
  • No new runtime dependencies in src//frontend/; tooling-only additions.
  • No comments on unchanged/context-only files.

Out Of Scope

  • The deterministic check:conventions --strict blocking gate (separate, recommended follow-up).
  • Extending depcruise to scan frontend/ (frontend uses a grep fallback for now).
  • Auto-fixing or committing suggestions; this only comments.
  • Dependency/security scanning, test execution, coverage gates.

Risk

Level: 2 (Concern) — external API + secret + non-deterministic output, but warn-only, internal repo, low blast radius.

Risks Identified

  • Workflow habit dependency. The review only runs if a PR is actually opened for dev/dave → main. If you direct-merge without a PR, it never fires. Mitigation: adopt the PR-into-main step; documented in the runbook produced at execution.
  • Model ID — resolved (Rev 1). Confirmed against Google's official Gemini API docs: the stable API model ID is gemini-3.5-flash (GA 2026-05-19; 1,048,576 input / 65,536 output tokens; preview variant gemini-3-flash-preview). Mitigation: still exposed as GEMINI_MODEL for one-line override.
  • Comment noise / false positives. AI review can be chatty. Mitigation: cap findings (default 12, prioritized), warn-only, exclude mechanical rules already covered by check-conventions.sh, focus the prompt on judgment-level issues.
  • Non-deterministic output breaking the poster. Mitigation: strict JSON output contract; the post step validates each finding (file in changed set, line in a diff hunk) and skips malformed ones rather than failing.
  • Secret exposure. Mitigation: pull_request trigger (same-repo only), least-privilege permissions (contents: read, pull-requests: write), GEMINI_API_KEY in repo secrets, never echoed.
  • Repeat comments on every push (synchronize). Mitigation: acceptable for v1; optional dedupe/minimize-stale-reviews noted as future enhancement.
  • depcruise frontend gap. Config scans src only. Mitigation: grep-based import + same-folder fallback for frontend/src changes; flagged as a known limitation.

Blast radius: All new files (.github/workflows/gemini-pr-review.yml, scripts/pr-review/*, .github/gemini/*). No existing workflow or runtime code is modified. New external touchpoints: Gemini API (via key) and the GitHub PR-comments API.

Pushback

  • AI review value depends on prompt discipline. If it just restates the grep checker, it adds noise. Recommendation: scope the prompt to what a model is uniquely good at — logic/edge-case/design/security reasoning — and let check-conventions.sh own the mechanical rules. Add the deterministic gate as the actual enforcement layer later.

Architecture

pull_request (base: main)
  job: gemini-review   [permissions: contents:read, pull-requests:write]
    1. checkout (fetch-depth: 0)        # need base for diff
    2. setup-node 22 + npm ci           # for depcruise
    3. collect-context.mjs              # changed files (git diff base...head)
         ├─ related files: depcruise JSON (src) + grep fallback (frontend)
         └─ build payload: diffs + changed/related file contents + Constitution
                                        → writes review-input.json + prompt.md
    4. run-gemini-cli (GEMINI_MODEL, GEMINI_API_KEY)
         └─ inference only, strict JSON out → findings.json
    5. post-review.mjs (github-script-style)
         └─ validate each finding (file in changed set, line in diff hunk)
         └─ create PR review, event: COMMENT, inline comments  # warn-only
