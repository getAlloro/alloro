Why

Alloro already records that an action happened — the metric_action_events rail stores an action's occurred_at, its active_until window, and the stage/metric it targets (src/models/MetricActionModel.ts:10-27, migration 20260715000000_create_metric_action_events.ts:26-53). On the Patient Journey card it surfaces a promise-to-watch: "Watching Google click-through through July 30" (src/services/MetricActionService.ts:89-96).

But nothing ever comes back and reads the metric to see if it moved. findLatestForJourney only re-displays the action; it never compares click-through before and after (src/services/MetricActionService.ts:140-164). The loop is open. A verified sweep of src/ confirms E2 is genuinely absent — no causalImpact, monteCarlo, interrupted-time-series, holdout, difference-in-differences, or attributed-lift code exists anywhere.

The naive way to close it — subtract last month's CTR from this month's and say "we caused +2.3%" — is a fabricated causal claim. A single practice's CTR moves for a hundred reasons (seasonality, a Google core update, more branded searches, a competitor's change). We cannot honestly attribute that swing to our action without a comparison that nets out those confounds. The funnel anchor is explicit: trend is the signal, never a promised number. E2 is the layer that closes the loop honestly.

What

A specification (no code this pass) for three connected pieces:

  • (a) Honest measurement. A pure, DB-free measurement module that takes an action event plus the daily metric time series and returns an ordinal verdict (not enough data / no detectable change / trending up / trending down), with the confound it cannot rule out named alongside it. It compares the metric against a baseline using a proven method chosen per available signal — never emitting a point-causation number.
  • (b) The "endures a month simulation" acceptance harness. A defined battery of fabricated 30-day daily series + an action, fed through the full connected chain (action recorded → measured → verdict surfaced). "Survives" = every scenario yields the honest expected verdict from the ladder, no scenario ever emits a promised causal number, and the chain stays connected end to end.
  • (c) A cited method-to-signal map. Which proven measurement method fits which stored signal, and which methods are explicitly rejected for now and why.

Done when: this spec is committed with (a), (b), (c) fully specified, every open decision documented with a recommendation, and every method grounded in a proven technique and an existing code anchor — with zero code written.

Measurement Method — Grounded in Proven Techniques

Alloro invents no method. Each below is an established, named technique from the causal-inference / marketing-measurement literature, mapped to the signal it fits.

Proven methodWhat it doesFits which signalVerdict
Difference-in-Differences (DiD) — treated vs untreated within the same site Compares the change on pages we edited against the change on pages we did not, over the same window. Site-wide and seasonal swings hit both groups, so subtracting one from the other nets the confound out. GSC per-page daily CTR/impressions — the untreated pages of the same project are the control (stageReaders.ts:239-244) Primary. Strongest method Alloro can run at single-practice scale.
Interrupted Time Series (ITS) / causal-impact-style counterfactual Fits the pre-intervention trend, projects "what the metric would have been" forward, and compares observed vs projected after the action, with an uncertainty band. (The canonical reference is Google's CausalImpact; we implement a transparent, auditable version, not the black box — see Decision D4.) Any single daily series with a clean pre-window — GSC site total, or the treated set when no untreated control exists Fallback when no usable untreated control exists.
Holdout / geo experiment / A-B The gold standard: randomize which units get the action, measure the gap. Requires deliberately withholding the action from a control group. Not available at single-practice scale today; DiD is the within-site stand-in. Revisit if Alloro ever holds pages out on purpose. Deferred.
Synthetic control Builds a counterfactual from a weighted basket of similar other practices (donors). Needs a large fleet of comparable practices — future, once the fleet is big enough to form donors. Deferred (fleet-scale).
Marketing Mix Modeling (MMM) Regresses an outcome across many channels/spend over long history. Rejected for now: Alloro captures no spend data and single-practice history is too short. Documented so it is not re-derived. Rejected.

The verdict ladder (never a promised number)

The measurement returns exactly one rung. It never returns "+X%" as a fact.

Not enough data yet — pre-window, post-window, or impressions volume is below the sufficiency floor (Decision D3). The default; the honest answer while the window is still filling.

No detectable change — the post-period movement sits inside the counterfactual's uncertainty band, or DiD nets it out. We watched and it did not clearly move.

Trending up / Trending down — the post-period moved outside the band in one direction, DiD-confirmed against the control where one exists. Reported as a direction, always paired with the confound we cannot rule out.

Honesty invariants (non-negotiable, inherited from the funnel anchor and funnelMath.ts):
  • Never present a causal point estimate ("+2.3% CTR because of us") as fact. Direction only. A magnitude band is allowed only when DiD control exists, only later (Decision D5), and always with the caveat.
  • Always name the confound the design cannot separate — e.g. "this window overlapped a Google update" when detectable, else the standing "single-practice measurement can't fully separate this from market shifts."
  • A drop is reported as honestly as a rise. Never suppress "trending down" to make Alloro look better (HONESTY bar).
  • Thin/absent data → "not enough data yet." Never a fabricated zero, never a fabricated lift (mirrors funnelMath.ts:47-54).

The "Endures a Month Simulation" Acceptance Harness

This proves the layer behaves honestly across a full 30-day action lifecycle (the METRIC_ACTION_VISIBLE_DAYS = 30 window, metricActions.ts:17) before any verdict reaches a customer. It feeds fabricated inputs through the whole connected chain: an action written to the rail → the daily series → the measurement module → the verdict as it would surface on the journey card. Synthetic fixtures only — no real customer data, no external calls.

What it feeds — the scenario battery

Each scenario is a fabricated per-page daily GSC series plus an action at day D. The honest expected verdict is fixed in advance:

#Scenario fedHonest expected verdictWhat it guards against
S1Treated pages' CTR clearly rises post-action; untreated flatTrending up (no number)A real lift must be seen — but never quantified as causal
S2Treated and untreated both flatNo detectable changeMust not manufacture a lift from noise
S3Seasonal confound: treated AND untreated both rise together (a holiday spike)No detectable change (DiD nets it out)The core honesty test — naive pre/post would falsely claim a lift; DiD must not
S4Regression to the mean: pre-window was an abnormal dip; post returns to normalNo detectable change / not enough dataOver-claiming a bounce-back as our win
S5Sparse: treated pages below the impressions floorNot enough data yetEmitting a direction from statistical noise
S6Action taken, but post CTR fallsTrending down (reported plainly)Hiding a decline to flatter Alloro
S7Action occurred_at precedes GSC coverage — no clean pre-windowNot enough data yetFabricating a baseline that doesn't exist

What "survives" means (the pass gate)

  • Correct rung: every scenario's emitted verdict equals its honest expected rung.
  • No fabricated causation: across the entire battery, the output never contains a promised point-causation number. (Asserted structurally — the verdict type cannot carry one in v1.)
  • Chain stays connected: the action is read back from the rail, measured, and surfaced end to end — no stage silently drops the signal (guards the "rooms with no doors" decay).
  • Confound always named: every up/down verdict ships with its un-ruled-out confound string.

The harness is specified here as two layers (both required at execution time, per §20): a Layer 1 vitest that drives the fixtures through the pure measurement function, and a Layer 2 test.html/test-results.json acceptance artifact asserting the connected-chain behavior. This spec pass writes neither — it defines what they must feed and assert.

Risk

Level: 3 — Structural / honesty-critical. This layer's entire purpose is to make a claim to a customer about cause and effect; a wrong or over-confident claim is a direct honesty-bar violation and erodes the trust that is the product.

Risks identified

  • Over-attribution. Presenting correlation as causation. Mitigation: the verdict type structurally cannot hold a causal number in v1; DiD nets out the biggest confound; every up/down ships its un-ruled-out confound; S3 in the harness fails the build if seasonality leaks through as a false lift.
  • Treated-set unknown (real data gap). DiD needs to know which pages were edited, but metric_action_events.metadata stores only counts, not page identifiers (MetricActionService.ts:132-136). Mitigation: capturing the treated page set at action time is a prerequisite (Task T1 / Decision D1); until it exists, E2 degrades to the ITS fallback on the site total, which is weaker but honest.
  • Thin-data false confidence. CTR on a handful of impressions is noise. Mitigation: the impressions floor (D3) forces "not enough data yet"; S5 guards it.
  • GSC ~2-day lag / partial windows. The current month trails reality (stageReaders.ts:185-215). Mitigation: the sufficiency check counts only settled days; a partial window stays "not enough data yet."
  • Silent chain breakage. A future refactor could drop the measured signal before it surfaces. Mitigation: the Layer 2 connected-chain assertion is the guard.

Blast radius: new read on GscDataModel (additive), a new pure util module, and a new call site inside MetricActionService/the journey assembler. Consumers of the journey response would gain an optional field; no existing field changes. No migration is required for the measurement itself; capturing the treated page set (D1) would add a column or metadata key — flagged as a schema decision, not assumed.

Pushback

  • Do not build the ITS counterfactual as the primary path. Single-series ITS attributes all post-period movement to the action — it cannot see a coincident Google update. Leading with DiD (within-site control) is the honesty-stronger design even though it costs the treated-set-capture prerequisite. Recommended: pay that cost (D1) rather than ship the weaker method as the headline.
  • Resist showing a number at all in v1. Even a banded estimate invites screenshots that drop the band. Ship the ordinal ladder first; earn the band later only when DiD control is reliably present.

Open Decisions

Each is documented with a recommendation so execution is not blocked on a guess. Decisions that are customer-facing or schema-affecting should be confirmed by a human before the execution pass.

D1 — Primary method & the treated-set prerequisite

DiD needs the identifiers of the pages an action edited; today only counts are stored (MetricActionService.ts:132-136).

Recommendation: DiD as primary, ITS as fallback. This requires capturing the treated page set at action time (new metadata key or join table — a schema decision, D6). Until captured, E2 runs ITS-only on the site total and labels its own weaker confidence.

D2 — Baseline definition

Options: (A) untreated pages, same window (DiD control) · (B) same pages, pre-action window (ITS) · (C) both, DiD primary with ITS as a cross-check.

Recommendation: C — DiD is the headline baseline; ITS runs alongside as a sanity cross-check. If the two disagree sharply, downgrade to "not enough data yet" rather than pick the flattering one.

D3 — Data-sufficiency floors

Unset: minimum settled pre-days, minimum post-days, minimum impressions/day per treated set, and how many trailing GSC days to treat as unsettled.

Recommendation (starting values, tune against real data before customer exposure): ≥14 settled pre-days, ≥7 post-days, treated-set ≥100 impressions across the window, drop the most recent 2 GSC days as unsettled. Encode beside METRIC_ACTION_VISIBLE_DAYS in metricActions.ts.

D4 — Estimator implementation: transparent build vs library

A full Bayesian structural time series (the CausalImpact approach) is an R/heavy-JS dependency and a black box to the customer.

Recommendation: build a transparent, auditable estimator in plain TS — a linear (or simple robust) pre-trend fit with a prediction interval for ITS, and a plain treated-minus-untreated delta for DiD. Auditable beats sophisticated for an honesty layer, and it adds no dependency (build-and-hold constraint).

D5 — Verdict granularity: ladder only vs ladder + band

Recommendation: ordinal ladder only in v1 (four rungs). Add a magnitude band later, DiD-backed only, always caveated. Never a point estimate.

D6 — Where the treated page set lives (schema)

If D1 is accepted: a JSON key on metric_action_events.metadata vs a dedicated join table.

Recommendation: start with a bounded array on metadata (reuses the existing jsonb column, MetricActionModel.ts:47) for the SEO-meta case; promote to a join table only if page sets grow large or need indexed lookups. Flag as a production-risk migration decision (additive, reversible) if a column/table is chosen.

D7 — v1 metric/action coverage

Recommendation: v1 covers only the one action/metric that exists today — seo_meta_update → impressions/ctr. The module signature stays metric-agnostic so visits/leads can be added when a daily control-bearing signal exists for them.

Rev 2 - 2026-07-18

Change: A fresh cross-model adversary broke the Rev 1 math and the "conservative floors" plan, so the design is corrected before any commit (the Rev 1 v1 math is superseded, never shipped). Executed proof at real scale (~30 impressions/day): a chance lucky run returned trending_up ≈ 1 action in 5 — a fabricated lift. Root cause (attributionMath.ts:183-187): the noise band was the empirical daily-CTR std pooled around one shared mean, which shrinks as data thins while true sampling noise grows — the band moved inverse to honest. Corrected design (supersedes D2/D4 emphasis, refines D1/D3):

  • Uncertainty from counts, not the observed series' variance — binomial standard error √(p(1−p)/N) on the pooled click/impression counts, so the band self-widens as volume thins. A wrong scale-guess can then only widen abstain — it fails silent, never dishonest.
  • Abstain is endogenous (MDE-gated), not a floor — compute the minimum detectable effect from actual window volumes; if it exceeds a materiality cap, the rung is not_enough_data before any direction logic. Distinguish "we measured, nothing moved" (no_detectable_change, a real null with power) from "we were blind" (not_enough_data) — never report a fabricated measured-null.
  • DiD demoted from "primary" to opportunistic-behind-a-hard-gate (minimum control impressions + coverage-stability + group-centered variance). Top-N JSONB control membership churns daily (selection bias, not missingness), so ITS/abstain is the honest default; DiD earns primacy only when real dev data shows the gate passes.
  • Missing ≠ 0 — a page dropping out of GSC top-N is an "unknown day" (reduces coverage → widens MDE → abstain), never a zero-CTR collapse (mirrors funnelMath.ts).
  • Windows derived from actual coverage dates — the new-business cold-start is then automatic (no pre-coverage → not_enough_data, structural), not a forgettable special branch. Scale-free formulas, not constants re-tuned to "hundreds."
  • Default-OFF / dark — the journey-card wiring stays behind a default-off flag; no verdict reaches a customer until calibrated against real dev-DB GSC series.
  • Fixtures labeled SYNTHETIC — passing proves verdict-logic shape, not honesty at production scale; add the adversary's real-scale + S8–S12 regression cases.

Rev 3 - 2026-07-18

Change: Second fresh cross-model adversary on the count-based v2. Verdict: the skeleton is right and the four v1 breaks + the real-scale lucky streak are CONFIRMED-FIXED — but it found 6 new confirmed holes (3 fabricate a direction, 2 hide/mislabel a real decline, 1 NaN). v2 is NOT committed. This is the turnkey fix-list for the next pass; re-adversary after, then it earns a commit:

  • F1 (fabricate, worst) — zero-variance at p=0/1. ctrVariance (attributionMath.ts:97-101) returns p(1−p)/N = 0 when clicks=0, so a 0-click thin window reads as infinitely precise 0% CTR → band collapses → fabricated trending_down (0/45 clicks has ~25% chance under an unchanged 3% CTR). Fix: Agresti–Coull / Wilson variance — p̃=(clicks+2)/(N+4) — or a variance floor, so SE stays positive at 0 and 1 clicks.
  • F3 (hide decline) — blind gate ordered before the observed effect. verdictFromDifference returns not_enough_data whenever band > material WITHOUT looking at diff, so a decisive collapse (4.6 SE) is silenced as "can't see a move this small." Fix: test the direction FIRST (|diff| > band AND ≥ material → direction); only if nothing clears, distinguish blind (band>material → not_enough_data) from measured-null. Still abstains on the v1 lucky streak (diff ≪ band).
  • F4 (hide decline) — a scrawny qualifying control floods the DiD SE. MIN_CONTROL_IMPRESSIONS=300 is absolute, never compared to treated volume, and DiD never falls back to ITS when the control blinds it — so a ~310-impression control re-hides a 40% decline on a 140k-impression treated series. Fix: only USE a control when it ADDS power (control volume comparable to treated); otherwise ITS. Never let adding a control turn a clear ITS signal into "not enough."
  • F5 (fabricate) — stationarity guard bypassable. (a) MIN_PRE_DAYS=4 < MIN_PRE_DAYS_FOR_STATIONARITY=6, so a 4–5-day baseline is never trend-checked; (b) the two-halves test misses a symmetric V (both halves average equal). Fix: require pre-days ≥ the stationarity minimum to run ITS at all (else abstain); strengthen the trend check beyond two-halves.
  • F2 (mislabel) — all-zero thin series claims "enough views." Same root as F1 (SE=0). The Agresti–Coull fix resolves it: SE>0 → band>material → not_enough_data, not a fabricated measured-null.
  • F6 (fabricate via NaN) — no input validation. clicks > impressions → p>1 → negative variance → NaN SE → all gate comparisons false → a direction slips through. Fix: validate 0 ≤ clicks ≤ impressions per point; malformed → treat as unknown/abstain, never a verdict.
  • F7 (fixtures) — the flagship lucky-streak fixture has NO streak (round(30×0.0433)=1=round(30×0.0333) — rounding erased it), and the suite can't detect the honesty band being narrowed 3× (BAND_Z÷3, materiality÷3 all pass).
