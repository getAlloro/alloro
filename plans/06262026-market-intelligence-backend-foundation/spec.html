Why

The current Patient Journey market-demand number reuses rank_keywords, which were built for competitor matching and usually contain only a few specialty terms. That is not a valid search-opportunity source and it creates impossible-looking funnels when compared with whole-site Google Search Console impressions.

This plan creates a separate backend Market Intelligence system. It leaves the ranking keyword pipeline untouched.

What

Create the backend foundation for a new org-level Estimated Search Opportunity metric. The system builds market keywords per location, auto-approves useful GSC query keywords, fetches fresh GBP context for all org locations, sends approved market keywords to DataForSEO, stores keyword-level volume separately from legacy keyword_search_volume, and exposes internal summary helpers for later Patient Journey use.

Constraints

Must

  • Keep rank_keywords, ranking, and competitor discovery behavior unchanged.
  • Keep legacy keyword_search_volume historical data intact.
  • Make all new keyword and volume tables tenant-scoped by organization_id and location_id.
  • Support org-level opportunity by summing all approved market keywords across all org locations.
  • Fetch fresh GBP profile data when identifiers and OAuth are available; degrade without failing the whole run.
  • Save GSC query keywords as candidates after normalization and quality filters. Do not make raw GSC rows harvestable until bootstrap/AI approves them.
  • Use nullable search volume to preserve the difference between provider no-data and real zero.
  • Batch DataForSEO requests with a named provider-limit constant verified during execution.
  • Store provider metadata, source month/report month, and location name used by DataForSEO.
  • When retiring the legacy monthly job, remove the existing BullMQ repeatable job key as well as the startup scheduler call.

Must Not

  • Do not change Patient Journey UI or labels in this spec.
  • Do not delete historical keywords. Archive instead.
  • Do not let the AI agent fetch GBP, GSC, website, or DB data directly.
  • Do not store secrets in code or plan files.
  • Do not run unbounded DataForSEO requests for every possible keyword.
  • Do not insert competitor names as market keywords.
  • Do not drop keyword_search_volume, KeywordSearchVolumeModel, or historical harvest logs in this retirement task.

Out Of Scope

  • Switching Patient Journey to the new metric.
  • Customer-facing coverage display.
  • Frontend work.
  • GBP search-query integration. No current app code stores GBP search queries.
  • Competitor keyword import.
  • DataForSEO Keyword Ideas API integration.
  • Deleting historical legacy search-volume rows.

Risk

Level: 4. This adds new persistent data, AI keyword generation, Google OAuth-backed GBP reads, DataForSEO cost exposure, and worker behavior.

Risks Identified

  • Cost amplification. A few rank keywords become hundreds of market keywords per location. Mitigation: cap approved market keywords at 300 per location in v1, batch requests, upsert idempotently, and log rows/cost metadata where provider response allows it.
  • Bad raw GSC queries. GSC contains brand, irrelevant, and accidental queries. Mitigation: save raw GSC queries as candidate evidence, demote old raw GSC-approved rows before bootstrap, and let the Market Intelligence Agent approve only the denoised keyword universe.
  • Tenant leakage. Org-level summaries must not include another org's locations. Mitigation: every model read requires organization id; tests prove cross-tenant isolation.
  • GBP OAuth gaps. Not every location may have usable account/location ids. Mitigation: context builder records missing context and continues with stored location/org/ranking/GSC data.
  • AI output shape drift. The Market Intelligence Agent can return malformed JSON or too many keywords. Mitigation: validate with a schema, clamp to the per-location cap, and store only normalized rows.
  • DataForSEO null versus zero confusion. Null means no provider data, not zero demand. Mitigation: keep nullable DB column and summary counts for null rows.
  • Zombie repeatable job. Removing the old scheduler code alone can leave a BullMQ repeatable job in Redis. Mitigation: add a startup cleanup or migration-safe script that removes the existing monthly-search-volume-harvest repeatable key from harvest-search-volume.

Blast radius: new migrations, new models, new backend domain under controllers/market-intelligence, new agent prompt, worker queue/processor registration, DataForSEO client extensions, and retirement of the legacy harvest-search-volume repeatable schedule. Existing Patient Journey and ranking behavior remain unchanged.

Pushback

  • Do not ship the UI switch in the same pass. Future-us will have no way to tell whether bad numbers came from keyword generation, provider fetch, summary math, or frontend labeling. The first pass must produce inspectable backend data only.
