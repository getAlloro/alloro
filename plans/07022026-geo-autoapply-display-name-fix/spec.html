GEO Auto-Apply Drops the Page Label

Status
Completed
Plan Folder
plans/07022026-geo-autoapply-display-name-fix
Size
Small
Risk
Level 1

Why

Earlier this session we found and fixed one instance of a bug class: creating a new page-version row from an existing published page can silently omit display_name, reverting the page to a bare-URL label in the admin Pages list. That fix covered createDraft() (click Edit on a published page). Auditing every other insert-into-pages call site turned up a second instance in the GEO auto-apply flow — same root cause, same fix shape.

What

applyGeoToPage() already loads the source page (page.display_name is in scope) but its insertData for the new draft version omits display_name, so any page whose GEO opening-content recommendation gets auto-applied loses its label. Fix: copy page.display_name into the insert, mirroring the pattern already used in createDraft. Extend the existing unit test that already asserts on this function's insertData shape so the fix has a regression guard.
