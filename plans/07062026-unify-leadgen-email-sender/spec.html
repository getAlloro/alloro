Unify Leadgen Email Sender

Why

Alloro has three independent outbound-email senders, each making its own HTTP call to n8n. The leadgen "email me when ready" report sender (sendAuditReportEmail) duplicates the transport, payload build, and environment interception that the central sendEmail already owns. Collapsing it removes that duplication and drops the sender count from three to two. This is step one; sendEmailWebhook (public website forms) is a later, separate task.

What

sendAuditReportEmail stops building its own payload, calling interceptEmailPayload, and posting to N8N_EMAIL_URL via raw axios. Instead it renders the leadgen HTML template (unchanged) and delegates to the central sendEmail, mapping the EmailResult back to its existing { ok, error? } contract. The two queue call sites are untouched. Done when: the leadgen report email still sends with a byte-identical n8n payload, interception still applies on non-prod, tsc is clean, and N8N_EMAIL_URL is no longer referenced in code.

Context

Relevant Files

  • src/controllers/leadgen-tracking/feature-services/service.n8n-email-sender.ts - the file being refactored (owns the leadgen report template + the { ok } wrapper).
  • src/emails/emailService.ts - the central sendEmail the wrapper delegates to (payload build + interception + axios.post + logging).
  • src/emails/index.ts - barrel; re-exports sendEmail. Import target.
  • src/controllers/leadgen-tracking/feature-services/service.email-notification-queue.ts - the only caller (2 sites, both check result.ok). NOT modified.

Payload equivalence (verified)

  • Both senders POST to the same n8n workflow alloro-email-service on dev AND prod (env compared: ALLORO_EMAIL_SERVICE_WEBHOOK == N8N_EMAIL_URL on both boxes).
  • Payload keys are identical: subject, body, recipients, cc, bcc, from, fromName. Current leadgen values (from: info@getalloro.com, fromName: Alloro, cc: []) equal sendEmail's brand defaults, so omitting them yields the same wire payload. bcc: ["info@getalloro.com"] and the leadgen subject are passed explicitly.

Patterns To Follow

  • §6.1 / §6.3 reference analog: keep the file in feature-services/; thin wrapper over the shared service.
  • §3.1 keep the defensive try/catch so the wrapper never throws (the worker drain relies on it).
  • §4.2 extract the leadgen subject and self-BCC to named constants.

Reference file: src/emails/templates/UserSendNotification.ts - closest analog of a template module that renders HTML then calls sendEmail.

Constraints

Must

  • Preserve the { ok, error? } return contract exactly — call sites unchanged.
  • Keep generateEmailHTML and the report template byte-for-byte.
  • Let sendEmail own interception — remove the wrapper's own interceptEmailPayload call (avoid double-interception / double subject prefix).
  • Keep the never-throws guarantee (try/catch around the delegate).

Must Not

  • No new dependencies. No new n8n webhook.
  • Do not touch service.email-notification-queue.ts or any other consumer.
  • Do not change sendEmail, the interceptor, or the website-form sender.
  • Do not remove the N8N_EMAIL_URL env var from the servers (harmless; only remove the dead code reference).

Out Of Scope

  • Narrowing / folding sendEmailWebhook (public website forms) — deliberate later task.
  • Adding retry/DB-logging to the main email path.
  • Removing the dead dist/-only checkup email pipeline.

Risk

Level: 2 (Concern) — an email path with no automatic retry; a broken send is a lost leadgen notification.

Risks Identified

  • Transport re-route: leadgen now sends via ALLORO_EMAIL_SERVICE_WEBHOOK instead of N8N_EMAIL_URL. Mitigation: verified both env vars resolve to the same alloro-email-service workflow on dev AND prod, and the payload shape is identical — no observable change on the wire.
  • Double interception if the local intercept call is left in. Mitigation: remove the wrapper's interceptEmailPayload call; sendEmail is the single interception point.
  • Log-tag change: intercept log line moves from [Leadgen Email] to sendEmail's generic line, and leadgen sends now also append to email.log. Mitigation: accepted — this is unified observability, an improvement.

Blast radius: sendAuditReportEmail has exactly one consumer, service.email-notification-queue.ts (lines 81, 137), both reading result.ok. Contract preserved → zero call-site impact. N8N_EMAIL_URL is read only in the file being edited.

Pushback

  • The file header currently says "consolidate sending behind the backend exclusively (flagged in Pushback)". This task is exactly that consolidation — no divergence. The one judgment call is omitting from/fromName to inherit central defaults vs. passing them explicitly; chose to inherit because the values are identical and it is DRY (§4.3), while noting the defaults are the Alloro brand address so future changes should flow through.
