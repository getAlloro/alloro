Why

The backend has no request-validation layer. This is the headline systemic breach of the conventions audit (check 17): 0 of 39 route files validate HTTP input, and the lone file in src/validation/ — minds.schemas.ts (25 lines) — validates LLM output, not client input. There is no validation middleware in src/middleware/. Client data reaches controllers untrusted, guarded only by scattered, inconsistent inline if (!x) return 400 checks (~77 controllers hand-roll body/query/param checks of varying shape; ~8 reference a VALIDATION_ERROR-style code).

This is a correctness and security exposure (audit dimension 12), and it compounds the security findings on the public and auth surfaces: the public websiteContact form, auth, auth-otp, auth-password, and billing all accept unvalidated input today. zod is already a dependency ("zod": "^4.3.6" in package.json; 9 files already import it for AI-response parsing), so the toolchain cost is zero — only the layer is missing.

What

A reusable request-validation layer, rolled out route-by-route, security/public-first — not big-bang. Concretely:

  • A generic validate(schema) Express middleware at src/middleware/validate.ts that validates req.body / req.params / req.query against a zod schema and, on failure, returns the standard contract shape { success: false, data: null, error: { code, message, details } } (the exact shape already shipped by gbp-automation — see Context).
  • Per-domain zod schema files under src/validation/ (e.g. auth.schemas.ts, websiteContact.schemas.ts, billing.schemas.ts), one per route group as it is onboarded.
  • A warn-only mode on the middleware: for the first pass on each route group it logs validation failures but lets the request through, so lenient/legacy clients are discovered before any request is rejected. Enforcement is flipped per route group only after a clean soak.
  • Incremental adoption in this priority order: auth → auth-otp → auth-password → websiteContact (public form) → billing → then high-traffic domains. Each route group is a self-contained, independently shippable unit.

Done (this unit overall): the middleware exists, the five security/public route groups validate input (warn-only first, then enforcing), and the pattern + schema location are established for every later domain to extend. Full repo-wide coverage of all 39 route files is explicitly a follow-on, not this plan's bar.

Context

The breach (verified)

  • src/validation/minds.schemas.ts (25 lines) — the entire validation/ dir. Validates an LLM ProposalsSchema via safeParse; not HTTP input. Do not repurpose it.
  • src/middleware/ — contains auth.ts, billingGate.ts, corsCustomDomains.ts, publicRateLimiter.ts, rbac.ts, scraperAuth.ts, superAdmin.ts, tokenRefresh.ts — and no validation middleware. validate.ts is a new file.
  • 9 files import zod, all for AI/LLM response parsing — never request bodies. No joi/express-validator/celebrate/yup is present.

Reference — the in-repo good pattern (copy this)

gbp-automation is the audit's reference domain (0 must-fix) and already ships exactly the two pieces this plan generalizes. Match its structure, naming, and error shape:

  • Error/response shape: src/controllers/gbp-automation/feature-utils/controllerResponses.ts — its private fail(res, status, code, message, details) emits { success: false, data: null, error: { code, message, details } }. This is the canonical failure shape validate() must return on a validation miss (use code VALIDATION_ERROR, status 400, and the zod issue list as details).
  • Input sanitizer: src/controllers/gbp-automation/feature-utils/GbpInputSanitizer.ts — typed length caps (GBP_INPUT_LIMITS), control-char stripping in sanitizeGbpText, and protocol-checked sanitizeGbpUrl. The new per-domain zod schemas should encode the same intent (max lengths, trimming, URL/email shape) declaratively rather than imperatively.
  • Typed error class: src/controllers/gbp-automation/feature-utils/GbpAutomationError.ts — (code, message, details) constructor; the model for a typed error if one is needed.

Rollout targets (real files, all small today)

  • src/routes/auth.ts (32 ln), src/routes/auth-otp.ts (19 ln), src/routes/auth-password.ts (34 ln), src/routes/websiteContact.ts (40 ln), src/routes/billing.ts (62 ln). All are thin routers, so adding a validate(schema) link per route is low-friction.
  • Auth handlers live under src/controllers/auth-otp/, auth-password/, auth/; the public form under src/controllers/websiteContact/formSubmissionController.ts; billing under src/controllers/billing/. The audit (check 09) flags these as the weakest input surfaces — exactly why they go first.

Patterns to follow

  • Middleware shape: standard (req, res, next) signature, mirroring the existing guards in src/middleware/auth.ts (early-return on failure, next() on pass).
  • Schema location/naming: per-domain files under src/validation/, named {domain}.schemas.ts, mirroring the existing minds.schemas.ts naming.
  • Failure shape: the controllerResponses.ts { success, data, error:{code,message,details} } contract — do not invent a new shape.

Reference file: src/controllers/gbp-automation/feature-utils/controllerResponses.ts — closest existing analog for the failure-response shape validate() must emit.

Dependencies (sequencing)

  • Soft dependency on the security hotfix. Best executed after or alongside the security-hotfix track (auth-gap + rate-limit + JWT-fallback fixes from the audit). Validation on the public + auth routes is naturally part of that hardening, and the same route files are edited, so coordinate to avoid churn. Not a hard blocker — this unit can land independently.
  • Foundational for later units. Every other domain's dim-12 remediation extends the validate() middleware + validation/ schema pattern established here. There is no existing shared layer to extend — this plan creates it.

Constraints

Must

  • Reuse the existing zod dependency; add no new validation library.
  • Return the canonical { success:false, data:null, error:{code,message,details} } shape on failure, matching gbp-automation/feature-utils/controllerResponses.ts.
  • Roll out route-by-route, security/public-first; commit per route group.
  • Ship each route group in warn-only mode first (log, do not reject), then flip to enforce only after a clean soak.
  • Place schemas under src/validation/ as {domain}.schemas.ts.

Must Not

  • Big-bang all 39 route files in one pass.
  • Add a new dependency (joi/express-validator/yup/celebrate).
  • Repurpose minds.schemas.ts — it validates LLM output, not input.
  • Refactor controller bodies, response shapes, or auth wiring while adding validation (no drive-by cleanup — those are separate audit items).
  • Flip a route group to enforce-mode in the same commit that introduces its warn-only pass.
  • Loosen or remove the existing inline guards before the schema covers the same fields.

Out Of Scope

  • The other 4 systemic breaches (Pino, DB-outside-models, response-shape normalization, transactions).
  • The auth-gap, rate-limit, and JWT-fallback security fixes (separate security-hotfix track) — except where this plan adds validation to the same route files.
  • Validating the remaining ~34 route files beyond the five security/public groups + first high-traffic batch (follow-on plans extend the same pattern).
  • Removing the now-redundant inline controller checks (a later cleanup, once schemas demonstrably cover them).

Risk

Level: 3 (Structural Risk) — validation that rejects input is behavior-changing. A strict schema can reject input that controllers previously accepted (extra fields, loose types, coercible strings), breaking lenient or legacy clients. The middleware is also cross-cutting: a bug in validate() affects every route it is mounted on.

Risks Identified

  • Rejecting previously-accepted malformed input. The current inline checks are loose and inconsistent; a tight zod schema may be stricter than reality. Mitigation: warn-only mode first on every route group — log every would-be rejection (with route + offending fields, never secret values) and soak before enforcing. Author schemas permissively first (e.g. .passthrough() / optional fields), then tighten once logs show real traffic shapes.
  • Cross-cutting middleware bug. A throw inside validate() would 500 every guarded route. Mitigation: the middleware must never throw — wrap safeParse, and on any internal error fall through (in warn-only) or return the standard 400 (in enforce), never an unhandled rejection. Verify with the manual endpoint checks below.
  • Logging sensitive input. Auth/OTP/password/billing bodies contain secrets; warn-mode logging could leak them (the audit already found plaintext OTP codes logged). Mitigation: log only field names + zod issue codes, never field values; redact by construction.
  • Schema/handler drift. A schema that omits a field the handler reads silently strips it (if using .strip()). Mitigation: derive each schema from the fields the controller actually reads; verify the happy-path request still succeeds end-to-end per route group before flipping to enforce.

Blast radius: new file src/middleware/validate.ts (no existing consumers); new schema files under src/validation/ (no consumers until wired); edited route files — initially only src/routes/auth.ts, auth-otp.ts, auth-password.ts, websiteContact.ts, billing.ts. Each added validate() link affects only that route's request path. Existing controllers are not modified.

Pushback

  • Do not enforce on day one. Flipping straight to reject-mode on the public form or auth routes risks locking out real clients during the exact window we are trying to harden. The recommended path is warn-only → soak → enforce, per route group, gated on clean logs. If the owner wants enforce-first for a specific internal-only route, that is acceptable case-by-case — but the public + auth + billing groups must soak.
  • Permissive-first schemas. Resist the urge to write maximally strict schemas immediately; an over-tight schema authored without seeing real traffic is the most likely way to break a lenient client. Tighten from data, not from imagination.

Tasks

Rollout order is security/public-first and incremental. T1 is foundational; every later task depends on it. Each route-group task is independently shippable and gets its own commit. P1 = do first.

T1: Generic validate(schema) middleware P1

Do
Create src/middleware/validate.ts exporting validate(schema, opts?) where schema targets body (default) / params / query and opts.mode is "warn" (default) or "enforce". Run schema.safeParse. On success: assign parsed value back and next(). On failure in enforce: return { success:false, data:null, error:{ code:"VALIDATION_ERROR", message, details } } at status 400, mirroring gbp-automation/feature-utils/controllerResponses.ts. On failure in warn: log route + failing field names + zod issue codes (never values), then next(). Must never throw.
