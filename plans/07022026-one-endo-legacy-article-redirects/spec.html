Why

The old One Endodontics WordPress site served its ~90 articles at root-level URLs (/{slug}/, Wayback-confirmed); the new site serves the same articles — same slugs — at /articles/{slug}. No redirect rows cover them, so today every legacy article URL soft-404s to the homepage. Why redirects and not just a Google reindex request: reindexing only makes Google re-fetch a URL — and the re-fetch still returns 200-homepage, teaching Google nothing about where the content went. The 301 is the actual signal that transfers each old URL's backlink history to its new home and sends real visitors clicking old links to the article instead of the homepage. GSC inspection submissions are also manual and quota-capped (~10/day — impractical for 90). Ordering matters: once the renderer's soft-404 fix (07022026-renderer-seo-hardening T5) ships, these URLs start returning hard 404s and their link equity dies — this plan must land first.

What

Insert exact-match 301 rows into website_builder.redirects for the One Endodontics project (0dcad678-2845-4c20-a298-e9c62aed9ebc): one per published article post, /{slug} → /articles/{slug} (91 rows, zero collisions with the existing 41 rows — verified 2026-07-02), plus /contact-us/gainesville-office-coming-soon → /locations/gainesville-office (Wayback-confirmed legacy URL). Done when every inserted from_path 301s to a 200 target on the live site.

Context

  • website_builder.redirects — project_id, from_path, to_path, type (301), is_wildcard; unique index on (project_id, from_path) makes the insert naturally idempotent with ON CONFLICT DO NOTHING.
  • Renderer resolution: redirects run before page lookup (website-builder-rebuild/src/routes/site.ts:408), cached 5 min in Redis (redir:*), flushed by ?nocache=1.
  • Row generation is a single INSERT ... SELECT from website_builder.posts joined to post_types (pt.slug='articles', published) — same direct-prod-data pattern as this session's canonical and og_image fixes.
  • Generating a row for a slug the old site never actually had is harmless: an exact-match redirect nobody requests does nothing.

Reference analog: the transaction + verify-count + cache-bust + live-curl procedure used in plans/07022026-seo-metatag-fixes T1.

Risk

Level: 1 — additive, idempotent, exact-match data rows; no schema change, no code change.

  • Level 1 A future article slug rename would leave a stale redirect. Mitigation: exact-match rows are inert unless requested; acceptable.

Blast radius: One Endodontics site only; 92 new redirect rows.

Revision Log

Rev 1 - 2026-07-02

Change: Executed as specified (92 rows, 0 conflicts, live-verified). One discovery during verification: the deployed renderer's ?nocache=1 pattern-flush is silently broken — the production Redis is a cluster with the KEYS command disabled, the flush's redis.keys(pattern) throws, and the catch swallows it. The redirects appeared only after the redirect cache's natural 5-minute TTL, not because of any cache-bust. Every prior "nocache-verified" change this session actually propagated via short TTLs.

Reason: Execution deviation: verification initially showed no redirects firing; root-caused to the broken flush + TTL rather than bad data.

Updated Done criteria: None here — the broken cache-flush fix was added to plans/07022026-renderer-seo-hardening's scope instead (SCAN-based, cluster-aware flush).
