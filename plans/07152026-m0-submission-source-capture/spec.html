Why

NS1 says every lever must be undeniable + attributed. Today a form submission on an Alloro-hosted site stores no source — the owner sees a bare count ("7 people reached out") with no honest "…and here's where they came from." Without a source column, every downstream lever is invisible: you cannot tell which channel earned the raised hand. M0 lays the one foundational brick — capture the channel at submit time — so no later lever ships without source captured. This is the finish line's track (the ① Proof check a lever-by-lever build hides).

Honesty cap (scope guard): M0 is the source column only — the capture half. It is not the E1 visitor→submission attribution + fleet A/B learning loop, which stays at slot #17 because it needs real traffic to measure. M0 stores the signal; E1 later reads and learns from it.

What (this delivery)

Not a new build — a reconcile + verify + refresh of PR #156 (+387/−0, was parked as premature/out-of-order). The work already existed and is clean; M0 lands it as an early foundational brick.

The 3-question coherence DoD

① Pioneer check

Pioneer = the CRO / attribution-analytics discipline (source/UTM attribution on an owned conversion surface — the practice behind GA/Plausible-style "by source" reporting).

Standing: At-or-above the pioneer on the specific mechanic M0 owns — server-side, source-of-truth channel classification with an honesty floor the analytics tools skip: unknown is preserved as null/"unknown," never bucketed into "direct" or a guessed channel, and whole-label host matching prevents the classic mis-bucketing (plumbing.com ≠ bing). Named honest gap: M0 captures one coarse first-touch channel; it does not yet do multi-touch journeys, campaign/medium/term breakdown, or a UI report — that is E1 (slot #17). Honest v1: the column is the correct, PII-safe foundation those richer views will read.

② Wire check

What feeds it: at submit, four signals — req.body.source, req.body.utm_source, req.body.first_touch_referrer (all frontend/site-supplied first-touch), and the Referer header (weak cross-site fallback). Honest incoming gap (verified): no code in this repo currently sends first_touch_referrer or utm_source/source to the endpoint — the public form lives in the generated hosted sites (outside src/). So until the hosted-site form is taught to forward the first-touch signal, source populates only from an external cross-site Referer, and stays null for the common "landed via Google, submitted from the practice's own page" path. That is honest (null = unknown), but the strongest signal is a backend contract awaiting a site-side sender — a named follow-on, not a silent hole.
What its output feeds: the source column → (a) honest source-broken-down submission counts and (b) E1's attribution→A/B loop (slot #17). Verified: nothing reads the column yet — E1 is the reader. Storing-now / reading-later is intended for the capture half.

Sender contract (the unbuilt other half — capture, not build)

M0 stores source server-side, but the strongest signal is the visitor's first touch, which only the hosted-site contact form can capture and forward. That form lives in the generated-site templates, outside this repo (same access blocker as B1-R) — so this is the contract the sender must honor, recorded so it isn't lost as a vague "someday." It is not buildable here; this is capture, not a build spec.

Endpoint: POST /api/websites/form-submission (mounted /api/websites, route /form-submission; controller handleFormSubmission). The public contact form on the rendered site.

Fields the sender should add to the POST body (all optional; all first-touch):

  • source — a first-touch channel label the site already resolved. Stored only if it matches M0's recognized channel vocabulary (else dropped to null). Use canonical channel words (google, facebook, email, sms, google_business_profile, …), never a raw or free-text value.
  • utm_source — the entry link's utm_source (same vocabulary rule).
  • first_touch_referrer — the raw referrer URL of the visitor's first landing page (captured on landing, persisted client-side, forwarded at submit) — not the submit page. Classified to a channel server-side (host→fixed label); internal/unparseable → skipped. This is the real entry channel for organic/referral traffic.

What M0 already does without the sender: it classifies the submit Referer header. For a same-site submit that is the practice's own page → internal → null. So without the sender, source is null for the common "landed via Google days ago, submitted from the own page" path — honest (unknown), but blind to organic/referral first-touch. The sender is what turns that null into a real channel.

Server-side precedence (already built, deterministic): explicit label (source/utm_source, vocabulary-checked) → first_touch_referrer (classified) → submit Referer (classified) → null. Never guessed.

Change: Dave's third-round finding fixed (§5.2 / §11.2) — the attribution bounds are now enforced, by sanitizing rather than rejecting. Root cause, stated plainly: Rev 6 added bounded Zod definitions for the three attribution fields and reported the §11.2 finding as fixed. Those definitions did nothing. The route mounts validate in warn-only mode (DEFAULT_MODE = "warn" unless VALIDATION_ENFORCE is truthy), so an over-cap value was logged and then handed to the controller untouched. Rev 6 shipped a guard that guarded nothing, and Rev 7's honesty sweep re-stated the gap without closing it — because both rounds treated warn-vs-enforce as the only two options. It was a false binary. The defect this actually hid (verified empirically against the pre-fix code, not reasoned about): a 3,032-char referrer returned {source:"google", method:"client_referrer"} and was persisted — the reviewer's exact claim. hostOf() classifies by hostname, which no length check touches. By contrast an over-cap source/utm_source returned {source:null}, because normalizeSource truncates to 100 chars and a 100-char string is never an allow-list key. So of the three named fields, only first_touch_referrer was actually reaching persistence out of contract; the other two were incidentally safe, not contractually safe — which is precisely how the referrer hole survived review. The decision — sanitize, not enforce. This is a PUBLIC lead-capture endpoint. Enforcing would 400 a submission whose only fault is a long tracking parameter, and the practice loses a real patient enquiry. A dentist losing a patient over a marketing tag is a far worse outcome than a null utm_source. The lead is the asset; attribution is a note about the asset, so the note is bounded and the lead always lands. Implementation: new sanitize(schema, keys) middleware in src/middleware/validate.ts — on a miss it DELETES the key and calls next(), so it can never set a status; new total attributionInputSchema in src/validation/websiteContact.schemas.ts (each field .catch(undefined), so it cannot fail and one bad field cannot discard its siblings); mounted on /form-submission after the warn-only soak. Values are DROPPED, never truncated — a truncated label or URL is one the visitor never sent. The bounds are now single shared constants used by both schemas, so the contract the soak LOGS and the contract sanitize ENFORCES cannot drift. Mounted as its own middleware on purpose: folding .catch() into formSubmissionSchema would have been cosmetic a second time — warn mode only assigns the parsed body back on SUCCESS, so any unrelated failure (an over-long hostname) would silently switch the sanitizing off. A13 is the regression for exactly that.
