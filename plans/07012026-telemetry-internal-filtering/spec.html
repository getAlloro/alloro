Why

Internal Alloro app usage — team logins, Pilot support sessions, and the internal Alloro Teams organization — is currently counted as real client engagement in the product telemetry system (app_usage_events). Corey flagged on the June 29 Weekly Reset that the Alloro Teams org accounts for 505 of 625 logged minutes, and that Garrison (zero logins) and Artful (90 seconds) are real engagement signals that are invisible until internal noise is removed. Jo confirmed this data will drive future CS outreach when clients go quiet, so it has to be trustworthy first. DentalEMR — fully offboarded — is also still showing in the organization list.

What

Stop internal activity from being recorded as telemetry going forward, filter it out of every existing rendered metric, archive DentalEMR, and add a zero-login / low-engagement flag to the Mission Control telemetry view so Corey and Jo can see real client engagement at a glance — no manual noise-subtraction required.

Done means: Mission Control's telemetry tab shows numbers with zero internal-team or Alloro-Teams contamination, DentalEMR is archived and off the org list, and orgs with zero logins or under 5 active minutes in the trailing 30 days are visibly flagged in that same view.

Context

What "Pilot" actually means here

is_pilot_session on app_usage_events does not mean a trial client account. It flags telemetry generated during Mission Control's embedded "Pilot" support feature (plans/06272026-embedded-organization-pilot-tab) — an admin viewing/acting as a client user in an iframe for support/debugging. Pilot-session rows are staff-generated support-tool activity, not client usage. This spec treats "pilotted accounts" in the original ask as this existing flag, confirmed with the user.

Constraints

Must

  • Exclude is_sandbox = true organizations and is_internal = true users unconditionally from every telemetry read path (not behind a toggle) — matches the existing unconditional exclusion in countInactivePaidOrganizations().
  • Block telemetry event writes at ingestion for: internal users (is_internal), sandbox/internal orgs (is_sandbox), and Pilot support sessions (isPilotSession).
  • Purge existing Pilot-session rows from app_usage_events via migration (per explicit user decision — see Revision context below).
  • Keep historical rows for internal users and the Alloro Teams org intact, filtered at read time only — do not delete them (distinct decision from Pilot rows; these may have other diagnostic value).
  • Add users.is_internal, backfilled by @getalloro.com email domain, following the is_sandbox migration pattern exactly.
  • Archive DentalEMR (org id 6) through the existing OrganizationArchiveService / admin UI Archive action — not a hard delete.
  • Low-engagement flag: zero sessions OR under 5 active minutes, trailing 30 days, computed only over eligible client orgs (archived_at IS NULL AND is_sandbox = false), surfaced in the existing Mission Control telemetry organization list.

Must Not

  • Do not drop the is_pilot_session column or remove backend query-param plumbing for includePilot — leave it inert. Full plumbing removal is out of scope (see Out of Scope).
  • Do not hard-delete DentalEMR's organization row or any FK-referenced data.
  • Do not fix the DentalEMR revenue hardcode in src/services/businessMetrics.ts:18 (ORG_MONTHLY_RATE) or the domain mapping in src/utils/core/domainMappings.ts:89-95 — real bugs, but unrelated to telemetry filtering. Flagged as a follow-up.
  • Do not fix the client-trusted isPilotSession boolean at ingestion (§5.4 concern, pre-existing) beyond what this spec already requires — flagged in Risk, not remediated here.
  • Do not introduce a new toggle/state-management pattern on the frontend for the low-engagement flag — render it as a derived badge from data already being fetched.

Out Of Scope

  • Removing the includePilot query-param plumbing from AppUsageRangeParams, MissionControlTelemetryData, and the route layer. The user asked only to remove the frontend toggle control; the backend param becomes inert (always effectively false-equivalent post-purge) but removing it end-to-end touches ~5 more files for no functional gain.
  • DentalEMR revenue-metrics hardcode cleanup (separate bug, separate ticket).
  • Server-side verification of the client-supplied isPilotSession flag (separate hardening ticket).
  • Automated CS-outreach trigger off the low-engagement flag — this spec only surfaces the flag; wiring it to an outreach workflow is future scope per the "Build — Journey Insights Dashboard" related card.

Risk

Level: 3 — Structural Risk

Risks Identified

Centralized but high-traffic blast radius. AppUsageEventModel.base() and buildOrganizationUsageQuery() are the sole query path for every Mission Control telemetry view (global summary, daily/surface/page usage, org list, org detail, user detail, inactive-org count). A filtering mistake in either function silently corrupts every one of those views at once, not just one.

Mitigation: Change is centralized to exactly 2 functions (confirmed by reading every consumer). Verify each of the 7 downstream read methods manually post-change, not just the org-list view Corey looked at.

Irreversible production data purge. Migration 2 (DELETE FROM app_usage_events WHERE is_pilot_session = true) permanently removes rows on the production database. Per this repo's migration-safety rules, this must be flagged explicitly before merge.

Mitigation: Rows being deleted are staff-generated Pilot-support-session telemetry (not client data, not billing data, not PII beyond internal staff identifiers already visible in Mission Control). Confirmed via the deliberate user decision to purge (see Pushback below). Runs on dev first per standard deploy path; verify row count before/after on dev before the same migration reaches prod via main.

Live ingestion endpoint behavior change. AppTelemetryController.recordEvents() is hit on every authenticated page navigation across the app. Adding an internal/sandbox lookup on the write path introduces a new failure surface (DB lookup latency, lookup errors) on a hot path.

Mitigation: Reuse the actor's already-resolved userId/organizationId (already fetched per-request elsewhere in the auth chain) rather than adding a second DB round-trip if avoidable during execution; if a lookup is unavoidable, wrap it in the same try/catch already present in recordEvents() and fail open (skip the block, log, don't 500) so a lookup failure never breaks page telemetry for real users.

Blast radius: AppUsageEventModel (7 public methods), AppUsageOrganizationTelemetryModel, AppUsageUserTelemetryModel, AppUsageOrganizationMovementModel query consumers, MissionControlTelemetryService.ts (4 exported functions), AppTelemetryIngestionService.ts / AppTelemetryController.ts (live write path), TelemetryToolbar.tsx, MissionControlTelemetryTab.tsx.

Pushback

  • Purging Pilot rows vs. keeping+filtering them. Recommended keeping all historical data and filtering at read time only (reversible, matches how internal-user and Alloro-Teams data is handled elsewhere in this same spec). User explicitly chose the harder line for Pilot specifically: hard-block writes, delete existing rows, remove the toggle entirely. Documented here as a deliberate, informed deviation from the safer default — not an oversight.
  • Client-trusted isPilotSession flag. The flag this entire Pilot-blocking mechanism depends on is currently set by the frontend, not verified server-side (§5.4: never trust client input). This spec does not fix that — it's pre-existing and out of scope — but it means the write-side Pilot block is only as reliable as the frontend's own reporting. Recommend a follow-up ticket to move Pilot-session detection to a server-verified signal (e.g., presence of an active embedded-pilot token) rather than a client-reported boolean.
