Why

The Practice Hub (/dashboard) is dense: a heavy dark Hero, a prose Trajectory card, an ActionQueue, and three fat metric cards each carrying their own sparklines, pie charts, and source lists. The redesign collapses this into a calm, scannable surface — greeting → one production chart → one "thing that matters" → four compact stat cards — and trims the data payload by removing fetches the simplified layout no longer needs.

This is the first of four dashboard simplifications (Practice Hub, Referrals Hub, Rankings, new Reviews & Posts). Practice Hub sets the visual + data pattern for the rest.

What

Rewrite DashboardOverview.tsx into the simplified layout while reusing existing data sources. Done when:

  • The Dashboard tab renders: header (PRACTICE HUB eyebrow + "YEAR TO DATE" label), greeting, one YTD production chart panel, one "1 thing that matters" banner, and a 4-card row (Referrals / Local rank / Reviews / Form subs).
  • DashboardAlertStack is preserved above the layout (upload / stale / onboarding nudges intact).
  • The data load is reduced: useAgentData, useActionQueue, and useLatestRanking are no longer fetched on this surface; PMSCard's referral-mix/sources rendering and Hero's heavy domain rendering are gone.
  • The backend surfaces reviews.total_review_count from already-fetched GBP allTime.totalReviewCount; the frontend type mirrors it.
  • The onboarding wizard's dashboard preview/tour is not broken.
  • npx tsc --noEmit is clean (frontend + backend) and lint passes.

Patterns To Follow

  • Self-fetching cards via React Query hooks (match existing focus cards).
  • Greeting time-of-day logic + name source: copy from Trajectory.tsx (reference analog for the header/greeting).
  • Chart rendering: reuse FocusTrendChart.tsx for the production line/area chart rather than a new chart lib.
  • Card navigation: useNavigate to existing routes, matching how current cards link out.
  • Status rules centralized in one helper (no thresholds scattered in JSX).

Reference file: frontend/src/components/dashboard/focus/PMSCard.tsx — closest analog for a self-fetching card that consumes both useDashboardMetrics and usePmsKeyData and renders a FocusTrendChart.

Constraints

Must

  • Reuse existing data sources; no new endpoints, no new fetch besides the additive backend field.
  • Keep DashboardAlertStack in full.
  • Centralize the 4-card status/dot rules in one statusRules.ts.
  • Label the production trend pill as month-over-month (e.g. "vs last mo") to avoid YTD misread.
  • Leave retired focus component files on disk; only stop importing them.
  • Re-validate the backend Zod schema after adding the field.

Must Not

  • No new dependencies.
  • Do not delete the old focus components or the wizard mock shapes.
  • Do not redesign the onboarding wizard (only keep it working).
  • Do not add an interactive period toggle (static "YEAR TO DATE" label only).
  • Do not touch the other dashboard tabs (PMS Stats, Rankings, Referral Engine, Patient Journey).

Out Of Scope

  • The Reviews & Posts page itself (/gbp-manager route) — Reviews card arrow links there but the page is a separate effort.
  • Referrals Hub, Rankings, Reviews & Posts redesigns (later passes).
  • YTD/YoY trend math, interactive period switching, product-defined health thresholds.

Pushback

  • The "YEAR TO DATE" framing with a MoM trend is a compromise. Recommended alternative once 2+ years of PMS data is reliable: switch the pill to true YoY YTD. Tracked as a follow-up, not this pass.
  • Status thresholds are invented here. If the product later defines real health bands, they belong server-side in the metrics dictionary, not in statusRules.ts. Flag for evolution.

Revision Log

Rev 1 — 2026-06-10 (execution)

Change: Two below-Level-3 architectural choices that refine, not expand, the spec.

  • Shared usePmsKeyData hook. Instead of each new card self-fetching like the PMSCard analog, the PMS series fetch was extracted to frontend/src/hooks/queries/usePmsKeyData.ts (mirrors PMSCard's inline query, same queryKey). ProductionPanel + StatCardRow share ONE request. Reason: avoids duplicating the fetch in two new files and a redundant network call.
  • Presentational StatCard + StatCardRow orchestrator. StatCard is pure/presentational; StatCardRow does the data wiring (real + wizard-demo) and renders the four tiles. Reason: centralizes the wizard-mode branch in one place rather than four, and keeps the four-card fetch to a single useDashboardMetrics call.
  • Wizard tour (T5). Trimmed the now-orphaned dashboard-trajectory and dashboard-queue steps, moved dashboard-pms (now Production) to follow the hero, and refreshed step copy. Targets dashboard-hero/pms/visibility/website were re-pointed onto the new elements. Demo-data fields in wizardConfig.ts were left intact (retired-but-retained cards still reference them).

Build: backend tsc clean; frontend tsc clean for all touched files (only pre-existing ai-seo-audit errors remain); eslint clean on touched files.

Pending: user visual QA of the live dashboard + a pass through the onboarding tour.
