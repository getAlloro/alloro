Why

PR 152 builds a useful receipts report, but several metrics are not safe to show to customers. The current implementation can claim credit for Google activity Alloro did not publish, reads a ranking table that does not exist in production, obscures Rybbit failures as zero values, and has no authenticated API route or automated tests.

The review is recorded as a requested-changes review at PR review 4692651075. This plan fixes every recorded issue without adding a migration, frontend surface, pricing assumptions, or unrelated refactor.

What

Deliver a production-compatible, read-only receipts-report backend that an authenticated super-admin can request for one organization and date range. The report will distinguish work Alloro actually published from synced Google state, use the production ranking source, report Rybbit users with explicit availability, and return a typed canonical API response.

Organization-level output

  • Website visitors from Rybbit users, with explicit availability state.
  • Leads captured from organization-scoped form submissions.
  • A total that includes only additive, attributable work metrics.

Location-level output

  • Published GBP posts and review replies from gbp_work_items.
  • Ranking movement from explicit practice_rankings.search_position observations.
  • Latest observed non-client competitor and client review-count context from search_results.

Success condition: the endpoint cannot turn synced Google activity, missing data, or source failures into a positive Alloro value claim.

Evidence Behind The Plan

Attribution mismatch

For organization 39 from 2026-04-01 through 2026-06-30, the synced-source queries counted 512 owner replies and 3 live posts. The Alloro work ledger recorded 5 published review replies and 0 published local posts for the same period.

Production ranking source

A read-only production check on 2026-07-14 found no weekly_ranking_snapshots table. practice_rankings exists and held 96 rows across 5 locations for organization 39, with a latest observation on 2026-07-01.

Architecture and test gap

The PR service performs direct Knex queries outside the model layer, and convention checks identify that violation. The PR adds no unit, integration, route, or source-failure tests.

Decisions

Attribution

Count an Alloro-published GBP action only when a matching gbp_work_items row has status = 'published', the requested organization and location, an allowed content_type, and published_at inside the half-open date range.

Dates

The API accepts ISO YYYY-MM-DD values, requires startDate <= endDate, and converts the inclusive end date into an exclusive next-day boundary. Observed ranking timestamps remain actual timestamps.

Rank movement

Use only explicit, completed search_position observations. Report first, last, best, and worst observed positions and their timestamps. Do not substitute rank_position or imply a calendar-period endpoint.

Competitor context

From the latest valid ranking observation, choose the non-client search_results entry with the lowest positive position. Use the isClient entry for client review-count context. Missing entries remain unavailable.

Rybbit

Add a typed users-period result: ok with a user count, not_connected, or source_unavailable. Do not use the comparison helper or convert a failed current request to zero.

API failure boundary

Database, authorization, and validation failures use canonical error responses. A Rybbit-only failure remains a successful report with an explicit field state so reliable database metrics are still usable.

Constraints

Must

  • Expose only GET /api/admin/receipts-report/organizations/:organizationId.
  • Require authenticateToken and superAdminMiddleware.
  • Require an organization ID in every report-model method and query.
  • Keep all Knex calls in the model layer and all orchestration in the service layer.
  • Use Zod for route params/query input and closed enums or discriminated unions for flags.
  • Use the canonical { success, data, error } response shape.
  • Return null or an explicit unavailable state when a metric cannot be supported.
  • Use synthetic fixtures in automated tests; do not depend on production data.

Must Not

  • Count gbp_local_posts or website_builder.reviews.has_reply as proof of Alloro-published work.
  • Read or recreate weekly_ranking_snapshots.
  • Add a migration, dependency, frontend page, PDF, email, billing logic, or replacement-rate estimate.
  • Use any, unbounded string flags, raw console.*, or direct DB calls from controllers/services.
  • Change existing Rybbit comparison callers while adding the receipts-specific users fetch.
  • Modify or stage unrelated Admin OS files or its active plan.

Out Of Scope

  • Restoring the retired weekly ranking snapshot pipeline.
  • Frontend or Alloro Docs work; this is an internal backend endpoint with no user-visible UI or copy change.
  • Downloadable reports, scheduled delivery, customer self-service, or production deployment.
  • Assigning monetary value to metrics or defining replacement-cost rates.
  • General cleanup of existing models, Rybbit helpers, analytics services, or ranking ingestion.

Risk

Level: 3 — Structural Risk. These numbers may be used in customer conversations, so a technically valid query with the wrong meaning is a product-integrity failure.

Risks And Required Mitigations

  • False attribution: synced Google state can include work done outside Alloro. Mitigation: use only the published gbp_work_items ledger for attributable GBP actions.
  • Production incompatibility: the PR queries a missing table. Mitigation: use existing practice_rankings rows and add no migration.
  • Ranking semantic drift: rank_position and search_position can disagree. Mitigation: expose explicit search_position observations, source/query context, and actual timestamps.
  • False zeros: an analytics outage can look like zero website use. Mitigation: return a discriminated Rybbit availability state.
  • Cross-tenant leakage: report aggregation spans several sources. Mitigation: require organizationId at every model boundary and test tenant isolation.
  • External latency: Rybbit can be slow or unavailable. Mitigation: add a bounded timeout only to the new receipts-specific request, contain the failure to the website-visitors field, and log through the project logger without secrets.
  • Date-filter performance: current indexes do not include gbp_work_items.published_at or practice_rankings.search_checked_at. Mitigation: preserve the approved no-migration scope, keep every query tenant/date/status scoped, inspect read-only query plans during verification, and record a separate index migration if production-scale plans require it.
  • Dirty-worktree contamination: the current checkout contains unrelated Admin OS work. Mitigation: execute from an isolated PR worktree or a clean PR branch and stage only files listed by this plan.

Blast Radius

  • src/app.ts gains one protected route mount.
  • src/utils/rybbit/service.rybbit-data.ts gains a new export; existing comparison exports and consumers must remain behaviorally unchanged.
  • Existing ranking, GBP work-item, form-submission, and location tables are read only; their schemas and ingestion paths are unchanged.
  • The receipts-report PR files are replaced or renamed; there is currently no runtime consumer to preserve.

Pushback

  • Do not revive weekly_ranking_snapshots to make the PR compile. That would add a second ranking pipeline and hide the source mismatch.
  • Do not count all Google posts or owner replies and relabel them as “managed.” That wording does not repair false attribution.
  • Do not add UI, delivery, or dollar-value presentation until the underlying metrics have passed API-level acceptance.
