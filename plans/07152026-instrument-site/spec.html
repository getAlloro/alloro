Provision the disabled preview‑analytics foundation

Provision Rybbit analytics for preview sites (*.sites.getalloro.com) on‑demand — the step that must exist before the renderer can instrument them. This PR does not measure previews. It ships disabled, and whether the renderer injects safely on a preview host remains unverified in a separate repo. The merge scope is only the disabled provisioning foundation; renderer, privacy, and live‑traffic checks are activation preconditions, not waived acceptance claims.

Why — whose life this lifts

A business owner is drowning: they built the practice to serve their community and get their life back, and instead they spend nights wondering whether the website Alloro built for them is doing anything at all. Right now Alloro cannot answer that question for a preview site — the moment a real neighbor lands on the owner's page, considers them, and decides, none of it is seen. The owner is asked to trust an invisible result.

B1 is the first disabled foundation needed before Alloro can later show an owner what happened on a preview site. It does not yet provide that evidence, and it does not promise a ranking or a customer (Value #6 — no rank/visibility/guarantee). The owner cannot know a lever worked until the separate renderer and privacy activation checks are complete.

What — the buildable brick

An admin‑initiated, gated, idempotent action that provisions a Rybbit site for a preview‑only project (one without a verified custom domain), reusing the proven provisionRybbitSite() path. Rev 6 makes that shared path safe under concurrent calls and provider‑success/local‑persistence‑failure: a project‑row lock serializes callers, and an exact‑domain provider lookup adopts an already‑created Rybbit site on retry. The expectation is that once the active Rybbit integration row exists, the already‑deployed website-renderer injects its tracking snippet on the preview hostname with no renderer change needed. That expectation is UNVERIFIED and this spec does not assert it — the inject condition lives in the website-renderer repo, which is not checked out on this machine (confirmed by search, 2026‑07‑16). It remains an activation precondition. If it turns out to be false, this PR provisions correctly and instruments nothing — which is why it ships disabled.

Load‑bearing scope correction (verified 2026‑07‑15). The B1 premise named publicRoutes.ts + instantWebsiteGenerator.ts as "the inject point." They are not. HTML serving and snippet injection live in a separate website-renderer repo (header comment service.rybbit.ts:5 — "The renderer owns script injection"; service.clarity-integration.ts:511 — "the renderer injects only status='active' rows"). This repo only stores the siteId and provisions it. So B1 in this repo = provision previews so the renderer instruments them, not "inject the snippet here."

The exact gap (confirmed against code, not the audit alone)

  • Automatic provisioning fires only on custom‑domain DNS verification: service.custom-domain.ts:178 → provisionRybbitSite(projectId, custom_domain).
  • generateInstantWebsite() (instantWebsiteGenerator.ts:483) never sets rybbit_site_id — preview projects are born un‑instrumented.
  • Result: a preview‑only project has rybbit_site_id = null and no active rybbit integration row → the renderer has nothing to inject. The audit's "instrumentation only after a custom domain + manual" is confirmed accurate this time.

The behavior, precisely

provisionPreviewAnalytics(projectId):
  1. GATE: if env PREVIEW_ANALYTICS_ENABLED !== "true"
        -> return { enabled:false, provisioned:false, reason:"gate_disabled" }   // provisions NOTHING
  2. Load project projection (id, hostname, generated_hostname, custom_domain, status, archived_at)
        -> not found        -> throw PROJECT_NOT_FOUND (HTTP 404)
        -> archived_at set   -> { enabled:true, provisioned:false, reason:"archived" }
        -> status !== "LIVE" -> { enabled:true, provisioned:false, reason:"not_live" }
  3. previewHost = (hostname ?? generated_hostname)
        -> missing          -> { enabled:true, provisioned:false, reason:"no_hostname" }
        previewDomain = `${previewHost}.sites.getalloro.com`
  4. provisionRybbitSite(projectId, previewDomain)
        -> transaction + SELECT ... FOR UPDATE on the project row
        -> re-read both local site-id sources under the lock
        -> if neither exists, GET the Rybbit organization sites and match exact domain
        -> no match -> POST one provider site; match -> adopt it (retry recovery)
        -> provider failure -> throw RYBBIT_PROVIDER_ERROR (HTTP 502)
        -> save project + integration in the same transaction
  5. Re-read the active integration row (report ACTUAL state, never assumed)
        -> siteId present   -> { enabled:true, provisioned:true, siteId, previewDomain }
        -> still null        -> throw RYBBIT_PERSISTENCE_FAILED (HTTP 500)

Idempotency is now explicit rather than assumed. The project row lock serializes concurrent callers across application processes. Exact‑domain provider reconciliation recovers a Rybbit site left behind when the provider succeeded but the local transaction rolled back, so retry does not POST a duplicate. Provider, project‑missing, and persistence failures are typed and reach the endpoint as non‑200 responses. The two local rows commit or roll back together (§10.5).

Constraints — the risk this feature actually carries

B1 is the first lever to observe real patients on live customer sites. Each guard below is load‑bearing, not polish.

1 · Default posture — OFF, with a named PII gate before flip‑on

Ships disabled via env PREVIEW_ANALYTICS_ENABLED (absent/≠"true" = off). This is deliberate (the A6 ships‑disabled lesson): on‑by‑default is wrong here because B1 cannot see the renderer's snippet (separate repo), so it cannot assert the snippet is cookieless / no‑PII. Enabling is therefore gated on a one‑time human verification: ⛔ GATE confirm the renderer's injected Rybbit snippet is cookieless and captures no PII (no form‑field contents, no patient identifiers in tracked URLs/query strings) before setting the env flag on. Rybbit's cookieless posture is assumed from its product design, not verified in this repo — treat it as unverified until the renderer snippet is read.

2 · Per‑tenant isolation — each site gets its OWN siteId

provisionRybbitSite() locks and reads only this project's local rows, while provider reconciliation matches the project's exact unique hostname/domain. One practice's traffic cannot land in another's analytics because different project domains do not reconcile to the same provider site. Proven by test: provisioning project A then project B yields two distinct siteIds, and A's row is never mutated by B's call.

3 · No PII in tracked events

B1 sends the domain only to Rybbit (${host}.sites.getalloro.com) — never form contents, patient names, or query strings. The event stream itself is emitted by the renderer snippet (out of this repo) — which is exactly why guard #1 gates enablement on reading that snippet. B1 introduces no new PII surface of its own.

4 · No live beacons in test — stub the Rybbit endpoint

Every test stubs global.fetch so the Rybbit site‑creation POST never fires against analytics.getalloro.com. A test run must not create real sites or pollute production analytics (the A6 "no live writes" analog).

5 · Consent / healthcare‑site privacy — named

Analytics on a healthcare‑adjacent site raises a consent/privacy‑policy question. B1's mitigation is structural: cookieless + no‑PII (pending guard #1), which is the posture that avoids a consent banner under GDPR/ePrivacy‑style rules. ⛔ ESCALATE: Corey whether the customer's site privacy policy must name analytics collection — a policy/canon call, not a build call. B1 does not decide it; it flags it.

Risk

Level 2 — additive, gated OFF, no migration, reversible (delete the integration row / clear rybbit_site_id to un‑provision). Production‑risk review:

  • Destructive schema change: none — no migration.
  • Data rewrite / backfill: none — on‑demand per project, explicitly no sweep.
  • Long lock: one project row is deliberately locked while the provider list/create call runs. This holds a DB connection across external I/O, but the endpoint is admin‑only, on‑demand, and disabled by default; the bounded lock is the schema‑free mechanism that serializes callers across app processes. Provider timeouts remain the operational backstop.
  • Irreversible: no — un‑provision by clearing the row/siteId; Rybbit site can be deleted admin‑side.
  • Dev‑only assumption: none — the same production provisioning path is used, with typed failure propagation and atomic local persistence added in Rev 4.

Whether the renderer actually injects for preview hostnames once an active rybbit row exists is UNVERIFIED — the injection condition lives in the website-renderer repo, which is not present in this worktree. It is plausible (the renderer resolves projects by hostname‑or‑domain and injects active rows), but plausible is not proven. Acceptance evidence is therefore a post‑enable customer‑walk: after Dave enables the flag for one project, load its *.sites.getalloro.com URL and confirm the Rybbit snippet is present in the served HTML. Until that walk passes, B1 is "provisions correctly," not "measures."
