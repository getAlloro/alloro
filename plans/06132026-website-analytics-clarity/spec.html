Why

Plan 5 of the dashboard-revamp feedback round — the largest surface. Covers PDF feedback #12–#21 ("Website" / Web Presence). The Website surface shows several numbers the owner cannot trust or read: a timeframe toggle that does not visibly change the data, an early-month month-to-date pace artifact rendered as a giant headline %, charts labeled "Last 12 months" that show ~3, an "Avg Position" that contradicts Local Rankings, post cards with an unexplained orange number, and a tab order / page-vs-post confusion. This plan makes every number honest and readable and brings the nav into the agreed order.

What

Done when:

  • Keywords (#12/#13): the range toggle's selected window is honestly labeled (it already filters server-side; the visible confusion is a data-availability ceiling — 3M/6M/12M all return the full stored history). Charts show the actual rendered span, expose per-point values on hover (axis labels + headline read-out), and totals reconcile with the chart. "Avg Position" is removed from the tab and tables. A one-line explainer tells the owner what Top Queries / Top Pages show.
  • Overview headline (#14/#17): the giant conversion % is reframed as last full month's conversion (a settled, retroactive figure) so no early-month pace artifact is shown as a headline. Conversion is removed from the left funnel chart (kept only on the card). The Leads line is green.
  • Traffic / Leads popouts (#15/#16): modal cards match the Overview numbers; chart titles match the rendered range (axis fix preferred over relabel); "Conversion" capitalized; "All Time" sits next to "Last Month"; leads framed retroactively.
  • GSC overview section (#18): consistent day-of-month x-axis labels; avg position removed; the clicks figure is labeled as coming from Google Search.
  • Pages (#19): the "/" page reads as "Home"; the label is editable from the Pages list.
  • Nav + Posts (#20/#21): tabs reorder to Overview · Pages · Posts · Menus · Keywords · Submissions; a short page-vs-post blurb is added; the orange SEO-score number on post cards is removed.
  • eslint clean on all changed files; backend tsc clean if backend is touched.

Context

Confirmed in code — the timeframe "not filtering" root cause (#12/#15/#17)

  • The backend DOES filter by range. service.gsc-performance.ts getDashboard() computes fromDate = latestReportDate − (rangeDays−1) and queries GscDataModel.findByProjectAndDateRange(projectId, fromDate, latestReportDate). service.rybbit-performance.ts does the identical thing via RybbitDataModel. The frontend caches per-range correctly (useWebsiteGscPerformance queryKey includes rangeDays; websiteAnalytics keyed by 365).
  • Why 3M / 6M / 12M look identical, and why "Last 12 months" shows ~3: a data-availability ceiling. The daily harvest only began storing rows a few months ago, so for any window ≥ the available history the date-range query returns ALL stored rows — the same set. dashboard.dataDays = actual stored rows (e.g. ~90), NOT the nominal window. The chart is faithfully plotting everything it has; the toggle and the "Last 12 months" titles are simply lying about the span.
  • The honest, low-risk fix is frontend labeling. Label the ACTUAL rendered span everywhere (dataDays / fromDate–toDate), not the nominal window. The Traffic card already does this (monthsRangeLabel(m.visitorSeries.length) → "Last N mo"); extend the same honesty to the Keywords tab, the GSC overview card, and the two detail modals. This makes totals reconcile with the chart (both describe the same real window) without any data-layer change. A deeper change (extending harvest retention/backfill) is OUT OF SCOPE and documented as a TODO finding.

Confirmed in code — the other items

  • #14 headline: WebsiteOverview.tsx score uses formatConversion(m.conversionRate) = month-to-date (verified leads ÷ MTD visitors). Early in the month this is a pace artifact. m.prevConversionRate (settled last-month figure) already exists in websiteMetrics.ts — reframe the headline to that. The funnel chart's third line (tertiaryKey="conversionN") is the conversion line to drop. The leads line is the funnel's secondaryKey="leadsN" with FUNNEL_COLORS.leads = navy — change to green.
  • #16 leads modal: ModalStat order is This month · Last month · Conversion · All time. The title-cased "Conversion" already capitalizes; the ask is to put All time next to Last month and make conversion the previous (settled) month. Modal chart title "last 12 months" → match rendered span.
  • #18 GSC card: WebsiteOverview.tsx "Search keywords" card shows ... avg position in the sub-line and "clicks from search" — relabel clearly as Google Search, drop avg position. Day labels via gscDayLabel already produce "Jun 10" form; ensure first/middle/last are shown.
  • #19 Pages: WebsitePagesTab.tsx renders page.path || "/". Add a "Home" label for / and an inline rename affordance. Page label persistence — see Risk R3 (no display_name on pages today).
  • #20 nav: WebsiteDashboardTabs.tsx options array order is the single source of nav order. Reorder to Overview · Pages · Posts · Menus · Keywords · Submissions. Page-vs-post blurb goes in DFYWebsite.tsx dashboardHeader (or the tabs wrapper).
  • #21 Posts: Admin/PostsTab.tsx ~line 870 renders {seoScore.pct} next to the bar (orange when 55–69% via colorClass). Remove the number text; keep the bar + the existing title tooltip so the data is still discoverable.

Risks Identified

  • R1 — Headline meaning change (Level 2). Reframing the giant % from MTD to last-full-month changes what a prominent number means; an owner used to "this month" may be briefly surprised. Mitigation: relabel explicitly ("Last month's conversion rate" + retroactive tooltip) so the new meaning is unambiguous; keep the MTD figure available in the leads modal.
  • R2 — Shared PostsTab edit (Level 2). PostsTab.tsx renders in both admin and the owner surface; removing the score number must not break the admin SEO workflow. Mitigation: remove only the trailing number <span>, keep the bar + title tooltip (data still discoverable), no logic change. The number is purely decorative.
  • R3 — #19 page rename has no storage (Level 2). Pages have path + status but no editable label column in the loaded Page shape. A true "rename the label" needs a backend field + endpoint + migration — beyond this plan's blast radius and into other agents' write paths. Mitigation (recommended): ship the display-only "Home" label now (honest, zero-risk) and document the rename-persistence as a TODO requiring a display_label column. Do not fake a rename that silently drops on reload.
  • R4 — Data ceiling could mislead even when labeled (Level 1). Once the harvest accrues 12 months, the toggle will visibly differ; until then 6M/12M legitimately equal the stored span. Mitigation: honest span label + a one-line note when the available history is shorter than the requested window ("Showing all N days available").

Blast radius: Website surface only. WebsiteDashboardTabs consumed solely by DFYWebsite. KeywordsTab/WebsiteOverview/WebsitePagesTab consumed solely by DFYWebsite. websiteMetrics.ts consumed by WebsiteOverview. PostsTab is shared (admin + user) — surgical edit. No backend writes; backend is read-only investigation.

Pushback

  • On #12 "make the toggle actually filter": it already filters — the data layer is correct. The real defect is dishonest labeling against a short data history. Building artificial differentiation (e.g. fabricating 12 months of zeros) would be worse than the truth. The right fix is honest span labels; the only way to make 6M ≠ 12M for real is to accrue/ backfill history, which is a separate data-layer effort (flagged, not built). Recommend shipping the honesty fix and tracking the retention work separately.
  • On #19 rename: recommend display-only "Home" now + a tracked TODO for a persisted label. Building a rename with nowhere to store it trains a broken interaction. If you want the persisted version this round, that's a separate small backend plan (column + endpoint) and should be its own spec.
