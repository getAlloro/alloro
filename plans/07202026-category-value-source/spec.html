Why

Alloro is the CMO of the owner's local presence: diagnose, prescribe the ONE move, do it once the owner approves, then measure the lift. The single strongest Get-Found lever is the Google Business Profile primary category ("Orthodontist" is found for far more intent than "Dentist"). Alloro already scrapes the current category into client_gbp and already reasons about category specificity in the audit (SearchConversion.md) — but no feature turns "your category should be X" into an owner-approved change. This closes that seam.

What

A recommendation service that, for a location, proposes a more specific, real, settable Google primary category from a curated Google GBP category catalog (a gcid + displayName, never free text), and an orchestrator that stages that proposal as an owner-approved A6 business_info draft through the existing GbpBusinessInfoDraftService.createDraft rail. Done when: (1) a better category yields a draft-status work item with patch.categories.primaryCategory and updateMask: ["categories"]; (2) no better category proposes nothing (returns a "no proposal" result, never a manufactured change); (3) a test proves the draft cannot reach Google without both owner approval and A6's master switch on; (4) npx tsc --noEmit is clean and the unit tests pass.

Honesty (Value #6): the output is a proposal ("a more specific category typically surfaces for more searches"), never "this will raise your rank."

Constraints

Must

  • Stage every proposal through GbpBusinessInfoDraftService.createDraft — inherit its master-switch, Google-ready, and owner-approval guards unchanged.
  • Propose only a real, settable category ({ name: "categories/gcid:*", displayName }) from the curated catalog.
  • Propose only a strictly more specific primary category than the current one; otherwise propose nothing.
  • Be deterministic and tenant-scoped; no any; typed errors.

Must Not

  • Never write to Google directly — no import of patchGbpBusinessInformation or any deploy path.
  • Never auto-approve or auto-deploy; the draft sits at draft for the owner.
  • Never emit free-text categories or manufacture a change when none is warranted.
  • No new dependencies; do not modify the A6 rail, the audit worker, or unrelated code.

Out Of Scope

  • An HTTP route / controller endpoint or a worker trigger that calls the orchestrator (the upstream door is the next seam, deliberately not built here).
  • Validating the seed gcid strings against Google's live categories.list (belongs at Dave's Google runtime gate — we hold no creds).
  • Additional-category (secondary) proposals; only the primary is in scope.

Risk

Level: 2 (Concern).

Risks Identified

  • Seed gcid accuracy. The catalog's gcid strings are the well-known public Google category IDs but are not verified here against Google's live taxonomy. Mitigation: the resolver logic is deterministic and fully tested; the gcid values are validated at Dave's Google runtime gate before any real write, and the whole write is already gated there. Flagged, not hidden.
  • Recommendation source is encoded, not called. SearchConversion.md is an LLM prompt returning free-text; I encode its stated specificity principle as a deterministic resolver instead of calling the LLM. Mitigation: determinism gives a stable, testable, settable proposal (an LLM emitting a free-text name cannot guarantee a settable gcid). Recorded as a deliberate deviation.
  • Orphan risk (Rube Goldberg). Nothing calls the orchestrator yet. Mitigation: its downstream door (createDraft) is real and tested; the upstream trigger is the explicit next seam. Stated plainly, not pretended end-to-end.

Blast radius: new files only. No existing file is modified. Consumers of the A6 rail are unaffected — I add a caller, not a change.

Pushback

  • A curated catalog covers only the ICP verticals it seeds. The right long-term source is Google's categories:search. Recommended: ship the deterministic resolver now (honest, testable), swap the catalog for the live taxonomy when the Google runtime is wired.

Revision Log

Rev 1 - 2026-07-20

Change: Fixed recommendation-logic honesty breaches found in adversarial review of PR #193 (write-safety rail untouched). (1) resolveCurrent now returns null when the current primary category is not in the catalog, so an already-specialized or unknown/absent baseline (e.g. "Oral and maxillofacial surgeon", null) proposes nothing instead of defaulting to specificity 0 and getting a lateral/downgrade. (2) A proposal is now only reachable for a catalog-verified generic→specialty pair, so the "more specific than" rationale is provably true; reworded the visibility-flavored tail ("typically surfaces for more of the searches…") to the descriptive "is a closer match to what this business does." (3) Pediatric tokens split into strong specialty words vs weak generic words ("child"/"kids"/"children"); a weak-only single signal no longer fires the board specialty. Hero status pill/card moved Completed → In Progress (PR still open).

Reason: User QA / adversarial review: manufactured false proposals off unknown baselines and a rationale that could assert a falsehood are Value #6 honesty breaches; the pediatric false-positive and the unmeasured visibility claim are the same class.

Updated Done criteria: Added the three Rev 1 checklist items above (unknown-baseline → nothing; rationale only on verified generic→specialty; family-dentistry-children does not propose Pediatric).
