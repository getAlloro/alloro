Why

The Patient Journey Insights page (/patientJourneyInsights) is locked to the current month, so owners cannot review how last month's funnel performed. Worse, the Google Visibility card says "Not connected yet" whenever the month window has zero stored GSC rows — which is false for connected practices early in a month (GSC data trails ~2 days behind, so on July 2 a fully connected practice shows "Not connected yet"). That message erodes trust in the product.

What

Two user-visible changes, both on the Patient Journey Insights surface:

  • Month navigation: prev/next chevrons beside the "Your Lead Pipeline · {Month Year}" label. Back up to 12 months (current month + 11 prior); forward clamped at the current month. Stepping months refetches the funnel for that month without flashing the full-page skeleton. The advisor banner copy becomes month-aware ("…this month." vs "…in June 2026.").
  • Honest GSC empty state: the Google Visibility card distinguishes three unavailable states:
    ConditionReason valueCard copy
    No active gsc integration for the projectnot_connected"Not connected yet" (unchanged)
    Active integration, zero rows in window, current monthpending"Google data is still pending" + a ? icon whose hover tooltip reads "GSC data trails ~2 days behind"
    Active integration, zero rows in window, past monthno_data"No Google data for this month"
    A fresh connection with no historical rows at all is treated as pending on the current month (owner decision 07-02).

Done when: a connected practice on day 1–2 of a month sees "Google data is still pending" with the tooltip; an unconnected practice still sees "Not connected yet"; users can step back through 12 months of funnels and the numbers match each month's window; both tsc gates and the patient-journey service tests pass.

Constraints

Must

  • Add unavailableReason?: "not_connected" | "pending" | "no_data" to both stage types (backend feature-utils/types.ts and frontend/src/types/patientJourney.ts) in the same execution — additive and optional so absent = legacy behavior.
  • Only the impressions stage emits the reason; visits/leads keep the generic fallback ("Not connected yet") when unavailable.
  • "Current month" for pending-vs-no_data is decided backend-side by comparing input.reportMonth to the current UTC month (same clock resolveReportMonth uses).
  • Frontend sends period only when viewing a past month (offset < 0); the current month keeps the period-less query key so the Practice Hub card's query stays deduped.
  • Month math on the frontend uses UTC (matching resolveReportMonth) so the label and the backend window never disagree around month rollover.
  • Chevrons clamp: next disabled at the current month, prev disabled at current-11 (12 visible months total). Both carry aria-labels.
  • Integration/history lookups run only on the already-empty path (zero rows in window) — no extra queries when data exists.

Must Not

  • No new dependencies — the ? tooltip is a lucide icon (already in use) with a native title/aria-label, and it must not be a nested <button> inside the stage-card button.
  • No backend route, controller, or API-client changes — the ?month= plumbing already exists end-to-end.
  • No changes to funnel math, leak selection, or conversion behavior — an unavailable impressions stage keeps available: false regardless of reason.
  • Do not modify the Practice Hub PatientJourneyCard or the detail-deck components — the new field is additive and they ignore it.
  • No refactors of unrelated stage readers or the Rybbit failure-path semantics.

Out Of Scope

  • Rybbit visits stage showing "Not connected yet" when the integration is active but the live fetch fails — known imprecision, separate fix if wanted.
  • Backend clamping of future/ancient months (authorized users get honest empties; frontend clamps the UI).
  • Month-scoped local-rank history (rank card stays "latest completed" on all months — owner decision 07-02).
  • A "reconnect needed" state for broken/revoked GSC integrations — those render as not_connected, matching the resolveGsc precedent.
  • Any month navigation on the Practice Hub summary card.

Risk

Level: 2 — additive contract change + new UI state on an owner-facing surface; no schema changes, no migrations, no auth changes.

Risks Identified

  • BE/FE contract drift — the stage type lives in two mirrored files. Mitigation: optional additive field, both files edited in T1/T2 of the same execution, verified by both tsc gates.
  • "Pending" mislabel on past months — a past month with no rows is not "pending"; that data will never arrive. Mitigation: the third no_data state ("No Google data for this month"), decided by report-month vs current-UTC-month. Flagged to owner 07-02; option A (three states) chosen over option B (one pending message everywhere).
  • Query-cache duplication — always sending an explicit month would split the cache from the Hub card's period-less key. Mitigation: send period only for past months.
  • Skeleton flash on month step — default React Query behavior blanks the page per new key. Mitigation: placeholderData: (prev) => prev in the hook + a dim/disable treatment while isFetching.
  • Extra reads on the empty path (Level 1) — two indexed lookups (website_integrations, max(report_date)) only when the month window is empty. Negligible.

Blast radius: consumers of the PatientJourney payload: PatientJourneyDashboard (this surface), components/dashboard/focus/PatientJourneyCard.tsx (Practice Hub — period-less query, ignores the new field), the detail-deck components (read stage metadata — additive-safe), and src/__tests__/patient-journey.service.test.ts. The GET /api/patient-journey endpoint has no other frontend callers.

Pushback

  • A per-stage unavailableReason string on the API is marginally more surface than a boolean, but it keeps copy decisions in the frontend and states typed — preferred over overloading the existing note field, which already feeds tooltips and whole-practice suffixes.
  • The 12-month window is a frontend-only clamp; the API will serve older months to anyone who crafts the query. That is acceptable (tenant-scoped, honest empties) and avoids duplicating the clamp server-side.
