GSC Scope Honors Admin Integration

Status
Completed
Plan Folder
plans/06112026-gsc-scope-honor-admin-integration
Size
Small
Risk
Level 2

Why

On /settings/integrations, the "Missing 1 Required API Access" banner and the GSC row's "Action needed" badge appear even when Search Console IS connected via an admin-managed integration. Root cause: getGrantedScopes only reads scopes off the org's own google_connections (One Endo conn 55, no GSC scope) and ignores the active admin-owned website_integrations GSC row (fed by super-admin conn 61). The client is asked to grant access Alloro already provides.

What

When an org's own connections lack the GSC scope but an active admin-owned GSC integration exists for the org's website project, treat gsc as granted. This drops the banner (missingCount → 0) and flips the GSC row to "Connected", which already renders the active admin source (sc-domain:..., Owner: Admin account). Frontend polish: when the active GSC source is admin-managed, show a calm "managed by Alloro" note instead of the contradictory "No Search Console account available" empty-state. Done when both render correctly and npx tsc --noEmit is clean.

Context

Relevant Files

  • src/controllers/settings/feature-services/service.scopes.ts - the per-org scope check; only change surface on the backend
  • src/controllers/admin-websites/feature-services/service.gsc-integration.ts - exposes getIntegrationForOrganization(orgId) (sanitized, org-safe; keeps connectionOwner, masks googleEmail to "Admin-managed account"); throws GscIntegrationError(404) if the org has no website project
  • frontend/src/pages/settings/IntegrationsRoute.tsx - MissingScopeBanner gated on missingScopeCount>0; GscSettingsSection derives hasGscScope = !missingScopes.includes("gsc") and already renders the "Active source" panel from useUserGscIntegration

Patterns To Follow

  • Service composition: settings service calling another feature-service (mirrors how user-website service delegates to gscIntegration)

Reference file: src/controllers/user-website/user-website-services/userWebsite.service.ts (delegates to gscIntegration.getIntegrationForOrganization).

Verified live (org 39)

  • website_integrations gsc row: status=active, connected_by=admin, connectionOwner=admin, conn 61, site sc-domain:1endodontics.com
  • org's own conn 55 has no webmasters.readonly scope → gsc currently reported missing

Constraints

Must

  • Only honor active + admin-owned integrations (an org-owned active integration already implies the org has the scope; don't mask a genuine scope loss)
  • Catch GscIntegrationError(404) for orgs with no website project — fall back to scope-only result

Must Not

  • No schema change
  • Do not change harvest/connection selection logic
  • Do not auto-grant gbp via any integration — GBP must remain a real OAuth scope check

Out Of Scope

  • The dashboard getting-started gating (separate, already fixed today)
  • Letting an org "take over" an admin GSC source

Risk

Level: 2

Risks Identified

  • Coupling settings/scopes to the website-builder integration model. Mitigation: one-directional dependency via the existing org-safe service function; wrapped in try/catch so a website-builder failure can't break the scopes endpoint.
  • Extra DB round-trip (project + integration lookup) on every scopes call. Mitigation: only runs when gsc is otherwise missing.

Blast radius: getGrantedScopes consumers — GET /api/settings/scopes (settings page banner + GSC row) and the dashboard GSC nag. Reducing false "missing" is the intended effect everywhere.

Pushback

  • Cleaner long-term: a single "effective integrations" resolver the dashboard, settings, and onboarding all share, so this logic isn't re-derived per surface. Out of scope today; noted for a follow-up.
