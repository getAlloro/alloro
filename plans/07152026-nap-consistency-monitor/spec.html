A4 — Citations & NAP Consistency Monitor

Get-found, in-lane. The measurement (does the practice's Name/Address/Phone match across the web?) already exists but runs only once, on-demand — the EMT taking vitals. A4 adds the hospital monitor: a recurring scheduled agent that runs the existing NAP measurement per onboarded location and records the result over time. Reuses the vitals kit; adds the schedule + persistence.

Plan: 07152026-nap-consistency-monitor  ·  Funnel: Stage A (get found), item A4  ·  Pioneer/guideline: Darren Shaw (Whitespark, 2010) — citation management; accurate NAP is a Google requirement
Lane: Corey+Claude spec → sandbox build + adversary → PR to dev/dave for Dave
Evidence: DIRECTIONAL · HYGIENE — NAP consistency is a recognized practice; its ranking magnitude is graded LORE (MENU). Monitor + flag to fix; never a rank promise, never a lore number as fact.

Why

When a practice's Name/Address/Phone is inconsistent across Google, directories (Yelp, Healthgrades, Zocdoc…), and its own site, it confuses both search engines and customers. Fixing that is the citation-management discipline Darren Shaw / Whitespark pioneered, and accurate NAP is required by Google's own Business Profile guidelines. It is a hygiene lever — real and recommended, but its ranking impact is honestly LORE (the "+16%"-style numbers are caged as lore in the graded menu, never stated as fact).

The gap (verified this session): the NAP measurement exists and works — a live simulation flagged a stale phone and a moved address, and correctly ignored a name-formatting variation. But it runs only on-demand (the admin audit route). The recurring schedules for customers are only proofline (daily) and ranking (15 days) — there is no ongoing NAP monitor. Vitals are taken at intake; the patient is never put on the monitor. A4 puts them on the monitor.

The acceptance bar (Corey, 2026-07-15): meet or exceed the proven example. For NAP/citations the proven example is the citation-management category — Whitespark, BrightLocal, Yext — whose core capability is exactly this: ongoing monitoring of NAP consistency across sources with specific conflicts flagged. A4 meets that bar (recurring monitor + the exact source/field that disagrees). Alloro's edge, and the reason this isn't a me-too: the monitor is one organ of the owned end-to-end connection engine — Alloro hosts the site and manages the GBP, so the same consistency check feeds surfaces Alloro actually controls, not a standalone report the owner has to act on alone. And it carries the honesty cap the incumbents don't state: no rank promise. If this lever ever fails to meet that bar, per Corey it becomes the focus.

What — in scope (the monitor, reusing the measurement)

Reuse first (verified against code this session). No new NAP comparison logic.

  • The vitals kit — compareExternalIdentity (per-field NAP comparison, fuzzy address/phone/name), collectExternalEntitySources (SerpApi + directory discovery), entitySignalScoring (the external-consistency scoring). Reused as-is; A4 does not re-implement matching.
  • The schedule + agent pattern — the schedules table + scheduler.processor (60s dispatcher) + scheduleExec.processor → getAgentHandler(agent_key) from services/agentRegistry (verified shape: { displayName, description, handler: () => Promise<{ summary }> }, iterating onboarded locations like proofline/ranking).
  • The audit entry — aiSeoAuditService (createOrganizationAuditRun / executeAuditRun) for the per-location run.

The monitor

  • New scheduled agent — register agent_key: "nap_consistency" in agentRegistry + seed a schedules row (interval_days: 14, aligned with ranking's cadence to keep SerpApi cost modest). Enabled flag honored — ships toggleable (can seed disabled until Corey sets it live).
  • Per-location NAP run — for each onboarded location, run the existing NAP measurement (the vitals kit) and compute the consistency result (consistent / conflicting per source, per field). Failures are isolated per location (one bad location never fails the batch).
  • Time-series persistence — record a snapshot per (location, run) into a new additive table nap_consistency_observation (org/location, run_date, sources_checked, consistent_count, conflict_count, conflicts JSON [source, field, values], observed_at). The monitor's chart — a log over time, so a customer/operator can see NAP drift and fixes.

Build decision to resolve at implementation (noted, not hand-waved): run the focused NAP measurement per location (lighter, cheaper) vs. trigger the full executeAuditRun and extract the NAP slice (more reuse, heavier/costlier per run). Recommendation: the focused NAP run to control SerpApi cost; confirmed against the code at build time.

What — explicitly NOT in A4

  • No new NAP matching logic — the comparison is reused, not rebuilt.
  • No rank promise, no lore-as-fact — never "fixing NAP raises your rank by X"; the honest claim is "your listings are consistent / here are the N that disagree, worth fixing."
  • No customer-facing dashboard in this slice — persist + operator-visible first; the display is a follow-on.
  • No auto-fix / no writing to directories — observe + flag only (writing NAP to external directories is out of lane / a separate future decision).

Constraints

Honesty (Value #6 + the LORE cap). Report NAP consistency and specific conflicts to fix — a data-hygiene/trust signal, never a rank promise and never a lore percentage stated as fact. A flagged conflict is "worth double-checking," not "confirmed error" (directory scraping is noisy — matches the existing scorer's own posture).

Anti-fabrication. Persist only real comparison results (keep the source + the two values that disagreed). Never infer a conflict the comparison didn't find; never invent a source.

Sandbox-safety + cost. Migration additive + reversible (tested down). The run hits SerpApi (an external read) — no Stripe/email/human side effect, but a per-run API cost, so cadence (14 days) is the cost knob and the schedule ships toggleable. No writes to any external listing. Prove DB paths with a rolled-back transaction in the sandbox.

  • Per-location MEASUREMENT failure isolation — one location's measurement failure (no website, SerpApi error) is logged + skipped, never aborts the batch. (Rev 4) A persistence failure is deliberately not symmetric: the measurement worked and the database dropped it, so write failures are aggregated and the run is failed visibly (NapPersistenceError) rather than reported as a completed run — every location is still attempted first.
  • Edge cases: location with no canonical NAP baseline → skip (nothing to compare); no external sources found on a FULL-coverage run → record "0 sources checked" honestly (not a false all-consistent); SerpApi unconfigured/over quota → skip the run, log it, never fabricate; (Rev 4) SerpApi answered only SOME queries → partial_coverage: record nothing, count it separately, never write a short count that reads like a finished one; a conflict that's really scraping noise → carried as "worth double-checking," never a hard error.
  • Idempotency — one snapshot per (location, run_date). (Rev 4) The write reports whether a row actually landed, so a same-day re-run is counted as locationsAlreadyRecorded, never as a new observation.
  • Lane: Corey+Claude author; build+adversary in the sandbox; PR to dev/dave; never push to dev/dave/main directly.

Risk

  • Recurring SerpApi cost. Mitigation: 14-day cadence, toggleable schedule (seed disabled if Corey prefers), per-location isolation so a failure doesn't retry-storm.
  • Directory-scraping noise → false conflicts. Mitigation: reuse the existing scorer's "worth double-checking, not confirmed" posture; never hard-fail on a possible mismatch.
  • Lore hardening into a promise. Mitigation: honesty cap in the output shape + copy; tests assert no rank/guarantee/lore-number language.
  • Heavy full-audit runs. Mitigation: prefer the focused NAP run over a full executeAuditRun per cadence.
  • (Rev 8) Shared scheduler migration lock. The additive schedule_runs.logical_run_key column needs an ordinary composite unique index so PostgreSQL can infer ON CONFLICT (schedule_id, logical_run_key). Building that index can briefly block writes to the scheduler run log. Mitigation: the column is nullable, historical rows need no backfill, the migration is reversible/resumable, and T14 records the live PostgreSQL proof as explicitly waived rather than passed.
