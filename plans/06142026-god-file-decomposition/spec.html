Why

The backend conventions audit (plans/06142026-backend-conventions-audit) confirmed 128 files over the 300-line cap — ~17% of src/. The damage concentrates in a handful of genuine god-files: AdminWebsitesController.ts at 5,567 lines / 166 handlers / ~10 domains, routes/checkup.ts at 2,140 lines holding a scoring engine inside route handlers, and the practice-ranking + user-website clusters. These files are unmaintainable, untestable, and the highest-friction surface in the codebase: every change risks a 5,000-line merge conflict and there is no unit small enough to reason about.

The audit already ruled the feature-services/ + feature-utils/ nesting ACCEPTED and named gbp-automation the reference shape to copy. This plan executes that ruling: break the god-files down into thin controllers + feature-services, behavior-preserving, one extracted unit at a time. Now — because the files only get harder to split the longer they grow.

What

Behavior-preserving decomposition of the over-cap backend files into the gbp-automation shape: thin *Controller.ts (validate → call service → respond) + domain feature-services/PascalCase*Service.ts + feature-utils/, with DB access living in models/ (already started by the upstream db-access-into-models phase). No endpoint, response body, status code, log line, or side-effect changes. The HTTP surface and the wire contract are frozen.

Done: the headliner god-files (AdminWebsitesController, checkup.ts, practice-ranking controller + ranking-pipeline, user-website trio) and the long tail of over-cap feature-services are decomposed; each extracted unit is committed individually behind a green checkpoint (smoke harness + npm run build + eslint); the wire contract is unchanged, verified per unit against the captured smoke baseline.

Context

Reference shape — copy this exactly

src/controllers/gbp-automation/ is the target shape (audit Check 05: reference domain, 0 must-fix). Structure to mirror for every decomposed domain:

  • gbp-automation/GbpAutomationController.ts, GbpReviewManagementController.ts, AdminGbpAutomationController.ts — thin controllers, one per sub-domain, each delegating to services.
  • gbp-automation/feature-services/Gbp*Service.ts — ~28 PascalCase single-responsibility services (GbpReviewReplyService, GbpLocalPostDraftService, GbpDeployPreviewService, GbpReadinessService, …). One service = one cohesive concern.
  • gbp-automation/feature-utils/ — GbpInputSanitizer.ts (input validation/sanitize), GbpAutomationError.ts (typed errors), controllerResponses.ts (the repo's only real {success,data,error} + sanitizer standard), googleApiErrors.ts.

This is the shape — thin controller, PascalCase feature-services, shared response/sanitizer/error utils. Decomposed domains land their new files into their own existing feature-services/ folders (already present in admin-websites, practice-ranking, agents), not into gbp-automation.

Dependencies — this phase runs LAST

Decomposing first would mean re-touching every file twice.
  • AFTER plans/06142026-backend-smoke-test-harness — the smoke harness is the only safety net for behavior-preserving extraction. Without a captured baseline of every god-file's endpoints, "behavior-preserving" is unverifiable. Do not start this phase until the harness exists and a green baseline is captured.

Reference file: src/controllers/gbp-automation/GbpReviewManagementController.ts + its feature-services/GbpReviewReplyService.ts — the closest existing analog for the thin-controller / feature-service split every task targets.

Constraints

Must

  • Behavior-preserving only. Same routes, same response bodies/shape, same status codes, same logs, same side-effects. This is a move, not a rewrite.
  • Target the gbp-automation shape: thin controller → feature-service → model; shared feature-utils/ for response/sanitize/error.
  • Commit per extracted unit — one domain or one god-function per commit, never a batch.
  • After each extracted unit: run the smoke harness, npm run build (tsc), and eslint. All green before the next unit.
  • Match each domain folder's existing filename prefix (service.*.ts in admin-websites/practice-ranking/agents; *.service.ts in user-website; pms-*.service.ts/Pms*Service.ts in pms) — do not introduce a new naming style mid-phase.
  • Preserve existing exports as re-exports where any out-of-domain consumer imports them, so the blast radius stays inside the domain.

Must Not

  • Never mix decomposition with behavior changes. No bug fixes, no auth additions, no response-shape normalization, no logging swap, no validation middleware — those are separate plans. If a bug is found mid-extraction, note it and keep moving.
  • Do not move db() calls in this phase if the db-access-into-models phase already owns them — consume its model methods; do not duplicate the migration.
  • Do not introduce new dependencies, new patterns, or a new response envelope.
  • Do not refactor or "clean up" code outside the file being decomposed.
  • Do not touch the frontend, migrations, or seeds.
  • Do not batch multiple god-files into one commit or one checkpoint.

Out Of Scope

  • The security holes (unauthed admin routes, IDOR, JWT fallback) the audit found — separate security-hotfix track, ahead of this one.
  • The 5 systemic breaches (no validation, no Pino, response shape, DB-outside-models beyond what the upstream phase moves, missing transactions) — their own plans.
  • Files already at or under 300 lines, and the gbp-automation reference domain itself.

Risk

Level: 4 (Major Impact) — the largest blast radius in the remediation programme. AdminWebsitesController alone backs ~210 route registrations; practice-ranking and agents back paid LLM/Apify pipelines and scheduled jobs. A silent behavior change here breaks the website builder, the ranking pipeline, or the monthly agent runs. Do not auto-execute. Run last, behind the test net, one unit at a time.

Risks Identified

  • Silent behavior drift during extraction. A moved function can change closure capture, import order, or error-propagation timing. Mitigation: smoke harness baseline captured before the phase; re-run after every unit; diff the response. No green diff → revert the unit.
  • Hidden cross-file consumers. A handler or helper may be imported from outside its domain. Mitigation: grep every moved symbol for importers before moving; keep a re-export shim at the old path for the duration of the phase if any out-of-domain importer exists.
  • checkup.ts may be dead weight. The audit found its default export checkupRoutes has no consumer in src/ (confirmed: the only grep hit for "checkup" is the file itself). Refactoring 2,140 lines of possibly-unwired code is wasted effort. Mitigation: T1 gates it — verify wired vs dead before any extraction; if dead, the decision is delete-or-archive (escalate to owner), not refactor.
  • Double-touch with the upstream DB phase. If decomposition starts before db-access-into-models finishes, every file is edited twice and conflicts compound. Mitigation: hard dependency — this phase does not start until that phase is merged.
  • Giant-function extraction (processLocationRanking ~1,159 lines, processMonthlyAgents ~417, analyzeBatch ~453) is the highest-risk move — shared mutable state across phases. Mitigation: extract phase-helpers in place first (same file, behavior identical), checkpoint, then relocate to modules in a second commit. Two small safe steps, not one big one.

Blast radius: routes/admin/websites.ts (~210 registrations) → AdminWebsitesController; routes/practiceRanking.ts + scheduled ranking jobs → practice-ranking; routes/agentsV2.ts + monthly scheduler → agents; routes/user/website.ts → user-website; routes/pms.ts → pms; frontend/DFYWebsite.tsx consumes the shortcode resolver output. Every one of these is a live consumer.

Pushback

  • This is the right work but the wrong thing to rush. Run it dead last in the remediation sequence, never interleaved with the security-hotfix or the systemic-breach plans, and only once the smoke harness gives a green baseline. The single highest-value, lowest-risk first move is T1 (settle whether checkup.ts is even live) — it may delete 2,140 lines outright instead of refactoring them.
  • Recommended alternative to a big-bang per-file rewrite: extract-helpers-in-place → checkpoint → relocate-to-modules → checkpoint. Two commits per giant function, each independently revertible, beats one 1,159-line move you cannot bisect.

Tasks

Ordered by dependency and risk: gate the dead-code question first (T1), then the highest-friction god-file (T2), then per-domain. Each task = one domain; each extracted unit within a task is its own commit + checkpoint. Sequential by default — these files share too much surface to parallelize safely, and the checkpoint cost is per-unit regardless.
