Why

PR 160 ships the Client Taste Profile spine — the composition service, the honesty gate, the taste_profiles table, and the model. A parallel review requested changes on four findings. Two were real defects in the code: the model imported its own persisted types from a controller (§7.1), and three id-based methods let one organization read or mutate another organization's profile (§11.7). The other two were honesty findings: the "compose + persist" contract is claimed but never exercised by production code, and the acceptance artifacts were missing.

This plan records the fixes and — just as important — narrows the stated contract so the PR stops describing a capability that no production path calls.

What

Fixed in code

  • §7.1 — persisted shapes moved to a neutral src/types/tasteProfile.ts; the model no longer imports from a controller.
  • §11.7 — organizationId is now a required argument on every id-based read and mutation, and the unscoped BaseModel entry points are sealed.
  • §20.2 — a cross-org isolation suite proving one org cannot read, approve, or delete another's profile.

Narrowed, not faked

  • The compose→persist path has no production entry point. Rather than invent one, the claim is narrowed in the code docstrings, here, and (proposed) in the PR body.
  • Layer-2 acceptance against a running app is recorded as pending, not waived to green — there is no surface to drive.

Success condition: no tenant can touch another tenant's profile, no model imports upward, and nothing in the PR claims a live capability that only tests exercise.

Narrowed contract

This is the section the code docstrings point at. Read it before describing this slice to anyone.

What this slice actually delivers: two independently unit-tested capabilities — a composition + honesty gate that turns already-computed extractor outputs into a gated TasteProfile, and a tenant-scoped model that can persist one. Plus the additive taste_profiles migration.

What it does NOT deliver: a production path that composes and persists. Verified on the branch head: composeTasteProfile() and composeFromExtractors() are called only by src/__tests__/taste-profile-composition.test.ts, and TasteProfileModel has zero importers anywhere in src/. No route, no job, no consumer.

Why no composeAndPersist() helper was added: a helper that nothing calls moves the unwired seam up one layer without creating an entry point — it would still be "called only by tests", which is the finding, not a fix. Real wiring needs a product decision this slice does not make: what triggers a compose (review/GBP sync? website build? an owner action?), which org/location context it runs under, and which surface consumes the result. Inventing that would be fake wiring on a public repo without a spec. It belongs to the slice that adds the owner preview/approve path.

Say this, not that. Say: "composition and persistence exist and are unit-tested; nothing calls them together yet." Do not say: "Alloro composes and persists a Taste Profile per business."

Rev 5 - 2026-07-16

Change: Third-round review fixes. (1) The owner signature was destructible (§5.4). markApproved() filtered only by { id, organization_id }, so calling it on an already-approved row re-ran the UPDATE and overwrote approved_by/approved_at. A later caller in the same org could silently reattribute the sign-off to a different person and time. Root cause: Rev 1 secured who may call the transition (the tenant predicate) but never constrained which rows it may act on — the status was treated as an output of the write, never as a precondition of it. Owner approval is the point where a human stakes what the AI drafted; a signature a later call can overwrite is not a signature, and the audit trail would name someone who never approved. Fixed by adding status: "draft" to the WHERE clause, making draft → approved a one-way transition: an approved row stops matching its own update predicate. The guard is positive rather than whereNot status = approved, so it fails closed — a future status (revoked, archived) cannot transition to approved and mint a fresh signature until the predicate is deliberately widened. It is a single atomic guarded UPDATE, not a read-then-write check, which would race two concurrent approvals. Design decision: a re-approval is a no-op returning 0, not a typed error — a retried job or a double-clicked Approve button re-issues the call and the desired end state already holds, so the idempotent answer is "0 rows changed". §21.1 settles it: "a job may run more than once (retries, at-least-once delivery); design every job so a repeat run is safe" — a guarded UPDATE is exactly the idempotency guard that Article calls for, and a throw would make a safe retry an error path. Whether a 0 means 404, 409, or a benign no-op is a policy question belonging to the calling surface, not to a thin DB-correctness layer with no business logic in it (§6.1). Returning 0 uniformly for wrong-org / no-such-id / already-approved also preserves the no-existence-leak property (§11.7). (2) The class, not the instance. Every prior round of this PR failed by fixing only the reported instance, so the write-path sweep was mechanized rather than eyeballed: a new test walks the prototype chain and asserts the model's entire runtime callable surface (20 names) equals a frozen, classified set — 2 scoped readers, 3 audited writers (create, markApproved, deleteByIdForOrg), 8 seals, 2 non-table delegates, 5 BaseModel internals. The sweep found no second instance: markApproved was the only unsealed path able to rewrite a signature, and Rev 3's seals and draft-only create() hold. A second test seeds an approved row and drives every candidate write at it with a forged signature, asserting the signature survives each. Both are mutation-verified. (3) §20.5 finalization resolved. Every non-pass item now carries an explicit waiver with a reason and a named authority, so finalization is unambiguous rather than drifting at "In Progress": T7 waived as out of scope (internal-only foundation, Dave's round-3 option A); T9 waived N/A on Dave's round-2 ruling that "the absence of PR CI itself is N/A, not a blocker"; T14 waived as an accepted upstream limitation per Dave's round-3 instruction. This PR ships a SQL migration, yet the artifact listed no item for executing it through three rounds — it asserted migration safety while never naming the run as an obligation. Added T18, recorded pending and waived under Dave's live-PostgreSQL owner waiver (PR #160 comment, 2026-07-16T17:54:36Z). Change: Approvals are now append-only (§5.4) — T20 is closed, not recorded. The finding: Rev 5's write-once guard made an approved row immutable, and that was not enough. A record's identity here is org+location, not row id — consumers read by org+location — so deleteByIdForOrg() + create() + markApproved(), every call typed, tenant-scoped and individually legitimate, produced a new approved row signed by whoever ran it with the previous signature gone entirely. The fix, in two guards that only work together: (1) deleteByIdForOrg() now carries status: "draft" in its DELETE predicate — an approved or superseded row cannot be deleted through this model at all, and the block lives in the query the database evaluates, not in caller discipline; (2) markApproved() retires the incumbent approved profile for the same org+location to a new superseded status in the same transaction, keeping its approved_by/approved_at intact. Why this shape. Rev 5 listed three options; this implements the first two together and deliberately rejects the third (an append-only audit table) for two reasons: an audit table observes a deletion rather than preventing it — it would have narrated this exact laundering instead of blocking it — and it is a second write recording the same fact, which can drift from the first, making the audit itself a claim without a receipt. The rows are the ledger: nothing approved is ever deleted, nothing approved is ever mutated except the one-way retire, and every row keeps the person who staked it and when, so the rows for an org+location ordered by approved_at are the approval history. The record is the row. Read the guarantee precisely: it does not prevent a different person from becoming the current approver — owner B composing a new profile and signing it is a legitimate staked act and is allowed. It prevents doing that invisibly. The threat was never "someone else approves"; it was "someone else approves and the previous stake disappears". Two adjacent defects found and fixed while here: findLatestByOrgAndLocation returns the newest row of any status, so a consumer reading it would have published an unapproved draft as though a human had staked it — defeating the entire approval gate. Added findCurrentApprovedByOrgAndLocation (status = approved only) as the consumer read, and the old method is now documented as a drafting-surface read that is explicitly not the published record. Rev 7 - 2026-07-16

Change: Adversarial round on Rev 6, plus an incident this round caused and must record. An independent adversary, tasked to refute the append-only fix rather than bless it, ran 8 mutants. Seven were caught. One survived — and it mattered. (1) The guard the docstring calls "the" guard was untested. Deleting status: "draft" from markApproved's final UPDATE (leaving it on the read) left the suite 38/38 green and tsc exit 0. That is the precise property the model stakes its concurrency argument on — "the guard is the UPDATE's predicate, not the read… a stale read costs 0 rows, never a signature". Root cause, and it is a lesson worth keeping: the test named for that property asserted that lastWhereConditions — a flat log of every WHERE the whole call built — contained {id, organization_id, status: "draft"}. Rev 6 added a draft-scoped READ to markApproved, and that read pushes the identical shape, so the assertion was satisfied for free. Rev 6 did not touch the Rev 5 fix; it silently removed its proof. A guard is a property of ONE statement, so it is now asserted on that statement: the mock records each UPDATE's and DELETE's own accumulated predicate, and the tests assert the approval UPDATE's, the supersede UPDATE's, and the DELETE's predicates individually. The guard is not redundant with the read — whenever ctx is not a real transaction, forUpdate() holds no lock past its own statement, two callers both pass the read, and without the clause the second reattributes the first's signature. (2) "The table is its own ledger" had no typed reader. The class docstring rejects a separate audit table on the grounds that these rows ARE the history — but every reader returned a single row and findMany/paginate are sealed, so no typed caller could read the history, through the very boundary the guarantee is scoped to ("no TYPED caller…"). A ledger nobody can read is retained bytes, not a ledger, and the justification for rejecting the audit table was hollow. Worse, T20 "proved" reconstructibility by reading the mock's internal rows array — an access no real consumer has, so the claim outran the evidence. Added findApprovalHistoryByOrgAndLocation (approved + superseded, newest stake first, drafts excluded because a draft was never staked); T20 now reconstructs the history through that typed reader. (3) Recorded, not fixed: QueryContext = Knex | Knex.Transaction, so markApproved(id, org, by, db) typechecks with a non-transactional handle, making supersede+approve non-atomic — weak, because the caller needs db, which is already strictly more power than this undermines. Axes the adversary found nothing on, stated plainly: location_id null-vs-0 handling (every comparison is === null, never falsy; scope is taken from the row, never a caller argument); resurrection of a superseded row (both positive predicates refuse it); over-supersession (each retired row keeps its own signature); and the T18 waiver, independently re-verified as legitimate (git diff 18742ebe -- src/database/migrations/ genuinely empty).
