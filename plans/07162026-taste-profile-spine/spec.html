Why

PR 160 ships the Client Taste Profile spine — the composition service, the honesty gate, the taste_profiles table, and the model. A parallel review requested changes on four findings. Two were real defects in the code: the model imported its own persisted types from a controller (§7.1), and three id-based methods let one organization read or mutate another organization's profile (§11.7). The other two were honesty findings: the "compose + persist" contract is claimed but never exercised by production code, and the acceptance artifacts were missing.

This plan records the fixes and — just as important — narrows the stated contract so the PR stops describing a capability that no production path calls.

What

Fixed in code

  • §7.1 — persisted shapes moved to a neutral src/types/tasteProfile.ts; the model no longer imports from a controller.
  • §11.7 — organizationId is now a required argument on every id-based read and mutation, and the unscoped BaseModel entry points are sealed.
  • §20.2 — a cross-org isolation suite proving one org cannot read, approve, or delete another's profile.

Narrowed, not faked

  • The compose→persist path has no production entry point. Rather than invent one, the claim is narrowed in the code docstrings, here, and (proposed) in the PR body.
  • Layer-2 acceptance against a running app is recorded as pending, not waived to green — there is no surface to drive.

Success condition: no tenant can touch another tenant's profile, no model imports upward, and nothing in the PR claims a live capability that only tests exercise.

Narrowed contract

This is the section the code docstrings point at. Read it before describing this slice to anyone.

What this slice actually delivers: two independently unit-tested capabilities — a composition + honesty gate that turns already-computed extractor outputs into a gated TasteProfile, and a tenant-scoped model that can persist one. Plus the additive taste_profiles migration.

What it does NOT deliver: a production path that composes and persists. Verified on the branch head: composeTasteProfile() and composeFromExtractors() are called only by src/__tests__/taste-profile-composition.test.ts, and TasteProfileModel has zero importers anywhere in src/. No route, no job, no consumer.

Why no composeAndPersist() helper was added: a helper that nothing calls moves the unwired seam up one layer without creating an entry point — it would still be "called only by tests", which is the finding, not a fix. Real wiring needs a product decision this slice does not make: what triggers a compose (review/GBP sync? website build? an owner action?), which org/location context it runs under, and which surface consumes the result. Inventing that would be fake wiring on a public repo without a spec. It belongs to the slice that adds the owner preview/approve path.

Say this, not that. Say: "composition and persistence exist and are unit-tested; nothing calls them together yet." Do not say: "Alloro composes and persists a Taste Profile per business."

Constraints

  • Node 22 required — Node 20 throws ERR_REQUIRE_ESM in vitest.
  • No live Postgres in the suite (vitest.config.ts, Option B). Isolation is proven against an in-memory table that applies the model's .where() filters.
  • TypeScript forbids widening a base static with a required parameter (TS2417), so organizationId cannot simply be added to findById/deleteById.
  • No migration change — the table already carries organization_id and its indexes.
  • No new public route or frontend surface in this PR.

Rev 3 - 2026-07-16

Change: Second-round review fixes, plus one independently-found defect. (1) The seals were incomplete (§11.7/§5.5). Rev 1 sealed only findById/deleteById; the rest of BaseModel's public surface was still inherited, so a caller could reach this tenant table with no organization predicate. Root cause: Rev 1 sealed the two entry points the first review named instead of enumerating the base class's public surface. Every inherited unscoped entry point is now sealed with the same compile-time technique — findOne, findMany, updateById, count, paginate, and createReturningId, which the review did not list and which was found by reading BaseModel directly (it is a write path that also bypassed create()'s draft-only contract). count() is the one honest exception: the base method is callable with zero arguments, so its seal is runtime-only, and that is documented in the model. Cross-org tests added for each, asserting both the throw and that no other tenant's row is read or mutated. (2) Approval could be minted at insert time (§5.4). create() accepted an optional status and wrote it directly, so an approved row could be created with no approved_by/approved_at — an unsigned approval that reads downstream exactly like a signed one. status is removed from the input type entirely; create() always writes draft, and markApproved() — which writes the status and the signature in one update — is the only route to approved. (3) The forged join (found here, not in the review). resolveReviewSource promised in its own docstring that "we never fabricate a source" while doing the opposite: it resolved a source on the reviewer name alone, never examining the quote, and its containment test ran in both directions, so a long claim that merely wrapped a short review ("Great!") inherited that review's receipt. Both produced a clean, followable link in the audit trail for words the review never said. Containment is now one-directional (the quote must be verbatim within the review's text — the same structural check extractPracticeFacts.processor.verifyFacts already applies to fact excerpts), a name is only ever a disambiguator or a veto, and an ambiguous quote is dropped rather than resolved to a guess. (4) isRealSource was string-presence only, so any invented label counted as provenance; it now accepts only the three forms the system actually produces (review:<id>, an http(s) URL, or a labeled excerpt from a known PracticeFactSourceField) and fails closed. (5) fact_text vs its excerpt — recorded as an open limit (T14), not closed. The excerpt is verified verbatim upstream in the extractor; this layer receives only the extractor's output and cannot re-derive whether the restatement is supported. No overlap heuristic was added on purpose: it would pass almost any restatement reusing a few of the excerpt's words, making an unverified claim merely look verified — the exact failure this gate exists to prevent. The boundary is documented at the mapping and the real fix belongs in the extractor.

Reason: User QA: Dave's second-round review pinned to 15b84963 requested changes on three findings (incomplete seals, the create() status hole, and §20.5 honesty). Scope addition: the forged join in resolveReviewSource, the isRealSource allowlist, and the fact_text limit were found by reading the code during this round and fixed here because they are the same defect class as the review's findings — a guarantee the docstring asserts but the code does not enforce.
