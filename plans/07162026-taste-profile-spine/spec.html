Why

PR 160 ships the Client Taste Profile spine — the composition service, the honesty gate, the taste_profiles table, and the model. A parallel review requested changes on four findings. Two were real defects in the code: the model imported its own persisted types from a controller (§7.1), and three id-based methods let one organization read or mutate another organization's profile (§11.7). The other two were honesty findings: the "compose + persist" contract is claimed but never exercised by production code, and the acceptance artifacts were missing.

This plan records the fixes and — just as important — narrows the stated contract so the PR stops describing a capability that no production path calls.

What

Fixed in code

  • §7.1 — persisted shapes moved to a neutral src/types/tasteProfile.ts; the model no longer imports from a controller.
  • §11.7 — organizationId is now a required argument on every id-based read and mutation, and the unscoped BaseModel entry points are sealed.
  • §20.2 — a cross-org isolation suite proving one org cannot read, approve, or delete another's profile.

Narrowed, not faked

  • The compose→persist path has no production entry point. Rather than invent one, the claim is narrowed in the code docstrings, here, and (proposed) in the PR body.
  • Layer-2 acceptance against a running app is recorded as pending, not waived to green — there is no surface to drive.

Success condition: no tenant can touch another tenant's profile, no model imports upward, and nothing in the PR claims a live capability that only tests exercise.

Narrowed contract

This is the section the code docstrings point at. Read it before describing this slice to anyone.

What this slice actually delivers: two independently unit-tested capabilities — a composition + honesty gate that turns already-computed extractor outputs into a gated TasteProfile, and a tenant-scoped model that can persist one. Plus the additive taste_profiles migration.

What it does NOT deliver: a production path that composes and persists. Verified on the branch head: composeTasteProfile() and composeFromExtractors() are called only by src/__tests__/taste-profile-composition.test.ts, and TasteProfileModel has zero importers anywhere in src/. No route, no job, no consumer.

Why no composeAndPersist() helper was added: a helper that nothing calls moves the unwired seam up one layer without creating an entry point — it would still be "called only by tests", which is the finding, not a fix. Real wiring needs a product decision this slice does not make: what triggers a compose (review/GBP sync? website build? an owner action?), which org/location context it runs under, and which surface consumes the result. Inventing that would be fake wiring on a public repo without a spec. It belongs to the slice that adds the owner preview/approve path.

Say this, not that. Say: "composition and persistence exist and are unit-tested; nothing calls them together yet." Do not say: "Alloro composes and persists a Taste Profile per business."

Context

  • §7.1 — "Routes → Controllers → Services → Models. Each layer talks only to the next." Mechanized by dependency-cruiser's models-no-upward-imports rule.
  • §11.7 — "Every model query against a tenant-scoped table filters by the organization/location from server context, passed as a required argument — not an optional filter a caller may forget."
  • §20.2 / §20.5 — tenant-isolation tests and the acceptance artifacts (test.html + test-results.json).
  • Reference analogs: GbpWorkItemModel.findByIdForScope (scoped by-id read), PracticeFactModel + PracticeFactModel.test.ts (thin model + in-memory isolation test).
  • Neutral shared-type precedent: src/types/adminReset.ts, src/types/pmsMapping.ts.

Note on Article numbering. §11.7, §20.2, §20.5 and §5.5 are cited by the review and by scripts/check-conventions.sh, but they do not exist in the repo's code-constitution.html viewer, which stops at Part 17 and ends §11 at §11.6 — and the canonical ~/.claude/skills/code-constitution/SKILL.md is absent from this machine. The viewer is stale relative to the contract. The work followed the rule text quoted in the review plus repo precedent, not the stale viewer.

Constraints

  • Node 22 required — Node 20 throws ERR_REQUIRE_ESM in vitest.
  • No live Postgres in the suite (vitest.config.ts, Option B). Isolation is proven against an in-memory table that applies the model's .where() filters.
  • TypeScript forbids widening a base static with a required parameter (TS2417), so organizationId cannot simply be added to findById/deleteById.
  • No migration change — the table already carries organization_id and its indexes.
  • No new public route or frontend surface in this PR.
