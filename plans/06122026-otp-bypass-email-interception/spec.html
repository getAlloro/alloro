OTP Bypasses Email Interception

Status
Completed
Plan Folder
plans/06122026-otp-bypass-email-interception
Size
Small
Risk
Level 3 (user-ratified)

Why

The email interceptor reroutes every non-production email to dave@getalloro.com (fail closed). That blocks the login flow for anyone but Dave on dev/local: their OTP code lands in Dave's inbox, not theirs. OTP codes must reach the person who requested them in every environment.

What

Add an opt-in allowLiveSend flag to SendEmailOptions. When set, sendEmail() skips the interceptor and sends live regardless of the box's identity. Only the OTP send path sets it. Done = an OTP requested on a non-prod box arrives at the requester with no [Intercepted] prefix, and every other email path still intercepts.

Context

Relevant Files

  • src/emails/types.ts - SendEmailOptions shape (the flag lives here)
  • src/emails/emailService.ts - sendEmail(), exit #1; calls the interceptor at line 169
  • src/emails/emailInterceptor.ts - the guard being bypassed (unchanged)
  • src/controllers/auth-otp/feature-services/service.otp-generation.ts - createAndSendOtp(), the only setter

Patterns To Follow

  • Bypass is applied inside the exit function (matches how interception is applied), not at the call site.
  • Opt-in per-send: an optional boolean on the existing options object — backward compatible, no other caller affected.

Reference file: src/emails/emailInterceptor.ts - the existing identity-based exit logic this rides alongside.

Constraints

Must

  • Default behavior unchanged when the flag is absent (interception still fail-closed).
  • Only createAndSendOtp sets the flag.

Must Not

  • Do not touch emailInterceptor.ts internals.
  • Do not set the flag on password-reset, invites, contact, or any other path.
  • No new env var, no NODE_ENV branch (keeps the identity-not-config design intact).

Out Of Scope

  • Password-reset and invitation emails (stay intercepted).
  • The two other email exits (emailWebhookService, service.n8n-email-sender) — unchanged.

Risk

Level: 3 (Structural / security exposure) — accepted by user.

Risks Identified

  • Inverts the fail-closed default on an auth-email path. Non-prod boxes (dev, localhost, CI) can now send a real OTP to a real inbox. Mitigation: bypass is opt-in per-send and set on exactly one call; blast radius is OTP only. OTP recipients are self-selected (the address typed into the login form), so the normal flow only mails the requester.
  • CI hitting POST /api/auth/otp/request with a real address now sends live mail. Mitigation: flagged to user; accepted. Follow-up if CI exercises this path: stub the send or use test accounts (isTestAccount already short-circuits before send).

Blast radius: SendEmailOptions is consumed by many templates and sendEmail() callers; adding an optional field is backward compatible and none of them set it. Only createAndSendOtp changes behavior.

Pushback

  • Recommended the narrower "internal-domains-only on non-prod" scope (protects client addresses from accidental dev/CI/local sends while still unblocking team dev-login). User chose full bypass with the exposure in view; honored.
