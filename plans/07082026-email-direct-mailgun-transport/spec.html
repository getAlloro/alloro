Why

Every outbound app email currently routes through an n8n webhook (ALLORO_EMAIL_SERVICE_WEBHOOK): emailService.sendEmail POSTs the payload to n8n, and n8n performs the real Mailgun send. The app has no direct control over delivery, and open/delivery tracking depends on n8n echoing back Mailgun's message-id — a fragile correlation.

A direct Mailgun transport was already written but is stranded as uncommitted WIP on the dave/email-manager worktree — ~365 commits (5 weeks) behind dev/dave, tangled inside a larger unfinished "Email Manager" (DB templates, suppressions, admin UI) built on a June email schema that conflicts with the email_logs table that shipped in July. It cannot be merged. This plan lifts only the transport layer onto current dev/dave.

What

Introduce a small transport abstraction inside the single email service point and add a direct Mailgun HTTP transport, so the app sends email itself instead of delegating to n8n.

  • Transport dispatch. emailService.sendEmail resolves a transport (mailgun | n8n) and dispatches to it, instead of hard-coding the n8n POST.
  • Direct Mailgun transport (ported from the worktree): sends via the Mailgun HTTP API with axios (no new dependency), returns the real provider message-id, never throws.
  • Mailgun is the active/default path; n8n is retained as an env-flippable fallback (EMAIL_DEFAULT_TRANSPORT) with automatic fallback when Mailgun is not configured — so a bad cutover cannot cause an email outage, and rollback is one env var.
  • Clean tracking. Store the normalized Mailgun message-id (<id@domain> → id@domain) as provider_message_id on the existing email_logs row, so the shipped mailgun-events webhook correlates delivery/open events without depending on n8n's response.
Transport resolution (full switch, safe rollout):

resolveTransport() = EMAIL_DEFAULT_TRANSPORT if set, else mailgun when MAILGUN_API_KEY+MAILGUN_DOMAIN are present, else n8n. So setting the Mailgun creds on an env flips it to direct send; before that (or by setting EMAIL_DEFAULT_TRANSPORT=n8n) it stays on n8n. No per-type allowlist — full switch.

Done when: with Mailgun configured on dev, all app email sends directly via Mailgun, the email_logs row carries the normalized provider id, and the mailgun-events webhook flips it to delivered/opened; flipping EMAIL_DEFAULT_TRANSPORT=n8n restores the n8n path unchanged.

Constraints

Must

  • Preserve emailInterceptor (non-prod reroute) and the OTP/allowLiveSend bypass — login/verification codes must still deliver.
  • Keep sendEmail's public contract (EmailResult) unchanged for all callers.
  • Write the existing email_logs row via the current recordEmailLog, storing normalizeMessageId(result.messageId) as provider_message_id.
  • Mailgun is the active/default path; n8n retained behind EMAIL_DEFAULT_TRANSPORT + auto-fallback when Mailgun is unconfigured.

Must Not

  • No new email schema or migrations — do NOT bring the worktree's email_log/email_events/email_suppressions/email_templates.
  • No suppression filtering, DB templates, or admin Email Manager UI.
  • No new dependency (no Mailgun SDK).
  • Do not modify the shipped mailgun-events webhook.

Out Of Scope

  • The full Email Manager (DB templates, suppressions, admin UI) — the rest of the stranded worktree.
  • Hard-deleting the n8n transport — retire it in a later cleanup once Mailgun is proven on prod.
  • Mailgun account + sending-domain provisioning (SPF/DKIM verification) and setting the per-env creds — an ops prerequisite (owner).

Prerequisites (ops — owner)

  • A Mailgun account with a DNS-verified sending domain (e.g. mail.getalloro.com).
  • MAILGUN_API_KEY + MAILGUN_DOMAIN (+ MAILGUN_API_BASE if EU region) on each env's server env file; then set EMAIL_DEFAULT_TRANSPORT=mailgun when that env is ready.
  • (For delivery/open tracking) MAILGUN_WEBHOOK_SIGNING_KEY + the Mailgun event webhook pointed at /api/webhooks/mailgun-events — separate, already-shipped receiver.

Risk

Level: 3 — Structural. The change sits in the single send point that every outbound email flows through.

Risks Identified

  • Cutover outage. A full switch to Mailgun means a misconfiguration (unverified domain, wrong region base URL, bad key) breaks ALL email silently. Mitigation: keep n8n as an env-flippable fallback + auto-fallback when Mailgun is unconfigured (resolution rule above); verify on dev with a real domain first; roll out per-env via EMAIL_DEFAULT_TRANSPORT. Do NOT hard-delete n8n in this plan.
  • Login/verification codes. Code emails use allowLiveSend to bypass the interceptor. Mitigation: transport dispatch is downstream of the interceptor, so allowLiveSend still applies; cover with a dispatch test.
  • Deliverability / spam. An unverified or cold sending domain lands mail in spam. Mitigation: prerequisite — DNS-verified domain (SPF/DKIM) before flipping an env to Mailgun.
  • Message-id correlation. Mailgun returns <id@domain>; events carry id@domain. Mitigation: store normalizeMessageId() output as the correlation key.

Blast radius: every sendEmail caller — support replies, website contact/lead forms, admin + user notifications, checkup results, and login/verification codes. All continue through the interceptor unchanged; only the delivery transport changes.

Pushback

  • "Retire n8n entirely" now removes the only rollback path for a Level-3 cutover and buys nothing over keeping the module dormant. Recommended: ship the switch with n8n retained behind the env toggle; delete n8n in a separate cleanup once Mailgun has run clean on prod for a defined window.
