Why

Alloro needs an Asia dev server that can receive the same dev/dave deployment artifact as the existing dev EC2 host while using a local Postgres copy of the dev database.

What

Provision ssh alloro-asia with the base runtime required by the app, clone the current dev database into local Postgres, place server-managed config files without printing secrets, and update the dev deploy workflow so one build artifact can deploy to both dev targets.

Context

Relevant Files

  • .github/workflows/dev.yml - current dev deployment pipeline for dev/dave.
  • ecosystem.config.js - PM2 process contract for backend and worker.
  • package.json - app build, start, and migration scripts.

Server Facts

  • alloro-dev uses Ubuntu 24.04, Node 22.18, PM2 6, Apache, Redis, and /etc/alloro/dev.env.
  • alloro-asia starts as a blank Ubuntu 24.04 DigitalOcean droplet with only SSH, git, curl, and rsync.
  • The dev source database reports PostgreSQL 17.9, so the Asia local database must run PostgreSQL 17 rather than Ubuntu 24.04's default PostgreSQL 16 package.

Reference file: .github/workflows/main.yml - closest deploy workflow analog for bundle, server config, migration, PM2 reload, and health check structure.

Constraints

Must

  • Do not print database credentials, API keys, tokens, or full env contents.
  • Keep existing alloro-dev deployment behavior intact.
  • Keep repo edits scoped to the deploy workflow and this plan folder.
  • Use local Postgres on alloro-asia for the cloned dev database.

Must Not

  • Do not copy prod-only services like MySQL, Docker, or n8n unless explicitly requested.
  • Do not run destructive commands against the existing dev database.
  • Do not manually deploy the Alloro app outside the GitHub Actions path.

Out Of Scope

  • DNS cutover, public domain setup, production traffic, and production database changes.

Risk

Level: 4

Risks Identified

  • Deployment topology changes can break dev/dave deploys. Mitigation: build once, publish an artifact, and keep the existing EC2 deploy as its own job.
  • Copying env files can leak secrets. Mitigation: transfer files over SSH/SCP only and never print values.
  • Database cloning can accidentally target production or destroy a local target. Mitigation: read DB source only from alloro-dev:/etc/alloro/dev.env and only drop/recreate the local Asia database.
  • The request conflicts with itself: "do not set up anything around Alloro yet" versus "deploy to Asia on dev/dave". Mitigation: set up server prerequisites and workflow readiness, but do not manually start a production-like app outside the workflow.

Blast radius: dev/dave GitHub Actions deploys, alloro-dev deploy reliability, alloro-asia local Postgres state, and any services that use copied dev env values.

Pushback

  • This is not a quick infrastructure tweak. It mixes server bootstrap, secrets movement, database cloning, and CI topology. Future-us will hate this if the deploy shape is not made explicit.

Revision Log

Rev 1 - 2026-06-12

Change: Updated the database runtime requirement from generic PostgreSQL to PostgreSQL 17 after the source dev database reported server version 17.9.

Reason: Execution discovery: pg_dump 16 cannot dump a PostgreSQL 17 source, and restoring a 17 dump into a 16 target would be an unsupported downgrade path.

Updated Done criteria: Asia local Postgres must be PostgreSQL 17 with pgvector available before restoring the dev dump.

Rev 2 - 2026-06-12

Change: Added the dev-asia.getalloro.com Apache vhost, issued a Let's Encrypt HTTPS certificate, enabled HTTP-to-HTTPS redirect, and configured the local Certbot default email as dave@getalloro.com.

Reason: User follow-up after pointing DNS for the new Asia dev hostname.

Updated Done criteria: The Asia dev hostname must resolve to the droplet, Apache must expose HTTP/HTTPS vhosts, and TLS must validate even before the app is deployed.

Rev 3 - 2026-06-12

Change: Changed the GitHub Actions handoff between the build job and deploy jobs from a raw uploaded directory to a compressed tarball that preserves hidden directories, symlinks, and executable bits.

Reason: First deploy run failed on both targets because the artifact handoff dropped node_modules/.bin, causing npm run db:migrate to fail with knex: not found.

Updated Done criteria: The deploy artifact must preserve node_modules/.bin so migrations can run on both dev targets.
