Why

Reading whether a website converts means reading a constellation, not one number. Microsoft Clarity is already connected and already harvested, but the extractor pulls only three metrics — sessions, dead clicks, and quickbacks — so two of the signals that actually describe visitor frustration were being thrown away on arrival: rage clicks and scroll depth.

Nothing needed to be requested differently to get them. The harvest already calls project-live-insights with no metric filter and the controller stores the raw payload, so the values have been landing in stored history all along and were simply never read out.

What

extractMetrics reads two more metrics; aggregateMetrics rolls them up honestly; both appear on the /clarity/getKeyData response. No API change, no harvest change, no trend-score change.

The honesty detail that drove the design

The two metrics do not share a shape, and the difference matters. RageClickCount mirrors DeadClickCount — a plain count. ScrollDepth carries a single field, averageScrollDepth, and it is null when Clarity has no reading.

The reflex — Number(x || 0), which is how every other metric in that file is read — would turn "we have no reading" into "visitors scrolled 0%". That is a fabricated measurement pointing the wrong way: it reads as a catastrophic engagement problem when the truth is an absent instrument. So the null is preserved as null through extraction and through the monthly rollup, which averages only the days that actually carried a reading. Counts still coerce to 0, because an absent count genuinely means none were recorded.

Constraints

Must

  • Preserve a missing scroll reading as null, never 0.
  • Preserve calculateTrendScore exactly.
  • Stay additive on KeyDataResult.

Must not

  • Change the Clarity API request or the harvest adapter.
  • Fold the new metrics into a score clients already see.
  • Claim a CRO diagnosis off these signals — that is a later brick.

Out of scope

  • Wiring these into the funnel diagnosis. The diagnostic gait (PR #214) is where a CRO constellation would eventually be read; feeding it these signals is a separate brick, downstream of the blocker below.
  • Bot stripping. Traffic.totalBotSessionCount is in the payload and nothing reads it. One live reading showed 5 bot sessions in 30 (~17%). The funnel rule says strip bots before optimizing anything. Its own brick.

Blocked — why this is not merging yet

The code is finished and its unit tests pass. It is being held anyway, deliberately, because it cannot be proven where it matters.

A read-only sweep of four mapped Clarity projects over a 3-day window found three returning zero sessions and one returning thirty. At least one of the silent three has real traffic in the same window by our own Search Console and Rybbit numbers, so "the site is quiet" does not explain it. The API call succeeds and the credentials work — the integration is fine and the instrument is dark. Cause is not diagnosed; the two candidates are a missing tracking snippet or a stale project-to-domain mapping. That is BUILD-QUESTIONS.md Q3, open for Dave.

So on the sites this most needs to work on, this change would return rageClicks: 0 and scrollDepth: null indefinitely, and look like working code. Merging it now would mean shipping a piece nobody can show working, and quietly building a CRO diagnosis on top of a dead instrument.

This lands when Q3 is resolved and a real receipt exists: capture restored on at least one proof site, and non-zero rage clicks plus a real scroll reading observed there through this code path. Not before.

What was verified, and what was not

Verified — with a receipt

  • The unfiltered project-live-insights response carries sixteen metric entries including RageClickCount and ScrollDepth, so no API or harvest change is required and stored history already contains them.
  • Both shapes, from live reads: RageClickCount mirrors the count metrics; ScrollDepth.averageScrollDepth returned 44.42 on a project with real traffic and null on projects with no captured sessions. Both paths observed, not inferred.
  • Typecheck clean; the unit suite passes, including the case distinguishing a genuine 0% from no reading.

Not verified — the reason this is blocked

  • This code path has never returned a non-zero value end to end. The live confirmation came from calling Clarity directly, not through getKeyData. T3 exists to close that, and cannot be closed until Q3 is.
  • The unit for averageScrollDepth is read as a percent from its range and Clarity's UI convention, not from documentation.
  • Whether these two signals actually improve a CRO diagnosis is untested and out of scope. This lands the data, not a conclusion drawn from it.
