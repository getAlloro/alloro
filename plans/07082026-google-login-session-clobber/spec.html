Why

Signing in through Google as dave@getalloro.com lands the admin in the app as info@getalloro.com. Root cause is confirmed with a live browser snapshot: cookie = dave (correct), localStorage = info (stale). The app reads identity from localStorage, so it shows info.

The backend is not at fault — it authenticates dave correctly and mints dave's JWT (dave's users row carries the bound google_sub). The wrong identity is produced entirely on the client.

The clobber, step by step:
  1. A stale info token sits in localStorage. It is older than 3.5 days, so it is past the sliding-refresh half-life (service.jwt-management.ts:22).
  2. Sign-in as dave: the backend callback sets cookie = dave and redirects to /auth/google/finish.
  3. The finish page loads fresh. The app shell (AuthProvider, telemetry) renders with the still-present info token captured in memory and fires info-authenticated background requests.
  4. authenticateToken sees info's token past half-life and returns a re-issued info token in the x-session-refresh header (middleware/auth.ts:36).
  5. The finish page copies cookie → localStorage (dave) and calls window.location.replace("/admin").
  6. The racing info responses resolve; the axios response interceptor blindly writes the header token back to localStorage.auth_token (api/index.ts:428) — overwriting dave. The cookie is never touched by the interceptor.
  7. /admin reads localStorage → info.

What

Two client changes plus one small shared helper and tests:

  • Identity-safe refresh (core fix). The response interceptor persists a re-issued token only when it belongs to the same user as the token currently in the target store. A sliding refresh may extend a session but must never change identity. A mismatched (or no-session) token is dropped.
  • Clear-then-set on the SSO finish page. Clear any prior localStorage session and lingering pilot sessionStorage before writing the new session, so no stale identity remains after the handoff.
  • One JWT decode path. Centralize atob payload decoding into utils/jwt.ts so the interceptor and currentUser.ts share it (today three sites hand-roll it).

Done when: with a stale info token in localStorage past half-life, logging in as dave lands as dave (localStorage = dave, cookie = dave); the exact failing snapshot no longer reproduces; the same-identity sliding refresh still updates the token; pilot-mode refresh still works.

Context

Relevant Files

  • frontend/src/api/index.ts — storeRefreshedToken() (the clobber site) + the axios response interceptor. The one layer that owns the JWT (§12.1).
  • frontend/src/pages/AuthGoogleFinish.tsx — the OAuth finish route; copies the cookie into localStorage, does not clear the old session.
  • frontend/src/utils/currentUser.ts — existing getCurrentUserId(); decodes the current session token's userId via atob. The decode analog to reuse.
  • frontend/src/hooks/useLocalStorage.ts — getPriorityItem() (sessionStorage → localStorage) that identity reads flow through.
  • frontend/src/utils/embeddedPilotSession.ts — embedded-pilot token store the interceptor also targets; must not regress.

Patterns To Follow

  • §12.1 / §17.5 — JWT through one path. Keep all token read/decode/write inside api/index.ts and shared utils/; do not scatter token logic into components.
  • §4.3 — DRY the decode. Extract the shared atob decode into utils/jwt.ts; have currentUser.ts delegate to it rather than adding a third copy.
  • §17.2 — no any. Decode returns a typed number | null; narrow from unknown, never any.
  • §16.2 — never swallow silently. A malformed token decodes to null (a typed, expected outcome), matching currentUser.ts; this is not a swallowed error.
  • §20.1 — test mirrors the unit. Add utils/jwt.test.ts beside the util, following existing utils/embeddedPilotSession.test.ts.

Reference file: frontend/src/api/index.ts (the §12.1 client triad) for structure; frontend/src/utils/currentUser.ts for the decode shape.

Constraints

Must

  • Keep the sliding-refresh feature working for same-identity sessions (token still extends; no forced logout at half-life).
  • Preserve pilot and embedded-pilot refresh behavior — the identity check compares against the correct target store per mode.
  • Reuse the existing atob decode approach; return typed results (§17.2).
  • Read the cookie value before clearing the old session on the finish page (clearing first would also wipe the new token).

Must Not

  • Do not change backend token issuance or the x-session-refresh emitter — the backend is correct.
  • Do not add a dependency (no jwt-decode library) — §4.4.
  • Do not refactor AdminTopBar.tsx's decode or the other login pages — out of scope.
  • Do not move identity reads into components or add a second token store — §17.5.

Out Of Scope

  • Clearing the session on the "Sign in with Google" button click (belt-and-suspenders; T1 makes it unnecessary).
  • Cookie-scope hardening on dev (.getalloro.com vs host-only duplicates) — the interceptor fix removes the clobber regardless; revisit only if a duplicate cookie is later observed.
  • Reworking how AuthGoogleFinish mounts inside the authed shell.
  • Backend changes of any kind.

Risk

Level: 3 — Structural. The core change is in storeRefreshedToken, which runs on every axios response (success and error) across the whole app.

Risks Identified

  • Every authenticated request passes through the changed code. A wrong identity comparison could either drop legitimate refreshes (users silently logged out after 3.5 days) or fail to block the clobber. Mitigation: the change is additive — a guard that only drops a write when identities differ or no session exists; same-identity path is unchanged. Cover with unit tests for same-identity (write), different-identity (drop), no-session (drop), and pilot mode.
  • Pilot / embedded-pilot mode uses a different token store. Comparing against the wrong store could break pilot refresh. Mitigation: compare the incoming token against the token in the same store the write targets (embedded → embedded, pilot → sessionStorage, normal → localStorage), branching exactly as the existing write does.
  • Malformed / unexpected token payload. Mitigation: decode is wrapped and returns null; a null incoming id is treated as "cannot confirm same identity" → drop the write (fail safe).

Blast radius: frontend/src/api/index.ts response interceptor — consumed by every apiGet/apiPost/apiPatch/apiPut/apiDelete call in the app. utils/currentUser.ts — consumers of getCurrentUserId() (author/uploader UI checks) must keep the same behavior after delegating to utils/jwt.ts. AuthGoogleFinish.tsx — admin SSO login only (P1).

Pushback

  • The deeper smell is that identity is driven through a localStorage slot that a background response can rewrite. A fuller design would make the freshly-authenticated session authoritative and not writable by an in-flight request from a prior identity. That is a larger refactor; the identity-safe guard is the correct, contained fix now and closes the vulnerability for every login flow. Recommend shipping this and noting the broader hardening as a future item.
