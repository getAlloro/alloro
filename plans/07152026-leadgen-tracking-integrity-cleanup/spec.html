Why

The admin Leadgen Submissions page currently presents several different facts as if they were the same. A derived email or domain match is displayed as “Account Linked,” abandoned sessions are hidden behind a stage label, and a raw 60-second browser timer is presented as “Spent 1+ Min on Report” even when the session has no audit to report on.

The local database also contains empty session shells produced when session creation succeeds but the first event does not. The cleanup must remove only those provably empty rows without erasing legitimate anonymous traffic or the invalid historical rows needed to prove the tracking defect.

Revision 6 extends that reviewed cleanup to the dev and production databases after live read-only checks found the exact same 15 strict-empty session IDs on both targets.

What

Make the backend the authority for valid leadgen state transitions, make the leadgen client measure real visible report time and preserve audit identity, make the admin read model show one honest row per session, and delete only strict-empty rows from the confirmed local, dev, and production databases.

Cleanup contract: a row is eligible only when it has zero events and every identity/link field is null: audit_id, email, domain, practice_search_string, user_id, and converted_at. Local cleanup is complete. Read-only live preflight found exactly 15 candidates in dev and 15 in production, with the same candidate-ID hash on both targets.

Pushback

  • Do not delete “blank-looking” rows. Most are legitimate anonymous sessions. The safe boundary is zero events plus zero identity/link data.
  • Do not keep the “Spent 1+ Min” wording. A mount timer does not prove active reading. The corrected claim is “Report Visible for 1+ Min.”
  • Do not turn derived matches into conversions. A coincidence on email/domain is evidence for review, not a persisted relationship.
  • Do not treat matching counts as sufficient. Production deletion is allowed only after the exact candidate hash also matches, the full-row backup is verified, and dev cleanup passes first.

Blast radius: public leadgen event ingestion, beacon ingestion, session summaries, admin list/stats/funnel responses, leadgen session persistence, report engagement instrumentation, email-gate unlocking, CSV/admin row rendering, and 15 strict-empty leadgen_sessions rows in each of dev and production. No event row, valid session, historical invalid-report row, customer dashboard, or database schema is authorized to change.
