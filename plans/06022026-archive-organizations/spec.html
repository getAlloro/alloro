Archive Organizations

Add a first-class organization archive lifecycle that hides archived organizations from normal admin views, archives connected website projects, disconnects custom domains, and prevents scheduled/manual automation from creating new work for archived organizations.

Status
Completed
Plan Folder
plans/06022026-archive-organizations
Size
Large
Risk
Level 4 - cross-cutting lifecycle and migration

Why

Deleting organizations is too destructive for normal offboarding, while billing lockout does not reliably stop public websites, domain resolution, GBP automation, PMS-triggered monthly agents, scheduled agents, or ranking runs. Alloro needs a reversible archive state with one clear operational meaning.

What

Admins can archive an organization from Organization Settings before the permanent Delete Organization danger zone. Archived organizations are excluded from default organization lists and visible only through an Archived tab/filter or direct admin URL. Archiving an organization also archives its connected website projects, disconnects custom domains, preserves historical data, and blocks new automated work.

Patterns To Follow

  • Use the website list view pattern from WebsitesList.tsx: default active view plus explicit Archive tab.
  • Use route-controller-service-model backend layering. Routes stay thin, controllers orchestrate, services apply business rules, models own Knex queries.
  • Use transactions for archive/unarchive because organization, projects, domains, integrations, and automation settings are updated together.
  • Use typed React Query hooks and API helpers for admin UI state. No direct API calls inside new components.
  • Use neutral admin styling already present in Organization Settings. Archive is cautionary, Delete remains the only red danger action.

Constraints

Must

  • Add real archive columns to organizations; do not overload subscription_status.
  • Archive all website projects connected by organization_id, not just the first project.
  • Disconnect custom domains by clearing custom_domain, custom_domain_alt, and domain_verified_at.
  • Persist previous domain values in archive metadata before clearing them.
  • Block new automation for archived organizations while preserving historical records.
  • Keep global BullMQ repeatable jobs installed; job processors must skip archived organizations.
  • Include Alloro Docs parity because admin UI behavior changes.

Must Not

  • Do not delete organization data, users, tasks, rankings, PMS rows, notifications, agent outputs, submissions, or website pages.
  • Do not revoke Google OAuth tokens during archive.
  • Do not silently reconnect domains on unarchive.
  • Do not change Stripe billing or cancellation behavior in this plan.
  • Do not modify unrelated PM/dashboard/ranking copy work already dirty in the worktree.

Risk And Pushback

This is Level 4. It changes organization lifecycle semantics, public site behavior, scheduled workers, admin UI, and database schema. Treat it as a platform lifecycle feature, not a small settings button.

Production Migration Risk

Adding nullable archive columns is low-lock risk, but indexes and backfill assumptions must be production-safe. The migration runs on dev first through dev/dave, then production after merge to main.

Mitigation: nullable columns, no destructive data rewrite, reversible down migration, explicit indexes.

Domain Disconnect Risk

Clearing custom domains is intentional but not truly reversible unless previous values are saved before mutation.

Mitigation: store prior domain values in archive metadata and require admin confirmation to reconnect.

Automation Drift Risk

Scattered checks will rot. Workers, manual endpoints, and queued jobs must share one lifecycle service/helper.

Mitigation: central OrganizationArchiveService and OrganizationLifecycleService plus model helpers.

Renderer Boundary Risk

If public rendering lives outside this repo, generated hostnames may still render unless that renderer also checks archive state.

Mitigation: include renderer verification and, if outside repo, get explicit permission before editing it.

Blast Radius

  • Admin organization management and organization detail settings.
  • Website project list/archive state and custom domain connection state.
  • Public website contact/newsletter submission paths and CRM push queueing.
  • GBP review sync, GBP local post sync/generation, review replies, and local post publishing.
  • Scheduled proofline and ranking agents plus manual monthly agent triggers.
  • PMS upload, approval, retry, restart, and monthly-agent trigger flows.
  • Practice ranking trigger, retry, batch retry, and competitor finalization flows.
  • Alloro Docs/admin documentation for organization management.
