Alloro Funnel Engine · Stage B (Get Considered) · Feature B2 · Spec Slice 2

CRO-lift rewrite — Taste-Profile-driven page copy

Make the hosted site's get-considered copy sound like the business patients already love — a draft the owner approves, with zero over-claim in their name. Reuse the Taste Profile spine (#160) and the AI-command approve→auto-publish rail; invent no new generator.

Status: In Progress — built, adversary-hardened, at PR gate Branch: claude/b2-cro-lift-rewrite → base claude/taste-profile-spine (#160) Migration: none Ships: DISABLED (TASTE_REWRITE_ENABLED off) + owner approval

Why (the service, not the feature)

Whose life this lifts: the overwhelmed local-business owner whose website is a thin scrapbook that sounds like every other clinic. They pay us because they cannot spend evenings writing web copy. B2 makes their page speak in their own proven voice — the exact phrases their happiest patients already use in reviews — without them writing a word, and without a single invented promise attached to their name.

Alloro is the guide; the owner is the hero (Value #2). B2 drafts; the owner approves; only then does anything publish. The honesty gate (Value #6) is what lets us hand an owner AI-written copy they can trust to be true about their business.

Funnel role: Driver 2 (Alloro Connect / presence) → Stage B "Get Considered." Audit: the website is BUILT + hosted but CRO/A-B is "absent"; it's a thin scrapbook. B2 is the get-considered rewrite that reads the true-voice spine.

What (behavior, precisely)

B2 adds a new taste_rewrite recommendation type that rides the existing AI-command rail. It is compose-and-wire: it reuses the Taste Profile spine, the batch/recommendation/approve rail, the LLM edit primitive, and the auto-publish path. It builds no new generator and does not touch instantWebsiteGenerator.ts.

  1. Input — the approved Taste Profile. Read TasteProfileModel.findLatestByOrgAndLocation(orgId, locationId). Enforce status === 'approved' in B2 (the model does NOT filter by status — see Context catch #2). No approved profile → generate nothing (honest empty), never a draft-seeded rewrite.
  2. Generate (LLM, sourced-facts-only). For each eligible page section, call the reused editHtmlContent primitive with a profile-grounded instruction that constrains the model to the profile's sourced claims only (headline, hero quote, praise themes, why-they-choose, credentials, practice facts) and forbids inventing facts, numbers, or promises. The LLM is an injected dependency so the gate + execution are tested without a live model.
  3. Gate the OUTPUT (this is the safety spine). Extract text from each rewritten section and run it through gateRewrite() = the spine's enforceHonesty() (banned rank/visibility/guarantee/invented-metric language) plus a new B2 subtle-over-claim scanner (unprovable superlatives / implied promises that are not banned keywords — "the gentlest dentist," "you'll love it," "best in town"). A section that trips either is DROPPED to an audit and never becomes a recommendation. Only clean rewrites become pending recommendations — so an over-claim is never even approvable.
  4. Owner approves. Reuse the existing approve routes and UI unchanged — draft → owner approves → publish. B2 adds no approval surface.
  5. Execute (deterministic, re-gated). The new executeTasteRewrite handler re-asserts gateRewrite() on the stored copy (defense-in-depth), then writes the stored gated HTML directly to the section via the existing pinned-draft mechanism → PageModel.updateSectionsById → end-of-batch auto-publish. No fresh LLM call at execution — the published bytes are exactly the approved, gated bytes.
The two structural guarantees (the adversary's targets): (a) over-claim is blocked in generated copy because the gate runs on the LLM output at generation and again at execution; (b) an unapproved rewrite cannot reach a live page because execution only processes status==='approved' rows and writes only the stored, pre-gated copy — never a fresh generation.

Constraints (honesty caps + hard rules)

  • Value #6 — no promises. Every rewritten section passes enforceHonesty() + the subtle-over-claim scanner before it is approvable AND again before it publishes. No rank / visibility / "get found" / guarantee / invented-metric language, and no unprovable superlative or implied promise.
  • Sourced-facts-only generation. The rewrite instruction constrains the LLM to the profile's sourced claims; it forbids inventing facts, numbers, testimonials, or outcomes. Invention that slips the instruction is caught by the output gate.
  • Value #2 — owner is the hero. Draft → owner approves → publish. Never auto-rewrite a live page. B2 reuses the existing approval gate; it adds no auto-publish that bypasses approval.
  • No new generator; no static-generator edits. Do not modify instantWebsiteGenerator.ts. Reuse editHtmlContent, the batch/recommendation/approve/execute rail, and the auto-publish path.
  • Do not mutate the spine's shared enforceHonesty(). B2's subtle-over-claim scanner is a new B2-local layer (util.taste-rewrite-honesty.ts) composed with enforceHonesty(), so #160's contract + tests stay stable. If the scanner proves general, folding it into the shared util is a fast-follow (noted, not done here).
  • No A/B testing in v1 (deferred). No conversion/lift promise.
  • Not this: booking, speed-to-lead, PMS write-back — out of the attraction lane.

Risk & migration safety

  • Migration: NONE. taste_rewrite reuses the existing ai_command_recommendations table (target_type free-form text; target_meta jsonb carries the stored rewritten HTML). Zero schema change → zero production migration risk.
  • Top risk — subtle over-claim slips the gate. enforceHonesty() catches banned keywords; a superlative/implied promise is not a keyword. Mitigation: the B2 subtle-over-claim scanner. Adversary's #1 job: try to slip a subtle over-claim past the combined gate in generated copy. If it succeeds, harden the scanner before B2 ships.
  • Gate-bypass at execution (the Catch #2 failure mode): mitigated structurally — execution writes only stored copy + re-asserts the gate; no fresh LLM.
  • Unapproved reaching live: mitigated — executeBatch reads only status='approved' rows; batch must be ready.
  • Ships behind owner approval — nothing publishes without the owner flipping a recommendation to approved. No feature flag needed; the approval gate IS the off-switch.
  • Cost: the LLM rewrite runs at generation time only (once per section), attributed via the existing costContext. No per-execution model spend.
