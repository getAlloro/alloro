A2 — GBP Own-Completeness Scoring

Get-found, in-lane. Score the practice's own Google Business Profile for completeness — deterministically, read-only — and return the exact set of missing fields as advisory recommendations. Extends the shipping get-found checker; scores data Alloro already has.

Plan: 07152026-gbp-own-completeness-scoring  ·  Funnel: Stage A (get found), item A2  ·  Source map: plans/07142026-alloro-funnel-engine/funnel-feature-sequence.md
Lane: Corey+Claude spec → sandbox build + adversary → PR to dev/dave for Dave to review + merge
Evidence: DIRECTIONAL — completeness correlates with crawl/trust eligibility, NOT placement. No rank claim.

Why

Alloro's job is Driver 2 — help people find the practice. A practice's Google Business Profile is a top get-found surface, and an incomplete profile (missing category, hours, or photos) leaves the practice less legible to Google and to a person deciding whether to visit. Today Alloro scores competitors' profile strength but never scores the client's own in a deterministic, testable way (verified: connect-lever-audit.md §discovery — "CLIENT's own completeness never scored/tracked (only competitors', util.competitor-profile-strength.ts)").

A2 closes that gap the same honest way Slice 1a closed the schema gap: a deterministic grader that returns the exact missing-field set as an advisory recommendation, so the owner (via the approved rail, in a later item) can complete their own profile. It improves eligibility and trust, never a ranking — Value #6.

What — in scope

Reuse first (reconciliation — verified against real code, not assumed).

  • Mirror the shipping completeness pattern: src/services/ai-seo-audit/schemaCompletenessScoring.ts — a pure grader that returns { presentFields, missingFields, completeness, …Incomplete } and whose incomplete-signal gates nothing. A2's scoreGbpCompleteness() is the same shape for GBP fields.
  • Plug into the shipping orchestrator: src/services/ai-seo-audit/getFoundChecker.ts — runGetFoundChecker already runs schema-completeness, GBP↔page consistency, answer-first, honesty, and observability, and assembles AdvisoryRecommendation[]. A2 adds one more piece + one more recommendation code: "gbp_completeness". No new orchestrator.
  • Reuse the data Alloro already fetches: the leadgen pipeline's condensed client_gbp record and the organization audit's existing getGBPAIReadyData().profile result. mapAiReadyGbpToCompletenessInput adapts the live organization-audit shape without another Google request. It grades only fields that source returned; the current AI-ready response has no photo count, so photos are not falsely reported missing.

The one new module + the wiring

  • New: src/services/ai-seo-audit/gbpCompletenessScoring.ts — a pure function scoreGbpCompleteness(clientGbp) grading the client's own GBP against the fixed set of real GBP fields the condensed record actually carries (business category, website, phone, opening hours, address, at least one photo). A GBP description is deliberately NOT graded — the condensed client_gbp record does not carry a description field, so grading it would emit a permanent false-“missing” (see Rev 2). Returns present/missing sets + a 0..1 completeness + an internal-only gbpIncomplete flag that gates nothing. Pure: no DB, no IO (matches the analog).
  • Wire: thread the optional completeness input into GetFoundCheckInput; in production, auditTargetExecutionService selects the target's matching location and adapts its already-fetched GBP profile. A single-location site may safely use its only profile when page mapping is absent; an unmapped multi-location page skips instead of borrowing another location's data. When present, run the score and add a gbp_completeness advisory recommendation via the existing buildRecommendations; extend the observability log line with the GBP missing-field count.

A2 is advisory + read-only. It writes nothing, touches no Google API for writes, and adds no autonomy — identical posture to Slice 1a.

What — explicitly NOT in A2

  • No GBP write-back (no category/hours/description patch to Google). That is item A6, a later slice with its own rollback design.
  • Not a replacement for the LLM audit. src/agents/auditAgents/GBPAnalysis.md already produces a narrative gbp_readiness_score. A2 is the deterministic, testable complement — an exact missing-field set the LLM prose can't guarantee — not a twin. If review shows the deterministic score should feed the LLM inputs, that wiring is a follow-up, not this PR.
  • No owner-facing "rank" or "visibility" language. Completeness ≠ placement.

Constraints

Honesty (Value #6). The completeness score and missing-field set improve eligibility/trust only. No owner-facing copy may claim or imply a rank/visibility outcome ("complete your profile to rank higher" is banned; "add your hours and services so people and Google can read your profile" is allowed). Any generated recommendation copy runs through the neutral shared honesty gate (GeneratedCopySafetyService.validateGeneratedCopy) exactly as corrected Slice 1a does.

Grade real fields only. Every graded field is a genuine GBP profile field the owner could actually fill (category, hours, website, phone, address, photo). Never invent a field to pad the score, and never mark a field "present" from inferred/guessed data — only from a real GBP source.

  • Read-only. No write to Google; no new autonomy. Advisory recommendations only.
  • Internal signal gates nothing. A gbpIncomplete flag is admin/internal, mirrors INTERNAL_AEO_INCOMPLETE_SIGNAL, and never appears in owner-facing copy.
  • Edge cases: no GBP connected → skip the GBP score entirely. A standard condensed record missing a key → grade it missing. A narrower adapter explicitly marks its gradable fields so a field the source never returned is ungraded, not falsely missing. Multi-location → use only the target's matching location; an unmapped multi-location page skips. A single-location site may use its only profile when page mapping is absent. Never guess a location or provider field.
  • Pure function. scoreGbpCompleteness takes the condensed record and returns the grade — no DB, no IO, no logging inside it (logging stays in the orchestrator, as in 1a).
  • Docs parity: A2 has no dashboard UI in this PR (advisory output only). If a later item renders it, that item adds the alloro-docs parity task; state so then.
  • Lane: Corey+Claude author; build + adversary in the sandbox; PR to dev/dave; Dave reviews + merges. Never push to dev/dave or main directly.

Risk

  • Duplicating the LLM audit (the main reconcile risk). Mitigation: A2 is explicitly the deterministic missing-field grader, not a narrative score; a reviewer should reject any A2 code that re-implements the LLM's judgment. It complements GBPAnalysis.md.
  • Provider-shape drift. The leadgen condensed record and the organization-audit AI-ready profile are different shapes. Mitigation: one pure adapter narrows unknown data, tests the live profile contract, and explicitly records which fields that source can grade; absent provider coverage is never converted into an owner-facing missing-field claim.
  • Directional evidence hardening into a promise. Mitigation: the module header states the tier (like the schema analog); the honesty gate rejects rank language; owner copy is eligibility/trust framed.
  • Scope creep toward a write. Mitigation: A2 stays read-only; the write is A6.
