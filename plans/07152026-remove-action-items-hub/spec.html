Why

The legacy Action Items system is no longer part of the product direction, but it still exposes admin and client pages, writes task rows as a side effect of agent runs, and leaks task-specific states and copy into PMS, Ranking, onboarding, notifications, and telemetry.

Removing only the visible hub would leave hidden generators and broken deep links behind. Removing the shared monthly-agent jobs would be worse: those jobs also produce Summary and Referral insights that active dashboards still use.

What

Retire the legacy tasks product surface end to end: remove its pages, navigation, API, task-specific model and writers, pipeline states, Ranking action plans, onboarding/demo data, notifications, telemetry entries, and task-oriented copy.

Safety boundary: keep monthly agents and their PMS triggers running, preserve Summary and Referral output, leave the physical tasks table and historical rows untouched, and keep Project Management pm_tasks fully operational. No live worker or cron process is stopped because exploration found no task-specific scheduler.

Practice Hub widgets that currently read Summary data through /api/tasks will be moved first to the existing latest-agent-output endpoint. Old URLs will redirect to active pages instead of becoming dead links.

Context

Verified runtime facts — July 15, 2026

  • The task writers are direct calls from monthly-agent processing, not independent scheduled jobs.
  • The registered minds-worker schedules are Proofline and Ranking. Neither should be stopped.
  • Production and dev have historical tasks rows. They are retained for rollback and audit; this plan has no schema or data migration.
  • Project Management uses the separate pm_tasks domain and /api/pm/tasks. It is explicitly excluded.
  • useTopAction currently parses Summary metadata from legacy task records, while GET /api/agents/latest/:googleAccountId already exposes the underlying latest Summary output.

Risk

Level: 3 — Structural Risk. This removes a cross-cutting legacy domain with UI, API, agent, PMS, Ranking, email, telemetry, and compatibility consumers.

Risks and required mitigations

  • Practice Hub regression: useTopAction currently depends on task records. Mitigation: implement and test the latest-Summary query first; do not remove /api/tasks until its widgets pass against the new source.
  • Stopping useful automation: task creation shares execution with monthly insights. Mitigation: delete only writer calls and task-only status/count logic; keep monthly/PMS/worker triggers and verify their outputs.
  • Project Management collateral damage: searches for “task” include pm_tasks. Mitigation: enforce an explicit PM exclusion list and acceptance-test the PM board/API.
  • Broken bookmarks and notifications: old task URLs may still exist outside the codebase. Mitigation: redirect /tasks to /dashboard, /admin/action-items to /admin/mission-control, and stale organization task tabs to a supported agent tab.
  • Historical payload drift: old PMS state may contain task_creation. Mitigation: ignore that field in rendering and state derivation; do not rewrite stored payloads.
  • Dead imports and stale contracts: broad deletions can leave TypeScript and query-key references behind. Mitigation: use compiler/search gates, focused tests, npm run check:all, and full frontend/backend builds.
  • Docs checkout overlap: unrelated docs files are already modified. Mitigation: re-check status and make only a narrow copy edit if parity requires it; otherwise record that no docs page exists.

Blast radius: monthly agents, PMS automation and retry flows, Practice Hub summary widgets, Admin organization detail, client routing, Ranking results, onboarding, notifications/email, telemetry, and Alloro Docs copy. PM tasks, worker schedules, and historical DB records are protected boundaries.

Pushback

Stopping the monthly-agent or worker process is the wrong control point. It would suppress active insight generation and Ranking/Proofline work, while task creation is only a side effect. The recommended path is to remove the writer calls and legacy domain, then prove the still-supported jobs run without inserting task rows.
