Why

Alloro is the CMO of the owner's local presence. Page title and meta description are the blue-link snippet — the get-considered / CTR lever (GC1): whether a searcher who sees you clicks you. A rewrite must never reach a live page without the owner reviewing current-vs-suggested and approving it (Value #6 — "designed to improve how you're described," never "you'll rank higher").

What

A contained backend rail that persists and stages a produced title/meta rewrite as an owner-reviewable proposal: stage → list (current-vs-suggested + plain rationale) → approve (stages) / reject. Publishing the approved metadata onto the live page's seo_data stays a separate, already-gated step and is out of scope here.

Context — measure-twice findings

  • Columns moved. Migration 20260308000001 dropped meta_title/meta_description from website_builder.pages; SEO title/description now live as seo_data.meta_title / seo_data.meta_description (JSONB).
  • The propose engine already exists. service.ctr-hypothesis.ts produces { before, proposed:{title,description}, rationale, prediction } — honest by construction — via POST /:id/seo/ctr-hypothesis. Its own header says: "Brick 3 owns the table; there is no persistence here." This slice is brick 3.
  • Surface is super-admin, not owner-facing. All /api/admin/websites routes sit behind [authenticateToken, superAdminMiddleware] and use the domain's { success, error, message } shape. There is no owner (client-dashboard) website-SEO surface; the cited GbpCategoryProposalPanel is not present in this branch's frontend. So the owner-facing UI is a genuine follow-up, not a quick mirror.

Constraints

  • Must re-read the page server-side and scope every query by project_id (§5.5/§11.7); validate at the route boundary in enforce mode (§11.2); keep the rewrite framed as description-improvement, never a ranking promise (Value #6).
  • Must not touch the publish / live-seo_data-write path; add a frontend surface (deferred); duplicate the existing CTR-hypothesis producer.
  • Out of scope: owner-facing UI; auto-producing proposals from a get-found/CTR audit run; publishing.

Risk — Level 2

  • New table on a hot schema. Additive, reversible, no data rewrite, no lock on an existing table; down drops it cleanly. Mitigation: dedicated table + indexes only.
  • Dangling half. A staged/approved proposal is inert until the separate publish step consumes it — by design, matching get-found-write's honest split. Named in code headers so no one reads it as end-to-end.

Blast radius: new files + one route file (seo.routes.ts) + one migration. No existing handler changed. Router-loading smoke tests stay green.
