Risks Identified

  • R1 — Brittle rationale cap (Level 2). A tight schema .max() on rationale is fail-closed: if the agent writes one char over, the WHOLE monthly summary rejects and the doctor sees no action. Mitigation: prompt-tighten to 1–2 sentences + update the worked example, AND add a generous schema guard (.max(400), a runaway backstop ~3 sentences) rather than a tight 280. Enforcement without realistic rejection.
  • R2 — Consolidation regression (Level 2). Over-correcting the consolidation rule could break legitimate same-entity merges. Mitigation: keep "same entity → may reference both signals"; add only a "ONE subject, ONE next step; never bundle multiple offices/sources/factors into a do-all action" constraint.
  • R3 — Agent change not retroactive (Level 1). Takes effect on the next monthly run per location. Mitigation: note it; existing cards reflect old output until the next run.
  • R4 — Form-subs field semantics (Level 1). Must confirm whether form_submissions.total is all-time vs this-month and where the latest-month count lives (the wizard branch uses timeseries.at(-1).total). Mitigation: read buildFormSubmissionsMetrics at execution; use the latest-month value for the card.

Blast radius: Summary.md + schema (monthly Summary agent output, all orgs, next run); StatCardRow/StatCard (Practice Hub only); ProductionPanel (one cue line). The canonical-review decision is documented here but its rankings-side wiring is Plan 3.

Pushback

  • You asked to "limit the sentence" — a hard schema cap is the literal enforcement, but rejecting a doctor's entire monthly action over prose length is worse than a slightly long action. Recommendation: tight prompt guidance (the real lever) + a generous schema backstop. If you want hard enforcement at exactly 2 sentences, the safe place is a deterministic truncate in the task-creator (never rejects) — flag if you want that instead.
  • Referrals card as "latest month" makes 853 (all-time) drop to a smaller monthly number. That's the honest, labeled figure — but confirm you'd rather see "April: 181" than "all-time: 853" on this card.

Rev 3 - 2026-06-13

Change: Reviews card window label CORRECTED — from "last 30 days" (Rev 2) to the current calendar month (e.g. "June 2026").

Reason: Execution deviation — Rev 2 mislabeled the window. A full trace of the data path shows reviews_this_month is a current-calendar-month-to-date count, not rolling 30 days.

  • The trace: DashboardController builds dateRange = { start: 1st-of-month, end: today } in UTC and passes it to computeDashboardMetrics; the GBP review window is filtered by createTime within that range (listAllReviewsInRangeREST). So the figure is "new reviews this month so far." Rev 2 instead trusted the reviewsLast30d alias in service.ranking-pipeline.ts:1598, which is a misnomer for the same calendar-month field — that was the error.
  • Fix: added currentMonthLabel() to utils/timeframe.ts (current month in UTC, to match the metrics window) and set the Reviews card sub to it in both data paths. The stale "rolling 30-day" comment in StatCardRow was rewritten to document the real window. This actually realizes the owner's original #3 ask ("show this month, label it like the next card").
  • Cross-surface note: Local Rankings' "N reviews last 30 days" is a DIFFERENT, genuinely-30-day number sourced from the Apify scrape (client_gbp.reviewsLast30d), correctly labeled there — no conflict with this card's month-to-date count. The standing rating shown remains current_rating (all-time Google average); still no month-scoped average stored.
