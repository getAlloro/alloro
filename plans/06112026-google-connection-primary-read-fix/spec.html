Why

Prod incident 2026-06-11: a GSC-only Google connection (info@getalloro.com) was created on org 39 (One Endodontics) during a pilot session. GoogleConnectionModel.findOneByOrganization runs LIMIT 1 with no ORDER BY, so Postgres returned the auxiliary row instead of the org's real GBP connection — blanking the client dashboard ("Let's Set Up Your Dashboard"), and risking client notifications routed to the admin email via recipientSettingsService. The duplicate row was deleted as a data fix; this plan fixes the code so auxiliary/override connections are never read as the org's primary connection.

What

findOneByOrganization returns only GBP-capable connections (scopes contain business.manage), deterministically ordered: rows with google_property_ids first, then lowest id. Orgs whose only connection is a GSC-only auxiliary row get undefined — the correct "no primary connection" answer. Done when the model is updated and npx tsc --noEmit is clean.

Context

Relevant Files

  • src/models/GoogleConnectionModel.ts - the model; findOneByOrganization at line 68 is the only change surface
  • src/controllers/auth/feature-services/OAuthFlowService.ts - initial signup always requests business.manage (REQUIRED_SCOPES), so every legit primary connection passes the filter
  • src/controllers/admin-websites/feature-services/service.gsc-integration.ts - GSC flows select connections explicitly by id (findById/findByOrgWithScope) and are untouched; auxiliary rows remain usable there

Prod audit (2026-06-11, post row-62 delete)

  • 8/8 google_connections rows have business.manage scope AND non-null google_property_ids — zero behavior change for existing data
  • 14 call sites of findOneByOrganization (OnboardingController, GbpOnboardingService, settings service.properties ×2, PilotSessionService, pms approval/retry ×2/finalize, LocationService ×4, recipientSettingsService) — all verified to want primary-GBP semantics

Constraints

  • Keep method name/signature — fix all 14 call sites in one place

Must Not

  • Do not touch GSC integration selection paths (findById, findByOrgWithScope, findByEmailsWithScope)
  • Do not change OAuth write paths

Out Of Scope

  • Reconnect-flow org resolution via unordered organization_users lookup (AuthController.ts:290) — flagged, separate fix

Risk

Level: 2

Risks Identified

  • Stringly-typed scope match could miss legacy rows with empty scopes. Mitigation: prod audit shows 0 such rows; matches existing findByOrgWithScope convention.
  • Orgs with only auxiliary connections now read as "no connection". Mitigation: intended semantics — onboarding correctly prompts for GBP connect.

Blast radius: 14 call sites listed in Context; all verified primary-GBP semantics. GSC harvest worker resolves connections by explicit id — unaffected.

Pushback

  • An explicit purpose column is the "cleaner" long-term model but duplicates what scopes already encode and drifts on scope re-grants; revisit only if a non-scope-distinguishable connection class appears.
