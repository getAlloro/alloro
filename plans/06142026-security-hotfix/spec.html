Why

The backend conventions audit (plans/06142026-backend-conventions-audit) surfaced a cluster of live, verified security exposures that outrank every other finding. They are exploitable today: admin and destructive endpoints are reachable with no authentication, the JWT secret fails open if an env var is missing, live third-party API credentials are committed to source, one-time auth codes are generated with a non-cryptographic RNG and written to logs in plaintext, a credential-free login bypass ships in production code, and the entire auth attack surface (login / OTP / password reset) has zero rate limiting. This is Phase 1 of the cleanup — it runs first, ahead of the slow conventions remediation, and is independent of every other cleanup spec.

What

A behavior-preserving (except where explicitly tightening security) hotfix that closes seven verified exposures:

  1. Default-deny auth. Mount authenticateToken as an app-level guard in src/index.ts with an explicit allowlist of genuinely-public routes — killing the class of unauthenticated admin/destructive endpoints at once instead of patching routers one by one.
  2. JWT fail-closed. One config getter for JWT_SECRET that throws at startup if unset; delete every || "..." fallback literal.
  3. Rotate & externalize the 16 hardcoded Clarity JWT tokens in domainMappings.ts.
  4. Crypto RNG for OTP codes and admin temp-passwords.
  5. Stop logging one-time codes in plaintext.
  6. Gate / remove the tester@google.com bypass.
  7. Rate-limit login / OTP / reset by wiring a new strict auth limiter into publicRateLimiter.ts.

How we'll know it's done: npm run build (tsc) clean, eslint clean, a manual endpoint smoke (an unauthenticated curl to a previously-open admin route returns 401; a real admin token still works; the admin UI pages still load), and the app boots with JWT_SECRET set and refuses to boot without it.

Patterns To Follow

  • Router-level auth. The healthy routers already do router.use(authenticateToken, superAdminMiddleware) (e.g. src/routes/minds.ts:21, src/routes/admin/organizations.ts). The default-deny mount in index.ts is the app-level version of the same guard.
  • Config getter, not inline env. Read a required secret once and throw at startup if unset (fail-closed). Mirror how src/config/stripe.ts centralizes Stripe config.
  • Crypto for secrets. Use crypto.randomInt / crypto.randomBytes — already the pattern in security-utils.ts (crypto.randomBytes(8) for the OAuth nonce).
  • Frontend auth. Route admin calls through apiGet/apiPost from frontend/src/api/index.ts (which attaches the Bearer token) rather than bare fetch.

Reference domain: gbp-automation (src/controllers/gbp-automation/*, src/routes/gbpAutomation.ts) is the repo's reference domain — thin controllers, the only mature {success,data,error} + sanitizer standard, 0 must-fix in the audit. Match its router-auth and error-handling shape for any new middleware/glue added here.

Constraints

Must

  • Run T1 (frontend caller audit) before flipping default-deny — it gates the auth change.
  • Default-deny + explicit public allowlist (fail-safe), not per-router patching.
  • Fail-closed on a missing JWT_SECRET (throw at startup), single config source for sign & verify.
  • Use crypto.randomInt/randomBytes for OTP, temp-passwords, and the OAuth state string.
  • Commit before each unit; one concern per commit; per-unit checkpoints (build + eslint + smoke) — not one unstoppable run.
  • Behavior-preserving except where the change is the security tightening (401 on previously-open routes, refusal to boot without the secret, rate-limit 429s).
  • Treat the 16 Clarity tokens as burned — rotate them, do not merely move the existing values.

Must Not

  • No unrelated refactors / drive-by cleanups. The conventions debt (god-controllers, DB-outside-models, response shape, Pino) is a separate plan — do not touch it here.
  • No new dependencies unless justified (express-rate-limit is already present; a new auth limiter reuses it — no new dep needed).
  • Do not rewrite the middleware guards themselves — they are sound; mount them.
  • Do not change the public renderer app, n8n webhooks, or migrations as part of this.
  • Do not silently broaden the public allowlist to make a test pass — every entry must be justified.

Out Of Scope

  • Per-route RBAC role-tightening beyond "authenticated + (super-admin where the router already implied it)" — granular role matrices are conventions work.
  • Input-validation middleware (dim 12), Pino adoption (dim 10), DB-in-models (dim 7), response-shape normalization (dim 9) — all later phases.
  • The websiteContact commented-out anti-abuse stack and reCAPTCHA fail-open — real, but a separate public-form hardening unit (flagged, not fixed here, to keep this hotfix tight). Re-confirm scope with owner if they want it folded in.
  • Constant-time API-key compare in scraperAuth.ts (low practical risk; nitpick).

Risk

Level: 4 (Major Impact). This changes the auth model and its blast radius is every route in the app. A default-deny flip that misclassifies even one legitimate public route, or misses one unauthenticated legitimate caller, takes down a user-facing or admin surface. Recommend the owner is aware before this lands; it is not a quiet patch. The mitigating factor: the guards being mounted are already proven in production on the healthy routers, so the risk is in classification (which routes are public) and caller readiness (does every legit call send a token), not in the guard logic.

Risks identified

  • Default-deny breaks legitimate unauthenticated callers. Already confirmed live: frontend/src/api/settings.ts, frontend/src/api/agentOutputs.ts, frontend/src/pages/admin/AppLogs.tsx, and the agent-insights callers (RecommendationCard.tsx, AIDataInsightsList.tsx) call soon-to-be-protected admin routes with no Authorization header. Mitigation: T1 is a hard gate — audit and remediate these callers (migrate to apiGet/apiPost or add the Bearer header) in the same unit, before the flip. The flip does not merge until T1's list is green.
  • A genuinely-public route gets caught by default-deny. Login, OTP issue/verify, password reset, Stripe webhook (signature-verified, no JWT), the public scraper (key-gated), public file serving, the public contact form, and the public skill/portal API are legitimately unauthenticated. Mitigation: build the allowlist from the audit's verified "public-by-design" set (worksheet 09 enumerates it) and assert each one with a smoke check after the flip.
  • rbac default-allow. rbac.ts:42-48 falls an orgless authenticated user through to viewer + next() rather than 403. So "authenticated" alone is weaker than it looks on org-scoped admin routes. Mitigation: for the admin routers in scope, layer the existing superAdminMiddleware/requireRole the healthy admin routers already use; do not rely on authenticateToken alone for super-admin surfaces.
  • Rotating the Clarity tokens breaks live analytics fetches. Mitigation: coordinate rotation with the Clarity integration owner; move values to env/secret store and verify a Clarity data fetch still succeeds before closing the unit. (Per repo memory, the renderer-side injection is separate and unaffected.)
  • JWT fail-closed could refuse to boot in an environment where the var is genuinely unset. That is the intended behavior, but it must be verified that dev/prod actually have JWT_SECRET set before deploy. Mitigation: confirm the env var on dev and prod hosts (PM2 ecosystem / host env) before merging; the throw is the safety net, not a surprise.
  • Auth rate-limiter false-positives (shared-IP clinics, NAT). Mitigation: tune max conservatively, per-IP + per-email, return the standard { success:false, error } body already used in the limiter file; note the single-instance in-memory caveat (worksheet 16 — counts don't aggregate across PM2 workers; acceptable for v1, Redis-back later).

Blast radius — concrete consumers

  • Default-deny flip (index.ts): every router mounted at lines 162-210 — gbp, clarity, tasks, auth, otp, pms, pm, dashboard, onboarding, rag, agents, notifications, all /api/admin/*, settings, profile, leadgen, support, scraper, places, audit, imports, websites (public contact), user/website, locations, minds (admin + public), skills, internal, billing, telemetry. Each must be classified public or protected.
  • Frontend: frontend/src/api/settings.ts, agentOutputs.ts, AppLogs.tsx, RecommendationCard.tsx, AIDataInsightsList.tsx (confirmed unauth callers) — plus any other module using bare fetch('/api/...') instead of apiGet/apiPost (T1 must grep the full set).
  • JWT secret change: every token-issuing and token-verifying site — auth.ts, billingGate.ts, AuthController.ts:15, PilotSessionService.ts:12, security-utils.ts:4, service.jwt-management.ts:14 (the six fallback sites the audit flagged). All must read the one config getter so sign and verify agree.
  • Clarity token externalization: every consumer of domainMappings.ts (the Clarity data-fetch path) plus the Clarity integration upstream.
  • Auth limiter: routes/auth.ts, routes/auth-otp.ts, routes/auth-password.ts and their real users (the login / verify / reset UI flows).

Pushback — default-deny vs per-router mounting

  • Option A — App-level default-deny + explicit public allowlist (RECOMMENDED). Mount authenticateToken once in index.ts ahead of the router mounts; maintain a small, reviewed allowlist of public path-prefixes that skip it. Pro: fail-safe — a newly-added route is protected by default; kills the entire class (admin/settings, app-logs, agent-outputs, agent-insights, pms admin block, agentsV2 IDOR, minds public surface) in one change; future routers can't silently reintroduce the hole. Con: the allowlist is now a security-critical artifact; a misclassification breaks a public route; requires the T1 caller audit up front. Recommended precisely because the failure mode is "something breaks loudly in testing," not "something stays silently open in prod" — and the T1 caller-audit is the gate that converts the loud-break risk into a pre-flight checklist.
  • Option B — Per-router router.use(auth) on each unguarded file. Add the guard to admin/settings.ts, appLogs.ts, adminAgentInsights.ts, admin/agentOutputs.ts, admin/imports.ts, admin/websites.ts (hoist), the pms admin block, agentsV2.ts, and the minds public API individually. Pro: surgical, smaller blast radius per change, no allowlist artifact. Con: fail-open by default — this is exactly how the holes appeared; you are guaranteed to miss one now and the next new router reopens the class. It treats symptoms, not the root cause (no global guard).
  • Verdict: Option A, gated by T1. Default-deny is the only option that closes the class; per-router patching is what produced the bug. Accept the allowlist as a deliberate, reviewed security surface and protect it with the smoke checks in Done.
