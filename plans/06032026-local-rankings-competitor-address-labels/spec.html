Local Rankings Competitor Address Labels

Status
Completed
Plan Folder
plans/06032026-local-rankings-competitor-address-labels
Size
Medium
Risk
Level 2

Why

Falls Church Local Rankings shows repeated competitor names like "Dominion Endodontics" without enough context, making distinct GBP or Maps locations look like duplicate rows. The ranking is not necessarily wrong; the identity label is incomplete.

What

Show competitor addresses as the secondary identity line in the Local Rankings competitor comparison table, including rows inside the Practice Health comparison modal/table. Keep ranking order, scores, and inclusion unchanged. Category remains a fallback only when no address is available.

Context

Relevant Files

  • frontend/src/components/dashboard/RankingsDashboard.tsx - owns the Local Rankings dashboard and renders the competitor comparison section.
  • frontend/src/components/dashboard/rankings/competitorComparison.ts - builds comparison rows and needs address propagation from every source path.
  • frontend/src/components/dashboard/rankings/CompetitorComparisonTable.tsx - renders the Practice column and already supports truncated secondary text with a tooltip.
  • frontend/src/api/practiceRanking.ts - frontend response types for selected and raw competitor payloads.
  • src/controllers/practice-ranking/feature-utils/util.ranking-formatter.ts - formats latest ranking responses and selected competitor search results.
  • src/controllers/practice-ranking/PracticeRankingController.ts - enriches latest ranking snapshots from active location_competitors.
  • src/controllers/practice-ranking/feature-services/service.ranking-pipeline.ts - persists future raw_data.competitors.
  • src/controllers/practice-ranking/feature-services/service.location-competitor-onboarding.ts - freezes selected competitor snapshots.

Patterns To Follow

  • Treat addresses as display-only metadata.
  • Prefer existing placeId-keyed enrichment from location_competitors over rerunning discovery or scraping.
  • Keep table truncation plus full-value tooltip behavior from the existing comparison table.

Reference file: frontend/src/components/dashboard/rankings/CompetitorComparisonTable.tsx - closest display analog for row structure and tooltip behavior.

Constraints

Must

  • Show a full-width truncated address under the practice name where available.
  • Preserve the full address in the tooltip/title for truncated text.
  • Keep category as fallback only when address is missing.
  • Support existing latest snapshots by enriching from active location_competitors where possible.
  • Persist address in future snapshots/raw data so the UI does not depend on fragile fallback paths.

Must Not

  • Do not de-dupe, hide, merge, or reorder competitors by name.
  • Do not trigger ranking reruns.
  • Do not mutate historical ranking math.
  • Do not alter competitor search or selector logic.
  • Do not touch unrelated dirty files beyond the planned scope.

Out Of Scope

  • Fixing bad or missing addresses at source if Google Places or SerpApi did not provide them.
  • Reranking Falls Church.
  • Changing competitor caps, sort defaults, comparison metrics, or modal layout.

Risk

Level: 2 - low behavioral risk, but this touches shared ranking response/display code and the current worktree is already dirty.

Risks Identified

  • RankingsDashboard.tsx is already modified in the working tree. Mitigation: avoid editing it unless necessary and keep any changes tightly scoped.
  • Existing rows may lack addresses in competitor_snapshot and raw_data.competitors. Mitigation: enrich the latest response from location_competitors by place id and preserve fallback category.
  • Adding address to future raw_data.competitors could be mistaken as a scoring change. Mitigation: keep it as display metadata only; no scoring inputs, sorting, or LLM prompt behavior changes.
  • Snapshot shapes may contain either placeId or place_id. Mitigation: normalize both keys while enriching and building display rows.

Blast radius: Local Rankings comparison table, Practice Health comparison modal/table, latest rankings API response shape, future ranking run raw_data.competitors payload shape.

Pushback

  • De-duping Dominion by name would be wrong. These are real distinct locations, and hiding them would damage accuracy.
  • A pure CSS/text tweak is not enough if the address is dropped before the table receives it.
