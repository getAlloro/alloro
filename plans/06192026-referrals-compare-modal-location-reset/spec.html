Why

On /pmsStatistics (Referrals Hub), the "Compare months" modal keeps the previous location's comparison after you close it and switch to a different location. Open compare → close → switch location → reopen still shows the prior location's picked months and AI insight paragraph. The comparison must reflect the location you are currently viewing.

What

The Compare modal resets when the active location changes. After switching location and reopening, the month pickers fall back to the new location's default months and the "Explain this comparison" AI paragraph is cleared — no carry-over from the previous location.

Context

Root cause

CompareMonthsModal is mounted permanently in PMSVisualPillarsModals.tsx; isOpen={showCompare} only toggles the inner AnimatePresence content, so the component instance never unmounts. Its local state survives close → location-switch → reopen:

  • monthAKey / monthBKey (useState) — the manually picked months.
  • useGenerateComparisonInsights(locationId) — a React Query useMutation with no mutationKey; changing locationId only updates the closure, it does not reset data/error, so the prior location's AI paragraph persists.

Relevant Files

  • frontend/src/components/PMS/PMSVisualPillars/PMSVisualPillarsModals.tsx - renders <CompareMonthsModal> (the one line that changes).
  • frontend/src/components/PMS/dashboard/CompareMonthsModal.tsx - holds the surviving state (no change needed).
  • frontend/src/hooks/queries/usePmsComparisonInsights.ts - the un-keyed mutation (no change needed).
  • frontend/src/components/PMS/PMSVisualPillars.tsx - owns showCompare, passes locationId + months={keyData?.months ?? []}.

Patterns To Follow

  • React identity reset via key: a changing key on a component unmounts the old instance and mounts a fresh one, clearing all internal useState and hook state at once.
  • §15.1 — server state stays in React Query (the AI insight remains a mutation, not mirrored into useState).

Reference file: frontend/src/components/PMS/PMSVisualPillars/PMSVisualPillarsModals.tsx - the sibling modals in the same fragment are the structural analog.

Constraints

Must

  • Reset the modal when locationId changes.
  • Keep the change to one file / one line; no behavior change inside CompareMonthsModal.

Must Not

  • No new dependencies, no new state system (§15.4).
  • No raw fetch/axios (§14.2), no any (§17.2), no console.* (§17.1).
  • Do not refactor CompareMonthsModal or the mutation hook.

Out Of Scope

  • Resetting on a same-location close/reopen (selections may persist within one location — desirable).
  • Other PMS modals in the same file.

Constitution Articles Touched

  • §15.1 — Server state is React Query (AI insight is a mutation; left as-is).
  • §13.2 / §13.3 — Lean component, one responsibility (no logic added to the component).
  • §12.3 — Feature folder (modal already lives under components/PMS/dashboard/).

Risk

Level: 1 (Suggestion — minimal, well-contained bug fix).

Risks Identified

  • Remount-on-key could cause a visible flash if location changed while the modal is open. Mitigation: the location selector sits behind the modal overlay, so location only changes while the modal is closed; the remount happens with isOpen=false (invisible).
  • If locationId is null (single-location / not loaded), the key is a constant and never remounts. Mitigation: acceptable — single-location practices never switch locations, so there is no cross-location bleed to reset.

Blast radius: only CompareMonthsModal rendered in PMSVisualPillarsModals.tsx. No other consumer of these files changes behavior.

Pushback

  • An alternative is an in-component useEffect keyed on locationId that calls setMonthAKey(null)/setMonthBKey(null)/insights.reset(). Rejected: more code, easy to get the dependency array wrong, and it duplicates what React's key identity already guarantees. The key approach is the idiomatic "this is a different modal for a different location" expression.
