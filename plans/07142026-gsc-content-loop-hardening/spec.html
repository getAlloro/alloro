GSC Content Loop Hardening

Completed

Why

PR 151 connects real Google Search Console queries to SEO target-query generation, but the current implementation is not safe to merge. It breaks the existing hermetic SEO test, exposes raw public search text at system-prompt priority, sends that context to every SEO section despite a GEO-only contract, and can force a niche page to choose an unrelated sitewide query.

What

Keep the real-demand feature, but make the trust and section boundaries deterministic. Normalize and cap GSC query text, serialize it as untrusted data in the GEO user message only, preserve content-based inference when no listed query is relevant, make optional GSC lookup failures fall back without breaking existing generation, and add hermetic tests for aggregation, isolation, failure, and adversarial inputs.

  • src/controllers/admin-websites/feature-services/service.seo-generation.ts — assembles single-section, generate-all, and bulk shared context. It is already 782 lines, close to the §2.4 hard ceiling, so this fix must not add another responsibility here.
  • src/controllers/admin-websites/feature-utils/util.seo-section-runner.ts — builds system/user prompts and runs all six SEO sections. PR 151 currently appends GSC data to the cached system prefix for every section.

  • claude/gsc-content-loop — existing PR 151 head. It is one unique commit while current dev/dave is 12 commits ahead; GitHub reports a clean merge, but final tests must exercise the combined result rather than the stale head alone.

Constraints

Must

  • GSC demand appears only in the geo_layer user message; it never enters cachedSystemBlocks or another section's user message.
  • Normalize control whitespace, trim and length-cap query strings, cap the query count, and serialize the payload as structured data (§5.2, §4.2).
  • Keep the system prefix stable for caching and preserve every existing non-GSC prompt field.
  • A missing or failed optional GSC lookup must log project context through Pino and continue with content inference (§3.1, §3.2, §9.1).
  • Non-GEO single-section generation must not perform the GSC read.
  • GSC reads remain in GscDataModel through the existing service path (§7.1, §7.4).
  • Use only synthetic data in tests (§20.4).

Must Not

  • Do not retain raw GSC strings in a system-priority block, even with stronger wording.
  • Do not add to the already 782-line SEO generation service without offsetting extraction (§2.4).
  • Do not change ranking, click, impression, or average-position calculations.
  • Do not change output schemas, API response shapes, auto-apply behavior, or worker job semantics.
  • Do not add a dependency, migration, frontend change, or Alloro Docs change.
  • Do not run a live client generation or auto-apply during execution without a separate explicit authorization.
  • Do not force-push or rewrite PR 151 history; fixes and the plan land as additive commits on the existing PR branch only after the execution and publish gates.

Out Of Scope

  • Page-specific GSC query/page joins or a new GSC ingestion schema.
  • SEO quality redesign outside target-query selection.
  • Changes to the GSC dashboard, integration flow, frontend, or production data.
  • Refactoring pre-existing controller and worker Constitution debt not introduced by PR 151.

Risk

Level: 3 — the input is public search text, the destination is an LLM prompt, and GEO output can be auto-applied to client content.

Risks Identified

  • Prompt injection or prompt-shape breakout. Mitigation: normalize and bound query text, serialize it as data in the GEO user message, label it untrusted, and test quotes, newlines, markup, and instruction-like strings.
  • Meta/schema contamination. Mitigation: enforce section === "geo_layer" in code and assert that every other captured prompt lacks the demand block.
  • Irrelevant sitewide query forced onto a niche page. Mitigation: the prompt must allow content inference when zero measured queries genuinely fit.
  • Optional GSC context breaks existing generation. Mitigation: a logged fail-open boundary returns no demand context; existing generation then follows its pre-PR fallback.
  • Near-ceiling service grows further. Mitigation: extract pure demand formatting to a focused feature-utils module and keep service.seo-generation.ts net-flat or smaller.

Blast radius: page and post single-section generation; page and post Generate All; the project-wide SEO bulk worker; the six section prompts assembled by util.seo-section-runner.ts; the GSC data read used only by the new SEO demand helper. The GSC dashboard endpoint, frontend, database schema, ranking agents, Summary agent, and unrelated client agent runs remain unchanged.

Pushback

  • Option A — recommended: normalized structured data in the GEO user message only. This creates a real trust and section boundary and matches the existing untrusted-content analog.
  • Option B — rejected: keep the demand block in the cached system prefix and add escaping/stronger instructions. This still grants public search text system priority and still exposes it to non-GEO sections.
  • Live agent generation is not required to prove isolation; mocked prompt capture is deterministic and cheaper. A dev generation with apply_geo_content: false is a separate acceptance step only after explicit authorization.
