Footer "Powered by Alloro" Credit Rollout

Status
Completed
Plan Folder
plans/07012026-footer-powered-by-alloro-credit
Size
Large (10 independent live-prod content edits)
Risk
Level 3

Why

Most live Alloro client websites don't credit Alloro in their footer. One site (One Endodontics) already carries a "Website Powered by Alloro" link; the owner wants that same credit rolled out to every other client site, except Alloro's own marketing site (which is itself rendered through this same CMS, on getalloro.com).

What

Directly edit the footer column of 10 live website_builder.projects rows in the production database, inserting the same "Website Powered by Alloro" credit block already live on One Endodontics's footer, placed consistently inside each site's existing footer layout. Verify each site renders correctly immediately after its own edit before moving to the next.

Context

Architecture

Footer content is not a repo file — it's stored per-project in the website_builder.projects.footer TEXT column (see src/database/migrations/20260214000000_restructure_templates_to_sections.ts for the column's origin, and src/models/website-builder/ProjectModel.ts for the model). The separate renderer repo (website-builder-rebuild/src/routes/site.ts) assembles the live page from wrapper + header + sections + footer at request time. No application code in this repo (src/) needs to change — this is a content/data operation against production, not a code change, so the Code Constitution skill does not apply here.

Insertion rule (confirmed by reading all 10 footers in full)

Every in-scope footer follows the same nesting shape: an outer <section>/<footer> wrapper → a max-w-7xl mx-auto (or equivalent width-constrained) content wrapper → a 4-column grid <div> (brand / services / quick-links / team-or-location). The credit block is inserted as a new sibling <div> immediately after the grid <div>'s closing tag and before its parent width-wrapper's closing tag — exactly where it already sits in One Endo's live footer. Two sites (Surf City, Artful) nest an extra background-color <div> layer around the grid; the rule still applies relative to the grid's own immediate parent.

Constraints

Must

  • Reuse the One Endo credit block verbatim (copy, link target, logo image, styling) on all 10 sites for brand consistency.
  • Read each project's current footer value fresh immediately before writing it (Read Before Write) — do not reuse the snapshots captured during planning, since prod data could have changed since.
  • Update one project at a time; verify the live domain renders correctly (HTML still valid, credit visible) before moving to the next project.
  • Run the writes as direct read/write UPDATE statements against the production DB — no schema change, no migration file.

Must Not

  • Touch project acc7cbfc-a3fd-4476-abbc-aace3a566fc5 (getalloro.com) under any circumstance.
  • Touch the two already-credited projects (One Endodontics, internal demo) or the unverified undomained Garrison duplicate (91a29a43...).
  • Modify any other part of a project's footer (colors, copy, columns) beyond inserting the credit block.
  • Batch all 10 writes before verifying any of them.

Out Of Scope

  • The header_footer_code snippet-injection system — owner explicitly chose direct footer-column edits instead.
  • Resolving whether the undomained Garrison project (91a29a43...) is a stale duplicate, staging copy, or something else — left untouched per owner decision.
  • Any template-level (website_builder.templates) footer change — this rollout only touches per-project overrides, since every in-scope project already carries a fully custom footer.

Risk

Level: 3 — direct writes to 10 live production rows, each rendering a real customer-facing website with no schema/migration safety net, and (per explicit owner decision below) no pre-write backup.

Risks Identified

  • A malformed insertion (unbalanced tag, broken nesting) could visually break or fully blank a live practice's website footer, or in the worst case the whole page if a tag imbalance cascades. Mitigation: sequential one-at-a-time updates with an immediate live-render check after each write, per owner's chosen verification cadence, so a bad insertion is caught and fixed before the next site is touched.
  • No backup of original footer HTML is being taken before the writes. If an insertion breaks a site and the exact original markup can't be reconstructed from memory/diff, recovery means manually rebuilding that one footer from scratch. Mitigation offered and declined: I recommended saving each project's pre-edit footer value to the plan folder before writing (cheap, ~10 SELECT statements). The owner explicitly declined this twice when asked directly. Accepted as-is per owner instruction — flagging again here as the standing risk record.
  • Mis-targeting the getalloro.com row would self-credit Alloro's own marketing site, which the owner explicitly asked to avoid. Mitigation: the project ID for getalloro.com (acc7cbfc...) is pinned in this spec and excluded by ID, not by guessing from domain pattern-matching at execution time.

Blast radius: 10 live, publicly-reachable production websites for real dental/orthodontic practices (Advanced Root Canal Specialists, Artful Orthodontics, Art of Sleep Dentistry, Caswell Orthodontics, Coastal Endodontics Studio, DentalEMR, Garrison Orthodontics, San Diego Center For Endodontics, Surf City Endodontics, TriCity Endodontics). No other tables, no schema, no application code.

Pushback

  • I pushed back on skipping the backup (asked twice) given this is an unrecoverable overwrite of live production content with no migration/version history behind it. The owner held the decision both times. Proceeding without a backup is the owner's explicit, informed call — not a default I'm choosing quietly.
This spec intentionally does not include a rollback artifact. If any site needs to be reverted after execution, recovery will require manually re-authoring that footer's HTML from a fresh read of its current rendered page (or from this spec's recorded full-footer scratch dumps from planning, which are NOT guaranteed current at execution time).

Revision Log

Rev 1 - 2026-07-01

Change: All 10 tasks (T1-T10) executed sequentially as specced, via a one-off Node script using a balanced-tag scanner (locate the `grid grid-cols-1` columns container, walk matching `<div>`/`</div>` pairs to find its true closing tag, insert the credit block as the next sibling). Every insertion previewed and visually diffed before commit, then the live domain curled and grepped for the credit string before moving to the next task. Root cause discovery mid-execution: while verifying T8 (San Diego Center For Endodontics), the live page did not show the credit despite a confirmed-correct DB write (footer column matched expected length and content). Root cause was a Redis-backed render cache in the website-builder-rebuild renderer (`pb:*`/`posts:*`/etc. key patterns) serving a stale pre-edit render. Resolved by hitting the renderer's existing `?nocache=1` cache-bust query param (already wired in `site.ts`, used here for the first time in this workflow), confirmed the credit then appeared and persisted on a follow-up plain request. Re-verified T1-T7 with the same `?nocache=1` pattern as a precaution; all were already clean (no stale cache present on those).

Reason: Execution deviation: an undocumented render-cache layer caused one site (of 10) to serve stale content immediately after a correct DB write, which the spec's verification step did not originally account for.

Updated Done criteria: none — original Done checklist items still apply and all passed. No spec task or scope changed.
