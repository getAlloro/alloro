Why

Plan 4 of the dashboard-revamp feedback round — covers PDF items #7–#11 ("Reviews and Posts" / Alloro Engage at /gbp-manager). The owner found the page's labels ambiguous (which stats mean what, "60d", the "safe" internal badge leaking to the client, two near-identical reply sub-tabs, a confusing standalone count, a bare Posts tab, and admin-only automation toggles exposed to the client). This plan clarifies copy + trims controls without changing any data pipeline.

What

Done when:

  • #7: page tab toggle uses the terracotta active pill (was navy); "Needs reply" stat reads "Reviews needing reply" with sub "received in last 60 days"; "Last review" reads "Days since last review" (received, not replied); "Coverage" gains a "% of reviews replied" definition; the Needs-Reply list defaults to a focused recent view capped ~10.
  • #8: the "Reply Drafts" sub-tab is renamed "Drafts Ready for Review"; the reassuring internal "Safe" badge is removed from the client view (Blocked / Safety review stay; admin still sees Safe).
  • #9: the standalone review-count number on the left of the range controls is removed; range buttons stay.
  • #10: the Posts tab gets a 1–2 sentence "why Google Posts help local visibility" blurb at the top (client only).
  • #11: Settings "Next post generation" line is clarified to explain it is the next auto-draft time; the two automation toggles (Review replies / Google post drafts) are removed from the CLIENT settings tab (admin keeps them); Diagnostics + sync controls stay.
  • tsc -p tsconfig.app.json + eslint (touched files) clean; admin GBP surface still renders.

Patterns To Follow

  • Terracotta active pill = the locked PeriodToggle recipe (bg-alloro-orange text-white active, muted inactive). Match it inline on the page toggle.
  • Prop-additive only on shared components; defaults preserve old (admin) behavior.
  • One-line cues / terse subs; spelled durations.

Reference files: PeriodToggle.tsx (terracotta active recipe), StatBox.tsx (label/value/sub shape), OrgGbpAutomationTab.tsx (admin render path to protect).

Constraints

Must

  • Stay in pages/GbpManagerPage.tsx + components/dashboard/gbp-automation/*.
  • Gate every shared-component change with a prop defaulting to old behavior; set the new behavior only from the GbpManagerPage → GbpAutomationPanel path.
  • Keep Blocked / Safety review badges; only the reassuring "Safe" label is suppressed on the client.
  • Read each file before editing.

Must Not

  • Do NOT edit index.css, statusRules.ts, or shared primitives (PeriodToggle/InsightCue/StatBox/timeframe) — other agents own those.
  • Do NOT break the admin GBP render (settings toggles, safety badges must persist for admin).
  • No data-pipeline / endpoint / schema changes.
  • No new dependencies. No git operations.

Out Of Scope

  • Backend automation-permission move to admin-only (this plan only hides the toggles in the client UI; enable/disable stays a settings field the admin tab still writes).
  • The inner Reviews sub-tab pill staying navy (only the page-level toggle is in the owner's #7 — noted, not changed).
  • Other surfaces (Plans 1–3, 5).

Risk

Level: 2 — four of the touched components are SHARED with the admin GBP surface; a non-gated removal would silently break admin.

Risks Identified

  • R1 — Admin breakage via shared components (Level 2). GbpSettingsSection, GbpReplyWorkItemCard, GbpReviewReplySlot all render in admin. Mitigation: prop-additive gating (hideAutomationToggles, hideSafeBadge) defaulting to current behavior; admin call sites pass nothing → unchanged.
  • R2 — Removing "Safe" hides genuine warnings (Level 1). Mitigation: suppress only the clean "Safe" label; keep Blocked + Safety review on the client too.
  • R3 — Removing the standalone count loses the "all loaded" total cue (Level 1). Mitigation: the panel already renders a "Showing {month}" line for the All-loaded range; the dropped count is redundant. No replacement needed.
  • R4 — Hiding client toggles diverges client vs admin state view (Level 1). Mitigation: the read-only Diagnostics box already surfaces "Review replies enabled / Google post drafts enabled" yes/no, so the client still sees status — just can't flip it.

Blast radius: GbpManagerPage (client only); GbpSettingsSection / GbpReplyWorkItemCard / GbpReviewReplySlot / GbpPostsManagerPanel (shared — guarded by props); GbpReviewRangeControls + GbpClientUnrepliedReviewsPanel + GbpClientReviewsPanel + GbpAutomationPanel (client thread-through).

Pushback

  • Item #11 says enable/disable "becomes admin-only." This plan only HIDES the client toggles; the underlying permission is still a writable settings field (admin writes it). True server-side role enforcement is a backend change out of this plan's boundary — flagged, not executed.
  • Item #8 offered "remove if truly redundant." Keeping "Drafts Ready for Review" as a rename is the safer call: the sub-tab shows AI-generated drafts awaiting human approve/publish, which is a distinct state from "Needs Reply" (no draft yet). Removal would strand in-flight drafts. Rename chosen.
