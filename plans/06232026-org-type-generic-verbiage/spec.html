Why

Alloro is locked to dental/endo/ortho language across the UI and the AI agents, which blocks sales beyond the dental vertical. We need a single platform that serves either the current healthcare experience or a generic local-service-business experience, driven by one field per organization — no codebase fork, no disruption to current clients. Target: a demo-ready generic account for LinkedIn outreach the week of June 30, 2026.

What

An organization gets a type — health (default, current verbiage) or generic (swapped verbiage). The type is chosen when an admin creates the org and is editable afterward. The type flows to the frontend (via the auth context) and to the AI agents (via the prompt loader), so every client-facing label, dashboard header, agent summary, and AI insight reflects the right vocabulary. Existing accounts are unaffected — they read and behave as health.

Done when: an admin can create a generic org from the Mission Control popup; every string in the explicit inventory below renders generic copy for that org; every client-facing agent prompt emits generic copy; health orgs are byte-for-byte unchanged; Jo completes a copy review in a staging generic account.

Constraints

Must

  • Resolve null → health in code (a resolveOrgType() helper) so any null-typed org behaves as health.
  • Rename the saas value to generic in all three type declarations and the admin validation array.
  • Write organization_type on the admin create path; relax the immutable guard so type is editable.
  • Deliver org_type per-request: add to /onboarding/status and the agent runner fetch. No JWT change.
  • Substitute prompt placeholders in prose only; leave every JSON schema key/example intact.
  • Swap only display strings via the label map; keep all data keys, API field names, and route paths.
  • Keep health output identical to today (placeholders resolve to the current words).

Must Not

  • No new column or table. No new dependency.
  • Do not rename DB columns (self_referrals, doctor_referrals, production, referrer_*) or API/interface keys.
  • Do not blind find-replace healthcare words across prompt files — only hand-placed {{tokens}}.
  • Do not bake org_type into the JWT.
  • Do not add the selector to self-serve signup (admin popup only).
  • Do not change the GBP safety rules — keep them; only swap their framing words.

Out of scope

  • N8N workflows — external (n8n.getalloro.com), not in this repo. The backend will pass resolved labels in the webhook payload (T5); the n8n prompt-node edits are a separate task on the n8n server.
  • Telemetry pageLabel values in routeTelemetry.ts — internal analytics, not shown to clients. Keep them stable so analytics stays consistent across org types, unless a label is also rendered as a visible breadcrumb/title (verify in T10).
  • Generic data ingestion / Stripe pricing by type — verbiage only; no new data pipelines this round.
  • Self-serve signup type selection.

Risk

Level: 3 (Structural) — wide blast radius across both stacks and a change to AI output; no destructive DB.

Risks & mitigations

  • Placeholder leaks into JSON schema keys → corrupts agent output parsing. Mitigation: tokens are hand-placed in prose only; T6 lists exact lines; schema keys (production_total, patient_records) explicitly excluded; verify each prompt still returns valid parsed output.
  • AuthContext contract change consumed by every authed page. Mitigation: the new field is optional (organizationType?); default resolves to health; no existing reader breaks.
  • Migration down() can't restore nulls. Mitigation: null and health are behaviorally identical (code default), so a no-op down is documented and harmless; migration is idempotent and forward-only normalization.
  • Over-replacement breaking data (renaming a key that looks like a word). Mitigation: the explicit Must-Not-Replace inventory; swaps go through the label map at render time, never at the data layer.
  • Generic ranking competitors leak dental specialties (Identifier default "orthodontist"). Mitigation: T7 parameterizes the specialty enum/default by org_type (health = dental list unchanged; generic = general business category).
  • "doctor" has two senses — the client/owner (prose → "you") vs. the external referring doctor (label → "Partner"). Mitigation: the dictionary separates {{provider_*}} (owner) from the Doctor Referrals → Partner Leads label entries.

Blast radius (known consumers)

  • AuthContext → every authed page/component that calls useAuth().
  • ~25 frontend display files (full list in the inventory).
  • 4 agent-prompt seams → monthly Summary/ReferralEngine/Opportunity, daily Proofline, GBP review reply + local post.
  • Admin create + set-type flows; Mission Control display of org type.

Pushback

  • Verbiage ≠ product fit (Level 2 concern). The swap makes Alloro read generic, but PMS ingestion is shaped around dental practice-management exports and the GBP prompts carry HIPAA-flavored safety rules. The generic version is a verbiage skin over a health-shaped product — good for a seeded demo, not a finished vertical. Recommendation: ship the verbiage layer now; seed the staging generic account with representative data for the demo; treat true generic data pipelines as a later plan.
  • New pattern introduced (justified). There is no central label map today. This plan creates one (backend config/orgLabels.ts + frontend constants/orgLabels.ts + useLabels()). Recommendation: adopt it as the standard — new client-facing strings go through the map, not inline JSX. Note it in the team conventions after merge.
