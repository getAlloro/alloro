Why

Alloro has the raw ingredients for a proprietary AI/SEO readiness product: website project data, page content, project identity, GSC harvests, GBP connections, and location records. The current gap is that these signals are not consolidated into a measurable audit product that can score an external URL or a full existing Alloro organization with evidence-backed criteria.

What

Build an internal-first AI/SEO Audit mini app inside Alloro that supports URL-only audits and organization audits. The app must produce a weighted score, data coverage, confidence level, hard caps, evidence, and prioritized remediation rather than a vague AI-generated grade.

The first execution should deliver the reusable backend scoring foundation, persisted audit history, admin organization audit surface, and an internal external-URL audit flow. A client-facing website tab can be exposed only after scoring output is stable and safe enough for customer display.

Scoring Contract

Full Organization Audit

Category Weight Data Needed Subcriteria
Access And Indexability 25 URL crawl, headers, rendered HTML, robots, sitemap
  • Final URL resolves with a successful status and sane redirect chain.
  • Search crawlers are not blocked by robots, CDN, headers, or meta tags.
  • Page is indexable and snippet-eligible unless a documented policy says otherwise.
  • Canonical is self-referencing or intentionally points to the correct canonical URL.
  • URL is discoverable through sitemap, internal links, and crawlable navigation.
  • Main content renders as readable text.
Page Source Readiness 20 Rendered page content, page type, headings, media, schema
  • Page has a clear primary intent and direct answer block.
  • Service, location, provider, and offer language is specific rather than generic.
  • Evidence units exist: definitions, FAQs, steps, comparisons, statistics, review snippets, or first-party proof.
  • Claims are supported by visible page content and do not overreach.
  • Structured data parses and reinforces visible content instead of contradicting it.
  • Page type requirements are met: home, service, location, provider, blog/post, or contact.
Entity And External Consistency 25 Submitted URL, JSON-LD, footer/contact content, Alloro identity, GBP profile, locations, web search results
  • Business identity is extractable from the submitted URL: name, domain, phone, address, hours, providers, services, and social/profile links where present.
  • Visible identity fields match schema fields on the submitted URL.
  • For organization audits, submitted URL identity aligns with Alloro project identity, GBP, and location records.
  • For URL-only audits, external sources are compared against the submitted URL baseline and flagged as consistent, conflicting, missing-on-site, or ambiguous.
  • External exact and near-exact web mentions are collected and weighted by source reliability.
  • High-authority or high-confidence mismatches are flagged: old phone, old address, wrong domain, stale hours, wrong provider, wrong location, or entity ambiguity.
Connected Performance Signals 20 GSC harvests, GBP profile/review/performance data, locations, audit history, later prompt-test history
  • GSC integration is active and has recent data for the site.
  • Audited URLs map to GSC page rows where available.
  • GBP profile data is available and location mappings are explicit.
  • Location-level GBP signals exist: reviews, rating, website clicks, call clicks, direction requests, impressions, categories, hours.
  • Audit history exists for before/after comparison.
  • Future AI prompt/citation tests can attach to the same audit target model without changing the scoring contract.
Authority And Market Signals 10 Reviews now; external mentions from consistency scan; citation/backlink provider later
  • Review volume, rating, freshness, and response posture support the entity.
  • External mentions corroborate the business rather than creating ambiguity.
  • Known profiles and directories point to the canonical domain.
  • Competitor citation, mention, and backlink gaps can be added when provider data exists.
  • Authority signals are reported separately from correctness signals; a high-authority wrong listing is a risk, not a win.

Patterns To Follow

  • Backend must follow Routes to Controllers to Services to Models; no DB queries in routes/controllers outside models.
  • Frontend API calls must live in typed API modules and React Query hooks, using QUERY_KEYS factories.
  • Scoring rules live in versioned code. Persist the rule version and results, not mutable rule logic only in the database.
  • Every score component must include evidence, status, remediation, data scope, and confidence; no unexplained scores.
  • External URL fetches must be admin/internal first with strict SSRF protections before any customer exposure.
  • External web consistency checks must distinguish missing-on-site, conflicting, candidate, and ambiguous states instead of pretending every external source is authoritative.

Reference files: src/services/integrations/gscHarvestAdapter.ts for integration validation/fetching shape; frontend/src/components/website/WebsiteDashboardTabs.tsx for client tab style; frontend/src/pages/admin/OrganizationDetail.tsx for admin embedded website routing.

Constraints

Must

  • Start as an internal/admin mini app before customer exposure.
  • Support both url_only and organization audit scopes.
  • Support multi-location organization audits and location-level score breakdowns.
  • Support external exact and near-exact web source discovery for entity consistency checks.
  • Persist audit runs, targets, results, evidence, and rule version.
  • Show score, data coverage, confidence, hard caps, evidence, and remediation.
  • Reuse existing GSC, GBP, project identity, and website page data boundaries.
  • Include docs parity for admin/client dashboard UI once implemented.

Must Not

  • Do not expose arbitrary URL audit to customers in the first implementation.
  • Do not create a standalone Next.js app or separate database for this feature.
  • Do not duplicate GBP collection logic already owned by getGBPAIReadyData.
  • Do not mark missing integration data as a failure; mark it unavailable and reduce data coverage/confidence.
  • Do not declare an external source correct in URL-only mode unless the submitted URL or a stronger connected source corroborates it.
  • Do not store secrets, crawler tokens, full OAuth payloads, or private credentials in audit evidence.
  • Do not treat llms.txt as a core scoring requirement.

Out Of Scope

  • Automated fixes to websites, schema, GBP, or GSC settings.
  • Production authority provider integration for backlinks/citations.
  • Live AI prompt/citation testing across ChatGPT, Perplexity, Claude, or Google AI Mode.
  • Customer-facing arbitrary external URL scanner.
  • Changing existing website generation prompts or templates.

Risk

Level: 4 - cross-system dashboard feature with new persistence, external URL fetching, scoring semantics, and future customer-facing impact.

Risks Identified

  • SSRF and network abuse: arbitrary URL audit can fetch internal network targets if naive. Mitigation: admin-only first, validate protocol/host, block private/reserved IPs after DNS resolution, disallow redirects to private IPs, enforce timeouts, byte limits, and rate limits.
  • Misleading scores: missing GBP/GSC/authority data can look like failure. Mitigation: separate score from data coverage and confidence; use unavailable and not_applicable result states.
  • Architecture drift: scoring can sprawl into controllers or UI. Mitigation: isolate rule evaluation in backend services, DB access in models, and UI rendering in small typed components.
  • Production migration risk: new audit tables touch production deployment. Mitigation: additive-only schema, reversible drops, indexed foreign keys, no backfill, no data rewrite.
  • Expensive audits: rendering pages and parsing sitemaps can become slow. Mitigation: queue long audits through a worker or explicit async run status before broad rollout; first implementation can support synchronous small runs only if timeouts are strict.
  • Data leakage: evidence can accidentally expose PII, OAuth state, or private GBP review text. Mitigation: evidence stores normalized facts and excerpts only, never tokens or full raw provider payloads.
  • External search false positives: web results can mix similarly named practices, old listings, lead-gen pages, and scraped directories. Mitigation: use entity disambiguation, source reliability weights, location/domain matching, and explicit ambiguous/candidate states.
  • Search provider dependency: external consistency scanning requires a search API or controlled search provider with cost, rate limits, and terms-of-use constraints. Mitigation: make the provider pluggable and keep the app useful without external scan data by marking that segment unavailable.
  • Synchronous audit timeout: URL fetching, sitemap discovery, external consistency checks, and scoring can exceed normal HTTP request time. Mitigation: audit launch must create a queued run/job immediately, process asynchronously, and expose current step/progress through polling.

Blast radius: Admin organization detail, admin website detail, client website dashboard tabs if exposed, GSC harvest model, GBP profile collector, project identity model, website pages model, new audit persistence tables, audit job runner, polling query hooks, query cache keys, dashboard docs parity.

Pushback

  • A standalone Next.js app is the wrong first move. It duplicates auth, org mapping, GBP/GSC access, and website project state already owned by Alloro.
  • A single opaque score is not enough. Future-us will hate this if every number cannot point to evidence and remediation.
  • External URL audit should not be customer-facing in phase one. That belongs behind admin controls until SSRF, rate limits, and output quality are proven.

Rev 9 - June 10, 2026

Change: Scoring-accuracy and comprehensibility pass. Accuracy: (1) entity.connected_canonical_match pass now awards its full weight (was 4 of 5, making 100 unreachable for orgs); (2) missing GBP (connected.gbp_profile, authority.reviews) is now unavailable — coverage/confidence impact only — instead of a partial score penalty, matching the spec's missing-integration rule; (3) added a rubric-integrity assertion: per-category check weights must sum to the declared category weight or scoring throws; (4) regex-based content checks relabeled deterministic (the llm_assisted method value is reserved for a future real LLM layer). Structure: (5) page selection now prioritizes home → content → legal/utility before the 12-page cap, the run score is an importance-weighted average (home ×2, content ×1, utility ×0.5 via target metadata), and the run summary records totalPages; (6) legal/utility pages take not_applicable on service-page content checks (specificity, evidence units, page-type fit). Reason: The score must be exactly reproducible from the published rubric, and a non-technical reader should understand what the number means and what to do next without decoding statuses. Equal page weighting and alphabetical page selection let boilerplate pages distort organization scores.

Updated Done criteria: Passing checks award full weight; missing integrations never lower the score; rubric drift fails fast; run score is importance-weighted; utility pages exempt from content rubric; results view leads with grade, top fixes, and per-page scores.
