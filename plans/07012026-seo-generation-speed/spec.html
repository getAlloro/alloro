Why

SEO generation ("Generate All" on a page/post, and the project-wide bulk-generate job) is slow: 10 fully sequential Claude calls per entity today (5 sections × generate + insight, each awaited before the next starts), with zero prompt caching despite a naturally cacheable system prompt. Separately, the model swap done earlier this session (claude-sonnet-4-6 → claude-sonnet-5) introduced a live regression — Sonnet 5 rejects the hardcoded temperature: 0 default with a 400, confirmed by a direct API test. That must ship as part of this work since nothing currently using Sonnet 5 in this codepath can succeed until it's fixed.

What

Four changes, landing together: (1) make the LLM runner omit sampling params for models that reject them, fixing the live Sonnet 5 breakage; (2) restructure section generation from 5 sequential rounds into 3 tiered rounds (3 sections run in parallel, then 1, then 1) via a single shared helper used by both the single-entity and bulk-generate flows; (3) stop blocking each section's progression on its own insight call — insight promises run concurrently and are awaited once at the end, with the API response shape unchanged so the frontend needs no changes; (4) set an explicit effort: "medium" on Sonnet 5 SEO calls instead of inheriting Sonnet 5's new silent adaptive-high-by-default behavior. Done when: SEO generation completes without erroring on Sonnet 5, "Generate All" for one page/post measurably drops from ~10 sequential round-trips to ~3, the bulk worker inherits the same speedup with no code duplicated, and a manual before/after quality check on a handful of real pages shows no regression in generated field quality.

  • src/workers/processors/seoBulkGenerate.processor.ts — 237 lines. Calls generateAllWithSharedContext() once per entity inside its own per-entity for loop (project-wide bulk job). Constraints

Must

  • The model-aware sampling-param fix in service.llm-runner.ts must be inert for every caller still on Sonnet 4.6 or older — verified by checking the request body sent is byte-identical to today's for those models.
  • The tiered-execution helper must be used by both generateAllSeoSections and generateAllWithSharedContext — one implementation, not two parallel rewrites (§4.3).
  • API response shape for "Generate All" (array of { section, generated, insight }) stays unchanged — no frontend changes required.
  • effort level lives as a named constant, not an inline string repeated at call sites (§4.2).
  • Error propagation behavior is preserved: today a single section failure already fails the entire generate-all call (no per-section catch exists) — Promise.all for the parallel tier preserves this exactly (first rejection propagates).

Must Not

  • Do not change DEFAULT_MODEL or touch any of the other 16 runAgent callers' behavior.
  • Do not defer/remove insight generation from the API response — keep it synchronous-from-the-caller's-perspective, just unblock the internal await chain.
  • Do not address the og_image_recommendation field-mismatch gap (already tracked separately as task_f5bd36bc) or the missing page-type signal that under-triggers LocalBusiness/Service schema — both out of scope here.
  • Do not add new database tables/columns — no schema change is needed for this work.

Out Of Scope

  • The og_image/og_image_recommendation field mismatch (separate background task already flagged).
  • Passing an explicit page-type/category signal into the prompt to fix LocalBusiness/Service schema under-generation (separate, larger scope decision raised earlier in this session — needs its own -s).
  • Automated test coverage for service.seo-generation.ts (pre-existing gap, not introduced or worsened by this change — not addressed here per the no-drive-by-cleanup rule).

Risk

Level: 3 — touches a shared file (service.llm-runner.ts) with 17 callers, and changes core generation behavior for a customer-facing feature.

Risks Identified

  • SEO generation on Sonnet 5 is currently broken (confirmed via live API test: temperature: 0 → 400 invalid_request_error). Mitigation: T1 below, gated strictly by model name so the other 16 callers (all on Sonnet 4.6 or older, where temperature: 0 is valid and may be intentional for determinism) see zero behavior change.
  • effort: "medium" is an informed guess, not yet validated against real output. Mitigation: Done checklist requires a manual before/after comparison on real pages before this is considered complete; the override is a single named constant, trivially revertable if quality regresses.
  • Tiered parallelization triples peak concurrent Claude calls per entity (1 → 3, for the first tier). Mitigation: bounded at 3 regardless of how many pages a bulk job processes (the worker still does one entity at a time, each entity's tier-1 burst is capped at 3) — well within standard Anthropic rate-limit tiers; no scaling risk with project size.
  • Shared-helper extraction means a bug affects both the single-entity editor flow and the bulk worker simultaneously (today a bug in one wouldn't necessarily exist in the other, since they currently duplicate the loop independently). Mitigation: thorough manual QA on both flows before --done, full TS build gate, and the duplication being removed is itself a net risk reduction going forward (one place to fix bugs, not two that can drift).

Blast radius: service.llm-runner.ts — all 17 current runAgent callers (PM AI synth, GBP draft generation, practice ranking, image processor, slot generator, URL scrape strategies, identity distillation, Mission Control insights, monthly/daily/webhook agent runners, PMS paste analysis/JSON parsing, website contact AI analysis, audit leadgen worker) — reviewed and confirmed inert for all except SEO generation, since none of the others reference claude-sonnet-5 or later models today.

Pushback

  • The bulk worker's per-entity uniqueness tracking (batchTitles/batchDescriptions) inherently caps how parallel bulk-generate can ever get across entities — entity N+1's title-uniqueness check depends on entity N having finished. This spec only parallelizes within one entity's sections, not across entities. Cross-entity parallelism would need a redesign of the uniqueness-tracking approach (e.g. checking against a snapshot taken at job start, accepting some title duplication risk across entities generated in the same burst) — flagged as a future option, not pursued here since it changes a real product guarantee (title uniqueness) rather than being a pure infra speedup.
