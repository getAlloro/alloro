Why

Mission Control org cards show a green Website pill but no signal of which analytics/integration platforms a practice has connected. The Websites tab already renders that signal as a row of small platform logos. Bringing the same logos onto the Mission Control card gives admins an at-a-glance integration health read without opening each org.

What

Render the active-integration logos (HubSpot, Rybbit, Clarity, Search Console) inline to the right of the Website pill on each Mission Control organization card, reusing the exact icon set, order, and styling from the Websites tab. Driven by real connection data (website_builder.website_integrations, status = 'active') for the org's latest website project.

Done when: an org that shows integration logos in the Websites tab shows the same logos, in the same order, inline with the Website pill in Mission Control — and an org with no active integrations shows none (no empty badge, no layout shift).

  • frontend/src/pages/admin/WebsitesList.tsx - source of the pattern: file-scoped logos (100-133), maps (46-53, 128-133), getActiveIntegrationPlatforms (135-142), ActiveIntegrationLogos (144-161), call site (724).

  • src/controllers/admin-websites/feature-services/service.project-manager.ts - backend reference analog: the active_integrations two-step query + dedupe (lines 120-146).

Patterns To Follow

  • Icon set, order (hubspot, rybbit, clarity, gsc), labels, and the 20px circular white badge come verbatim from WebsitesList.tsx — do not invent or restyle icons.
  • Backend integration fetch mirrors service.project-manager.ts:120-146: collect project ids, one whereIn(...).where("status","active") query, dedupe into a per-project map. No per-org N+1.
  • Adding a field to MissionControlOrganization is additive and follows the existing camelCase service-mapping convention (e.g. hasGbpConnection, websiteStatus).

Reference file (frontend): frontend/src/pages/admin/WebsitesList.tsx lines 100-161 - the component being extracted. Reference file (backend): src/controllers/admin-websites/feature-services/service.project-manager.ts lines 120-146 - the integrations query to mirror.

Constraints

Must

  • Extract logos/maps/component into one shared module and have BOTH WebsitesList and the Mission Control card consume it (single source).
  • Join integrations to the org's latest project only (the same project whose status already drives websiteStatus).
  • Reuse the existing 4-platform set, order, labels, and badge styling exactly.
  • Keep icons display-only and inside the existing stopPropagation wrapper so they never trigger card navigation.

Must Not

  • No DB schema change, no migration (website_integrations already exists).
  • No new dependencies.
  • Do not add a third copy of the logos — no parallel abstraction.
  • Do not aggregate integrations across all of an org's projects.
  • Do not touch GBP logic (hasGbpConnection / google_connections) — separate source, separate chip.

Out Of Scope

  • De-duplicating IntegrationProviderList.tsx's copy of the logos (pre-existing debt; noted, not fixed here).
  • Any "not connected" / revoked / broken state styling on the Mission Control card — show active only, matching the Websites tab.
  • Click-through / linking from a Mission Control logo to that integration's settings.

Risk

Level: 2 (Concern)

Risks Identified

  • Backend data plumbing required. The card has no integration data today. Mitigation: add activeIntegrations to MissionControlProjectSummary via a single batched query mirroring the Websites tab; additive field, no consumer breakage.
  • Wrong-project join. An org may have multiple projects; integrations must come from the latest one shown. Mitigation: capture the chosen project's id inside getLatestWebsiteSummaries and query integrations for exactly that id.
  • Feature + refactor mixed. Reuse forces editing WebsitesList.tsx. Mitigation: pure lift-and-shift into a shared module + one call-site swap; no behavior change to the Websites tab. Verify the tab still renders identical icons.
  • Icons without a Website pill. A project with active integrations but null status would hide the row. Mitigation: render the row when hasConnectedWebsite || activeIntegrations.length > 0.
  • Extra query per load (L1). One additional batched query on Mission Control load. Mitigation: single whereIn, no N+1; negligible, identical cost to the Websites tab.

Blast radius:

  • MissionControlProjectSummary - consumed only within MissionControlModel.ts (def 26, MissionControlBaseData.websites 62, builder 188/202). Additive.
  • baseData.websites - read only at MissionControlService.ts:129. Additive.
  • MissionControlOrganization - consumed by the card + Mission Control page. Additive field, safe.
  • Extracted logos/maps/component - currently file-scoped in WebsitesList.tsx (not exported); no external importer today, so the move is safe.

Pushback

  • The 4 logos already exist in two files (WebsitesList.tsx, IntegrationProviderList.tsx). This plan removes WebsitesList's copy into a shared module but intentionally leaves IntegrationProviderList's copy alone to avoid scope creep. Recommendation: file a follow-up to point IntegrationProviderList at the shared module so all three surfaces share one source.

Revision Log

Rev 1 - 06052026

Change: Restored the ?? [] guard inside getActiveIntegrationPlatforms and made ActiveIntegrationLogos's integrations prop tolerate undefined/null; guarded the card's render condition with (organization.activeIntegrations?.length ?? 0) > 0.

Reason: Runtime crash (Cannot read properties of undefined (reading 'filter')) on Mission Control. The app uses PersistQueryClientProvider, so a persisted react-query payload from before the backend change rehydrates with orgs lacking activeIntegrations. The TS type marked the field required, so the compiler did not catch the missing-at-runtime case. The extraction had dropped the guard that the original Websites-tab code carried.

Verification: Browser-verified on /admin/mission-control (dave@getalloro.com): 9 cards render, no error boundary, icons inline with the Website pill; Garrison = Rybbit + Search Console; empty org (McPherson) shows no icons and no crash. tsc + eslint clean.
