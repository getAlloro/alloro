Why

Patient Journey currently exposes Searching your market from legacy rank-keyword volume. After the backend foundation proves the new market keyword data, the API needs to expose org-wide Estimated Search Opportunity instead.

What

Change the Patient Journey backend contract so the first funnel stage reads from the new Market Intelligence summary, aggregated across all organization locations. The existing selected location still drives location context cards, but the market-opportunity stage becomes an org-level demand estimate that matches the whole-site scope of GSC impressions.

Dependency: plans/06262026-market-intelligence-backend-foundation must be executed, tested, and have real pilot rows before this spec executes.

Context

  • src/controllers/patient-journey/feature-services/stageReaders.ts currently calls KeywordSearchVolumeModel.getMarketVolumeForLocation().
  • src/controllers/patient-journey/feature-services/PatientJourneyService.ts builds the market_demand stage label and source.
  • src/controllers/patient-journey/feature-utils/types.ts is the backend response contract mirrored by the frontend.
  • src/__tests__/patient-journey.service.test.ts covers current stage order and labels.

Patterns To Follow

  • §6.1 and §7.1: keep the existing controller/service/model chain.
  • §7.4: read Market Intelligence data only through models/services from the foundation spec.
  • §8.1: preserve the existing { success, data, error } response envelope.
  • §11.7: org-level aggregation must still be tenant-scoped by server-derived organization id.

Reference file: src/controllers/patient-journey/feature-services/stageReaders.ts - closest current stage-reader shape.

Constraints

Must

  • Keep the stage key stable as market_demand unless frontend compatibility requires a coordinated versioned change.
  • Return an org-wide estimated opportunity value across all locations.
  • Return metadata for keywordCount, clusterCount, sourceBreakdown, coverage, confidence, null-volume count, and source freshness.
  • Mark the first stage as shared/org-wide in the response so the UI can label it honestly.
  • Fall back to an honest unavailable stage when the foundation has no approved market keyword volume yet.

Must Not

  • Do not keep calling legacy KeywordSearchVolumeModel for the first card after this spec executes.
  • Do not label the metric as people or live searches.
  • Do not make the whole response fail when Market Intelligence data is missing.

Out Of Scope

  • Frontend visual changes. Those live in plans/06262026-market-intelligence-dashboard-ui.
  • Customer-facing coverage display.
  • New keyword generation or harvest logic.

Risk

Level: 3. This changes the backend contract of a live dashboard screen.

Risks Identified

  • Contract drift. Frontend types mirror backend types. Mitigation: update backend tests first, then frontend types in the UI spec.
  • Scope mismatch. Selected location context cards remain per-location while opportunity becomes org-wide. Mitigation: return explicit scope metadata.
  • False precision. A larger keyword set can look exact. Mitigation: response includes confidence and coverage, not only a number.

Blast radius: Patient Journey backend stage readers, response types, service tests, and any frontend consumer once UI spec executes.

Pushback

  • Do not execute this spec until foundation data has been inspected on at least one pilot org. Otherwise we are only replacing one uncertain number with another.
