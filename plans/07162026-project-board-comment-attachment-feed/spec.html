Why

The task sidebar splits comments and files into isolated views. A file uploaded from Attachments is invisible in the conversation, and the Comments tab has no file drop path, so the task history is incomplete and upload progress depends on which tab is open.

What

Turn Comments into a chronological task feed that interleaves real comment rows and real task attachment rows. Reorder the tabs to Details, Comments, Attachments; allow file drag-and-drop plus a keyboard-accessible picker from Comments; and keep completed, pending, failed, and deleted attachment state synchronized across both tabs.

Core contract: pm_task_comments and pm_task_attachments remain the sources of truth. The frontend creates a typed read model sorted by created_at; it does not create a synthetic comment or a new attachment relationship.

Constraints

Must

  • Render Comments as an ascending chronological feed of comment and attachment records, with a deterministic tie-breaker for equal timestamps.
  • Append a pending upload item immediately after the current latest feed item, then replace it with the server attachment record on success.
  • Show the same upload queue and failure state in Comments and Attachments regardless of where the upload started.
  • Accept drag-and-drop anywhere in the Comments upload target and provide an equivalent labeled file-picker control for keyboard and assistive-technology users.
  • Keep the Comments badge limited to real comments and the Attachments badge limited to completed task attachments.
  • Preserve existing MIME limits, file-size limits, preview/download behavior, delete permissions, comment editing, mentions, and markdown safety.
  • Removing an attachment from either tab must remove it from both after the successful server mutation.
  • Use QUERY_KEYS, React Query invalidation/cache updates, typed props, visible focus states, and the existing PM theme variables.

Must Not

  • Do not add a database migration or comment_id relationship.
  • Do not create fake comment rows to represent uploads or trigger comment notifications for attachment-only activity.
  • Do not use pm_activity_log as the feed source; that log is clearable and is not the canonical comment/file record.
  • Do not cherry-pick the stale codex/project-board-comments-ux commit wholesale; its attachment filtering contradicts this feature.
  • Do not add new API endpoints, dependencies, state libraries, raw fetches, inline styles, or unrelated PM refactors.
  • Do not leave failed uploads looking completed or add them to the attachment count.

Out Of Scope

  • Real-time cross-user updates through WebSockets or polling.
  • Email or in-app notification changes.
  • Changes to permitted file types, size limits, S3 storage, or backend authorization.
  • Threaded comments, attachment-to-comment grouping, comment attachments that delete with a comment, or rich inline file embeds inside markdown.
  • A full visual redesign or full decomposition of legacy PM sidebar components beyond extracting the logic needed for this feature.

Risk

Level: 2 - Concern

Risks Identified

  • Two independently fetched copies of attachment state can drift across tabs. Mitigation: route both tabs through the same React Query keys and one shared upload queue.
  • Comments are returned oldest-first while attachments are returned newest-first. Mitigation: normalize both into a discriminated feed union and sort once by timestamp plus a stable id tie-breaker.
  • Drop events can fire more than once when nested elements handle them. Mitigation: centralize the drop handler, prevent default propagation, and give each queued file a unique client id.
  • An upload can reach S3 but fail before the UI receives the completed metadata response. Mitigation: keep the pending row in an explicit failed state, do not increment the attachment count, and allow the user to retry by selecting or dropping the file again.
  • The existing sidebar components are already large. Mitigation: place all new feed rendering, sorting, and query orchestration in focused child files/hooks; touched legacy components must not gain another independent data path.

Blast radius: Project Board task detail tab ordering; comment loading, editing, deletion, mention rendering, and counts; attachment upload, progress, preview, download, deletion, and counts. Backend routes, models, migrations, notifications, S3 keys, and other PM surfaces remain unchanged.

Pushback

  • The previous comment_id design is the wrong model for this request because it intentionally removes comment uploads from the Attachments library. Recommendation: keep one task attachment collection and render it in two views.
  • Using the activity log would appear simpler, but clearing that log would erase the conversation view while the comments and files still exist. Recommendation: merge the canonical records in the frontend.
  • Drag-and-drop alone excludes keyboard users. Recommendation: the Comments tab gets both a drop target and a labeled file picker.
