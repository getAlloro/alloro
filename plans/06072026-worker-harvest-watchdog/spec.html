Worker Stall Watchdog & Redis Hardening

Why

On 2026-06-07 the minds-worker process was alive but wedged — pm2 reported online (idle CPU, flat memory) while it had processed zero jobs for ~42h. Every scheduled job silently stopped, so the 5 AM data harvest missed Jun 6 and Jun 7 and Clarity/Rybbit/GSC data froze at Jun 5. Nothing alerted; it was found only by manual inspection. The worker's Redis connections have no error/close/reconnect handlers, so a dropped or hung connection fails silently — the exact signature observed.

What

Two complementary changes: (1) an external watchdog that detects a stalled worker / missed harvest and emails a dev alert to dave@getalloro.com via the existing n8n email webhook, and (2) Redis connection hardening so a dropped/hung connection logs loudly and self-heals instead of silently freezing all queues. Alert-only (no auto-restart, per decision).

Done when: the worker writes a processing heartbeat every scheduler tick and on each harvest completion; an out-of-process checker emails dave@getalloro.com (error-format) when the heartbeat goes stale or a daily harvest is missed, de-duped so it doesn't spam; and worker Redis connections emit visible error/reconnect logs and recover from a transient drop.

Context

Relevant Files

  • src/workers/queues.ts - getRedisConnection() (L9-19, shared producer connection, same gap). Hardening target.
  • src/workers/processors/scheduler.processor.ts - processSchedulerTick (L16). Note: it early-returns at L19 when no schedules are due — the heartbeat must be recorded BEFORE that return so it fires every 60s.
  • src/workers/processors/dataHarvest.processor.ts - processDataHarvest; record harvest completion after the Daily harvest complete log (L84).
  • src/emails/emailService.ts - sendEmail({ subject, body, recipients }) (L123) over ALLORO_EMAIL_SERVICE_WEBHOOK (n8n). The alert sender — the "other implementations" analog.
  • New src/workers/workerHealth.ts - heartbeat read/write helper (file-based).
  • New src/scripts/worker-watchdog.ts - the out-of-process checker (compiled to dist/scripts/worker-watchdog.js, run by system cron).
  • Ops (alloro-app): pm2 at /home/ubuntu/.nvm/versions/node/v22.18.0/bin (shebang needs node on PATH); env at /etc/alloro/app.env; worker logs ~/.pm2/logs/minds-worker-*.

Patterns To Follow

  • Send the alert with emailService.sendEmail exactly as other senders do (subject + HTML/text body + recipients); do not introduce a second email path.
  • Keep maxRetriesPerRequest: null on worker connections (BullMQ requirement); hardening is additive (event handlers + bounded retryStrategy), not a rewrite.
  • Heartbeat must reflect actual queue processing (tie it to the scheduler tick + harvest completion), not mere event-loop liveness — the stall had a live loop but dead job processing.

Reference file: src/emails/emailService.ts (alert sender) and any existing caller of sendEmail for body/subject shape. Decision recorded: the Clarity/Rybbit/GSC harvest itself is left as-is (it is already correct for what the Clarity API allows — numOfDays 1-3 returns an aggregate, not per-day; verified 2026-06-07). This plan does not change harvest cadence.

Constraints

Must

  • The checker runs outside the worker process (system cron) — an in-process self-check is worthless (pm2 showed "online" while wedged).
  • Heartbeat written on every scheduler tick (before the early-return) and on harvest completion.
  • Alert via emailService.sendEmail to dave@getalloro.com, error-format subject/body, including last-tick / last-harvest times, hostname, and remediation (pm2 restart minds-worker).
  • De-dupe alerts: notify on healthy→unhealthy transition and re-alert at most every N hours while still unhealthy.
  • Redis connections: add error/close/reconnecting/end handlers that log, plus a bounded retryStrategy; keep maxRetriesPerRequest: null.

Must Not

  • No auto-restart (alert-only, per decision).
  • No change to harvest cadence / the Clarity adapter (out of scope; correct as-is).
  • No new heavy dependency (use fs + existing emailService).
  • Do not store the heartbeat in Redis (it is the failing dependency) — use the filesystem so detection survives a Redis outage.
  • Do not block or slow the scheduler tick / harvest (heartbeat write must be cheap and non-throwing).

Out Of Scope

  • Trailing-3-day Clarity rollup (deferred per decision — no consumer yet).
  • Auto-restart / self-healing (explicitly alert-only this round).
  • A full resilience audit of all ~20 workers beyond the shared connection-handler change.
  • Backfilling the already-lost data (handled operationally on 2026-06-07).

Risk

Level: 3 (Structural — touches shared worker Redis infrastructure used by all ~20 queues)

Risks Identified

  • Redis-handler change affects every worker (L3). A bad retryStrategy could worsen reconnection. Mitigation: additive handlers only; conservative bounded backoff (e.g. min(times*200, 5000)); keep maxRetriesPerRequest: null; deploy the worker and watch ~/.pm2/logs/minds-worker-* for a clean boot + first scheduler tick before walking away.
  • Root cause unconfirmed. Idle CPU + silence strongly implies a hung Redis connection, but it was never proven. Mitigation: the hardening adds visibility (loud logs) so a recurrence is diagnosable, and the external watchdog is the guaranteed safety net regardless of cause.
  • Alert-only means data can still be lost (L2). A missed Clarity day is unrecoverable (API has no historical fetch). If a stall happens overnight and the email isn't actioned before the next day rolls over, that day is gone. Mitigation: check every ~5 min and alert fast; revisit capped auto-restart later if alert latency proves too slow.
  • Watchdog needs an ops step (L2). The system cron entry is not part of the app deploy. Mitigation: document the exact crontab line + env in this spec and in the runbook; verify it is installed post-deploy. Alternative: run the checker as a separate small pm2 app.
  • Alert spam (L1). A 5-min checker could email every tick while down. Mitigation: de-dup state file (transition + N-hour re-alert).

Blast radius:

  • makeConnection() / getRedisConnection() - used by all worker + producer queues. Additive handler/retry changes; verify normal processing after deploy.
  • processSchedulerTick / processDataHarvest - one cheap, non-throwing heartbeat write added; no behavioral change to dispatch/harvest logic.
  • New workerHealth.ts / worker-watchdog.ts - net-new; no existing importer.

Pushback

  • The watchdog treats the symptom; the Redis hardening targets the (suspected) cause but can't be confirmed a cure. Recommend shipping both, then watching the new connection logs for the next disconnect to confirm the mechanism before considering it closed.
  • You chose alert-only. Given Clarity days are unrecoverable, consider a fast check cadence now and keep capped auto-restart as a documented follow-up if a human can't reliably action the email within a few hours.
