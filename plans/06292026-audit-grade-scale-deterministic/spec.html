Why

The audit tool (audit.getalloro.com) shows letter grades that do not match the approved scale. Grades are inflated and inconsistent: a 58% has shown as C (should be F), a 72% as B or C+ (should be C-), a 68% as B (should be D+). Jo flagged this on June 17 and documented the full scope in a reference sheet. It is a blocking trust issue for using the audit as a sales asset — every prospect who received a report has an inflated grade on it.

Root cause, confirmed in code: the three grade cards are produced three different ways, and two of them let the LLM freehand the letter instead of deriving it from the score.

CardFieldToday's sourceDefect
GBP Readinessgbp_gradescoreToGrade() in gbpAnalysisAggregator.tsDeterministic but a coarse 5-band scale (A/B/C/D/F, no +/-). 88→B, 77→C.
Website Performanceoverall_gradeLLM output (WebsiteAnalysis.md), passed throughLetter never derived from score. 72→"C+", 76→"C+".
Local Rankingrank_gradeLLM output (gbp/CompetitorAnalysis.md), passed throughCohort-relative letter, untethered from score. 72→"B", 68→"B", 62→"C".

What

Make the letter grade a pure, deterministic function of the percentage score, applied uniformly to all three cards, using the approved 12-band scale. The LLM stops emitting letters; code owns the mapping.

Approved scale (implement exactly):

Done when: for every card, the letter shown matches this table for the displayed percentage. No score 0–59 shows anything but F. No score 70–72 shows anything but C-. Jo re-runs ≥2 documented audits and confirms.

Decisions locked with the user: (1) Fix-forward — no historical backfill script / no DB rows rewritten. (2) Also update the prompts — the website and competitor agents stop emitting a letter grade.

Constraints

  • Single shared scoreToGrade() in audit-utils/, imported by every consumer (§4.3, §6.3).
  • Derive the letter from the rounded score (Math.round), so the letter matches the integer the UI displays.
  • Guard non-finite input: if a score is missing/NaN, fall back to the stored letter — never blank or mislabel (§3.x).
  • Keep the served response shape identical (frontend contract unchanged: overall_grade / gbp_grade / rank_grade still strings).

Must Not

  • No new dependencies. No DB migration. No backfill script (fix-forward only).
  • Do not touch GBPAnalysis.md (legacy/dead) or the four non-competitor GBP pillar prompts.
  • Do not change the numeric scores or the scoring prompts' scoring logic — only the letter.
  • No drive-by refactor of the aggregator beyond swapping the grade function.

Out Of Scope

  • Rewriting historical audit_processes rows (declined).
  • The practice-ranking / Journey Insights feature, which has its own separate rank_score logic (§4.3 — different domain, not the same concept).
  • The leadgen frontend's mock-data fallback leak and dead /checkup flow — tracked separately.

Risk

Level: 2 — Concern. Contained logic change, blast radius fully mapped, no schema/auth change. Elevated above Level 1 only because it touches the sales-critical audit pipeline and changes a grade's meaning.

Risks Identified

  • rank_grade semantic change. CompetitorAnalysis.md:41 intentionally makes rank_grade cohort-relative (A = top quartile), independent of the score. Deriving it from rank_score abandons that intent. Mitigation: rank_score is already a 0–100 cohort score (top 25% ⇒ 90+, bottom 25% ⇒ <40), so the letter stays cohort-meaningful and now matches the displayed %. This is exactly what Jo's card requires and makes all three cards consistent. Proceed.
  • Serve-time derivation also corrects historical links. Deriving in the normalize (serve) layer means old audit links render corrected letters too — without rewriting any DB row. This is a superset of "fix-forward," not the backfill script that was declined (no writes, no migration). Mitigation / decision needed: recommended, because the old inflated letters are the problem. If you instead want historical links frozen at their original letters, we fix write-time only and leave the serve path passing stored letters through — flag it and the spec adjusts.
  • Prompt field removal vs. output validator. Removing overall_grade / rank_grade from the two prompt schemas could break a parser/validator that hard-requires them. Mitigation: T4 verifies audit-utils/validationUtils.ts and agents/service.llm-runner.ts tolerate the absent field before the prompts ship; code populates the letter regardless.
  • Rounding edge. A raw 72.6 displays "73" but must grade as the band for its rounded value. Mitigation: round once, derive from the rounded integer; unit-tested at boundaries (59/60, 69/70, 72/73, 92/93).

Blast radius: gbpAnalysisAggregator.ts (gbp_grade write + log line at processor 584), normalizationUtils.ts (serve path → leadgen frontend), the two prompt files, scripts/admin-audit.ts log. No email, no admin frontend, no other API consumer.

Pushback

  • The real lesson is architectural: a 12-band rubric is a lookup table; an LLM will never honor it by hand. Even after this fix, keep the rule "the letter is always scoreToGrade(score)" — never re-introduce an LLM-authored grade. The migration notes' "≥80% grade parity" tolerance was the original mistake; this replaces tolerance with determinism.
