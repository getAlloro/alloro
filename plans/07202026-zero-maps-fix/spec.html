This spec is not the fix yet — and the fix is contingent on an unconfirmed hypothesis. A low-risk, additive, logging-only diagnostic ships first (branch claude/zero-maps-diagnostic) to confirm or refute the root cause with real API values. Only after the next real daily run confirms the cause should the fix below be executed. Do not treat the zeros as fixed by this document.

Why

Alloro is the CMO of the owner's local presence — it must report the Get Found number (Google impressions = the top funnel gate) honestly. Right now the daily agent stores 0 for the Google Maps impressions term, which understates or zeroes the owner's visibility gate. A wrong "0" at the top of the funnel poisons every downstream rate (CTR, CRO) and breaks the honesty that is the product.

Leading hypothesis (UNCONFIRMED): the daily agent fetches only yesterday and day-before-yesterday (getDailyDates). The Google Business Profile Performance API trails several days for impression metrics, so the two most-recent days legitimately return no data (which the code sums to 0). Nobody has yet verified against the live API that Maps is 0 for yesterday but non-zero for older dates, or whether Search is also 0. The diagnostic below produces that evidence.

What

The robust fix: instead of fetching only the two most-recent single days, fetch a small trailing window (default ~7 days) in one API call per metric set, then use the most-recent day(s) that actually carry data. This is robust to whatever the real API lag turns out to be — no hard-coded "subtract N days" offset guess. If the whole window is empty, the payload says "no recent data" honestly rather than silently reporting 0.

Done when: for a location with known non-zero recent Maps/Search impressions, the daily Proofline payload's visibility.* values reflect the most-recent day that has data (not 0), the value carries the date it came from, and an all-empty window is reported as "unavailable", not 0. Confirmed by re-running the diagnostic and by Dave's live-API check (see Dave's confirmation test).

Constraints

Must

  • Fetch a trailing window in one API call per metric set, not N single-day calls (bounds the extra API cost — see Risk).
  • Pick the most-recent day that has data; do not average across the window unless the product explicitly wants a window sum.
  • Distinguish "0 interactions on a day that reported" from "no data for this date". §4.2: window size is a named constant in config/.
  • Keep the DB writes in service.proofline-executor.ts consistent — the stored date_start/date_end must describe the window actually used.
  • §9.1 Pino logging; §3.1/§3.2 every async wrapped, errors logged with context, never swallowed; §20.1 tests mirror the changed units.

Must Not

  • Hard-code a "subtract N days" offset guess for the lag — the window + most-recent-with-data is the point.
  • Add new dependencies.
  • Touch the monthly agents' date logic (getPreviousMonthRange, getCurrentMonthRange) — out of scope.
  • Refactor the GBP fetch chain beyond what the window change needs (no drive-by cleanup — §Refactoring Rule).
  • Execute this fix before the diagnostic confirms the cause.

Out Of Scope

  • Monthly agent windows and any dashboard/frontend rendering of the number.
  • Back-filling historically-stored 0 rows in google_data_store (separate data task if wanted).
  • The diagnostic itself — it ships on this branch now and is the gate, not part of the fix execution.

Risk

Level: 3 — Structural risk: broad blast radius plus a behavior change to what every daily agent sees and stores.

Risks Identified

  • Shared date source. getDailyDates is the single date source for the whole daily pipeline. Changing its shape changes what the daily processor fetches AND what the proofline executor stores. Mitigation: keep getDailyDates's existing return shape intact for backward-compat and add the window as a new, separately-consumed helper; migrate consumers deliberately, not implicitly.
  • Extra API cost / rate limits. A 7-day window is more data per call; if implemented as N calls it is 7× the requests. Mitigation: one fetchMultiDailyMetricsTimeSeries call spanning the window (the API already takes a date range), not a loop.
  • Semantic drift of stored dates. Downstream rows (agent_results.date_start/date_end, google_data_store) currently mean "yesterday/day-before". After the change they mean "the window / the day actually used". Mitigation: store the actual resolved date(s) and document the meaning change in the Revision Log at execution time.
  • Hypothesis may be wrong. If the diagnostic shows Search is ALSO 0 for older dates, or the API returns data for yesterday, the root cause is elsewhere (auth scope, wrong location, metric name, or a real zero). Mitigation: the diagnostic gates the fix; do not execute until it points at lag.

Blast radius (known consumers of getDailyDates / the daily fetch):

  • service.daily-agent-processor.ts — param typed ReturnType<typeof getDailyDates>; drives both fetch windows and the google_data_store raw row (date_start/date_end).
  • service.proofline-executor.ts:96 — reads the dates for the agent_results + google_data_store transaction writes.
  • service.agent-input-builder.ts — extractMetricTotal / flattenDailyGbpData shape the values every daily agent (Proofline) reads.

Pushback

  • Because this is Level 3, the fix must not auto-execute. Confirm the cause with the diagnostic first, then execute deliberately with a Revision Log entry recording the stored-date semantic change. A cheaper-but-wrong alternative — hard-coding "fetch 4 days ago" — is explicitly rejected: it bakes in a guess at the lag that will silently rot when Google changes it.

Dave's confirmation test (live API — the exact check)

This is the check Dave runs against the live Google Business Profile Performance API to (a) confirm the lag is the cause and (b) later validate the fix. It needs a real OAuth2 client and a location with known recent activity.

Step 1 — Confirm the cause (before the fix)

  1. Pick an active location (accountId, locationId) that you know had Maps/Search impressions in the last two weeks.
  2. Call the same path the agent uses, but over a 14-day range: getGBPAIReadyData(oauth2Client, accountId, locationId, <today-14>, <today-1>), or hit fetchMultiDailyMetricsTimeSeries directly with the four BUSINESS_IMPRESSIONS_* metrics for that range.
  3. Inspect performance.series[].dailyMetricTimeSeries[].timeSeries.datedValues[] per metric. Read off, per calendar date, the value for DESKTOP_MAPS, MOBILE_MAPS, DESKTOP_SEARCH, MOBILE_SEARCH.

Hypothesis is CONFIRMED if: the most-recent 1–3 dates are empty/absent for all metrics (Maps AND Search), and older dates carry non-zero values — i.e. it is an API lag, not a Maps-specific zero. Hypothesis is REFUTED if: yesterday already has data (cause is elsewhere), or Search has data while Maps is 0 across the whole window (a Maps-specific problem — auth scope, metric name, or a genuine zero), in which case do not ship the trailing-window fix; re-diagnose.

Step 2 — Validate the fix (after execution)

  1. Run the daily agent (or the diagnostic) for that same location.
  2. Confirm the Proofline payload's visibility.impressions_maps_* now reflect the most-recent day that had data (matching what you read manually in Step 1), and each value carries the date it came from.
  3. Confirm a location that genuinely had zero activity is reported as such (not conflated with "no data").

Revision Log

Rev 1 - 2026-07-22

Change: Executed T1–T3. T1 — getDailyTrailingWindow() added beside getDailyDates (whose shape is untouched), with the window size as the named constant DAILY_TRAILING_WINDOW_DAYS = 7 in src/config/dailyAgents.ts (§4.2). T2 — the two single-day fetches became ONE window fetch, and a new pure module feature-utils/gbpWindowSelector.ts resolves the most-recent day(s) the API actually published, plus an explicit "no data in window" signal (an empty result). No fixed offset is subtracted anywhere — the mechanism is most-recent-with-data, so it survives Google changing the lag. T3 — the Proofline payload now carries period.window_start/window_end, period.latest_data_date, period.previous_data_date and period.has_recent_data, with visibility.latest/engagement.latest set to null (never 0) when the window published nothing; the stored google_data_store row carries a per-side data_date and OMITS its visibility object entirely when no day reported.

Reason: Dave's Step-1 live-API check on production CONFIRMED the hypothesis — the Business Profile Performance API trails roughly 3–4 days, so the two most-recent dates were legitimately absent and the old code summed an empty datedValues array to a reported 0. That is the gate this spec set for executing the fix, and it is met.

Updated Done criteria: Stored-date semantics changed — agent_results.date_start/date_end and the google_data_store row's dates now describe the days Google ACTUALLY published, not calendar yesterday and the day before. The yesterday / dayBefore keys in the stored blob keep their names (renaming them is a wider migration than this fix needs) but their MEANING is now "most-recent published day" and "the one before it"; each side's data_date records which day that actually was. Consumers reading those keys get real values instead of fabricated zeros. The omitted-visibility case is deliberate and load-bearing: stageReaders.mapsImpressionsForVisibility already treats a side with no visibility object as a MISSING metric excluded from day-coverage, and a present-but-zero object as a real measured zero — so "no recent data" and "zero impressions" stay distinct all the way to the dashboard. Still open: Dave's Step-2 live validation after deploy.

Rev 2 - 2026-07-22

Change: Hardened after an independent adversary pass on the Rev 1 code. (a) Cross-metric lag skew. The selector originally treated a date as "covered" if ANY of the seven metrics reported it — which silently assumes Google publishes all metrics for a day atomically, something nobody measured. If the interaction metrics (CALL_CLICKS, WEBSITE_CLICKS, BUSINESS_DIRECTION_REQUESTS) publish even a day ahead of the impression metrics, the newest "covered" day would carry interactions but no impressions, an absent metric reads 0, and we would stamp a fabricated measured zero impressions onto a real date — the original zero-Maps bug wearing a verified-looking timestamp. Metrics are now split into IMPRESSION_METRICS and INTERACTION_METRICS and each family resolves against its own dates, so the assumption is removed rather than relied on. (b) Review window regression. The GBP fetch scopes its review list to the requested range, so widening the fetch from 2 days to 7 silently widened "new reviews" to 7 days — inflating the daily narrative roughly 3.5x and re-reporting the same review for a week. The list is now re-narrowed to the prior 2-day meaning (filterReviewsSince); a review with an unreadable timestamp is KEPT, since dropping a real review to tidy a list is the worse error. (c) Removed the now-dead extractMetricTotal (zero callers after T2) rather than leave the bug's own mechanism in the file as an attractive nuisance.

Reason: Adversary review of the Rev 1 implementation. (a) is the finding that mattered: the fix's whole discipline is "never assume publication timing", and Rev 1 assumed it once, for cross-metric atomicity — in the one place the merged diagnostic (impressions-only) is blind to.

Updated Done criteria: The stored row now carries data_date = the impressions day (the gate this row is read for) and a separate engagement.data_date when the interaction family resolved to a different day. Regression tests cover the skew case, the review-window narrowing, and the unreadable-timestamp fallbacks. Known gaps, stated: the metric-family split is now pinned at the processor too — a processor-level test drives processDailyAgent with a skewed response and asserts the stored row resolves impressions on their own day, so collapsing the families inside service.daily-agent-processor.ts fails the suite (closed in Rev 3). Also unchanged and pre-existing: only locations[0] is read, and the setDate/toISOString idiom shifts the REQUESTED window by a day across a DST fall-back on a non-UTC server (never the reported dates, which always come from the API's own date fields).

Rev 3 - 2026-07-22

Change: Closed the Rev 2 known gap. Added a processor-level test (daily-agent-processor.test.ts) that drives the real processDailyAgent with a skewed API response — interactions published two days ahead of impressions — and asserts the stored row resolves impressions on the impressions' own newest day, with the interaction family carrying its own later date. Mutation-verified: collapsing the two families inside the processor's wiring now fails the suite.

Reason: The Rev 2 tests pinned the family split at the selector and builder, but called the selector directly, so the processor's wiring was unpinned — a regression there would have passed. Requested to close the gap rather than carry it.

Updated Done criteria: the metric-family split is now enforced end-to-end (selector, builder, AND processor wiring); no remaining known-gap in the split.
