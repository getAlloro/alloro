Why

A client (One Endodontics, org 39, all locations in US Eastern) uploads May data into the May slot and the dashboard labels it "April." Investigation (plans-less -b, 2026-06-22, 16-agent trace + adversarial verification) proved the data is stored correctly as "2026-05" — this is a frontend display bug, not data corruption. Several PMS labels render a "YYYY-MM" key with new Date(key + "-01"), a date-only string the JS engine parses as UTC midnight and then formats in the browser's local time. For any negative-offset (US) user, midnight-May-1-UTC is the evening of April 30 locally, so the label rolls back a month. It is invisible in dev because the dev machine runs Asia/Manila (UTC+8, positive offset). Every US-timezone client is affected; the first-of-month dates that trigger it are common in DentalEMR exports.

What

Route all 11 broken month-label sites (7 files) through the codebase's existing timezone-safe labeler, frontend/src/utils/timeframe.ts formatDataMonth() (string-only, builds from a MONTH_NAMES array via parseYM, never touches Date) plus a short-format sibling and the exported month-name arrays for the pickers. Done when: every PMS month label shows the correct month for a US-timezone user (proven by a timezone unit test), the persisted month is untouched, and no new Date(key + "-01") pattern remains in frontend/src/components/PMS.

Context

The bug pattern

Unsafe (parses as UTC, renders local → shifts for US users): new Date(m.month + "-01").toLocaleDateString(...). Safe, already used in the same tree: new Date(`${m}-01T00:00:00`) (local parse) and, the canonical one, formatDataMonth() which never constructs a Date from the key at all.

Patterns To Follow

  • The shared labeler lives in frontend/src/utils/ (§12.2 — pure, framework-free helpers), already proven by timeframe.ts + timeframe.test.ts.
  • Pickers render the month name by loop index — use a name array, never a Date, so the visible tile always matches its committed "MM" value.

Reference file: frontend/src/utils/timeframe.ts + frontend/src/utils/timeframe.test.ts — the closest existing analog (timezone-safe month labeler with a unit test).

Constraints

  • Keep formatMonthLabel signatures and outputs identical so the 11 consumers are untouched (§4.3, §13.3).
  • Add a timezone unit test (§20.1) proving the fix under TZ=America/New_York and unchanged under Asia/Manila.
  • Leave the persisted month, all backend code, and all data flow untouched.

Must Not

  • No new dependencies; no new state; no any / console.* / raw fetch (§17.1, §17.2, §14.2).
  • Do not change the month-selection logic or the submitted payload key — only the rendered label.
  • Do not "fix" the pre-existing utils/ → components/ import of parseYM (no drive-by refactor).
  • Do not touch the backend parseDateToMonth duplication — separate concern.

Out Of Scope

  • DentalEMR standard-format hardening (production basis billed-vs-collected, header-signature brittleness, status filter, Doc-space guide) — separate plan, needs an owner decision first.
  • Backend parseDateToMonth 4× dedup.
  • Moving parseYM out of components/ into utils/.

Risk

Level: 2 (Concern)

Risks Identified

  • Format parity — sites use different formats (short MMM YYYY, long Month YYYY, month-only). A blind swap would change formats beyond the month value. Mitigation: add formatDataMonthShort; map each site per the table above; drive pickers off month-name arrays by index.
  • Helper delegation drift — the two formatMonthLabel copies differ (long vs short). Mitigation: delegate each to the matching variant so output bytes are identical; covered by the unit test.
  • utils → components boundary (§12.4, Level 1) — timeframe.ts already imports parseYM from components/. Mitigation: reuse as-is; do not worsen; advisory note only.

Blast radius: 7 PMS component files (labels only), 2 PMS util helpers (delegation, signatures unchanged → 11 consumers unaffected), timeframe.ts (additive — new short variant + exported arrays), timeframe.test.ts (additive). No backend, no data, no payload changes.

Pushback

  • The cheaper fix (append T00:00:00 at each site) was rejected: it leaves the same footgun in place and three parallel labelers. Routing through one tested helper kills the bug class (§4.3).

Docs parity

  • This corrects an existing label; it adds no controls, copy, or navigation. Likely no Alloro Docs change. At -d, check whether any PMS "Manage Data" screenshots in ~/Desktop/alloro-docs bake in a wrong month label; update only if so.
