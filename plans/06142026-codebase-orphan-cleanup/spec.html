Why

The repo has accumulated tracked assets, backup files, unreferenced frontend components, unmounted route modules, and dead backend helper islands. Leaving them around makes search results noisy, invites false reuse, and makes future agents guess which path is current.

This cleanup should remove files that are actually unreachable while preserving anything that is intentionally parked or product-ambiguous. Future-us should not have to spelunk old prototypes to find the active implementation.

What

Delete confirmed orphaned files in staged risk groups, rerun type/build verification, and leave product-ambiguous dead islands untouched unless the spec is revised. The output is a smaller tracked source tree with no broken imports and a clear list of intentionally retained dead islands.

Current Audit Evidence

  • Static graph roots: src/index.ts, src/workers/worker.ts, frontend/src/main.tsx, Vite config, scripts, migrations, seeds, and config entrypoints.
  • Tracked code scan found 110 unreachable app-code files after excluding plan and Friyay artifacts.
  • 68 unreachable app-code files have zero inbound code references.
  • Known dead islands: Checkup route island (17 files, about 5.9k lines), Monday route island (10 files, about 953 lines), Documentation endpoint island (4 files, about 186 lines), old focus dashboard frontend island (14 files, about 2.9k lines), and legacy PMS dashboard island (7 files, about 1.3k lines).
  • Sibling repo scan across website-builder-rebuild, alloro-leadgen-tool, alloro-site, and other Desktop alloro*/orange* folders found no runtime dependency on the planned Alloro deletions. Non-blocking findings: historical plan/docs references, one alloro-docs source comment naming ActionQueue.tsx, one independent alloro-site/src/components/Button.tsx, and copied stale files under alloro-email-manager.

Scope Ledger

  • Objective: remove orphaned tracked files that have no current runtime/code references.
  • Intended user: internal Alloro engineering team and future agents working in the repo.
  • Surface: repository source tree only; no runtime behavior should change.
  • In scope: unreferenced assets, backup files, orphaned frontend component islands, orphaned backend services/models, and unreferenced barrels.
  • Out of scope: deleting product-ambiguous unmounted route features without explicit confirmation; migrations; database/data cleanup; docs copy updates unless an active UI surface changes unexpectedly.
  • System of record: current tracked files and import graph from the app, worker, scripts, migration, seed, and config roots.
  • Existing analogs: no new runtime files should be created; deletion-only cleanup follows current import/export patterns by removing consumers and dependencies together.
  • Open questions: are Checkup, Monday, and Documentation intentionally retired or accidentally unmounted? This spec assumes they are not deleted.
  • Drift warnings: static import graphs do not prove external HTTP clients are absent. Unmounted routes are unreachable from this app but may reflect parked product work. Sibling repos can hold duplicate stale files; those are separate cleanup surfaces, not blockers for deleting files from this repo.

Context

Relevant Files

  • src/index.ts - backend app root and Express route mount source of truth.
  • src/workers/worker.ts - BullMQ worker root and processor reachability source of truth.
  • frontend/src/main.tsx and frontend/src/App.tsx - frontend runtime roots and route ownership.
  • frontend/vite.config.ts - Vite alias and frontend build root.
  • package.json and frontend/package.json - build/typecheck commands.
  • src/database/migrations/* and src/database/seeds/* - treated as tooling entrypoints, not import-graph orphans.
  • scripts/* and src/scripts/* - treated as manual entrypoints, not import-graph orphans.

Patterns To Follow

  • Delete whole dependency islands together so no import points at a removed file.
  • Do not remove migrations, seeds, scripts, or plan artifacts solely because app code does not import them.
  • Do not remove unmounted product routes unless the feature is explicitly retired.
  • For frontend cleanup, run the frontend build because TypeScript and Vite are the actual integration gate.
  • For backend cleanup, run root TypeScript build because backend compilation catches broken route/service/model imports.

Reference file: src/index.ts - route registration is the backend reachability source of truth.

Constraints

Must

  • Rerun the static orphan check immediately before deleting files.
  • Read any file group before deleting it if its status is not purely artifact/backup.
  • Keep deletions staged by risk so failures are easy to isolate.
  • Run npx tsc --noEmit, npm run build, and cd frontend && npm run build after deletion.
  • Inspect git diff --name-status and verify only planned files were deleted.

Must Not

  • Do not introduce new dependencies, abstractions, or replacement code.
  • Do not refactor active files during cleanup.
  • Do not delete migrations, seeds, scripts, or plan artifacts as part of this plan.
  • Do not delete Checkup, Monday, or Documentation route islands without a spec revision and explicit user confirmation.
  • Do not touch runtime env files, server config, database state, or deployment workflows.

Out Of Scope

  • Re-mounting dead routes.
  • Product decisions about whether Checkup, Monday, or Documentation are retired.
  • Alloro Docs parity updates unless execution discovers an active dashboard/admin/client UI surface is affected.
  • Database migrations or data cleanup.

Risk

Level: 3 - structural cleanup touching many files across frontend, backend, assets, and parked feature areas.

Risks Identified

  • Static analysis false positives. Some files can be invoked by tooling, routes, or external clients rather than imports. Mitigation: treat scripts, migrations, seeds, and configs as roots; do not delete unmounted product route islands without confirmation.
  • Product surface removal disguised as cleanup. Checkup, Monday, and Documentation are not mounted today, but they look like real features. Mitigation: mark them parked and out of scope for this deletion pass.
  • Frontend build drift. Removing orphaned components can expose stale barrels, route assumptions, or type-only imports. Mitigation: delete whole islands and run frontend build.
  • Backend compile drift. Removing model barrels or services can expose stale imports. Mitigation: run root TypeScript build and inspect all compiler errors by cause.
  • External public asset hotlinks. Repo search cannot prove nobody externally links public assets. Mitigation: only delete assets with no internal references and accept that public URL compatibility is not guaranteed by this repo alone.
  • Cross-repo stale references. Other Alloro folders may mention deleted files in historical plans, source comments, or copied old trees. Mitigation: treat runtime/source imports as blockers; treat comments and historical plan references as non-blocking notes unless they affect active docs copy or builds.

Blast radius: frontend SPA build, backend TypeScript compilation, public asset bundle size, developer search/navigation, and any external client relying on unmounted route files if they are later re-mounted from history.

Pushback

  • Deleting the Checkup island in this cleanup would be sloppy. It is 2k+ lines at the route root, has email templates and analytics-like services, and prior plans mention it being intentionally ported. If it is retired, say that directly and make it a separate revision.
  • Deleting rankingPeriod.ts is tempting but wrong for this pass. It is intentionally documented as the future enable point for a disabled Local Rankings period toggle. Keep it unless product decides the toggle is dead.
  • Do not convert this into a refactor. If an active file looks ugly while deleting dead files, leave it alone. Cleanup plus refactor is how accidental regressions sneak in.
