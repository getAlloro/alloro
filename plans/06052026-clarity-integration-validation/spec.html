Why

The Clarity integration UI collects a Project ID (outbound tracking) and an optional API token (inbound Data Export). Admins have no way to confirm the tracking tag is actually live on the published site, and the integration can sit in a connected-but-not-working state. Live evidence: Caswell has an active Clarity record but no tag on the page; Garrison's live tag is a legacy snippet whose Project ID (x27lax5nj4) does not match the record (r9diusipt9). The pipe is connected but unfed, and nothing surfaces it.

What

Add a Validate installation action and a completeness gate to the Clarity integration page. Validation checks three things server-side: (1) Project ID is valid, (2) the stored API token still authenticates against Clarity, and (3) the live published page actually serves the Clarity tag for the record's Project ID (flagging a mismatched/legacy tag). The integration is marked Complete only when all three pass. The script tag itself is derived from the Project ID (shown read-only/copyable) — never stored — and admins may paste a full snippet to auto-extract the Project ID.

Done when: clicking Validate on a properly connected site shows all three checks green and a "Complete" badge; validating Garrison surfaces a Project ID mismatch; validating Caswell surfaces "tag not found"; and no renderer changes are required.

Patterns To Follow

  • Token validation already exists as validateApiToken(projectId, token) — reuse it; classify the result into valid | invalid | error. Do not write a second Clarity API client.
  • HTTP handler + envelope response shape: mirror the existing clarity handlers in WebsiteIntegrationsController.ts and the Envelope<T> client funcs in api/integrations.ts.
  • UI status chips: reuse the StatusPill pattern already in ClaritySettingsCard.tsx.
  • Live-HTML parsing: reuse isClaritySnippetCode / extractClarityProjectId logic — collect all Clarity Project IDs found on the page, not just the first.

Reference file (backend): src/controllers/admin-websites/feature-services/service.gsc-integration.ts - closest analog for an integration service that validates against a live external surface. Reference file (frontend): the existing ClaritySettingsCard.tsx StatusPill block for chip styling.

Constraints

Must

  • Derive the script tag from the Project ID (canonical clarity.ms/tag/{projectId} bootstrap). Show read-only + copyable. Never persist a tag string.
  • Mark Complete only when: Project ID valid AND token authenticates AND live tag present for the record's Project ID.
  • Probe ONLY the project's own stored hostnames (custom_domain → custom_domain_alt → generated_hostname → hostname), over HTTPS, with a timeout and response-size cap.
  • Use the decrypted token internally only; the API must never return the token. Surface only valid | invalid | missing | error.
  • Distinguish live-tag outcomes: present (record ID found), mismatch (only other Clarity IDs found — return them), absent (no Clarity tag), error (fetch failed / no URL).

Must Not

  • Never mutate the integration status column during validation. The renderer injects only status='active' rows; flipping to broken on a token/probe failure would stop live tracking. Write only metadata.validation, last_validated_at, last_error.
  • No renderer changes (website-builder-rebuild already injects Clarity).
  • No DB schema change / no migration — persist the validation snapshot in the existing metadata JSONB.
  • No new dependencies. Use the HTTP client already in the service (axios).
  • Do not run the live probe on every getStatus load (network cost) — only on explicit Validate.

Out Of Scope

  • Reconciling the live data itself (deciding Garrison's correct Project ID, applying the backfill migration in prod). Validation will surface these; fixing the data is a follow-up operational task.
  • The Data Export "sample harvest" check (not selected). Validation does not run a harvest.
  • Surfacing "Complete" on the Mission Control card (separate feature: mission-control-integration-icons).
  • Rybbit/HubSpot/GSC validation parity (Clarity only for now).

Risk

Level: 3 (Structural Risk — a naive implementation can silently disable live tracking)

Risks Identified

  • Status-flip kills tracking (L3). If validation sets status='broken' on a failed token/probe, the renderer stops injecting Clarity for that project. Mitigation: validation is read-only w.r.t. status; it writes only metadata.validation + last_validated_at + last_error. Add an explicit code comment stating why.
  • Server-side fetch abuse / SSRF (L2). The probe fetches a URL. Mitigation: URL is built only from the project's own stored hostnames (never free user input); enforce HTTPS, a short timeout, a max-content cap, and minimal redirects; admin-only route.
  • Token leakage (L2). Mitigation: decrypt internally via getDecryptedCredentials; return only a status enum; never echo the token or full credentials.
  • Token-mandatory Complete blocks tracking-only setups (L2, accepted). Per decision, Complete requires a valid token even though injection works without one. Mitigation: document in the UI ("API token required to complete"); the separate Data Export pill still communicates the token's real purpose.
  • CDN/cache lag & client-only quirks (L2). A freshly saved tag may not appear immediately if the page is cached; the probe reads server-rendered HTML (renderer injects server-side, so normally fine). Mitigation: treat absent as "not detected yet," not "broken"; surface the probed URL + timestamp so admins can recheck.
  • Frontend/backend snippet-util duplication (L1). The derive/extract helpers exist in the backend util and will be mirrored in a small frontend util (cross-runtime). Mitigation: keep the frontend copy minimal and reference the backend util in comments.

Blast radius:

  • service.clarity-integration.ts / ClarityStatus - consumed by WebsiteIntegrationsController.ts (clarity handlers) and, via the API, by api/integrations.ts → ClarityTab / ClaritySettingsCard. The completeness field is additive.
  • New route + handler - additive; no existing endpoint changes.
  • ClaritySettingsCard props - additive (onValidate, completeness, validating); update the single call site in ClarityTab.
  • website_integrations.metadata - additive validation key; renderer ignores unknown metadata keys.

Pushback

  • Requiring the API token for "Complete" conflates outbound tracking with inbound Data Export. Recommended alternative was "Project ID + live tag" with the token as a separate capability. Proceeding token-mandatory per decision; revisit if practices that only need tracking get stuck "incomplete."
