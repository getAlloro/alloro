Why

The DataForSEO-backed "Search Opportunity" feature (the first card of the Patient Journey funnel and the whole Market Intelligence engine behind it) is being retired. It carries a recurring per-client cost, the numbers are directional at best (impressions vs. estimated search volume), and the owner has decided the feature is not worth keeping. We are removing it completely — code, database tables, scheduled jobs on dev and prod, server credentials, and the planning artifacts — so nothing residual is left behind.

What

A full teardown of both generations of the search-volume system, plus the visual removal of the first funnel card, while the rest of the Patient Journey funnel keeps working.

  • Backend: delete the entire Market Intelligence module, the legacy gen-1 search-volume layer, the DataForSEO integration, and the worker processor. Unregister the worker and its cron schedules.
  • Patient Journey: remove the first funnel stage (market_demand / "Potential patients searching") from the backend contract and re-anchor the funnel so "Saw" (GSC impressions) becomes the head. Keep stages 2–6 and the rankings/reviews context intact.
  • Database: a new reversible Knex migration that drops keyword_search_volume and the market_* tables. It runs on dev first, then on prod at the next main deploy.
  • Frontend: remove the first stage card from the pipeline view, strip search-opportunity wording, drop the stage from the API types, and remove any dedicated Search Opportunity / Market Intelligence screen if one exists.
  • Servers: cancel the BullMQ repeatable jobs and obliterate the orphan queues in Redis on alloro-dev and alloro-app (prod), and remove the DataForSEO credentials from the server env files.
  • Artifacts: remove the five related plan folders and the build memory, leaving a short "removed" tombstone marker and a changelog entry for the audit trail.

Done looks like: the funnel renders without the first card; grep for DataForSEO / Market Intelligence / search-volume across src/ and frontend/src/ returns nothing; the drop migration is applied on dev; no harvest-market-intelligence / harvest-search-volume jobs or queues remain in dev or prod Redis; backend + frontend type-checks and tests pass.

Pushback

  • Level 4 — do not auto-execute. Sequence dev-first: land the code + drop migration on dev/dave, verify dev (funnel renders, migration applied, dev Redis clean), then merge to main for prod. Cancel prod Redis jobs only as part of that promotion, and re-verify on prod after deploy.
  • Order of operations matters. Cancel the running jobs / pause the worker before or during the deploy that removes the worker code, so an in-flight harvest doesn't error against dropped tables. Recommended: cancel jobs → deploy code (worker gone) → migration drops tables, in that order per environment.
  • Reversibility is shallow. The migration down() recreates empty tables, not the harvested data. If there's any chance the feature comes back, take a quick dump of the tables before the prod drop (optional; the owner has chosen full removal).
