Why

A read-only audit (2026-07-02) of One Endodontics, Artful Orthodontics, Garrison Orthodontics, and getalloro.com found the SEO-generation pipeline producing several confirmed defects that plausibly cost real leads: an invented schema.org business type that Google likely can't parse ("Endodontist" / "Orthodontist" don't exist in schema.org's vocabulary), zero social-preview images on every one of 55 published pages across all 4 sites despite the generator writing a detailed image brief for each, no star-rating structured data despite these practices having hundreds to thousands of real 5-star reviews, and generated FAQ content that never becomes usable FAQ schema. These are pipeline bugs, not per-page mistakes — the same defect repeats on every page/post the generator has touched.

What

Two independent deliverables:

Task A — page-level social image (quick, low-risk): set og_image on every published page of all 4 sites to an owner-supplied photo (one photo per site, all pages on that site share it).

Task B — fix the SEO-generation pipeline, then apply it to existing content: correct the four confirmed generator defects (invalid schema type, missing og_image, missing aggregateRating, unconverted FAQ candidates) as targeted, deterministic code fixes — then run that fix against the existing published posts/articles of One Endodontics, Artful Orthodontics, and Garrison Orthodontics (not getalloro.com, which has no posts). getalloro.com's page-level image is still covered under Task A.

Done when: all 55 pages show a real og:image on the live site; every post on the 3 named sites has a valid schema.org @type, a real aggregateRating sourced from actual GBP data, an og_image sourced from the post's own featured_image where one exists, and an FAQ schema block wherever FAQ candidates exist — with title/description/canonical/target-query fields on those posts byte-for-byte unchanged from before the run.

  • §3.1/§3.2 — enrichment must handle one post's malformed/missing data (e.g. no GBP link, no featured_image) without failing the whole batch, matching the existing try/catch-per-entity shape in seoBulkGenerate.processor.ts.

Constraints

Must

  • Preserve every existing correct field on a post's seo_data during enrichment — read-patch-write the full object, given the write path is a full replace.
  • Source aggregateRating from real, currently-synced GBP review data — never LLM-authored.
  • Validate schema.org @type against a hardcoded allowlist before it can reach the database, regardless of what the prompt/LLM produces.
  • Test the enrichment pass against a single post on one site and manually verify before running the 3-site batch.
  • Verify after the batch run that Artful's 74 and Garrison's 33 already-correct og_image posts are unchanged.

Must Not

  • Do not re-run full LLM section generation (title/description/target-query tiers) against already-published posts as part of this plan — those fields aren't broken and re-running them risks regressing live, indexed copy for no benefit.
  • Do not invoke the existing POST /:id/seo/bulk-generate endpoint as-is for this fix — confirmed unsafe without the enrichment guarantees above (full-column replace + no featured_image awareness today).
  • Do not touch getalloro.com posts (none exist) or the two archived ghost projects (Garrison orphan, DentalEMR) — out of scope, tracked separately.
  • Do not add a new dependency for image handling or schema validation — plain JS/TS is sufficient for an allowlist check and an FAQPage object transform.

Out Of Scope

  • Fixing the same generator defects for pages (only posts are covered by Task B's regeneration; pages' schema/FAQ/rating gaps are a known follow-up, not in this plan).
  • Automatic og_image sourcing for future brand-new pages (pages have no featured_image concept — a future page still needs a manually-chosen photo via the editor).
  • The renderer soft-404 / archived-project-still-live / sitemap-and-robots fixes — already tracked in a separate background task.
  • Prod canonical_url repair for posts — already completed in a prior session for these same 3 sites.

Risk

Level: 3 — Structural Risk (cross-cutting change to a shared, platform-wide generation engine, plus a live-content write against real client sites)

Risks Identified

  • Level 3 The post seo_data write path (PostModel.ts:407, updateSeoDataByIdJsClock) is a full-column replace. Naively re-running the existing generation/bulk-generate flow risks silently dropping Artful's 74 and Garrison's 33 already-correct og_image values. Mitigation: build the fix as a dedicated enrichment function that always starts from the row's current seo_data, patches only the 4 targeted keys, and writes the merged object back — verified with a before/after diff on a sample post from each site before the full batch.
  • Level 2 The invalid schema-type bug is a prompt-authoring problem (SeoGeneration.significant.md literally instructs "'Endodontist'-equivalent specialty type"), not just an LLM slip. A prompt fix alone doesn't guarantee compliance from a non-deterministic model. Mitigation: rewrite the prompt to use only "Dentist" (schema.org's actual most-specific valid dental type, with specialty conveyed via knowsAbout/description as the generator already does well) AND add a code-level allowlist guard/fallback as defense in depth.
  • Level 2 Star rating and review count are factual claims about a real business; letting an LLM author them risks the same hallucination failure mode that produced the invalid schema type, with a worse trust/compliance downside if wrong. Mitigation: inject aggregateRating in code from the already-synced GBP data path (mirroring buildReviewsMetrics), never ask the generation prompt to produce it.
  • Level 2 service.seo-generation.ts / util.seo-section-runner.ts are shared by every org on the platform, not just these 3 sites — any regression in the enrichment logic affects future generation everywhere. Mitigation: the enrichment step is additive-only and guarded (schema-type allowlist with safe fallback, featured_image/rating injection only when source data exists) so it can't make an already-working generation worse; test on one post before any batch run.
  • Level 1 Two of the four owner-supplied page images have unusual aspect ratios (Garrison: square 600×600; getalloro.com: a 1128×191 banner strip) that will likely crop awkwardly in a standard 1.91:1 social-preview card. Mitigation: use them as supplied (still strictly better than no image at all) and flag to the owner as a possible future asset swap — not a blocker for this plan.

Blast radius: SeoController.ts (single-section generate, generate-all, bulk-generate routes — used by every org's admin SEO tooling), seoBulkGenerate.processor.ts (shared BullMQ queue), any future page/post SEO generation platform-wide. Directly written: website_builder.posts.seo_data for ~254 published posts across 3 live client sites (One Endo 121, Artful 84, Garrison 49), and website_builder.pages.seo_data for 55 published pages across 4 live client sites.

Pushback

  • The user's own phrasing was "regenerate correct SEO data" — the literal reading is a full re-run through the existing bulk-generate endpoint. This plan deliberately narrows that to a targeted enrichment pass instead, because the full re-run (a) risks the regression described above and (b) rewrites title/description/target-query copy on live, presumably-performing pages that were never identified as broken. If a full regeneration is actually wanted (e.g. because the existing titles/descriptions are also considered weak), that is a materially different, larger, costlier task (LLM calls across ~254 posts × 6 tiers) and should be scoped as its own decision, not bundled into this fix.
