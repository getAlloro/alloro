Website Detail: Trim Page-List Payload

Why

The admin website-detail endpoint GET /api/admin/websites/:id (service.project-manager.ts getProjectById → PageModel.findByProjectOrderedPathVersion → findByProjectOrderedPathVersionQuery) runs SELECT * over every page row, including all version history. Each row carries the full sections HTML blob, so a large site (One Endodontics: 345 version rows) returns a ~16 MB payload in ~36 s (measured live on dev, responseTime: 35983). The pages list never renders page content — it only shows metadata (path, version number, status, SEO indicator). Page content is already loaded separately, on demand, when a page is opened for editing.

What

Trim the list query to a lightweight column projection — keep all version metadata (so the list's per-path version history and version numbers still work) but drop the heavy content JSONB columns (sections, edit_chat_history, generation_progress). Content continues to load only when a page is opened (unchanged fetchPage path). Fix the one list-level consumer of sections — the version revert button — to re-fetch that version's content on demand. Done when the website-detail payload no longer carries page content, the pages list + version history render unchanged, revert still works, and opening a page for editing still loads its content.

Context

Relevant Files

  • src/models/website-builder/pageQueries.ts:511 — findByProjectOrderedPathVersionQuery, the SELECT * to trim. §7.4 (query lives in the model layer — correct home).
  • src/controllers/admin-websites/feature-services/service.project-manager.ts:271 — the sole caller (getProjectById), confirmed via grep. No other consumer of this query.
  • frontend/src/pages/admin/WebsiteDetail/PagesTab.tsx:474 — the revert handler that reads page.sections from the list row; re-point to an on-demand fetch.
  • frontend/src/api/websites/page-editor.ts:19 — existing fetchPage(projectId, pageId) returns a page/version row's full sections (each version row has its own id), the on-demand re-fetch for revert.

Consumer verification (blast radius)

  • The admin page editor loads content via GET .../pages/:pageId (usePageEditor.ts fetchPage), not from this payload's pages[]. Confirmed.
  • The client surface (useDfyWebsite) uses a different endpoint (/user/website) — unaffected.
  • Only the revert at PagesTab.tsx:474 reads sections from the list row — the single dependency this spec fixes.

Constraints

  • Must keep every metadata field the list + version-history UI renders: id, project_id, path, version, status, created_at, updated_at, generation_status, template_page_id, seo_data, display_name, page_type, artifact_s3_prefix, change_source, revision_note.
  • Must keep returning all version rows (metadata) — the list's expandable per-path history depends on them. This is not a one-row-per-path change.
  • Must not change the editor's content-load path, the client /user/website endpoint, or the response shape beyond dropping the 3 heavy columns.
  • Out of scope: collapsing the list to one-row-per-path; pagination; touching seo_data size.

Risk

Level 2. Hot, shared endpoint — but the query has a single backend caller and the only list-level sections reader (revert) is fixed in the same change. Mitigation: explicit column whitelist (not a fragile SELECT * minus), so a future column addition can't silently re-bloat the payload; verify live on dev against org 39 that payload size and load time drop and the list + revert + editing all still work. Blast radius: getProjectById consumers only — WebsiteDetail.tsx/PagesTab.tsx (list) and usePageEditor.ts (reads project chrome + a pages.find that never touches sections).

Follow-up (not in this spec)

The remaining ~2.7 MB is almost entirely seo_data, which the list keeps because the SEO score is computed client-side in computeSeoScore (websiteDetail.utils.ts:47) and reads most of the blob, including schema_json (it scores each schema by @type). Dropping it would break the per-page SEO score. A larger future optimization: compute the SEO score server-side and return only the number (+ title/description for the uniqueness check) per page — that would let the list drop seo_data entirely and take the payload to a few hundred KB. Separate, bigger change; deliberately out of this spec.
