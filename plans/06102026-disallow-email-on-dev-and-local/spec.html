Why

The dev server (dev.getalloro.com) points at the same production n8n email webhooks as prod, and its CONTACT_FORM_RECIPIENTS contains real recipient addresses. Any email-triggering action on dev or localhost (form submission, OTP, password reset, PMS notification, leadgen report) sends real email to real people. No runtime environment discriminator exists: ecosystem.config.js hardcodes NODE_ENV=production for both PM2 apps and both servers deploy with that same file, so dev and prod are indistinguishable to the code.

What

A runtime sender-identity guard: email dispatches normally only when the sending machine's own public IP is among the DNS A records of app.getalloro.com (verified: prod box 52.203.199.155 matches; dev box 3.210.41.226 does not; DNS points directly at the boxes, no proxy). On any other machine — and on any failure to determine identity — every email is intercepted: recipients rewritten to dave@getalloro.com strictly (cc/bcc emptied), subject prefixed with [Intercepted] , original recipients logged. No env variable, no NODE_ENV, no Host header. Applied inside all 3 email exit functions so every current and future caller is covered.

Done means: all 3 exit points guarded, npx tsc --noEmit clean, lint clean, and a local manual trigger produces an [Intercepted] email addressed only to dave@getalloro.com with originals logged.

Patterns To Follow

  • Module-level config consts + exported async functions + structured logging, as in src/emails/emailService.ts (logEmail levels INFO/ERROR/WARN).
  • axios with explicit timeout for outbound HTTP, as in emailService.ts; node:dns/promises for resolution (built-in, no new dependency).
  • Guard errors must never throw out of the exit functions — mirror sendEmail()'s return-an-EmailResult error style.

Reference file: src/emails/emailService.ts - closest existing analog for the new module's structure (consts, logging, exported config for testing).

Constraints

Must

  • Fail closed: any DNS failure, IP-fetch failure, timeout, or mismatch → intercept.
  • Intercepted recipients are exactly ["dave@getalloro.com"]; cc and bcc emptied.
  • Subject becomes [Intercepted] {original subject}.
  • Guard applied inside the 3 exit functions, not at call sites.
  • Log original recipients on every interception (via logEmail in emailService; console.log in the other two, matching their existing logging style).
  • Cache the live/intercept verdict with a 10-minute TTL; dedupe concurrent in-flight checks behind one promise; fire-and-forget warm-up at module load.
  • Use dns.promises.resolve4("app.getalloro.com") and https://checkip.amazonaws.com (axios, ≤5s timeout, trimmed body) for own-IP discovery.

Must Not

  • No new dependencies (axios and node:dns already available).
  • No env-variable gating, no NODE_ENV reads, no request Host header inspection.
  • Do not modify any caller, template, or email body content.
  • Do not change ecosystem.config.js, knex config, or server env files.
  • Do not refactor unrelated code in the touched files.
  • Never drop an email entirely — interception reroutes, it does not discard.

Out Of Scope

  • Leadgen tool client (separate repo, audit.getalloro.com) posting directly to the n8n webhook from the browser.
  • Emails sent by n8n flows on their own triggers; Stripe's own receipt emails.
  • The Mailgun-based email-manager branch (worktree dave/email-manager) — must adopt this guard when it merges.
  • NODE_ENV / per-environment config normalization (captured as separate background task).

Risk

Level: 2

Risks Identified

  • Infra coupling: adding a CDN/proxy in front of prod or swapping the Elastic IP makes prod stop matching and divert all email to dave. Mitigation: fail-closed is loud — [Intercepted] prod mail lands in dave's inbox immediately; documented here and in agent memory.
  • Transient DNS/checkip failure on prod intercepts mail for up to one TTL (10 min). Mitigation: short TTL, re-check on next send, every interception logged with originals so nothing is lost.
  • checkip.amazonaws.com is an external dependency on the send path. Mitigation: 5s timeout, verdict cached, failure → intercept (never block/drop). EC2 IMDS fallback deliberately omitted to keep the hotfix small.
  • First send after boot/TTL expiry pays up to ~5s of network checks. Mitigation: module-load warm-up primes the cache before the first email.

Blast radius: every email feature — OTP login codes, password reset, billing, support tickets, user invites, admin temp passwords, tier notices, checkup emails, user notifications, worker-watchdog alerts, website contact forms, newsletter, admin form resend, leadgen audit reports. On prod (IP match) behavior is byte-identical to today. On dev/local, all of it reroutes to dave — the purpose of this change. Known consequence: nobody but dave can receive OTP codes on dev; the JWT test shortcut covers dev auth.

Pushback

  • Env-var gating (e.g. ALLORO_PUBLIC_HOST) was rejected: env hygiene already failed once (uniform NODE_ENV, prod webhooks copied into dev.env), and a copied env file could silently re-enable live sending. Self-identification by IP cannot be lied to by configuration.
  • Request Host-header gating was rejected: forgeable (fail-open on dev via spoofed header) and nonexistent for worker/queue/cron sends.
  • Accepted trade-off: email delivery is now coupled to DNS/IP infrastructure. If infra changes, the failure mode is visible interception, not silent client email.
