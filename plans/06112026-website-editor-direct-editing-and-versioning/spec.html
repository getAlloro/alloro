Website Editor: Direct Editing & Versioning

Why

The website editor is AI-first by accident: only elements the generator happened to tag with an alloro-tpl-* class are selectable, so plain <p>/<span>/<a>/<img> elements force users into the AI chat. Saves never create snapshots anywhere, so "revert to version" has nothing meaningful to revert to — and restore silently drops seo_data. The admin editor additionally lacks redo, a refresh guard, and any version history UI, while autosave fires every 800ms making save points meaningless.

What

A fully functional manual-editing workflow across both editor surfaces (admin PageEditor and customer DFYWebsite):

  • Every basic content element (p, span, a, h1–h6, li, button, blockquote, figcaption, img, video) inside a page section is click-selectable and directly editable (text, link href, media swap, font-size step, hide) — no AI required. No element deletion or reordering.
  • Admin editor: working undo and redo (buttons + Cmd/Ctrl+Z / Shift+Cmd/Ctrl+Z), covering AI edits, direct edits, and code-view edits.
  • Admin editor: autosave removed. Explicit Save only. Every content write snapshots the prior state, creating restorable versions.
  • Admin editor: History tab (preview + restore-into-draft) wired into the existing EditorSidebar.
  • Both surfaces: refresh/close with unsaved changes prompts confirmation; admin Back link gets an in-app dirty guard.
  • User-side restore fixed to carry seo_data, display_name, template_page_id; snapshot growth pruned.
  • Constrained rich-text editing: bold/italic/link formatting inside mixed-content paragraphs via a sanitized contentEditable session (Rev 1).
  • Cmd/Ctrl+S save shortcut, localStorage crash recovery, and optimistic-concurrency conflict detection on both surfaces (Rev 1).
  • Save notes + provenance badges in version history, backed by two nullable columns on pages (Rev 1, migration).
  • Version diff badges + per-section restore in admin History (Rev 1).
  • Image upload-in-place + alt-text editing; admin site-wide find & replace; pre-publish lint warnings (Rev 1).

Done when: an admin can edit any paragraph/link/photo by clicking it, undo/redo freely, save deliberately, see each save in History, preview and restore any version into the draft, and never lose work to a refresh.

Patterns To Follow

  • Undo/redo stacks + beforeunload: copy the working implementation in DFYWebsite.tsx.
  • Admin endpoints: thin controller + feature-service pattern, error objects with {status, code, message}, as in service.page-editor.ts.
  • Version model: page rows at the same project_id + path, status ∈ draft|published|inactive, monotonically increasing version (see restorePageVersion for the insert pattern).

Reference file: frontend/src/pages/DFYWebsite.tsx — closest existing analog for undo/redo, beforeunload, and version preview/restore wiring.

Constraints

Must

  • Reuse the existing pages-row version model — no schema change, no migration.
  • Auto-tagging only fires inside [data-alloro-section] (LayoutEditor stays untouched).
  • Admin restore loads the version into the current draft — never auto-publishes.
  • User-side save/restore keeps its publish-live semantics (only fixed + snapshotted).
  • Snapshot rule: on every content write (save/restore/find-replace), snapshot the about-to-be-overwritten state iff it differs from the newest history entry.
  • Keep VersionHistoryTab backward-compatible with DFYWebsite.
  • Rich-text commits pass through an allowlist node-rebuild serializer (never trust raw innerHTML); a[href] normalized via normalizeEditorHref.
  • Migration is additive + nullable only (change_source, revision_note); repo migration file follows the src/database/migrations TS analog.
  • Find & replace writes to draft rows only (creating drafts where needed); never touches published rows directly.

Must Not

  • No new dependencies.
  • No element deletion or drag-reorder capability.
  • No changes to AI edit flow, prompts, or generation pipeline.
  • No modification of the renderer handoff (sections JSONB contract unchanged).
  • No refactor of unrelated editor code.

Out Of Scope

  • AI-as-reviewer "suggest improvements" mode — needs its own design pass (suggestion source, cost, accept/reject UX); separate plan.
  • LayoutEditor (header/footer/wrapper) version history.
  • SEO-data versioning UI (snapshots carry seo_data, but History UI shows section content only).
  • Removing/redesigning the AI chat sidebar.
  • Regex / case-transforming find & replace (plain text + tel/mailto hrefs only).
  • Lint as a publish blocker — warnings are advisory, publish always allowed.

Risk

Level: 2 overall — one Level 3-class item (in-house HTML sanitization = security exposure) mitigated by design, see below.

Risks Identified

  • Auto-tagging persists editor-generated classes into saved content. Mitigation: classes are already the addressing scheme for saved sections; use a deterministic alloro-tpl-m- prefix, assign only on user click, only inside section markers.
  • Removing autosave shifts data-loss risk to the user. Mitigation: beforeunload guard + in-app Back-link guard + existing "Unsaved changes" badge; undo/redo stacks survive until navigation.
  • Page-row snapshots grow the pages table. Mitigation: differs-check dedupe + prune inactive rows beyond 20 per path (never touch draft/published rows). Alternatives considered: (B) dedicated snapshots table — cleaner but needs migration + new UI; (C) provenance column — needs migration. A chosen for zero schema change and full reuse of existing history UI/endpoints.
  • replace-text sets textContent and destroys inline children (e.g. <a> inside <p>). Mitigation: auto-tagging lets users select the child element directly; canvas editing already refuses unsafe nested content. Documented limitation; rich-text editing deferred.
  • Stale-draft refresh clobbers saved draft edits (compares published updated_at vs draft created_at). Mitigation: T8 changes comparison to draft updated_at.
  • (L3-class, Rev 1) Rich-text editing introduces an HTML injection surface — committed markup is persisted and rendered on live customer sites. Mitigation: allowlist serializer that rebuilds nodes (strong, em, b, i, u, small, sup, sub, mark, br, span, a), strips all attributes except href (normalized) and existing classes, and drops anything else; Escape always restores the original; covered by an explicit sanitizer test in Done.
  • (Rev 1) Find & replace fans one action out across many pages. Mitigation: drafts only, per-page snapshot + concurrency check, explicit per-match selection before applying, summary of pages touched.
  • (Rev 1) localStorage backups could resurrect stale content after a publish/restore. Mitigation: backup keyed by draft page id, cleared on save/publish/restore, offered only when newer than the server row's updated_at.
  • (Rev 1) Image-size lint needs HEAD requests to S3 which CORS may block. Mitigation: best-effort — skip the size check silently on failure; other checks are pure DOM.
  • (Rev 1) Additive migration on the shared website_builder.pages table. Mitigation: nullable columns, no backfill, no index — zero impact on existing readers.

Blast radius: useIframeSelector is consumed by PageEditor, LayoutEditor, and DFYWebsite (auto-tag gated to section markers so LayoutEditor is unaffected). VersionHistoryTab + EditorSidebar are consumed by DFYWebsite (props stay backward-compatible). updatePageSections API is called only by the admin editor. savePageSections endpoint is called only by DFYWebsite. createDraft is called from admin page-load and publish flows. (Rev 1) MediaBrowser is shared by both surfaces — upload-in-place must degrade gracefully if a surface's media API lacks upload. The publish confirmation in PageEditor is extended locally; the shared settings/ConfirmModal is not modified. canvasTextEditing eligibility changes affect PageEditor, LayoutEditor, and DFYWebsite equally — the rich-text path must be additive (plain-text elements keep the existing textarea flow).

Pushback

  • Reusing page rows as snapshots overloads one table with two meanings (live workflow states + history). A dedicated snapshots table is the textbook answer — but it costs a migration, duplicate endpoints, and a merged-history UI for marginal benefit at current scale. With dedupe + pruning, row reuse is the pragmatic call; revisit if snapshot volume or listing performance becomes a problem.
  • Keeping user-side save = publish-live is a product decision being preserved, not endorsed; a draft workflow for customers would be safer but is a separate feature.
  • (Rev 1) This is now a release-sized plan (~25 files, 17 feature tasks). Recommend executing in two milestones with a commit each: M1 = T1–T9 (core editing + versioning) and M2 = T10–T18 (feature layer). M2 depends on M1 landing clean; cramming both into one pass raises regression risk for no benefit.
