Why

Once the backend returns a true market-opportunity estimate, the Patient Journey UI must stop implying that the first card is a live count of people searching. It should describe an estimated monthly opportunity across all org locations.

What

Update the Patient Journey frontend and Alloro Docs to show the new first-stage wording, scope, and detail explanations. The visual pipeline remains the same unless the API metadata requires small layout adjustments.

Dependency: plans/06262026-market-intelligence-patient-journey-api must be executed and verified first.

Context

  • frontend/src/types/patientJourney.ts mirrors the backend response contract.
  • frontend/src/components/dashboard/patient-journey/PatientJourneyStageCard.tsx renders the stage card value and label.
  • frontend/src/components/dashboard/patient-journey/PatientJourneyDetailDeck.tsx and detail helpers explain source and stage context.
  • frontend/src/components/dashboard/patient-journey/patientJourney.utils.ts handles funnel helper copy.
  • /Users/rustinedave/Desktop/alloro-docs must be checked for dashboard documentation parity.

Patterns To Follow

  • §12.1: keep frontend API, React Query hook, and feature-folder flow.
  • §14.2: no raw fetch or hand-rolled auth.
  • §15.1: server state stays in React Query.
  • §17.2: no any for new response metadata.

Reference file: frontend/src/components/dashboard/patient-journey/PatientJourneyStageCard.tsx - closest UI card analog.

Constraints

Must

  • Use Potential patients searching or equivalent approved wording.
  • Show that the estimate is monthly and org-wide across all locations.
  • Explain that the number is an estimate from market keywords and DataForSEO.
  • Preserve empty states when no Market Intelligence data exists.
  • Update docs parity if visible dashboard wording changes.

Must Not

  • Do not say people searched.
  • Do not imply month-to-date observed demand.
  • Do not expose raw internal keyword coverage to customers unless explicitly approved.
  • Do not redesign the whole Patient Journey screen in this spec.

Out Of Scope

  • Backend data generation.
  • Admin keyword review screens.
  • Service-cluster charts.

Risk

Level: 3. This changes user-facing dashboard meaning and documentation.

Risks Identified

  • Misleading wording. Users may still read the estimate as observed traffic. Mitigation: use monthly-estimate wording and source detail.
  • Multi-location confusion. Sidebar location selection remains visible while opportunity is org-wide. Mitigation: show all-locations scope in the first-stage detail.
  • Docs drift. Dashboard wording changes require docs parity. Mitigation: update Alloro Docs in the same finalization path.

Blast radius: Patient Journey frontend types, API consumer, pipeline card, detail deck, tests, and Alloro Docs.

Pushback

  • Do not show keyword coverage to customers in this spec. Coverage is a data-quality control first, not a customer promise.

Revision Log

Rev 1 - 2026-06-26

Change: Reframe the Patient Journey surface as four gates: Search Opportunity, Google Visibility, Website Visitors, and Website Leads. Simplify each modal into measured summary fields, insights, recommended improvements, and a short source footer. Wire through existing GSC and Rybbit detail metrics where the backend already has source data.

Reason: User QA: the prior funnel labels and giant percentages made the first-stage estimate look like observed live traffic and made the dashboard read more like raw analytics than an advisor.

Updated Done criteria: The surface uses the new gate labels, arrows avoid first-stage percentage math and "biggest drop-off" copy, the action card reads like an opportunity recommendation, and modals do not present unavailable metrics as measured facts.

Rev 2 - 2026-06-26

Change: Add top approved tracked keywords and search volumes to the market opportunity metadata, then show those top 10 keyword rows in the Search Opportunity modal.

Reason: User QA: the cluster-level Top demand list is too broad for this modal; the owner wants the actual tracked keyword phrases with their numbers.

Updated Done criteria: Search Opportunity insights list reads Top tracked keywords and displays up to 10 keyword phrases with estimated monthly search-volume numbers.

Rev 3 - 2026-06-26

Change: Declutter the Search Opportunity modal by removing confidence, service clusters, primary market, and recommended improvements from that stage, leaving the estimate, tracked keyword count, and top tracked keyword-volume rows.

Reason: User QA: the extra fields invite questions that do not help the customer understand this card; this modal should act as a simple keyword tracker expansion.

Updated Done criteria: Search Opportunity detail shows only the monthly estimate, tracked keyword count, top tracked keywords with volumes, and source footer.

Rev 4 - 2026-06-26

Change: Move the Patient Journey detail scroll behavior into the modal card with a custom thin scrollbar, and change keyword value copy from bare numbers to monthly estimate labels.

Reason: User QA: the browser-edge scrollbar looks detached from the modal, and bare keyword numbers can be misread as exact people counts.

Updated Done criteria: All stage detail cards scroll inside the modal surface when needed, and Search Opportunity keyword rows clearly read as rounded monthly search-volume estimates.

Rev 5 - 2026-06-26

Change: Hide the visual scrollbar on the Patient Journey detail card while preserving scroll behavior.

Reason: User QA: the orange scrollbar is too loud and visually cuts through the rounded right edge of the modal.

Updated Done criteria: The detail card keeps its rounded right edge without a visible scrollbar, and long modal content remains scrollable.

Rev 6 - 2026-06-26

Change: Simplify the Google Visibility, Website Visitors, and Website Leads modals by removing recommendation lists, unavailable fields, and technical metrics that are likely to create confusion.

Reason: User QA: the remaining modals should explain the card, not introduce extra analytics concepts or unsupported details.

Updated Done criteria: Each non-search-opportunity modal shows only clear source-backed summary fields, optional plain-language top search detail where available, and short source copy.

Rev 7 - 2026-06-26

Change: Remove the Google Visibility summary metric cards for impressions, clicks, and average position.

Reason: User QA: the modal should not repeat or add extra metric blocks when the customer only needs the main visibility number and the top Google searches.

Updated Done criteria: Google Visibility detail shows the main header value, description, top Google searches, and source footer without the circled summary-card grid.

Rev 8 - 2026-06-26

Change: Add animated help tooltips to the Patient Journey metric cards.

Reason: User QA: customers need short explanations for metric meaning, especially visitors versus sessions and why Google clicks may not always become recorded website visitors.

Updated Done criteria: Metric cards show a circle-question icon with hover and keyboard-focus tooltip copy for each displayed summary value.

Rev 9 - 2026-06-26

Change: Remove the standalone Growth Opportunity action card beneath the growth gates and tighten the Patient Journey page bottom padding.

Reason: User QA: the section adds noise after the pipeline and leaves too much empty scroll space at the bottom of the page.

Updated Done criteria: The Patient Journey screen ends after the growth-gates/context section without the Growth Opportunity card and without the large bottom overscroll area.

Rev 10 - 2026-06-26

Change: Add an accessible circle-question tooltip above the Search Opportunity to Google Visibility arrow.

Reason: User QA: users may expect the left card to be higher than the right card, but these two cards are different measurements from different sources.

Updated Done criteria: The first arrow explains that Search Opportunity is a tracked-keyword estimate while Google Visibility is Search Console impressions across more queries and repeated appearances.

Rev 11 - 2026-06-26

Change: Shorten the first-arrow help copy, move the tooltip above the icon, and increase the pipeline row's top room so the tooltip is not clipped.

Reason: User QA: the tooltip was too long and visually cut off when it opened downward inside the horizontal pipeline row.

Updated Done criteria: The first-arrow tooltip opens upward, uses concise copy, and fits cleanly within the visible pipeline area.

Rev 12 - 2026-06-28

Change: Further compact the first-arrow tooltip and add more top-safe spacing to the growth-gates row.

Reason: User QA: the Rev 11 tooltip was still clipped at the top edge in the visible dashboard.

Updated Done criteria: The tooltip is short enough and positioned with enough row padding that it opens without clipping.
