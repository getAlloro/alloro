Why

Plan 0 of the dashboard-revamp feedback round. The owner's 24-item review (PDF 6-12) lands three cross-cutting requirements that every later plan depends on: #22 one time-representation standard across all boards, #23 the data-validity ship-blocker (numbers "change by screen" because timeframes are inconsistent and unlabeled), and #24 add back light one-line insight cues. This plan builds the shared primitives so Plans 1–5 consume one source of truth instead of each reinventing labels/toggles/cues. Nothing convincing ships until these exist.

What

Three presentational/util primitives — no surface is rewired here (adoption happens in Plans 1–5). Done when:

  • A timeframe util exports the canonical labels per the locked standard: durations spelled out ("3 Months", "28 Days", "30 Days") and to-date abbreviated (MTD, QTD, YTD), plus formatDataMonth(monthKey) → "April 2026" (the named-month labeler, dual-format like pmsPeriod.parseYM).
  • A shared PeriodToggle component: terracotta active pill, muted inactive (the locked "terracotta active-pill" decision), label-agnostic (options: {key,label}[]), accessible.
  • A shared InsightCue component: one line, fed a trend direction (up | down | flat) + optional action text, renders the locked copy pattern — up = "This is strong", down = "Spend time here" + the one action — in the shared status tones.
  • tsc + eslint clean. No visual change to any existing page in this plan (primitives are added, not yet wired).

Context

Locked decisions feeding this plan

  • Time format: spelled durations + MTD/QTD/YTD (one scheme, all boards).
  • Period anchor: latest uploaded month, named explicitly — never a bare "this month".
  • Toggles: terracotta active pill, muted inactive (preserves "terracotta = attention", still pops).
  • Insights: one-line cues only (no AI blurb yet).

Relevant Files

  • frontend/src/components/PMS/dashboard/pmsPeriod.ts — already exports parseYM (dual "YYYY-MM" / "Apr 2026" parsing). The new formatDataMonth reuses this rather than re-implementing.
  • frontend/src/utils/ — shared util home (existing: pmDateFormat.ts etc., but those are PM-module-specific; this is a general dashboard util). New timeframe.ts here.
  • frontend/src/components/dashboard/ — home for the new PeriodToggle.tsx + InsightCue.tsx (alongside the existing shared ActionBanner.tsx / StatBox.tsx from the consistency pass).
  • frontend/src/components/dashboard/focus/statusRules.ts — TONE_COLOR (positive/warn/critical/neutral) is the tone source InsightCue consumes.
  • Existing ad-hoc period toggles the shared one will eventually replace (NOT in this plan): PmsHubSurface PeriodToggle (Plan 2), the dormant RankingsHubSurface PeriodToggle (Plan 3 removes it).

Patterns To Follow

  • Shared dashboard component idiom: typed props exported, presentational, tokens only (matches ActionBanner/StatBox).
  • Tones via TONE_COLOR import — never raw hex.
  • Terracotta = alloro-orange token.

Reference files: ActionBanner.tsx + StatBox.tsx (component idiom), RankingsDashboardViewTabs.tsx (pill-toggle markup to generalize), pmsPeriod.ts (parseYM).

Constraints

Must

  • Primitives only — do NOT rewire any existing surface in this plan.
  • Label-agnostic PeriodToggle (each consuming plan sets its own option labels).
  • Tokens only; terracotta-active per the locked decision.
  • formatDataMonth reuses parseYM (no third month parser).
  • Ship together with Plan 1 (Practice Hub) so the primitives have a live consumer on first commit.

Must Not

  • No backend changes.
  • No AI/agent-fed insight content (cues are client-derived from trend direction).
  • No change to the existing toggles' behavior yet (Plans 2/3 own that).
  • No new dependencies.

Out Of Scope

  • Wiring the primitives into surfaces (Plans 1–5).
  • The backend month-key util already exists (src/utils/pms/monthKey.ts) — this is the frontend counterpart for display only.

Risk

Level: 2 — low individual risk, but these primitives are consumed app-wide, so their API shape needs to be right the first time.

Risks Identified

  • R1 — API churn (Level 2). If PeriodToggle/InsightCue props are wrong, every later plan eats a refactor. Mitigation: keep them minimal + label-agnostic; model on the proven StatBox/ActionBanner shapes.
  • R2 — Unconsumed exports (Level 1). PeriodToggle's first consumer is Plan 2; if Plan 0 commits alone it's an exported-but-unused component (not a noUnusedLocals error — exports are fine). Mitigation: ship Plan 0 + Plan 1 together; InsightCue + timeframe get consumed immediately by Practice Hub.

Blast radius: additive only — three new files + (optionally) re-exporting from an index. No existing import changes, so nothing can regress in this plan.

Pushback

  • "Month" in the Referrals toggle is neither a spelled duration nor a to-date measure — it's a single named month. The standard (#22) doesn't cover it. Resolution deferred to Plan 2: the toggle pill stays "Month" but the tile/label shows the named month via formatDataMonth. Foundation stays label-agnostic so Plan 2 can make that call without touching the primitive.
