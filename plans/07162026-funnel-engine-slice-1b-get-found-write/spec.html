Slice 1b — Get-Found Write Path

Write an owner-approved schema.org JSON-LD array to a hosted page's seo_data.schema_json through the edit-and-publish rail that already ships — gated by an honesty check that no rank, placement, or visibility claim can pass, and by a human approval that already exists.

Why

Slice 1a can tell a hosted practice page that its structured data is incomplete, but it cannot fix it. 1a is read-only by design: it analyses and emits an advisory, and nothing it finds ever reaches the page. Without a write path the get-found lever produces a report nobody acts on.

1b closes that half: it turns an owner-approved schema into a real edit on a real page, and it does so through the rail that already exists — a human approves each recommendation, publish auto-follows on approval, and the change lands as one reversible page version. No new autonomy is introduced.

The lever is directional, not bedrock: complete structured data improves crawl eligibility, and Alloro must never let that harden into a placement promise. That is why the honesty gate is not a nice-to-have here — it is the constraint that lets the write path exist at all (Value #6).

What — in scope

Reuse first. 1b adds one handler and one target type. It builds no new rail:

  • Edit-and-publish rail — service.ai-command-execute.ts already executes only approved recommendations, pins a draft per batch, and auto-publishes. 1b registers a new target_type in it; it does not fork it.
  • Version-reversibility — rides the same mechanics as the shipping page_section handler. Publishing retires the prior row (with its prior seo_data) as restorable history; restoreVersionQuery already carries seo_data back. No new rollback code.
  • Honesty gate — consumes the neutral GeneratedCopySafetyService that Slice 1a extracted to src/services/content-safety/. 1b adds the schema-specific selection of what to scan, not a second pattern set.

The write handler

  • New page_seo_schema target type + execute handler writing seo_data.schema_json via PageModel.updateSeoDataById. It is the only one of the 13 target types that touches seo_data; the rest edit HTML or structure.
  • Strict shape validation §5.2 — schema_json must be a non-empty array in which every member is a non-null, non-array object. The whole codebase reads it as an array of JSON-LD objects and dereferences each entry; a bare object, an empty array, or a scalar/null member fails the recommendation and writes nothing.
  • Honesty gate before the write — every claim-bearing free-text value is scanned; any rank/placement/visibility/will-rank/outcome claim fails the recommendation and writes nothing.
  • No-clobber merge — the write merges into whatever seo_data the draft already held.
  • Live verification — after publish, the approved array is confirmed present in the published page's seo_data.schema_json; anything short of proof-of-presence fails the recommendation rather than reporting success.
  • Batch honesty §3.2 — a batch whose page failed to publish terminates failed, and its recommendations are downgraded executed→failed, so stats never claim a change is live while it sits on a draft.

What — explicitly NOT in 1b

  • The producer. Nothing in the application creates a page_seo_schema row. 1b ships the execute half plus the row contract; queueSeoSchemaRecommendation has no production caller, so the type is inert until a producer lands. This is stated in the module header and recorded as a failing item (T10) rather than faked.
  • The live dogfood. Running one real hosted page through the rail needs a live page and a human approval — a runtime step, not a code step. Deferred with the producer.
  • Any GBP write. Slice 1's GBP touch stays read-only.
  • Approval-queue UI. 1b adds a summary icon for the new type only; no new admin surface, so no alloro-docs parity item is triggered (see Constraints).

Constraints

Honesty (Value #6) — the gate excludes on the VALUE, never on the key. A key name is caller-supplied input §5.2, so a gate that trusts one lets any claim ride inside it: {"url": "<claim sentence>"} is claim text, not a URL. A value is skipped only when it is actually structural for the key it sits under — a whole-value URL/mailto:/tel:/urn: under any key, a type token for @type/@context, a fragment or blank-node ref for @id, a phone shape for telephone/faxNumber, an addr-spec for email. Unknown key ⇒ scanned. Known key with a non-structural value ⇒ scanned. Both directions fail closed.

The identity carve-out is value-shaped too. A practice genuinely named "Pain-Free Dental Studio" is stating its name, not making a claim, so a medical/outcome-only match does not block it. That softening applies only to a value that is genuinely name-shaped (no first/second-person pronoun, at most six words) — otherwise the same defect returns one level down and name: "We cure gum disease permanently" publishes. Prose wearing a name's key is judged as prose. The rank/placement/visibility/will-rank families are never softened, so name: "Rank #1 Dental Implants" still blocks.

  • Owner-approval preserved: rows are created pending (table default — the builder sets no status); a human approves each before executeBatch runs it; publish auto-follows on approval (existing behavior). No new autonomy.
  • Schema truthfulness: only claims the practice genuinely holds. The handler re-checks owner-approved input through the gate before writing — approval is not trust.
  • Reversibility without new code: the write must land on the batch's pinned draft so publish retires the prior version as restorable history. A direct write to the published row would break this.
  • Strictness asymmetry is deliberate: HTML edits verify loosely (LLM-authored, fuzzily comparable — a false "executed" beats a false "failed"); the schema verifies strictly (exact JSON, presence is decidable — a false "failed" costs one idempotent re-run, a false "executed" would claim structured data is live when it is not).
  • Docs parity: 1b adds no admin control, label, or page copy — only a summary icon for the new recommendation type on an existing surface. No alloro-docs update is required; the parity item arrives with the approval UI that the deferred producer needs.
  • Public repo: probe/claim strings live in test files only.

Risk

  • A claim reaching a live page is the only unrecoverable-reputation risk here. Mitigation: the gate scans by default and fails closed on both axes (unknown key, non-structural value); regressions cover every excluded key, not only the ones a reviewer happened to name. Residual: the gate is pattern-based, so a novel phrasing outside the pattern set still passes — the human approval remains the second line, and the patterns are shared with GBP so improvements compound.
  • Over-blocking honest schema would make the write path useless. Mitigation: whole-value URLs stay structural under any key (a practice's own painfreedental.com link must not trip pain-free), the identity carve-out protects real names, and a full honest LocalBusiness fixture is a standing no-false-positive guard.
  • Inert type read as shipped. The contract can be mistaken for a live capability. Mitigation: the module header, T10, and this spec all state that no producer exists.
  • Stacked-PR drift. 1b depends on #158's neutral safety service; #158's head moves. Mitigation: 1b imports the neutral service by its stable path and does not vendor a copy.
  • Malformed schema crashing a consumer. Mitigation: every member is validated as a JSON-LD object before the write.

Tasks

  1. New page_seo_schema target type + execute handler — write seo_data.schema_json via PageModel.updateSeoDataById onto the batch's pinned draft; merge, never clobber; own file for §2.4.
  2. Strict shape validation §5.2 — non-empty array, every member a non-null non-array object; otherwise fail the recommendation and write nothing.
  3. Honesty gate on the VALUE axis — scan every claim-bearing string by default; skip only values that are actually structural for their key; bound the identity carve-out to name-shaped values. (Rev 1 — round 3)
  4. Row contract — buildSeoSchemaRecommendationRow / queueSeoSchemaRecommendation produce a PENDING row carrying the schema array. Contract only; no production caller.
  5. Batch-failure honesty §3.2 — never complete a batch after a failed publish; downgrade its recommendations executed→failed.
  6. Post-publish verification — confirm the approved array is live in the published seo_data.schema_json; fail on absence, on no published row, and on an unreadable read.
  7. Acceptance artifacts §20.5 — this spec, test.html, test-results.json.

Done

  • The page_seo_schema handler writes seo_data.schema_json as an array, merged into existing seo_data, on the batch's pinned draft — version-reversible with no new rollback code.
  • schema_json that is not a non-empty array of JSON-LD objects fails the recommendation and writes nothing.
  • The honesty gate blocks a rank/placement/visibility/outcome claim under any key — including every structural key (@context, @id, @type, identifier, url, sameAs, image, logo, telephone, faxNumber, email) and inside an array member — and writes nothing.
  • A full honest LocalBusiness schema still publishes; a real practice name ("Pain-Free Dental Studio") publishes; name: "Rank #1 Dental Implants" and name: "We cure gum disease permanently" both block. All four directions are test-proven.
  • Rows are created pending; a human approves each before execution; no new autonomy.
  • A batch whose page did not publish terminates failed and downgrades its recommendations; a healthy batch still completes.
  • An approved schema that did not reach the live page is failed, not reported as a success.
  • npx tsc --noEmit, npx vitest run, and npm run check:conventions -- --strict all exit 0, with 0 clear structural violations.
  • Deferred with the producer: a production path that creates a page_seo_schema recommendation (T10), and the live dogfood on one real hosted page — both need runtime pieces outside this PR.
