Why

Alloro-owned transactional emails currently use multiple unrelated layouts. Invitations use a blue CTA, auth and billing messages are raw HTML, and the transport test email has no shared shell. The support emails are the closest branded analog, but the typography still does not match Alloro's serif-display and sans-body system.

The result is a fragmented customer experience across account creation, verification, password recovery, billing, notifications, and internal transport checks.

What

Move every in-scope Alloro transactional email onto one code-owned HTML system built from src/emails/templates/base.ts. Headings use Spectral, Georgia, "Times New Roman", serif; body copy uses "Plus Jakarta Sans", "Segoe UI", Arial, sans-serif; verification codes and temporary passwords use a monospace stack. The layout keeps the existing Alloro navy, orange, white, and light-gray palette, with the orange CTA used consistently.

Done means all listed messages render the shared logo, canvas, content card, typography, spacing, CTA, and footer; dynamic content is HTML-escaped; behavioral delivery contracts are unchanged; and automated rendering assertions cover the variants.

Open questions: None. The user selected serif headings and sans-serif body copy.

Design Contract

Typography

  • Headings: Spectral, Georgia, "Times New Roman", serif.
  • Body: "Plus Jakarta Sans", "Segoe UI", Arial, sans-serif.
  • Codes/passwords: "SFMono-Regular", Consolas, "Liberation Mono", monospace.
  • No external font download is required; fallbacks must look intentional.

Layout and color

  • 600px responsive email canvas with logo, white content card, and restrained footer.
  • Navy headings, dark-gray body, light-gray page background, orange CTA.
  • Verification and password values render in a dedicated high-contrast code card.
  • No blue CTA or unrelated one-off color tokens in in-scope messages.

Constraints

Must

  • Keep HTML email-safe: presentation tables, inline critical styles, conservative CSS, responsive spacing.
  • Keep each existing category, subject, recipient, code/expiry rule, URL destination, logging path, and best-effort failure behavior.
  • Escape organization names, roles, user names, locations, metadata keys/values, recipients shown in HTML, and other dynamic copy.
  • Use synthetic values in tests and acceptance previews.

Must not

  • Add dependencies, a rich-text editor, a template table, migrations, or a second rendering system.
  • Change sendEmail, Mailgun/n8n selection, non-production interception, or allowLiveSend behavior.
  • Mix in copy rewrites unrelated to clarity or accessibility.
  • Modify the source checkout's unrelated dirty files.

Docs impact: None. This changes outbound HTML only; it does not change dashboard controls, labels, permissions, navigation, or user guidance in Alloro Docs.

Risk

Level: 2 — Concern

Risks and mitigations

  • Shared-shell blast radius: typography changes also affect existing base-wrapped support, checkup, inquiry, admin, and notification messages. Mitigation: change only shared typography/layout tokens, render representative existing messages, and avoid copy or delivery changes.
  • Email-client variance: web fonts and head CSS are not uniformly supported. Mitigation: use inline critical styles, table layout, and deliberate Georgia/Segoe UI/Arial fallbacks; do not depend on remote font loading.
  • HTML injection: several current templates interpolate organization, role, location, recipient, and metadata values directly. Mitigation: central escaping plus hostile-input tests; this is a must-fix §5.2 requirement.
  • Auth delivery regression: template cleanup could accidentally alter a security-sensitive send path. Mitigation: pure builders only; preserve subjects, recipients, codes, expiry windows, and interception behavior exactly.
  • Concurrent local work: the source checkout contains uncommitted project-board email work, and another email-manager worktree overlaps these files. Mitigation: execute only in this clean worktree; rebase and rerun the full email suite before integration.

Blast radius: every consumer of wrapInBaseTemplate, createButton, and shared brand tokens, including support, checkup, inquiry, admin, and user-notification emails. Direct call-site wiring also affects auth registration/resend/reset, settings invitations, admin temporary passwords, billing services, location lifecycle services, and the admin transport test endpoint.

Pushback

  • Do not revive the database-backed email-manager approach for this visual consistency fix. It adds schema, runtime editing, permissions, and fallback behavior to a problem that the existing code-owned template layer already solves.
  • Do not force client-site form/newsletter emails into the Alloro product shell. Those messages represent the dental practice, not the Alloro application.

Test Plan

  • Semantic shell: doctype, logo, shared page/card/footer, navy/orange tokens, serif heading stack, sans body stack, and no legacy blue CTA.
  • Variants: invite initial/resend; verify initial/resend; password reset; temporary password; billing add/remove and four lifecycle states; five notification types; system transport test.

  • Regression: representative support and checkup emails still render correctly after the shared-base change.
