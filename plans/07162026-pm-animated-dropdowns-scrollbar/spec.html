Why

The PM Backlog and Assignee views still render browser-native HTML select menus, which break the visual language of the animated dark PM interface. The notification card also exposes a thick native scrollbar that dominates the compact list instead of supporting it.

What

Replace all five native select controls on the pictured Backlog and Assignee views with the PM animated dropdown, harden that shared dropdown for accessible keyboard use, and add a thin PM-scoped scrollbar to notification and dropdown lists. Existing filter values, task queries, navigation, and API behavior remain unchanged.

Scope boundary: “all dropdowns” means the five controls on the two pictured views. The eight native selects in secondary PM modals and task utilities are recorded as adjacent work and are not part of this plan.

Constraints

Must

  • Remove native <select> elements from the pictured Backlog and Assignee view controls.
  • Preserve exact filter values, default values, URL behavior, and callback types.
  • Support string, number, and null values in the shared PM dropdown without coercion bugs.
  • Expose accessible labels, aria-haspopup, aria-expanded, listbox/option roles, selected state, visible focus, and keyboard navigation.
  • Support Escape, Enter, Arrow Up, Arrow Down, Home, End, outside click, disabled trigger state, and skipped disabled options.
  • Respect prefers-reduced-motion while retaining the normal open/close animation.
  • Support compact and regular trigger sizes and explicit top/bottom placement so menus remain usable near scroll boundaries.
  • Use PM theme tokens in dark and light mode.
  • Use a PM-scoped thin scrollbar with a transparent track, muted thumb, and orange hover state in Firefox and WebKit browsers.
  • Preserve the existing Task Detail dropdown behavior after the shared component changes.

Must Not

  • Add a dropdown library or another state system.
  • Change PM API calls, backend routes, database state, filter rules, or task behavior.
  • Modify the global .scrollbar-thin class; that would affect unrelated surfaces.
  • Introduce a second PM animated dropdown abstraction.
  • Add inline hardcoded colors for the new dropdown or scrollbar behavior.
  • Refactor unrelated notification loading/error behavior during this visual task.
  • Claim every PM native select has been removed; eight secondary-flow controls remain outside this plan.

Out Of Scope

  • Secondary PM modals and per-task utility selectors.
  • Notification content, polling, mark-read, delete, click-through, or API changes.
  • Backend, database, migrations, permissions, and deployment configuration.
  • A new Alloro Docs Project Management page.

Risk

Level: 2 - Concern

Risks Identified

  • Replacing native controls can regress accessibility and keyboard behavior. Mitigation: harden the shared dropdown first, mirror the accessible OS listbox analog, and cover the interaction contract with focused tests.
  • Absolute dropdown menus can be clipped by page or card overflow. Mitigation: add explicit top/bottom placement, keep menu widths configurable, and verify every pictured control at desktop and narrow viewport sizes.
  • Changing the shared dropdown can break Task Detail, its only current consumer. Mitigation: retain generic value behavior and add a Task Detail regression check to automated and manual verification.
  • Broad CSS scrollbar changes could alter unrelated screens. Mitigation: add a new .pm-scrollbar class and apply it only to the PM dropdown option list and notification list.
  • BacklogTabView.tsx is near the single-purpose target and AssigneeTabView.tsx already contains several child controls. Mitigation: extract only the touched filter/picker components into focused sibling files; do not add more embedded components.
  • The phrase “all dropdowns” could be overclaimed while secondary PM selectors remain native. Mitigation: scope acceptance to the two pictured views and record the remaining eight controls as adjacent work.

Blast radius: Backlog filtering, Assignee user/project/priority filtering, Task Detail assignment dropdown, PM dark/light theme tokens, and notification list presentation. There is no API, backend, database, auth, or tenant-data blast radius.

Pushback

  • A module-wide replacement is not part of this plan. It would pull in four additional PM flows, including a 733-line AI synthesis modal, and turn a focused polish pass into a structural refactor. Recommended path: finish and verify the pictured views first, then open a separate plan for the secondary-flow sweep if it remains valuable.
  • A custom dropdown without full keyboard and ARIA behavior is worse than the native select it replaces. Accessibility hardening is a prerequisite, not optional polish.
