Why

Only 8 of 67 Constitution Articles are mechanically enforced, and all 8 are cheap grep/wc heuristics in scripts/check-conventions.sh. Worse: the backend has no ESLint at all (no config, no lint script, eslint not installed) — so a ~130-line bash script is single-handedly the entire backend mechanized gate. A large tier of Articles is mechanizable with the right tools and simply hasn't been wired up. Raising coverage stops violations from growing silently and narrows what the §-citing review agent must catch by hand to genuinely semantic Articles.

What

Two tiers of new enforcement, each rule annotated with the Constitution Article (§N.M) it enforces so tool output cites the document:

  • Tier A — quick greps into check-conventions.sh: §17.4 dangerouslySetInnerHTML, §17.5 scattered token reads, §10.2 knex.raw, §15.4 stray state libraries, §17.3 non-VITE_ env in the bundle, §5.1 basic secret patterns.
  • Tier B — a real linter layer: stand up ESLint on the backend (it has none), enable an Article-mapped ruleset on both stacks (§2.2 max-lines-per-function, §2.3 max-depth, §17.2 no-explicit-any, §4.1 no-commented-code, §4.2 no-magic-numbers, §1.x naming-convention), and add dependency-cruiser to enforce the layering Articles (§7.4, §7.3, §7.1, §14.2, §14.1) as hard import-boundary rules.

Done = both tiers run, every rule cites its §, the suite is warn/baseline (CI stays green), and the code-constitution.html viewer reflects the expanded mechanized set.

Context

  • frontend/eslint.config.js — frontend ESLint flat config already exists (Tier B tunes + extends it).
  • Backend: no ESLint config exists — Tier B creates eslint.config.js + a lint script at the repo root.

Reference analog

The backend remediation's warn-only validation pattern (plans/06142026-validation-layer) is the model for introducing enforcement without breaking the build: report, don't block, until the debt is cleared. This plan is the guardrail for [[project-frontend-remediation-plan]] — it should land first or alongside so the remediation has a ratchet to hold the line.

Constraints

Must

  • Map every rule to a specific Article and annotate it with the §N.M in config + output.
  • Start warn-only / baseline-ratchet — fail only on NEW violations, never the existing flood.
  • Keep the bash detector zero-dependency; Tier B tools are dev-deps only.

Must Not

  • Block CI on day one (the backend has never been linted — it will light up).
  • Dump eslint:recommended/typescript-eslint:recommended wholesale — enable only Article-mapped rules.
  • Auto-fix / mass-rewrite source as part of this plan — that is remediation, not mechanization.
  • Change any application logic or runtime behavior.

Out of scope

  • Fixing the violations the new rules surface (backend is already clean; frontend debt is [[project-frontend-remediation-plan]]).
  • Tier C (semantic Articles) — those remain the review agent's job via the Enforcement Protocol.

Risk

Level: 3 — touches tooling and CI across both stacks; blast radius is the dev/CI workflow and lint config, not runtime.

Risks identified

  • Never-linted backend floods on first run. Mitigation: warn-only + a committed baseline; CI fails only on new violations (ratchet), exactly like the validation warn-only rollout.
  • Noisy rules (no-magic-numbers especially) drown the signal. Mitigation: tune allowlists (-1,0,1,2), or ship those advisory; start with high-signal rules (size, depth, any, layering).
  • Generic ruleset dilutes the contract. Mitigation: enable only Article-mapped rules, each commented with its §; the lint output then literally cites the Constitution.
  • dependency-cruiser false boundaries. Mitigation: encode the 3–5 clearest layering rules first (§7.4, §7.3, §14.2), advisory, validate against the certified-clean gbp-automation domain before tightening.

Pushback

  • The single highest-value piece is dependency-cruiser for layering — it enforces §7.x/§14.x structurally, which grep can only approximate. Prioritize it over the noisier ESLint rules.
  • This plan should precede or run alongside the frontend remediation so new code can't add to the pile while old code is being cleaned.
