Why

Three trust/usability problems in the Mission Control telemetry tab, found right after the internal-filtering work (plans/07012026-telemetry-internal-filtering) shipped to dev:

  • Idle time counts as engagement. The frontend heartbeat (useAppTelemetry.ts) reports ~30s of "active" time every 30s whenever the tab is merely visible — there is no mouse/keyboard/scroll check. A dashboard left open on a second monitor for 3 hours records 3 hours of engagement. Existing recorded minutes are inflated by this (e.g. Local Rankings: 84.8 minutes over 5 views).
  • The layout wastes a screenful of white space. The Usage Trend card sits in a CSS grid row beside the very tall Surfaces & Pages card; grid items stretch to equal height, so the 256px chart card is stretched to ~2,000px of blank white, pushing Organization Usage below the fold.
  • The chart shows the wrong series and has no time grouping. All three views (aggregate, org detail, user detail) plot active users + page views daily. The owner wants view-specific series (see mapping below) and range grouping including month-to-date and monthly-aggregated 12-month/year-to-date views.

What

Four deliverables:

  1. One-off idle-data cleanup (first): snapshot app_usage_events, then delete heartbeat rows beyond a 5-minute cap per continuous same-page run — dev first with before/after verification, then prod.
  2. Layout restructure: left column = Usage Trend (content height) + Organization Usage; right column = Surfaces & Pages + User Drilldown. No stretched empty space; Organization Usage visible near the top.
  3. View-specific chart series per the mapping table below.
  4. Range picker becomes six options: 7D · 30D (default) · 90D · MTD plotted daily; 12M · YTD aggregated monthly.

This plan does not fix the heartbeat itself — that is a separate follow-up (see Out of Scope).

Cleanup SQL (T1 — executed manually, dev then prod)

Rule: within each continuous heartbeat run (same session_id + route_template, gaps > 2 minutes start a new run), keep heartbeat rows until 5 minutes (300s) of recorded time is reached; delete the rest. Only app.page_active_heartbeat rows are ever deleted — session_started and page_viewed rows are untouched, so session counts and page-view counts do not change. The rule is idempotent: re-running after deletion removes nothing further.

-- 0) SNAPSHOT (verify count matches app_usage_events before proceeding)
CREATE TABLE app_usage_events_backup_20260702 AS TABLE app_usage_events;
SELECT (SELECT count(*) FROM app_usage_events) AS live,
       (SELECT count(*) FROM app_usage_events_backup_20260702) AS backup;

-- 1) BEFORE numbers (rerun after delete for the comparison)
SELECT o.name, round(sum(e.active_seconds)/60.0, 1) AS minutes, count(*) AS rows
FROM app_usage_events e JOIN organizations o ON o.id = e.organization_id
GROUP BY o.name ORDER BY minutes DESC;

-- 2) PREVIEW the rows the cap would delete (sanity-check the count first)
WITH hb AS (
  SELECT id, session_id, COALESCE(route_template, '') AS route_key,
         occurred_at, active_seconds,
         CASE WHEN LAG(occurred_at) OVER w IS NULL
                OR occurred_at - LAG(occurred_at) OVER w > INTERVAL '2 minutes'
              THEN 1 ELSE 0 END AS is_new_run
  FROM app_usage_events
  WHERE event_name = 'app.page_active_heartbeat'
  WINDOW w AS (PARTITION BY session_id, COALESCE(route_template, '') ORDER BY occurred_at)
), runs AS (
  SELECT id, session_id, route_key, occurred_at, active_seconds,
         SUM(is_new_run) OVER (PARTITION BY session_id, route_key ORDER BY occurred_at) AS run_id
  FROM hb
), cumulative AS (
  SELECT id,
         COALESCE(SUM(active_seconds) OVER (
           PARTITION BY session_id, route_key, run_id ORDER BY occurred_at
           ROWS BETWEEN UNBOUNDED PRECEDING AND 1 PRECEDING), 0) AS seconds_before
  FROM runs
)
SELECT count(*) AS rows_to_delete FROM cumulative WHERE seconds_before >= 300;

-- 3) DELETE (same CTE chain, DELETE ... WHERE id IN (...))
-- 4) AFTER numbers: rerun step 1, compare per-org drops; spot-check known
--    inflated rows (e.g. Local Rankings 84.8m, dr.kargoli 86.6m) shrank sanely.

Exact statements are finalized and captured with their outputs during execution. Prod run only after dev per-org numbers are eyeballed and approved.

Risk

Level: 3 — Structural Risk

Risks Identified

Irreversible heuristic rewrite of production telemetry. True active time was never recorded (no interaction signals), so the 5-minute cap is an estimate, not a recovery. Deleted rows are gone from the live table.

Mitigation: full snapshot table before deletion (restorable by swap/insert), dev-first with per-org before/after verification, preview count before delete, and the cap documented here so nobody later mistakes post-cleanup numbers for measured truth.

5-minute cap is aggressive (user's explicit choice over the recommended 10). Genuine long single-page work sessions (e.g. 20 real minutes in the website editor) get truncated to 5 recorded minutes per run. Engagement will read lower than reality for legitimately heavy users.

Mitigation: accepted and documented; the snapshot preserves the originals; the low-engagement flag threshold (<5m/30d) is far below any realistic truncation impact for an active client.

Cleanup may newly flag orgs as low-engagement. Trimming inflated minutes can push borderline orgs (e.g. Artful at 1.5m) under thresholds — that is the intended, honest outcome, but the numbers Jo/Corey see will drop noticeably. Communicate the cleanup when sharing the new baseline.

Mitigation: the before/after per-org table from T1 becomes the explanation artifact.

Range/granularity contract change spans BE + FE. Six range values and a granularity field flow through the params object, three model queries, the API types, hooks, toolbar, and chart.

Mitigation: all range logic is already centralized in MissionControlTelemetryRangeUtils.ts + the three getDailyUsage methods; parseRange defaults unknown values to 30d so stale clients can't break the API.

Blast radius: every minute total in Mission Control telemetry (summary cards, org rows, user rows, surface/page lists) drops after T1 — intended and consistent, since all read the same app_usage_events rows. Session and page-view counts are unchanged by design. AppUsageOrganizationMovementModel (recent movements) does not aggregate active_seconds — unaffected. The low-engagement flag (previous plan) reads the same data — may flag more orgs post-cleanup, which is correct.

Pushback

  • Cleaning data before fixing the leak. The user chose cleanup-first, idle-fix-later. Fine — but inflation resumes immediately, so the post-cleanup baseline starts drifting the same day. The cap being idempotent makes periodic re-runs possible, but the real fix should follow quickly; a chip is filed for it.
  • 5 minutes over the recommended 10. Recorded and accepted above — flagging once here, not relitigating.
