Why

The current pilot flow opens a separate browser window from Mission Control and Users & Roles. Dave wants Pilot to become a first-class organization detail tab, with Mission Control's Pilot action deep-linking into that tab instead of spawning a popout.

What

Add a top-level Pilot tab to /admin/organizations/:id. The tab lets an admin choose an organization user and launches a same-origin, contained iframe showing the client app as that user, scaled inside the page. New pilot launches do not call window.open, do not put pilot tokens in the URL, and do not write pilot auth into parent-tab localStorage or shared sessionStorage.

Assumptions captured: place Pilot after Users & Roles; selecting a user from a Mission Control card navigates to /admin/organizations/:id?section=pilot&userId=:userId; the Pilot tab auto-starts the embedded session when a valid userId is present.

Scope Ledger

Objective

Move pilot from popout behavior into a contained organization-detail tab.

Intended User

Internal Alloro admin or super admin.

Surface

Admin Mission Control cards and /admin/organizations/:id.

System Of Record

Existing admin organization detail users plus existing /api/admin/pilot/:userId token endpoint.

Constraints

Must

  • Add pilot as a valid organization section key and visible tab.
  • Make Mission Control user selection navigate to the organization Pilot tab with the selected user encoded as userId.
  • Make Users & Roles pilot action route to the same tab, not a popout.
  • Keep the admin parent session authenticated as the admin while the iframe acts as the pilot user.
  • Keep the embedded pilot token in iframe memory only, set before client providers mount.
  • Use postMessage with exact targetOrigin and origin validation for token handoff from parent to iframe.
  • Use a non-persisted Query Client in the iframe to avoid showing parent/admin cached data.
  • Handle token refresh by updating embedded iframe memory, not browser storage.

Must Not

  • Do not put pilot JWTs in iframe URLs, query params, localStorage, shared sessionStorage, cookies, or logs.
  • Do not let iframe logout/session-expired cleanup clear parent admin auth, cookies, or query cache.
  • Do not add a new auth model or duplicate API client outside the existing api/index.ts path.
  • Do not change normal client login, legacy direct pilot URL handling, or admin auth semantics outside the embedded flow.
  • Do not touch unrelated organization tabs or refactor the dashboard while adding the pilot tab.

Out Of Scope

  • New backend impersonation model, new DB tables, audit log schema changes, or a separate pilot subdomain.
  • Replacing the client dashboard UI. The iframe should show the existing client app.
  • Changing customer-facing Alloro Docs unless final docs parity check finds an admin guide that names the old popout behavior.

Risk

Level: 3 - auth/session boundary and same-origin iframe behavior.

Risks Identified

  • Admin session contamination. A naive same-origin iframe using PilotHandler would write pilot token state into storage that the parent can read. Mitigation: embedded route stores the pilot token in iframe module memory only and sets it before providers mount.
  • Parent cached data shown in pilot frame. The normal app persists React Query cache to localStorage. Mitigation: iframe route uses a non-persisted Query Client and starts cold.
  • Iframe logout clears admin login. Existing logout/session-expired code clears localStorage, cookies, query cache, and broadcasts logout. Mitigation: embedded pilot mode branches cleanup to iframe-only memory reset and parent notification, never global auth cleanup.
  • JWT leakage. Token in URL, storage, logs, or telemetry would be a security issue. Mitigation: parent fetches token, passes by origin-checked postMessage, stores only in React state and iframe memory, and clears state on close/restart.
  • Provider order bug. If AuthProvider mounts before embedded token is set, iframe may authenticate as the admin from localStorage. Mitigation: /pilot-embed route waits for a valid token handoff before mounting the normal client providers.

Blast radius: App.tsx provider tree, api/index.ts auth resolution and refresh handling, AuthContext, SessionProvider, SessionExpiredModal, PilotBanner, telemetry pilot detection, organization detail navigation, Mission Control cards, Users & Roles pilot action, and client dashboard routes rendered inside the iframe.

Pushback

  • A simple <iframe src="/?pilot_token=..."> does not belong here. It reuses the legacy storage-based pilot path and risks flipping the parent admin tab into pilot mode. The recommended path is an embedded pilot runtime with memory-only auth.
  • Using a separate subdomain would isolate storage more naturally, but it adds deployment and CORS surface. For this plan, same-origin is acceptable only with the memory-token and non-persisted-query mitigations above.
