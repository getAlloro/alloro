Why

Third of four dashboard simplifications (Practice Hub → Referrals Hub → Local Rankings → Reviews & Posts). The current /rankings Overview is dense: a narrative hero, a 0–100 Health gauge, a Visibility score, an 8-factor breakdown, drivers/gaps modals, and a Next Moves grid. Both scores "kept drawing questions" and are removed entirely. The redesign anchors on the thing owners actually ask about — where do I rank, against whom, on the map — plus three vitals and one action.

What

Build a slim RankingsHubSurface rendered by RankingsDashboard's Overview branch in place of PerformanceDashboard. Done when:

  • Header reads "LOCAL RANKINGS" with a MONTH/QTR/YTD toggle rendered disabled (tooltip "Not enough ranking history yet"), with the history-trend wiring built behind it ready to enable.
  • Hero card: "YOU RANK #N of M nearby" + the standard searchQuery shown ("measured the same way every time") + a leaflet competitor map — rank-numbered dot markers, highlighted YOU marker, auto-fit bounds.
  • Three stat boxes: REVIEWS (total · rating★ · +N/30d), LAST POST (Nd · overdue/current), RATING VS MKT (you / market avg).
  • "1 ACTION" banner shows llmAnalysis.top_recommendations[0].
  • Bottom strip: comparison insight ("You lead all 10 tracked competitors") → existing Manage Competitors action.
  • Visibility Score and Practice Health are gone: HealthGauge, DriversPanel, FactorBreakdown, ScoreCardCtas, both detail modals, the MeaningHero narrative, and the NextMoves grid no longer render.
  • The ranking agent emits exactly ONE top_recommendations entry (prompt + guardrails backfill).
  • In-flight banner, competitor-onboarding banner, refresh, and the Engage tab keep working. Wizard tour not broken. tsc + lint clean.

Patterns To Follow

  • Surface-swap pattern from the previous two hubs: slim surface file, orchestrator keeps plumbing, retired internals stay on disk.
  • Header + disabled segmented toggle: mirror PmsHubSurface's PeriodToggle (1080px container, same typography).
  • 1-ACTION banner: same warm strip used on both other hubs (#FAF1EC/#EFDED4, eyebrow #B3503E).
  • Map: copy the onboarding leaflet config verbatim (divIcon numbered markers — avoids leaflet's default-icon asset trap; fitBounds helper).

Reference files: PmsHubSurface.tsx (surface shape/toggle/banner) and LocationCompetitorOnboarding.tsx (map).

Constraints

Must

  • Reuse react-leaflet (already a dependency); no new packages.
  • Filter competitors with null lat/lng; render a graceful empty state when no coordinates exist.
  • Preserve hero error states (not_in_top_20, bias_unavailable, api_error) — searchPosition can be null.
  • Keep in-flight banner, onboarding banner, refresh, Engage tab fully functional.
  • Build the period toggle's history wiring (fetchRankingHistory → rank-at-period-start delta) but ship the control disabled with a tooltip; one documented enable point.
  • Reuse the existing Manage Competitors handler/destination from SearchPositionSection.

Must Not

  • No new dependencies. No rewrite of RankingsDashboard beyond the Overview render swap (+ minimal lifted derivations).
  • Do not delete retired internals (HealthGauge, DriversPanel, FactorBreakdown, modals, NextMoves) — stop rendering only.
  • Do not remove the Engage tab (sole GBP-automation access until /gbp-manager exists).
  • Do not render rankScore / practiceHealth anywhere on the new surface.

Out Of Scope

  • Enabling the period toggle (needs accumulated ranking history).
  • The Reviews & Posts page (/gbp-manager) and moving the Engage tab there.
  • Backend changes to ranking storage/history; competitor onboarding flow changes.

Risk

Level: 3 (Structural — surgery in a ~2000-line monolith + an agent-output change with task-creation knock-on).

Risks Identified

  • R1 — No clean surface seam. Unlike PMS, the rankings internals live inside the monolith. Mitigation: new RankingsHubSurface file; orchestrator edit limited to the Overview render branch; retired internals stay in-file unrendered.
  • R2 — Leaflet in a dashboard card. Tile loads, null coords, icon assets. Mitigation: copy the proven onboarding divIcon/fitBounds setup; filter null coords; empty state.
  • R3 — Agent trim knock-on (accepted). 1 recommendation → 1 category="USER" ranking task per run (per service.ranking-llm.ts:450, recommendations are the sole writer of those tasks) and a 1-item ranking_recommendations pool for the Summary agent.
  • R4 — Dormant toggle wiring. Shipped-but-disabled capability can drift. Mitigation: small pure helper + hook, single documented enable constant.
  • R5 — Rank is the only anchor post-scores. Null/20+ positions must render the existing fallback copy, not a broken hero.

Blast radius: RankingsDashboard.tsx (render branch), new surface + map + helper files, wizardConfig.ts, ranking prompt + guardrails (backend). Consumers of top_recommendations: ranking task creator (1 task now), Summary agent inputs (service.ranking-recommendations.ts, service.agent-input-builder.ts — additive-safe, arrays just get shorter), wizard demo data untouched.

Pushback

  • The Engage tab contradicts the mockup's chrome-free header but is kept deliberately — removing it orphans GBP automation. Revisit when Reviews & Posts lands.
  • Shipping a visible-but-disabled toggle is honest about roadmap but is still dead UI; if history accumulates slowly, consider hiding it instead. Flagged, owner chose disabled-visible.
