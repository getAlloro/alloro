Why

Across ~24h and four sessions, one pattern held: the work gets done, but the knowledge does not transfer, so every session re-derives state from scratch and the only thing reliably catching errors is a human (Corey). The registry's own metric names it — on 2026-07-16, ~20 fires were caught 4 by a gate, 16 by a human/adversary (~/.claude/skills/LESSONS.md header). Every session has diagnosed "the coordination state must be self-maintaining" and then written another advisory doc instead of a gate — the fix for "nothing lands" had not itself landed until this session shipped the write-side warn-hook. The deepest driver: the recurring defect is a claim that does not hold (a guard that guards nothing, a source that fabricates, a green that is wrong while rank is 10) — and a wrong green is invisible by construction, so it survives to review or to a customer. Corey is blind during deep coding and Dave is blind to intent, so the human net has a hole exactly where code errors happen.

What

A three-layer system whose single success metric is moving the caught by column from human to gate. Nothing in it is a new advisory rule — every element is generation, a hook, a gate, or a step (DOCTRINE build-pillar #1: the rule-to-remember is the weakest, accreting form).

  • Layer 1 — Single source, generated (the map). One front door: the NOW block atop the auto-loaded MEMORY.md → generated PR-LOG.md (build/PR state, from git) + staked build-checklist.md (seam map) + Dave's global CLAUDE.md (process). Done this session. Honest limit: the seam/connection map is not fully generatable (it needs agent analysis + touches an external repo) — so it is staked-with-a-commit-stamp + a freshness flag (the build-checklist poka-yoke), never claimed "generated."
  • Layer 2 — Enforced read + write (hooks). Read: the SessionStart hook that already injects canon-grounding.md gains a freshness check (warn if a NOW-named live doc is older than NOW's last-change date, or PR-LOG.md is stale vs git HEAD). Write: the warn-only Stop hook shipped this session becomes a hard gate — block session-end if the session shipped/changed state but did not re-point NOW + run alloro-harvest. The hard gate ships only after a test proves it can never wrongly block.
  • Layer 3 — Autonomous verification (the human-replacement net). Because neither human catches code errors, the machine must, before a PR is "ready for Dave" and before promotion: a spec-code parity gate (status matches branch/PR/test — the #185/#186 class); an honesty-scan gate (no fabricated/placeholder numbers reach a customer surface); a no-"fixed"-without-a-mutation-receipt gate (revert the fix, prove the test fails — the vacuous-green class, rows 25/28); class-guard tests in CI (a test that fails on the whole class, not the named instance — row 28, PR #174); and sandbox acceptance on real data (taste in the test kitchen so "surfaced in use" becomes "surfaced in the sandbox"). Plausible-but-wrong is caught by an independent-framing adversary (rows 27/28 method: raw artifacts with no verdicts attached, framing varied per agent, ≥1 agent tasked to defend the killed option, never cite agreement among identically-briefed agents).

The compounding loop: alloro-aar fires on every correction → one LESSONS.md (path disambiguated so no twin) → a per-seat Read LESSONS.md pre-flight auto-load (the standing OPEN fix, rows 1/22/26) so the next run inherits the last correction; alloro-harvest at session end, forced by the Stop hook, routes learnings to homes as committed changes. Done = the caught by ratio trends toward gates; if it does not, the system is not working and this spec failed.

  • One source of each kind — one registry, one front door. Disambiguate alloro-aar's "LESSONS.md in this repo" to an exact path so no future twin.

  • Do not author the build from a deep-context tail — a fresh, short-context session builds; Corey stakes.

Out of scope

Risk

Level: 3 — touches every session's config; a wrong blocking hook could lock sessions out.

Risks + mitigations

  • The meta-process trap. A prior draft this session was a fancier repeat of the 6/15 approved-never-built fix; an adversary killed it. Mitigation: every element must be a gate/generation/step or it is cut; the plan explicitly deletes duplicate advisory rules rather than adding more.
  • Blocking hook locks sessions. Mitigation: warn-only first (shipped); hard gate only after a test proves it can't wrongly block.
  • Over-engineering — all gates at once. Mitigation: ship the 2–3 highest-leverage gates first (parity, honesty-scan, mutation-receipt), each proven, not the whole set.
  • Un-generatable seam map claimed as generated. Mitigation: commit-stamp + freshness flag, never "generated."

Blast radius: every Alloro session (hooks in settings.local.json); CI on every PR (PR #174); the seat skills (per-seat pre-flight). All read-only/warn until each hard gate is individually tested + staked.

Pushback

  • The author is the deep-context tail this system exists to protect against — so this spec is a draft to be refuted by a Fable adversary, not a self-approved plan. If the adversary finds it is (a) another advisory layer, (b) a repeat of a landed-and-failed approach, or (c) buildable more simply, cut to the simpler thing.
