Why

Alloro needs first-party product telemetry to understand which organizations and users are actually using the dashboard, how often they return, which pages hold attention, and which paid accounts are going quiet. Third-party replay tools can help with qualitative behavior, but Mission Control needs operational facts owned by Alloro.

What

Create a dedicated app-usage event layer and a new Telemetry tab inside /admin/mission-control. V1 tracks authenticated app sessions, route/page views, focused active-time heartbeats, and aggregated usage by organization, user, page, feature surface, and date range.

The tab must show overall usage first, then narrow down to per-organization and per-user usage without mixing product telemetry into the existing revenue/client grid view.

Patterns To Follow

  • Backend flow must remain Routes -> Controllers -> Services -> Models; all Knex queries live in model files.
  • API responses must use { success, data, error }, matching Mission Control controller responses.
  • Frontend server state must use typed API functions plus React Query hooks and QUERY_KEYS, not raw string keys or component-level fetch calls.
  • UI should reuse the current Mission Control visual language: compact stat cards, white cards on the light admin canvas, navy text, orange/teal accents, tabular numbers, and Recharts for trends.

Reference files: MissionControlService.ts for aggregate service shape, MissionControlSummary.tsx for KPI cards, MissionControlRevenueTrend.tsx for chart behavior, and LeadgenTrackingController.ts only for strict event-name validation concepts.

Constraints

Must

  • Create a dedicated app_usage_events table instead of expanding behavioral_events.
  • Derive user_id, organization_id, and user_role server-side from JWT/RBAC context.
  • Use a strict allowlist for event names, route templates, feature surfaces, and property keys.
  • Record active time only while the browser tab is visible/focused; clamp heartbeat durations server-side.
  • Separate normal user sessions from admin pilot sessions so internal activity does not distort client usage.
  • Support aggregate, per-organization, and per-user drilldown in Mission Control.
  • Keep the Telemetry tab visually consistent with the existing Mission Control page.
  • Index every admin query path used by the telemetry dashboard.

Must Not

  • Do not track keystrokes, hover/mouse movement, raw form values, PMS row contents, patient names, emails in event properties, or full URLs with sensitive query strings.
  • Do not ingest arbitrary frontend-provided event names or arbitrary JSON blobs.
  • Do not build a replay/heatmap tool or a PostHog clone in V1.
  • Do not mix product telemetry cards into the existing revenue overview grid.
  • Do not introduce a new analytics dependency unless explicitly approved later.

Out Of Scope

  • Session replay, heatmaps, dead/rage click detection, screen recordings, and external Clarity/PostHog installation.
  • Historical backfill before deployment. V1 starts collecting from deployment forward.
  • Full funnel-builder UI. V1 may show basic route/surface adoption, not arbitrary funnel construction.
  • Detailed workflow event instrumentation across every feature surface. The event contract should support it, but broad action tracking is a follow-up.
  • Client-facing documentation changes unless we choose to document internal admin tooling. A docs parity check is still required at finalization.

Data Contract

V1 Event Catalog

  • app.session_started - first event for a generated browser session id.
  • app.page_viewed - route template, feature surface, and page label when a route becomes active.
  • app.page_active_heartbeat - active seconds for the current route while the tab is visible and focused.
  • mission_control.telemetry_viewed - optional explicit tab view event when the Telemetry tab opens.

Storage Shape

The table should store one row per accepted event with server-derived identity and compact event metadata: id, event_name, event_category, source, user_id, organization_id, user_role, session_id, route_template, surface, page_label, active_seconds, is_pilot_session, properties, occurred_at, created_at.

Risk

Level: 4 - database migration, cross-cutting frontend tracker, production data growth, privacy implications, and admin dashboard changes.

Risks Identified

R1: Privacy and sensitive data capture

Risk: Product telemetry can accidentally collect patient-adjacent data, emails, PMS values, or sensitive URLs if the frontend sends arbitrary payloads.

Mitigation: Server-side event allowlist, property allowlist, no raw URL query strings, no free-form text values, no emails in properties, and server-derived identity only.

R2: Production data volume

Risk: Heartbeats and page views can grow quickly and slow Mission Control if unindexed or queried naively.

Mitigation: Batch ingestion, heartbeat interval no shorter than 30 seconds, server clamps, indexed queries, date-range filters, aggregate SQL in model methods, and no raw event table rendering in V1.

R3: Misleading duration metrics

Risk: Browser exit events are unreliable. Exact time spent claims would be false precision.

Mitigation: Use visible/focused heartbeat totals and label UI metrics as active time or estimated active minutes, not exact dwell time.

R4: Admin pilot pollution

Risk: Super-admin pilot sessions may look like client usage and distort account health.

Mitigation: Send a client-side pilot-mode marker, persist is_pilot_session, and default Mission Control Telemetry aggregates to exclude pilot traffic while making that filter explicit.

R5: Scope creep into product analytics platform

Risk: Adding funnels, replay, action taxonomy across every page, and arbitrary drilldowns turns this into an analytics product.

Mitigation: V1 stops at route/session/heartbeat usage plus aggregate/org/user drilldown. Workflow event instrumentation is backlog unless the spec is revised.

Production Migration Safety

The migration is additive: it creates a new append-only telemetry table and indexes. It does not rewrite or delete existing production rows. Production risk is storage growth and index creation time, not data mutation. Rollback drops the telemetry table and loses collected telemetry; there is no business-data recovery risk, but there is analytics-data loss.

Deployment path: migration runs on dev/dave first through the dev workflow, then production after merge to main.

Blast Radius

Known consumers touched or affected: authenticated app routes mounted under App.tsx, Mission Control page/components, Mission Control API/query hooks, admin Mission Control route/controller, a new telemetry model/service, and the database migration pipeline. Existing revenue overview payload should remain compatible.

Pushback

  • Do not reuse behavioral_events. It already carries checkup, referral, engagement-score, and agent-finding concepts. Product telemetry belongs in its own table.
  • Do not build broad click tracking now. Track intent-bearing route/session/activity facts first, then add explicit workflow events where the product questions justify them.
  • Do not present time spent as exact. The honest metric is active seconds from visible/focused heartbeats.
