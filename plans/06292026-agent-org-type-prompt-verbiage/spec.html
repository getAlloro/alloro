Why

The generic org-type feature still has a generation-layer risk. Even after UI labels are fixed, in-repo agent markdown can teach the model healthcare wording and then persist that wording into tasks, summaries, local posts, review replies, or prompt-driven parsing output.

The goal is to make agent prompts org-type aware at runtime so generic organizations do not receive medical, dental, patient, doctor, referral, PMS, or production language in human-readable generated output, while health organizations keep the current experience.

What

Extend the existing backend prompt vocabulary system instead of adding a parallel prompt framework. Every active in-repo markdown prompt that can affect customer-facing output must either receive resolved org-type placeholders before runAgent() or be documented as out of scope because it has no org context or is intentionally vertical-specific website-builder work.

Tests must render the generic prompts and fail when generic prose still contains banned healthcare terms outside allowed internal schema keys. Existing JSON field names and database/API contracts stay stable.

Patterns To Follow

  • Use src/config/orgLabels.ts as the single vocabulary source. Extend it when a new grammatical role is needed.
  • Use loadPrompt() followed by substitutePromptPlaceholders() before runAgent() when the caller has an organization id or organization row.
  • Keep legacy JSON keys and enum values in schemas. Generic instructions can explain that field names are legacy and must not appear as prose.
  • Add tests against rendered prompts, not just individual placeholder values. A prompt can pass token substitution and still leak "Dr. Joe" in an example.

Reference analog: backend §6.1 src/controllers/gbp-automation/ for prompt call structure and typed service boundaries; src/controllers/agents/feature-services/service.monthly-agent-runner-core.ts for the closest prompt substitution path.

Constraints

Must

  • Generic rendered prompts must instruct generated prose to use customer, business, revenue, lead/source, records, owner/team, visit, and local-service wording.
  • Health rendered prompts must preserve the current healthcare/dental wording.
  • Monthly, daily, GBP automation, PMS column mapping, and PMS sanitizer prompt callers must use the same org-type vocabulary mechanism when an organization context exists.
  • Prompt tests must allow legacy schema keys such as doctor_referral_matrix, patient_records, production_total, and referring_doctor, but fail healthcare words in freeform instruction/example prose for generic prompts.
  • Internal role enums, Zod schemas, database fields, API payload keys, and task persistence columns must stay unchanged.
  • Existing healthcare accounts must be unaffected.

Must Not

  • Do not rename legacy parser roles or agent output schema keys during this work.
  • Do not run live monthly/daily agents or create client tasks during execution unless explicitly approved for a test account.
  • Do not edit external N8N workflows in this plan.
  • Do not broaden into generic website-builder or public audit report verticalization.
  • Do not rewrite existing agent_results or historical task rows.
  • Do not use blind search-replace across src/agents.

Out Of Scope

  • N8N workflow prompt updates.
  • Lead-generation audit report prompt verticalization.
  • Website-builder prompt verticalization.
  • Schema migrations or persisted data rewrites.
  • Changing model vendor from Claude to Gemini. Current code uses runAgent() with Anthropic models.

Risk

Level: 3. This is not a schema migration, but it changes the instructions that generate client-facing tasks, summaries, review replies, and posts.

Risks Identified

  • Schema breakage. Prompt cleanup could rename keys that parsers and validators require. Mitigation: only change prose and examples; explicitly preserve legacy JSON keys and enums.
  • False confidence from simple tests. A placeholder unit test can pass while examples still leak "doctor" or "patient". Mitigation: add rendered-prompt tests with allowlists for schema keys and strict checks for freeform prose.
  • Overcorrecting safety rules. Healthcare safety rules are important for health orgs, but generic orgs should not be prompted with clinical terms. Mitigation: make the safety vocabulary itself org-aware instead of deleting safety constraints.
  • Missing org context. Some prompt callers do not receive an organization row today. Mitigation: fetch org type server-side from trusted account/job context where available; document callers with no org context as out of scope.
  • Generated output drift remains for old rows. Prompt fixes affect future runs only. Mitigation: rely on the display-time stale-copy handling from plans/06292026-generic-revenue-data-verbiage for old tasks; do not mutate history.
  • Instruction mismatch. The original requirement mentioned Gemini 2.5-flash, but this repo currently uses Claude via runAgent(). Mitigation: implement against the current runner and do not invent a Gemini path.

Blast radius: src/config/orgLabels.ts, prompt substitution tests, active markdown prompts under src/agents/monthlyAgents/, src/agents/dailyAgents/, src/agents/gbpAgents/, src/agents/pmsAgents/, and prompt callers for monthly agents, Proofline, column mapping, and paste sanitization.

Pushback

  • This should not become a full "make all agents generic" sweep. Some prompt folders are intentionally dental or do not have org context. Forcing generic wording there would create a worse bug.
  • Prompt tests need an allowlist. A hard ban on words like doctor would fail valid schema keys and break the parser contract.
