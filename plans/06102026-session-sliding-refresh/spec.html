Session Sliding Refresh

Status
Completed

Size
Medium (5 files)
Risk
Level 2

Why

Client sessions hard-expire 7 days after login regardless of activity — a daily-active user gets kicked out mid-session once a week. Admin pilot sessions die after 1 hour, which is too short for real support work.

What

Sliding session expiry: when an authenticated request arrives with a token past half its 7-day life, the backend re-issues a fresh 7-day token in an x-session-refresh response header; the frontend axios interceptor stores it transparently. Pilot tokens move from 1 hour to 7 days and ride the same refresh path (preserving the isPilot claim). Done = an active user is never logged out; a user inactive for 7+ days still expires; pilot sessions behave identically.

Context

Relevant Files

  • src/controllers/auth-otp/feature-services/service.jwt-management.ts - single source of session-token signing (generateToken, 7d)
  • src/middleware/auth.ts - authenticateToken, the single verification gate (46 consumer files; pilot + admin + client traffic all pass through it)
  • src/controllers/admin/auth/services/PilotSessionService.ts - signs pilot tokens (isPilot: true, currently 1h)
  • src/index.ts - hand-rolled CORS middleware; needs Access-Control-Expose-Headers for the refresh header
  • frontend/src/api/index.ts - global axios interceptor + getCommonHeaders(); token storage is localStorage.auth_token (normal) vs sessionStorage.token (pilot)

Patterns To Follow

  • All login flows (password, OTP, checkup) issue via generateToken — extend that service rather than scattering new signing sites.
  • Frontend pilot detection mirrors getCommonHeaders(): sessionStorage.pilot_mode === "true" or sessionStorage.token present.
  • Axios normalizes response header names to lowercase — read x-session-refresh lowercase.

Reference file: src/controllers/auth-otp/feature-services/service.jwt-management.ts - the refresh helper lives beside generateToken/verifyToken, matching their style.

Constraints

Must

  • Re-sign with a clean payload (userId, email, optional isPilot) — never spread the decoded token (jwt.sign throws when exp exists in payload alongside expiresIn).
  • Preserve isPilot across refresh so pilot UI guards keep working.
  • Expose x-session-refresh via CORS for absolute-URL deployments.

Must Not

  • No new dependencies.
  • No change to the 401/403 → session:expired modal flow (explicitly kept as-is).
  • No refresh-token table / server-side session state — stays stateless JWT.

Out Of Scope

  • Server-side revocation / logout-everywhere.
  • Proactive client-side expiry checks (frontend keeps reacting to 403s).
  • httpOnly-cookie migration.

Risk

Level: 2

Risks Identified

  • (L2) Pilot tokens grow from 1h to 7d — a week-long impersonation credential. Mitigation: token lives only in sessionStorage (dies with the tab), isPilot claim survives refresh, and rotating JWT_SECRET remains the kill-switch. Accepted by Dave 2026-06-10.
  • (L2) Without Access-Control-Expose-Headers, cross-origin deployments silently never see the refresh header. Mitigation: added in the same CORS block in this change.
  • (L1) Parallel requests past the threshold each mint a token. Stateless JWTs — all remain valid; last writer wins. Harmless.

Blast radius: authenticateToken guards 46 files — change is additive (a response header on success paths only). generateToken callers: AuthPasswordController (3 sites), AuthOtpController, routes/checkup.ts (2 sites) — signature unchanged. Frontend: every API call passes the global interceptor — additive header read.

Pushback

  • Sliding JWTs mean a stolen token can be kept alive forever by replaying requests. The stateless design has no revocation; if that ever matters, the fix is a server-side session/deny-list — out of scope today, flagged for the record.
