Why

Patient Journey Insights shows what is happening and how the practice is performing, but it does not show work Alloro completed to improve those numbers. The first trustworthy signal is a completed SEO metadata regeneration tied to Google click-through rate.

The feature must build trust without implying that Alloro caused a result that has not been measured. It should say what changed, when it changed, and which metric is being watched.

What

Add a durable, tenant-scoped metric action record and surface the latest active SEO metadata action inside the Google Visibility detail view beside a newly visible Google click-through rate metric.

Customer-facing example: Updated Google search titles and descriptions on 6 pages. Watching Google click-through through Aug 14.

The action row remains in the database as internal history. Its dashboard note is eligible for display for 30 calendar days, then disappears automatically. Version 1 does not calculate or claim SEO lift.

Behavior Contract

Trigger and aggregation

  • Compare the stored meta_title and meta_description with the generated values for each logical page or post before the write.
  • Count only entities whose title or description actually changed and was successfully persisted.
  • Create one action per completed bulk job, not one note per page or post.
  • Use the SEO job id as the idempotency source. A retry or duplicate finalization must not create a second action.
  • A partially successful job may produce an action for its successfully changed entities. A job with zero successful metadata changes produces no action.

Visibility

  • The API returns only the latest active action for the impressions stage and ctr metric.
  • An action is active when active_until >= now.
  • It is shown for a selected report month only when the 30-day measurement window overlaps that month and the action occurred no later than the end of that month.
  • An expired action remains stored but is not returned to the client, including when browsing an older month.

Copy

  • Use plain terms such as Google search titles and descriptions, not an engineering changelog or raw job terminology.
  • Use a separate Alloro did this label so the action is visually distinct from the metric.
  • Say the metric is being watched through a date. Never say the action increased CTR unless a later measurement feature proves it.

Data Contract

Create metric_action_events in the public schema with:

  • id uuid primary key.
  • organization_id integer required; cascade on organization deletion.
  • location_id integer nullable; null means organization/project-wide. Location-scoped rows cascade on location deletion.
  • project_id uuid nullable; cascade on project deletion.
  • action_type text, initially seo_meta_update.
  • stage_key text and metric_key text, initially impressions and ctr.
  • source_type text and source_id text, initially seo_bulk_generation_job plus its job id.
  • entity_type text (page or post) and affected_count integer greater than zero.
  • occurred_at timestamptz, active_until timestamptz, metadata jsonb, and timestamps.
  • Unique idempotency key on (action_type, source_type, source_id).
  • Read index covering organization_id, project_id, stage_key, metric_key, active_until, and occurred_at.

metadata stores structured counts such as title changes, description changes, and failed job items. It must not store customer-facing prose, secrets, page content, or PII. The service builds display copy from structured values so wording can change without rewriting history.

The migration is additive and has no backfill. Past SEO jobs must not be converted into actions because the old job records do not prove which metadata fields actually changed.

Constraints

Must

  • Keep tenant identity server-derived and require organization/project scope in every action read (§5.5, §11.7).
  • Keep DB access in MetricActionModel (§7.4) and the shared behavior in MetricActionService.
  • Use named constants for action type, stage, metric, 30-day window, and display limit (§4.2).
  • Make event creation idempotent for BullMQ retries (§21.1).
  • Log action-write failures with job/project context through Pino, without exposing content or secrets (§9.1, §21.4).
  • Keep backend and frontend Patient Journey types in lockstep and preserve the existing success envelope (§8.1, §16.1).
  • Add focused model/service/worker and presentation tests with synthetic data (§20.1, §20.2, §20.4).

Must Not

  • Do not infer an action from CTR movement, commits, audit logs, or incomplete jobs.
  • Do not claim measured lift or causal attribution.
  • Do not show a permanent dashboard feed or stack multiple notes into changelog-like noise.
  • Do not log manual page/post SEO saves until actor identity can prove the edit was made by Alloro.
  • Do not add a new endpoint, frontend state system, package, or raw API call.
  • Do not refactor unrelated legacy any, response-shape, worker, or query-key drift while adding this feature.
  • Do not overwrite unrelated dirty work in Alloro Docs.

Out Of Scope

  • GBP post, review reply, website content, and new-page publishing triggers.
  • Manual SEO saves and actor attribution.
  • A customer-visible action history or global changelog.
  • Baseline capture, before/after trend calculation, notifications, and proof that an action changed CTR.
  • Backfilling action rows from historical SEO jobs.

Risk

Level: 3 - structural risk. This adds an additive production migration, a shared action contract, a background-job producer, and customer-visible attribution language.

Risks Identified

  • False attribution: job completion alone does not prove a metadata change. Mitigation: compare persisted before/after titles and descriptions and count only successfully written differences.
  • Duplicate actions: BullMQ may retry a job or finalization. Mitigation: unique source idempotency key plus model upsert/ignore behavior (§21.1).
  • Action logging failure after SEO success: retrying the entire generation job could rewrite metadata unnecessarily, while swallowing the error could lose the trust signal. Mitigation: isolate final action recording from generation success, retry the idempotent write a small bounded number of times, log a structured error if exhausted, and never display a note without a durable row. Do not re-run successful generation solely to repair a missing note.
  • Cross-tenant visibility: organization-wide website actions use a null location. Mitigation: require organization and project filters in the model, then allow either the authorized location or null within that tenant; add isolation tests (§5.5, §11.7, §20.2).
  • Wrong month context: a current action beside an unrelated historical CTR would mislead. Mitigation: require both current activity and overlap with the selected report month.
  • Production migration: a new indexed table changes the shared database. Mitigation: additive reversible migration, no backfill, indexed read path, dev migration and rollback before production (§10.3, §10.4).
  • Docs conflict: Alloro Docs is already dirty in PracticeHubReplica.tsx and practice-hub.ts. Mitigation: keep this plan's docs changes to new Patient Journey files and the required registry/loader/sidebar mappings; preserve and exclude unrelated dirty files.

Blast radius: bulk SEO generation for every project, GET /api/patient-journey, the Practice Hub query cache that shares that payload, the Google Visibility detail deck, the public database migration path, and the separate Alloro Docs dashboard registry.

Pushback

  • seo_generation_jobs should not become the customer action store. It is shaped for queue progress, lacks the reusable metric/stage contract, and would couple future GBP/review actions to SEO internals.
  • A general Alloro activity feed is not part of this first version. Showing one current, metric-linked signal is clearer and less noisy.
  • Measured CTR lift must be a later feature with a captured baseline and comparison window. Adding a green/red result now would overstate what the data proves.
