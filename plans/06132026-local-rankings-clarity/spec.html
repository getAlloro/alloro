Context

Confirmed in code (the findings driving this plan)

  • Dead toggle: RankingsHubSurface.tsx defines a local PeriodToggle() rendered greyed (lines 41–63, render at 132–134); rankingPeriod.ts exports PERIOD_TOGGLE_ENABLED=false. The toggle is decorative — no period state is wired. Removing the toggle UI + its imports is safe; rankingPeriod.ts stays (it is also the future enable point and is referenced read-only).
  • Refresh button is real: the sole "Refresh rankings" control lives in RankingsDashboard.tsx (header block, lines 859–870), calls handleRefreshRankings() → reselectAndRun() which kicks off a re-rank batch and shows the in-flight banner. It is disabled only when !canRefreshRankings (no locationId or zero saved competitors). For an onboarded client it is fully functional and NOT greyed. Per the locked instruction ("if a real refresh action exists, leave it functional") it is left untouched.
  • Map zooms out on a far competitor: RankingsMapCard.tsx FitBounds calls map.fitBounds(points, { padding:[36,36] }) over the practice + all competitor pins with NO max-zoom. Competitor discovery uses a 25-mile radius (DEFAULT_COMPETITOR_DISCOVERY_RADIUS_METERS = 40234), so one distant competitor (the "Saif outside Fredericksburg" case) drags the fit to a city-wide zoom-out.
  • Geocoding is Places-sourced, not a custom geocoder: practice coords come from locations.client_lat/client_lng, written by LocationModel.setClientIdentifiers after a Google Places lookup (getClientPhotosViaPlaces) / ranking-history resolution. Competitor coords come from the Places Place Details response (placeDetails.location.latitude/longitude, see buildAddCompetitorInput ~lines 482–489) and are surfaced verbatim by formatLocationCompetitor (lat: Number(c.lat)). There is no lat/lng transform, rounding, or custom geocode step that could be "off." A pin that looks far out is either a genuinely distant competitor selected from the 25-mile radius, or a Places record whose own coordinates are imprecise — neither is an Alloro geocoding bug. No backend change in scope.
  • Review mismatch 163 vs 152: Rankings currently reads result.rawData.client_gbp.totalReviewCount/averageRating — a scrape snapshot from the last ranking run (the 152). Practice Hub reads dashboard-metrics.reviews.total_review_count/current_rating — the live GBP aggregation (the 163). Switching Rankings to dashboard-metrics makes both surfaces show one number (Plan 1 declared it canonical).
  • Velocity source: the 30-day count is rawData.client_gbp.reviewsLast30d. dashboard-metrics.reviews has no equivalent rolling-30d field (reviews_this_month is a calendar-month count, not a 30-day window), so the velocity keeps its existing source and is only relabeled.

Risk

Risks Identified

  • R1 — Review number visibly changes (Level 2). Switching to dashboard-metrics makes the Rankings total jump from the scrape snapshot (152) to the live aggregate (163). Mitigation: that is the intended reconciliation — both hubs now agree. Documented so it isn't mistaken for a regression.
  • R2 — Metrics still loading (Level 2). useDashboardMetrics is async and is disabled when orgId/locationId are null. Mitigation: fall back to the existing rawData.client_gbp values (or "—") while metrics are undefined, so the card never renders blank.
  • R3 — Map maxZoom too tight (Level 1). A hard cap could over-zoom when competitors are genuinely spread out. Mitigation: use a moderate cap (≈13–14) that still respects fitBounds when points are close; it only prevents the pathological zoom-out, not legitimate wider fits.
  • R4 — Parallel-agent collision (Level 2). Three agents edit sibling surfaces. Mitigation: all edits confined to rankings-hub/* + the documented no-op read of RankingsDashboard.tsx; no shared files touched.

Blast radius: RankingsHubSurface + RankingsMapCard (Local Rankings hub only). RankingsDashboard.tsx is read-only here. The canonical review source is consumed read-only from useDashboardMetrics; no provider change. Backend untouched.

Pushback

  • The owner flagged "Refresh Rankings" as confusing — but it is a real, working re-rank. Removing it would delete a client capability. The honest fix to the confusion is a better disabled-state hint (e.g. "Add competitors to enable"), which is a separate, deferred change. Flagging rather than silently deleting a functional control.
  • The "Saif far outside Fredericksburg" concern is not a geocoding bug — coords are Places-verbatim. If the practice wants only nearby competitors, the lever is the 25-mile discovery radius or the curated competitor selection, not a coordinate fix. The zoom cap makes the map readable regardless; the radius is a product decision left to the owner.
