Why

The backend conventions audit confirmed dimension 16 (transactions for multi-table writes) is a systemic breach: db.transaction / trx appears nowhere in src/workers, and the top-level src/services tier writes across tables with no atomicity. These are not style nits — they are live data-corruption paths. websiteRestore destructively wipes ~13 tables plus all S3 media with zero transaction, and because the job re-throws, a BullMQ retry re-runs the wipe against already-wiped data: permanent, unrecoverable loss of a client's website. compilePublish swallows its top-level failure so BullMQ records a half-published run as succeeded. Referral reward and instant-website creation can each leave half-written state on a mid-flow throw.

Now, because the audit verified the exact file:line targets, and because the repo already has a correct in-house pattern to copy (gbp-automation threads trx through 19 call sites). This is mechanical, bounded, and behavior-preserving — the right kind of cleanup to land before it bites a real client.

What

Wrap each verified multi-table, non-idempotent write path in db.transaction(async (trx) => …), threading trx through every write in the set so a failure rolls back to the pre-write state instead of committing a partial. Two reliability companions ship with it: compilePublish must re-throw so BullMQ can retry/dead-letter, and the websiteRestore queue must be set to attempts: 1 with S3 deletion deferred until after the DB commit so a non-idempotent destructive job never auto-retries against wiped data.

Done: every target below either (a) writes atomically inside one transaction, or (b) is documented in this spec's Revision Log as verified-already-safe (idempotent upsert / single-table / intentional non-blocking) and intentionally left unwrapped. npm run build is green and each affected job/flow has been run manually with no behavior change on the success path.

Patterns to follow

  • The repo's connection exports db from src/database/connection and supports db.transaction(async (trx) => { … }) (the gbp-automation + ai-seo-audit consumers prove this). Thread the trx handle into each write in the set; do not open a transaction around a slow external call.
  • For paths that re-use existing inline db("table") queries, the minimal behavior-preserving change is to swap db → trx inside the wrapped block. Pushing those queries into models/ (dim 7) is a separate remediation and is explicitly out of scope here.

Constraints

Must

  • Be behavior-preserving — the success path produces identical rows/side-effects; only the failure path changes (rolls back instead of committing partial).
  • Verify each target is genuinely multi-table AND non-idempotent before wrapping. If a write is already idempotent-by-upsert (onConflict().merge()), single-table, or an intentional non-blocking fire-and-forget, do not wrap it — record the finding in the Revision Log and move on.
  • Commit before each unit, one concern per commit (one target / one transaction boundary per commit).
  • Mirror the in-house reference pattern (gbp-automation / ai-seo-audit): thread trx; never hold a transaction open across an external API call (Stripe, Anthropic/OpenAI, Google, S3) — wrap only the local writes.
  • For websiteRestore: reorder so S3 deletion happens after the DB commit (or upload-new-keys-first / delete-old-last) so media stays recoverable, and set the restore queue to attempts: 1.
  • Run npm run build and manually exercise each affected job/flow before declaring done.

Must Not

  • No new dependencies, no new logging/response-shape patterns, no schema/migration changes — pure transaction wrapping + the two reliability fixes.
  • Do not refactor unrelated code: no model extraction (dim 7), no function splitting (dim 2), no naming renames (dim 14), no console→Pino, no magic-number hoisting (dim 11). Those are other plans.
  • Do not wrap a slow external call inside the transaction (lock-held-across-network = the trap gbp-automation deliberately avoids).
  • Do not change the shape of what each function returns or the events it emits.

Out Of Scope

  • Moving inline db("table") queries into models/ (dim 7 systemic) — separate remediation.
  • The agentsV2 IDOR / unauthenticated-route fixes (dim 13) — that is the security-hotfix track; this plan sequences after it.
  • The gbp-automation dim-16 concerns (post-Google bookkeeping in GbpLocalPostDeploymentService / GbpReviewReplyDeploymentService / GbpPublishedReplyService) — already partially mitigated with partial-failure fallbacks; revisit only if cheap, otherwise defer.
  • The minds regenerateEmbeddings delete-then-insert (checks/03) — related class, but lives in the minds remediation track; not included here.
  • Function-size, file-size, validation, and logging items recorded in the audit.

Risk

Level: 3 (Structural Risk). These are write paths. Introducing transactions changes locking and connection-pool behavior, can surface latent ordering bugs, and — done wrong — can hold locks across slow I/O and cause pool exhaustion or deadlocks under concurrency. The change is behavior-preserving by intent, but the blast radius is data-write correctness, so it does not auto-execute without the verify-before-wrap gate.

Risks Identified

  • Some writes may already be idempotent / single-table and wrapping them adds risk for no benefit. Mitigation: the Must constraint — confirm multi-table + non-idempotent per target before wrapping; log a Revision-Log entry for any target found already safe and leave it untouched.
  • Lock held across an external call (Stripe in referralReward, the LLM/webhook in the agent executors, S3 in websiteRestore) would convert a correctness fix into a perf/deadlock regression. Mitigation: wrap only the local DB writes; keep external calls outside the trx block, exactly as gbp-automation does. For websiteRestore specifically, S3 deletion moves to after the commit.
  • websiteRestore retry semantics — even wrapped, a destructive restore must not auto-retry. Mitigation: set attempts: 1 on that queue; rollback + no-retry together make a failed restore safe (project unchanged) rather than corrupting.
  • compilePublish re-throw changes job outcome — failures that currently report "success" will start reporting "failed" and retry/dead-letter. Mitigation: this is the intended correction; verify the run still completes on the happy path and that a forced failure now surfaces as failed (manual run).
  • No backend test suite to catch regressions. Mitigation: gate is npm run build + manual run of each affected job/flow; commit-per-unit so any one change is trivially revertible.
  • trx + existing transaction nesting — if any target is already called within an outer transaction, nesting could double-wrap. Mitigation: grep each call site's callers for an enclosing db.transaction before wrapping; if found, accept the trx as a parameter instead of opening a new one.

Blast radius (known consumers of the touched paths):

  • websiteRestore.processor ← src/workers/worker.ts:277 (BullMQ restore worker) and its queue/enqueue site.
  • compilePublish.processor ← the minds-worker compile/publish queue (BullMQ); changes which jobs are marked failed.
  • referralReward.ts ← imported directly by routes/checkup.ts (per checks/11) and any referral-conversion / Stripe webhook flow that calls applyReferralReward.
  • instantWebsiteGenerator.ts ← routes/checkup.ts (instant-website creation flow).
  • The four agent executors ← AgentsController run-triggers (proofline / monthly / ranking / process-all) and the scheduler.

Pushback

  • Verify before you wrap — do not wrap reflexively. The audit flagged these as transaction-less, but "transaction-less" ≠ "corruptible". A single-table write or an onConflict().merge() upsert is already atomic. configStore.setConfig (checks/11) is a read-then-insert TOCTOU that wants an upsert, not a transaction — that distinction matters. Confirm the failure mode is real (partial multi-table state on throw) for each target; if it isn't, skip and document.
  • This is a Level-3 set; resist scope-bundling. The same files carry dim-7 (DB-in-service), dim-2 (fat functions), and dim-11 (magic numbers) findings. Pulling those in would balloon the blast radius and break behavior-preservation. One concern per commit, transactions only.
  • Sequence after the security-hotfix. No hard dependency, but the agent executors overlap files/attention with the agentsV2 auth work; doing security first keeps each diff focused and reviewable.

Pattern used

Mirrors ai-seo-audit/auditTargetExecutionService.ts / gbp-automation: await db.transaction(async (trx) => { … }) with trx("table") for every write in the set; external calls (S3, Stripe, Google, LLM) kept strictly outside the trx block. For id-returning inserts inside a transaction, the row is awaited into a local and returned (returning the bare .returning() builder from the async callback trips TS2488 — the callback infers void).
