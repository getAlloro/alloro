Superseded

This original Patient Journey plan was replaced by the later Market Intelligence plan set from June 26, 2026. Its acceptance checklist remains Not Run because validation moved to the newer specs: plans/06262026-market-intelligence-backend-foundation, plans/06262026-market-intelligence-patient-journey-api, plans/06262026-market-intelligence-dashboard-ui, and plans/06262026-market-intelligence-enrichment-ops.

Why

Corey wants one owner-facing screen that shows the monitored lead-generation journey as a funnel — from market demand down to website leads — and makes the single biggest leak obvious. It is the Alloro translation of the TrainerRoad "Career" dashboard concept reviewed on the 6/15 Weekly Reset. Today this insight is scattered across Rankings, Reviews, Submissions, and Referrals; no screen tells the doctor "you are losing people here."

The two mock concepts (~/Desktop/alloro-owner-dashboard/funnel-flow.html vertical, funnel-pipeline.html horizontal) are validated. Decision locked: build the horizontal card + arrow pipeline as the entire content of the screen, ending at website leads. Production/revenue is intentionally excluded from the pipeline until Alloro has fresh, attributable conversion data.

What

A new Patient Journey Insights dashboard at the existing /patientJourneyInsights route (currently a placeholder), rendering a four-stage lead-generation funnel as horizontal cards joined by arrow connectors, plus rank/reviews context cards and a "one thing that matters" action. Clicking one of the four pipeline cards opens a compact slideable detail deck that explains where that number came from, using the existing stage value/source/as-of/conversion data. It is powered by a new backend funnel-assembly endpoint that stitches together existing data (GSC, form submissions, rankings, reviews) and one net-new source (market search volume). A compact summary card linking to the screen is added to the Practice Hub.

Done when: a real pilot organization loads the screen and sees the monitored lead-generation stages with live numbers (or honest "not connected yet" states), correct conversion %s and a highlighted biggest-leak, and a working summary card on Practice Hub — with no tenant-data leakage and all Constitution gates green.

Conversion chain: volume → impressions → visits → leads. "Saw your listing" is GSC search impressions; "Visited your site" is Rybbit visits across all channels (search + direct + social), so "Saw → Visited" is a directional relationship, not a strict search CTR — the GSC click number is kept as the search-specific cut and the step is labelled "visits". The "Searching → Saw" step is a directional visibility share (impressions ÷ market volume); impressions count appearances, not unique searchers, so it is indicative, not exact. Scope: every stage scopes to the selected location; for single-location practices this is exact, and for multi-location practices the shared website-traffic stages (impressions / visits / leads) show whole-practice totals, labelled. Production: intentionally excluded from the pipeline because PMS production can lag the 30-day lead-generation window and cannot be attributed only to website leads.

Constraints

Must

  • All DB access in models/ (§7.4); reuse existing models, do not duplicate query logic.
  • One backend response contract { success, data, error } (§8.1); typed domain error + central status map (§8.3).
  • Frontend data only through api/index.ts (§14.2), fetched in a hook (§14.3), server state in React Query (§15.1).
  • Tenant scope derived from server context, passed as a required arg to every model query (§5.5/§11.7).
  • Org wording via useLabels() / orgLabels — never hardcode "patient"/"practice" (reuse the existing system).
  • New search-volume API key in env/config only (§5.1, §5.6); adapter follows the harvest contract (§21).
  • Honest empty / "not connected" states for any stage whose source is missing (GSC unconnected, no PMS upload, no search-volume yet) — never render a misleading zero.
  • Per-stage "as of" freshness shown (GSC ~4-day lag, rankings twice-monthly, PMS possibly months old).

Must Not

  • Present the Rybbit visit count without flagging it as all-channel and best-effort bot-filtered (Rybbit blockBots + edge-block); do not read "Saw → Visited" as a strict click-through.
  • Invent a composite "Clarity Score" or any prediction — we have no forecasting model; the headline is descriptive only.
  • Add a new state library (§15.4) or hand-roll fetch/auth (§14.2).
  • Block the whole screen on the one sparse/paid stage — degrade gracefully per stage.
  • Build per-location separate funnels in v1 (data is one website per org).

Out Of Scope (v1)

  • A composite score / prediction headline; GBP-impressions variant of "Saw your listing" (use GSC; GBP local-pack impressions are a later enhancement).
  • A stored patient_journey_snapshots history table (v1 computes trends live from existing date-series; snapshot table is Phase 2 for a dedicated "journey over time" chart).
  • True per-location attribution of shared website traffic (GSC pages / Rybbit paths / form page) for multi-location practices — deferred fast-follow; v1 scopes to the selected location, exact for single-location practices and practice-wide-but-labelled for multi-location. Real-time refresh.

Risk

Level: 4 (new external paid data pipeline + cross-cutting new screen + multi-source aggregation). Build ships the monitored lead-generation pipeline only; production/revenue remains outside the Journey pipeline until attribution and freshness are trustworthy.

Risks Identified

  • Search-volume provider cost — RESOLVED (Rev 2). Pilot measured $0.075/practice/refresh, ~$0.82/month fleet-wide (~$10/year) on DataForSEO. Negligible; the stage is un-gated. Residual build care: metro-level geo + a curated patient-intent keyword set + null handling so "your market" is an honest local estimate (see Rev 2).
  • Production attribution is not trustworthy as a funnel stage (PMS data can lag the 30-day lead-generation window and is not caused only by website leads). Mitigation: exclude production/revenue from the Journey pipeline; show it only in separate PMS/production surfaces where period and attribution caveats are explicit.
  • Visitor analytics carry bot risk (owner chose Rybbit for "Visited"; ~10% repeat-visitor inflation when summing daily rows, plus datacenter-bot surges like One Endo). Mitigation: rely on Rybbit blockBots + the deployed renderer edge-block, prefer fetchRybbitOverview() deduped totals over summed daily rows, and label it all-channel visits; residual bot inflation is a flagged limitation — a dedicated in-app bot-filter is not v1 scope.
  • Entity scope — website-traffic stages come from one shared website per org; rank/reviews/PMS/search-volume are per-location. Mitigation: funnel scopes to the selected location — exact for single-location practices (the common case); for multi-location practices the per-location stages stay location-specific while the shared website-traffic stages show whole-practice totals, clearly labelled, with true per-location attribution deferred to a fast-follow.
  • Visibility-share % is directional (impressions ≠ unique searchers). Mitigation: label "Searching → Saw" as an estimate; the stage detail deck explains it.
  • Freshness mismatch across sources. Mitigation: per-stage "as of" date; single window with caveats, never one global "as of".

Blast radius: new backend domain controllers/patient-journey/ + route + new migration(s) + new worker adapter; frontend new api/patientJourney.ts, hook, surface component, and edits to Dashboard.tsx (tab renderer), Sidebar.tsx + MobileBottomNav.tsx (nav), and DashboardOverview.tsx (Practice Hub card). Existing consumers of those three frontend files must keep working — additive edits only.

Pushback

  • Including production in the same card-and-arrow sequence would imply website leads solely caused that production. Recommended alternative: keep Journey focused on lead generation and show production only in separate period-aware PMS/production views.
