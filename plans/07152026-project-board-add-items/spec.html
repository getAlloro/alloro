Why

The cross-project PM workload board lets admins review To Do and In Progress items, but it has no way to add a new item directly from that view. Adding from this surface must require a project because the real PM columns belong to projects, not to the global board.

What

Add manual create controls to the To Do and In Progress columns on the assigned workload board shown by /admin/pm?view=me and /admin/pm?view=assignee&userId={id}. The create flow requires a project selection, locks the target column to the column where the user clicked add, and assigns the new task to the current user for Mine or the selected user for People.

Scope Ledger

In Scope

  • Add controls only to the cross-project assigned board's To Do and In Progress columns.
  • Require an explicit project pick before create from the cross-project board.
  • Use the selected project's real To Do or In Progress column from the existing PM project detail endpoint.
  • Refresh the assigned board after create so the new item appears without a full page reload.

Out Of Scope

  • No Done-column creation.
  • No Backlog changes.
  • No new database tables or migrations.
  • No broad PM state-management refactor.
  • No changes to project-specific board quick add behavior except shared modal support needed for this flow.

Constraints

Must

  • Require a project selection before creating from the assigned board.
  • Lock the create target to To Do when launched from To Do and In Progress when launched from In Progress.
  • Assign created Mine-view tasks to the signed-in admin user.
  • Assign created People-view tasks to the selected PM user.
  • Disable or block create if the assignee id cannot be resolved.
  • Use the existing POST /api/pm/projects/:id/tasks path and existing assigned_to payload field if it verifies cleanly.
  • Refresh the assigned workload after create via the existing loadData path.

Must Not

  • Do not create unassigned tasks from Mine/People views.
  • Do not fake global columns. The selected project's real column id is required.
  • Do not auto-pick a hidden default project in the cross-project create flow.
  • Do not change Backlog movement, bulk delete, notification, comment, attachment, or project board drag behavior.
  • Do not add new dependencies.

Risk

Level: 2 - Concern

Risks Identified

  • Unassigned created tasks would disappear from Mine/People after refresh. Mitigation: require a resolved assignee id and send assigned_to on create.
  • Auto-selecting the first project would violate the user's required project attachment intent. Mitigation: add an explicit project-required mode for cross-project create and keep the submit action disabled until a project is selected.
  • The global assigned board has no real global column ids. Mitigation: use the selected project's detail response to find its real To Do or In Progress column id.
  • PM frontend files already have older direct state patterns. Mitigation: keep changes narrow, reuse existing typed API/store surfaces, and extract any new UI state into small props or helpers when files approach size limits.

Blast radius: /admin/pm?view=me, /admin/pm?view=assignee, the shared task creation modal, and the existing PM task create API payload. No schema or migration impact is expected.

Pushback

  • Do not add a simple plus button that creates a task with no assignee. Future-us will chase a "created but missing" bug because the assigned board filters to assigned work only.
  • Do not build a separate create endpoint unless execution proves the existing assigned_to create contract fails. The current API already supports the payload; duplicate endpoints would add drift.
