Why

Admin access today is email + OTP gated by a SUPER_ADMIN_EMAILS allowlist. The owner wants admin sign-in to be Google-only, restricted to verified @getalloro.com Google accounts, and to drop the OTP admin path entirely (no break-glass). Separately, regular users (dental practices) currently sign in with email + password only; the owner wants to add "Sign in with Google" as an option beneath the password form, and let invited users link their Google account from Settings so they can use SSO. Primary/owner accounts already use their Google address (it was required for GBP access from day one), so SSO should be seamless for them.

What

Deliverable, two phases.

  • P1 — Admin Google SSO (replaces OTP). The admin login screen becomes a single "Sign in with Google" button. The backend runs the OAuth authorization-code flow, verifies the Google ID token, asserts the email is a verified @getalloro.com account, finds-or-creates the user, and mints the same bearer JWT the app already issues. The admin OTP login endpoints and UI are retired. superAdminMiddleware gates admin routes on the @getalloro.com domain (with SUPER_ADMIN_EMAILS retained as an optional extra grant).
  • P2 — User "Sign in with Google" + account linking. DEFERRED (Rev 2, owner 07-05) — not in this execution. The client-facing user login is parked for now. When resumed, the client sign-in page gains a "Sign in with Google" button below email/password; on first Google sign-in, if Google's verified email exactly matches an existing verified Alloro user, we auto-link and sign them in, otherwise we tell them to link Google in Settings first; Google sign-in never creates a brand-new account; a "Sign-in methods" card in Settings lets any logged-in user link/unlink Google. P2 requires an External consent screen (this project is Internal); resume by using the External project confirmed in Rev 1, or flipping a project to External.

Done when (this execution — P1 only): an @getalloro.com Google account can reach the admin dashboard with no OTP anywhere; a non-@getalloro account cannot reach admin (Google's Internal consent screen blocks non-Workspace accounts, and the code check backs it up); password login, signup, email-verify, and the JWT session model are unchanged. (P2 client/user Google login is deferred — see Rev 2.)

Design — flows

Admin sign-in (P1)

AdminLogin "Sign in with Google" → GET /api/auth/google/start?flow=admin (set signed state+nonce cookie, redirect to Google, hd=getalloro.com hint) → Google consent (any account; hd pre-selects @getalloro) → GET /api/auth/google/callback?code&state (exchange code, verify id_token, check nonce) → assert email_verified && email endsWith @getalloro.com (else redirect /admin/login?error=AUTH_DOMAIN_FORBIDDEN) → UserModel.findByGoogleSub | findByEmail → find-or-create user; upsert google_sub + avatar_url → token = generateToken(user.id, user.email) (SAME JWT as today) → set auth_token cookie (buildAuthCookieOptions, httpOnly:false) → redirect /auth/google/finish?flow=admin → Frontend finish route: read auth_token cookie → setAuthSession() → redirect /admin

User sign-in (P2)

Signin "Sign in with Google" → GET /api/auth/google/start?flow=login → Google → callback → assert email_verified (any domain) → user = findByGoogleSub(sub) found → sign in not found, email_verified && email == existing VERIFIED user.email → auto-link google_sub → sign in otherwise → redirect /signin?error=GOOGLE_NOT_LINKED (never create an account) → token = generateToken → auth_token cookie → /auth/google/finish?flow=login → /dashboard

Link from Settings (P2)

Settings › Sign-in methods › "Link Google account" → GET /api/auth/google/link/start (authenticateToken; stash req.user.userId in signed tx) → Google → callback (flow=link): attach sub to the tx userId → guard: sub not already bound to a different user (unique index) → error GOOGLE_ALREADY_LINKED → redirect /settings?googleLinked=1 ; "Unlink" → POST /api/auth/google/unlink (authenticateToken)

Constraints

Must

  • Reuse the existing bearer-JWT session (generateToken) — no second session model.
  • Verify Google identity server-side: email_verified === true, ID-token signature + nonce; hd is a hint only (§5.4).
  • Admin gate enforced server-side on every /api/admin/* route via superAdminMiddleware (§11.1, §5.4).
  • Distinct env names GOOGLE_LOGIN_CLIENT_ID/SECRET/REDIRECT_URI (§5.1) validated at startup (§5.6).
  • Auto-link only when email_verified AND exact match to an existing verified user; never create an account from Google alone.
  • All google_sub reads/writes go through UserModel (§7.4).
  • Rate-limit /api/auth/google/start (§11.3); CSRF via signed state+nonce (§11.4).

Must Not

  • Do not reuse the GBP OAuth env vars / client (GOOGLE_CLIENT_ID stays owned by controllers/auth/ + googleauth/).
  • Do not reuse google_connections for login identity.
  • Do not delete shared helpers still used by password auth — generateToken (jwt-management) and generateSixDigitCode (otp-generation).
  • Do not remove /api/auth/otp/validate (website-builder depends on it).
  • Do not put OAuth token/JWT in a redirect URL — keep it in the auth_token cookie, finish-route copies to localStorage.
  • Do not change password login, signup, or email verification.

Out of scope

  • Migrating the whole app to httpOnly cookie sessions or a refresh-token table (alloro-os's model). Explicitly not ported.
  • Google login for the OS admin tab / sibling apps beyond what already exists.
  • Removing password auth. It stays as the default user method.
  • SCIM / Workspace directory sync, group-based admin roles.

Risk

Level: 3 (auth surface change + broad admin grant).

Risks identified

  • (L3) Domain-only admin grant — owner-chosen. Every verified @getalloro.com Google account becomes full super-admin over all client data (sales, support, contractors with a Workspace seat). Mitigation: superAdminMiddleware still enforces the gate server-side; SUPER_ADMIN_EMAILS retained as an optional extra grant so the owner can tighten to a list later with no code change; offboarding a Workspace account removes admin immediately. See Pushback.
  • (RESOLVED 2026-07-05, see Rev 1 & Rev 2) Consent screen type. Admin (this execution) uses the Internal "Alloro" Google project: only @getalloro.com Workspace accounts can authenticate — ideal for an admin-only door, with no user cap, no "unverified app" screen, and no verification review. The code domain check remains as defense-in-depth. The External + In-production project confirmed in Rev 1 (100-user cap irrelevant to non-sensitive login scopes) is the path for the deferred P2 client login.
  • (L2) OAuth env collision. Reusing GOOGLE_CLIENT_ID would clobber GBP OAuth. Mitigation: distinct GOOGLE_LOGIN_* names; grep-verify at execution.
  • (L2) Account takeover via auto-link. Linking by unverified or mismatched email could hijack an account. Mitigation: require email_verified===true + exact match to an existing verified user; never link a sub already bound to another user (partial-unique index).
  • (L2) Retiring OTP endpoints blast radius. requestOtp/verifyOtp call onboardUser and linkAccountCreation. Mitigation: at execution, grep every consumer of service.user-onboarding and the OTP routes; retire only /request+/verify and the admin OTP UI; keep /validate, jwt-management, otp-generation.
  • (L2) Login broken if config missing. Mitigation: validate GOOGLE_LOGIN_* at startup (§5.6) with a clear message; a clean error on the callback (redirect to login with ?error=config), never a 500 stack.
  • (L1) Non-httpOnly cookie bridge. Accepted deviation from §11.5 — it follows the app-wide existing pattern (buildAuthCookieOptions), so it introduces no new exposure and avoids a divergent second pattern (§4.3 / consistency).
