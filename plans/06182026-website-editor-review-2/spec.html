Risks Identified

  • Item 2 — deduplicated window total + per-month trend (Level 2). [Rev 1] The bottom Traffic card's "visitors over the window" must be the deduplicated count for that exact window (e.g. last 3 months), not the sum of monthly uniques (which double-counts repeat visitors). Sessions, page views, and leads sum cleanly. Mitigation: source the window visitor total from a Rybbit overview for that window; if only the 365-day overview + monthly series are available client-side, add a small backend window-overview (reusing fetchRybbitOverview) rather than summing monthly uniques. Per-month hover values + month-over-month % come from the monthly series. Confirm at execution; record the outcome here. (Supersedes the earlier "force the modal tiles to this-month" approach — the modal now stays as-is per Rev 1.)
  • Item 3 — surface detection correctness (Level 2). If the surface prop is wired wrong, the Import button could vanish on admin (regression) or persist on client (no fix). Mitigation: default the prop to "admin" (current behavior) and have only the client page (DFYWebsite.tsx) opt into "client"; verify both surfaces in the acceptance run.
  • Item 4 — nothing to fix (Level 1). The duplicate button already renders unconditionally and is deployed to dev. The screenshot likely predates deploy. Mitigation: verify on current dev first; only open a code fix if it is genuinely absent for the Doctors type on the client surface.
  • Item 6 — fragile name matching (Level 2). A form maps to a location only via its name string ("Sterling - referral"). A form whose name omits the location won't float. Mitigation: case-insensitive contains-match on the selected location's name; forms with no match keep their existing relative order; document the limitation in Docs. This is the owner's chosen trade-off over a schema change.
  • Item 8 — "insufficient history" threshold (Level 1). Disabling a toggle needs a clear rule. Mitigation: available-days = latest−earliest report_date; disable a range whose required days exceed available-days by a small margin (e.g. > 7 days short). Tooltip explains why.

Blast radius: PostsListView.tsx / PostsTab.tsx render on both the admin WebsiteDetail and client DFYWebsite surfaces — the surface prop touches both render sites (intended). SeoPanel.tsx is shared by the post and page editors — gate only the Posts/client path. KeywordsTab.tsx is client-only; the admin GscPerformanceDashboard is a separate consumer of the same service (item 7 rename touches it; item 8 disable is client-only unless the admin view shares the toggle — confirm at execution). GscDataModel / service.gsc-performance.ts feed the GSC dashboard on both surfaces; the new field is additive and backward-compatible.

Pushback

  • Item 6: name-string location matching is genuine tech debt — it will silently fail for any form not named after its location, and break if naming conventions drift. The cleaner answer is a structured location reference on the form/submission. Recommended: ship the sort now (owner's call), but log a follow-up plan for a real location field so this doesn't calcify. Surfaced, not blocking.
  • Item 7: "Impressions" is the Google Search Console term of art; renaming it hub-wide (to e.g. "Search appearances") diverges from what owners may see in Google's own tools. Recommended term: "Search appearances" with a one-line definition that names Google Search, but the exact wording is the owner's to finalize before merge (low risk, easily tweaked).
