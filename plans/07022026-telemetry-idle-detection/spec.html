Why

The product-telemetry heartbeat counts idle time as engagement. frontend/src/hooks/useAppTelemetry.ts:68-100 flushes an app.page_active_heartbeat (~30s of active_seconds) every 30 seconds whenever the tab is merely visible — there is no mouse/keyboard/scroll check. A dashboard left open on a second monitor records unlimited "active" minutes, poisoning the CS-outreach signal that the internal-filtering plan (plans/07012026-telemetry-internal-filtering) and the data cleanup (plans/07022026-telemetry-trend-revamp-idle-cleanup) are building toward. The cleanup only patches history — this fix stops the inflation at the source.

What

The heartbeat stops counting time once the user has gone 2 minutes without a real interaction (pointer move, key press, wheel/scroll, pointer down, touch), and resumes counting on the next interaction. Visibility handling stays as-is. No backend or schema change — ingestion's per-event 120s cap (MAX_ACTIVE_SECONDS) is untouched.

Rule: on each 30s tick, count elapsed time only up to lastInteractionAt + IDLE_GRACE_MS. Fully idle ticks report 0 seconds and are not sent at all (the existing activeSeconds === 0 early-return). Worst-case idle overcount per parked page: ~2 minutes, well under the 5-minute historical cleanup cap.

Assumptions self-resolved (recorded per -i-style autonomy): idle grace = 2 minutes; interaction sampling throttled to 1s (a timestamp compare, no timers); route changes and tab-refocus count as interactions.

Context

Relevant Files

  • frontend/src/hooks/useAppTelemetry.ts - the only file changed. flushHeartbeat() (line ~69) computes activeSeconds = now - lastActiveAt; gains the idle clamp. A new lastInteractionAt ref is updated by throttled passive listeners registered in the same effect that owns the interval.
  • src/controllers/app-telemetry/feature-services/AppTelemetryIngestionService.ts - context only (not modified): server already caps a single event at 120s.

Patterns To Follow

  • §13.2/§13.3 — logic stays in the hook, no new component/state system; refs not state (no re-renders from mousemove).
  • §4.2 — named constants (IDLE_GRACE_MS, INTERACTION_SAMPLE_MS) beside the existing HEARTBEAT_INTERVAL_MS.
  • Listeners are passive and capture (to catch inner scrollables), registered/removed symmetrically in the effect cleanup — same shape as the existing visibilitychange/pagehide wiring.

Reference file: frontend/src/hooks/useAppTelemetry.ts itself — the change extends its existing effect structure.

Risk

Level: 2 — Concern

  • Under-counting passive reading. A user reading a long page without touching mouse/keyboard for >2min stops accruing time until they interact. Accepted: minutes become a floor of provable activity, matching how the owner wants the metric read. Mitigation: threshold is one named constant, trivially tunable.
  • Listener overhead. pointermove fires constantly. Mitigation: handler is two timestamp reads + occasional ref write (throttled to 1/s), passive, no React state.
  • Behavior shift in recorded minutes. Post-deploy per-user minutes will read lower than the old regime. Communicated together with the data-cleanup baseline change.

Blast radius: the hook runs app-wide (mounted once); only affects what active_seconds values get sent. No consumer parses heartbeat cadence.
