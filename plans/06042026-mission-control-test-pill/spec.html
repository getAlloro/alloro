Mission Control: Test Filter Pill & De-Hardcode

Why

Mission Control currently hides "sandbox" orgs by excluding them from the org list based on a hardcoded name match (isSandboxOrganization in MissionControlModel.ts). That means a rename silently flips visibility, new test orgs aren't caught without a code edit, and the orgs vanish with no way to inspect them. We want to stop hard-excluding them: keep them in the data, mark them with a real DB flag, and surface them behind a new Test filter pill. While here, remove the unused Risk filter pill.

What

Two user-visible changes to the admin Mission Control page (/admin/mission-control):

  • Test pill added. Test orgs are hidden from every default view (All, No Method, Stripe Active, Admin Granted, Archived) and from the "N Alloro Clients" count, revenue, summary, and movement metrics — and appear only when the new Test pill is selected.
  • Risk pill removed. The payment-risk pill and its filter branch are deleted. riskFlags data is untouched — it still powers the org-card risk chips, the "No Method" pill, and the Payment Attention panel.

Done when: a new organizations.is_sandbox column drives membership (backfilled from the old name list as a one-time migration); the runtime no longer contains the hardcoded sandbox name list; the Test pill shows exactly the backfilled orgs; the headline count is unchanged; tsc and builds are clean.

Constraints

Must

  • Add organizations.is_sandbox boolean NOT NULL DEFAULT false; backfill orgs whose normalized name ∈ {test, hamiltonwise'sorganization, alloroteam'sorganization} to true.
  • Surface the flag as isTest on the API org object (backend interface + frontend type).
  • Exclude test orgs from: the card list (all non-Test filters), the headline count memo (MissionControl.tsx L28), and backend activeOrganizations/activeBaseOrganizations (summary, revenue, movement).
  • Show test orgs only under the new test pill.
  • Remove the hardcoded sandbox name list from runtime code (isSandboxOrganization); names survive only inside the one-time migration backfill.

Must Not

  • Do not remove or alter riskFlags data, the "No Method" pill, the org-card risk chips, or the Payment Attention panel.
  • Do not change organization_type (it stays the business vertical: health/saas).
  • No new dependencies. Do not refactor unrelated Mission Control code.
  • Do not let a test org leak into any default view, count, or revenue/summary aggregate.

Out Of Scope

  • UI to toggle is_sandbox (e.g., a "test org" checkbox on Create Org, or an admin toggle) — future enhancement.
  • Archiving/deleting the existing test orgs (Hamilton Wise org 36, etc.) — they stay, just relabeled.
  • Changing what riskFlags means or how risk is computed.

Risk

Level: 2 (Concern)

Risks Identified

  • Leak risk: backend now returns test orgs (previously excluded), so any consumer that doesn't filter isTest will surface or count them. Mitigation: the only aggregate consumers are activeOrganizations/activeBaseOrganizations (summary, revenue, movement) and the frontend count memo + card filter — all enumerated in Tasks. Verify the count stays at its current value after the change.
  • Migration on a live DB: active environment is live. Mitigation: column is additive, NOT NULL DEFAULT false, no destructive change; backfill touches only name-matched rows. Confirm the migration has not been applied before running; run via the project's knex migrate command.
  • Minor perf (Level 1): Stripe revenue lookup (getStripeRevenueByOrg, Service L93) now also runs for test orgs. Mitigation: accept it — Test-pill cards need real Stripe data; the test-org count is tiny.

Blast radius: consumers of the Mission Control payload — MissionControl.tsx (count, card list), MissionControlSummary, MissionControlRevenueTrend, MissionControlPaymentWatch, OrganizationMissionCard. All read from the same payload; only the count memo and filterOrganizations need new isTest handling. Removing payment-risk touches exactly 3 lines (Header union L6, FILTERS L25, filter branch L187) — confirmed no other references.

Pushback

  • The migration backfill still hardcodes the three names — but that is correct: seeding existing data is a one-time act. The durable mechanism becomes the column. If you would rather not enumerate names even once, the alternative is to flag them by hand in the DB; the migration is the reproducible path and is recommended.
