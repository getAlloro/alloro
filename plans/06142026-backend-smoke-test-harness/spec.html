Phase 0 — The Enabler

This is the first plan in the backend-cleanup arc, and nothing else in that arc should start until it lands. The conventions audit (06142026-backend-conventions-audit) produced a ranked backlog whose two heaviest items — DB access out of models/ (~125 files / ~507 call sites) and god-file decomposition (128 files over the 300-line cap, incl. checkup.ts 2,140 ln and admin-websites 5,567 ln) — are large, behavior-preserving refactors across the hottest endpoints in the app.

There is no safety net. Verified during recon: 0 *.test.ts / *.spec.ts files in src/; no jest, vitest, mocha, or supertest in package.json; no test runner config on disk; no test npm script. The only verification gate today is tsc, which proves the code compiles, not that an endpoint still returns the right thing. Refactoring 500+ DB call sites with nothing but a type-checker is how silent regressions ship.

This plan introduces the test net the rest of the cleanup needs. It is deliberately a smoke net, not exhaustive coverage: enough HTTP-level assertions over the highest-value endpoints to catch a refactor that breaks a route's status code or response shape. Coverage breadth is explicitly a later concern.

Why

The backend is ~743 files / ~142K lines with zero automated tests. The cleanup backlog requires moving DB queries into models/ and breaking apart god-files — both touch live, paid, auth-bearing endpoints. Without a regression net, every refactor is a blind edit verified only by "it compiles" and manual clicking. We need a fast, cheap smoke suite that fails loudly when a refactor changes observable endpoint behavior, so the big refactors can proceed safely.

What

A runnable backend test harness:

  • Vitest + Supertest added as devDependencies (pinned), with a vitest.config.ts and a test + test:watch npm script.
  • A small set of smoke / integration tests over the highest-value endpoints — auth login, OTP verify, a couple of admin reads, the PMS file/job list, a practice-ranking read, and a GBP read — each asserting HTTP status + response shape (not deep business correctness).
  • A documented, agreed data strategy (model-layer mocking first; disposable test DB deferred — see Pushback).

Done (this plan): npm test runs green locally; the listed endpoints each have at least one passing smoke assertion; the suite runs without a live Postgres or external network. Out (later): per-route coverage, integration-depth DB tests, CI wiring.

Context

Current state (recon 2026-06-14)

  • package.json scripts: dev = tsx watch src/index.ts, dev:worker = tsx watch src/workers/worker.ts, build = tsc && rsync ... src/agents/ dist/agents/, start = node dist/index.js, plus db:migrate / db:rollback / db:seed (all knex). No test script.
  • Runtime/dev toolchain is tsx (ESM-aware TS executor) + tsc for build. typescript ^5.9.2, tsx ^4.20.6. Test runner must be ESM/TS-native to match — Vitest fits cleanly; Jest would need extra ESM/ts transform config.
  • zod ^4.3.6 is already a dependency (used today for LLM-output parsing). Available for free as the response-shape assertion tool in tests — no new schema lib needed.
  • Zero existing tests: find src -name '*.test.ts' -o -name '*.spec.ts' → 0; no vitest.config.* / jest.config.* on disk. Confirmed independently by the audit worksheet checks/17-auth-core-shared.html.

The blocker: app is not importable without binding a port

  • src/index.ts (273 ln) constructs the Express app inline, wires every router, and calls app.listen(port, ...) at index.ts:244 inside a startServer() that runs on import. It has no export of the app object. Supertest needs the app handler without a live socket. This requires a small app/server split before any test can mount the app — see T2. (Flagged as a dependency/risk; not assumed trivial.)

Endpoint targets (real paths, mounted in src/index.ts)

  • POST /api/auth/login — routes/auth-password.ts → authPasswordController.login
  • POST /api/auth/otp/verify (and /validate) — routes/auth-otp.ts → controllers/auth-otp/AuthOtpController
  • GET /api/pms/file-manager (job/file list) — routes/pms.ts, guarded by protectedPms
  • GET /api/practice-ranking/... — routes/practiceRanking.ts (client-facing /latest; mounted at both /api/practice-ranking and /api/admin/practice-ranking in index.ts:194-195)
  • GET /api/gbp/... — routes/gbp.ts
  • An admin read or two, e.g. under /api/admin/app-logs or /api/admin/agent-insights (both bare-mounted, useful as auth-boundary smoke cases).

The mock seam

  • src/models/ — 77 model files; per audit dim-7 the intended contract is that all DB access flows through this layer. That makes models/ the clean seam to mock: stub the model functions a controller calls, and the endpoint runs end-to-end through routing + middleware + controller with no Postgres. (Caveat: the same audit found ~125 files bypass models/ with raw db("table") — endpoints whose controllers query db directly cannot be mocked at the model seam and either need a DB or get deferred. This is itself a signal of what to refactor first.)
  • Auth middleware (authenticateToken, rbacMiddleware, protectedPms) lives in src/middleware/; the audit rates the guards themselves as sound. Tests mint a valid JWT (the signing helper already exists — generateToken in controllers/auth-otp/feature-services/service.jwt-management) rather than mocking auth out, so the guard stays in the tested path.

Patterns to follow

  • No existing test convention to match — this plan establishes it. Pick one layout and document it so the cleanup PRs extend it instead of inventing parallel ones: src/__tests__/<domain>.smoke.test.ts, a single src/__tests__/helpers/app.ts (builds the app), and src/__tests__/helpers/auth.ts (mints a test JWT).
  • Reference file: src/index.ts — the app it builds is exactly what the test app-factory must reproduce minus listen(); the router/middleware wiring there is the source of truth for what to mount.

Constraints

Must

  • Pin every new dependency to an exact version (no ^/~) — the net must be reproducible across machines and CI.
  • Keep it a smoke net: assert HTTP status + response shape only. No deep business-logic assertions, no snapshotting whole payloads.
  • Tests run with no live Postgres and no outbound network (model-layer mocks; external SDKs — Anthropic, Google, Stripe, S3 — never hit in test).
  • Establish exactly one test layout + helper set and document it for the cleanup PRs to extend.
  • Mint a real JWT for protected-route tests so the auth guard stays in the tested path.

Must Not

  • Change any application behavior. The app/server split (T2) is the only source edit and must be behavior-preserving — npm run dev / npm start still boot and listen identically.
  • Add runtime dependencies — test tooling is devDependencies only.
  • Introduce a parallel assertion/validation lib — reuse the already-present zod for shape checks.
  • Stand up a real/throwaway test database in this phase (deferred — see Pushback).
  • Drive-by refactor controllers while writing their tests. Tests observe; they don't fix.

Out Of Scope

  • Exhaustive per-route coverage, error-path matrices, and edge-case fuzzing.
  • A disposable/migrated integration test DB (Strategy A) — revisited once the model-layer leaks are closed.
  • CI pipeline wiring, coverage thresholds/gates, mutation testing.
  • Worker (bullmq) and renderer-app tests — different harness, separate plan.

Risk

Level: 2 (Concern). The change is additive — new dev tooling plus a behavior-preserving app/server split. The two flagged risks are the new dependencies (justified below) and that the split touches the entry point.

New dependencies — justification (required by the conventions contract)

The contract discourages unjustified dependencies. These are justified and are the explicit point of the plan:

  • Vitest — the test runner. Chosen over Jest because the repo is ESM + TS executed via tsx; Vitest is ESM/TS-native and needs near-zero transform config, whereas Jest requires ts-jest/babel ESM plumbing that fights this toolchain. devDependency only.
  • Supertest — drives HTTP assertions against the in-process Express app with no bound port. It is the standard Express integration-test tool and is what makes the smoke net possible. devDependency only (with @types/supertest).

Both exist to be the refactor safety net. No runtime surface changes. Net new runtime deps: 0.

Risks Identified

  • App/server split touches the entry point. index.ts currently builds + listens in one file with no export; tests need the app without listen(). Mitigation: extract the app construction into src/app.ts that exports the configured app; reduce index.ts to import app from './app' + the existing startServer()/listen()/signal-handler block. Behavior-preserving; verify npm run dev and npm start still boot. Mounting order in app.ts must match index.ts exactly.
  • The model-mock seam leaks. ~125 files query db() directly (audit dim-7), so some target endpoints won't be fully isolatable by mocking models/. Mitigation: pick first-pass targets whose controllers already route through models/; for the rest, assert only the auth-boundary / validation response (e.g. 401/400 without a token/body), which needs no DB. Document which endpoints are "mock-blocked" — that list seeds the db-into-models plan's priority order.
  • OTP/login paths have crypto + side effects (email send, token mint). Mitigation: mock the model + email seams; assert the shape of success/failure, not real code delivery. Never call the live email sender.
  • Flaky env coupling — code reads process.env (JWT secret, etc.). Mitigation: a setupFiles entry sets deterministic test env (incl. an explicit JWT_SECRET) so signing/verifying is stable and the auth.ts fail-open fallback is never exercised in tests.

Blast radius: source — only src/index.ts (reduced) and new src/app.ts. Consumers of index.ts: the dev and start scripts and PM2 (signals-backend) — all invoke it as an entry point, none import from it, so the split is contained. New test files import from src/app.ts and existing models/middleware; they ship no runtime exports.

Pushback

  • Data strategy: A vs B. (A) disposable test DB — run knex migrations against a throwaway schema/database, hit real SQL. Highest fidelity; catches query-level regressions (exactly what db-into-models risks). Cost: Postgres in the test env + CI, migration run per suite, slower, more setup. (B) mock the models/ layer — stub model functions per test; no DB. Fast, hermetic, trivial to run anywhere. Cost: doesn't exercise real SQL, and is undermined wherever controllers bypass models/.
    Recommendation: B for this first pass — a fast, zero-infra net unblocks the refactors now and forces us to lean on the models/ seam (which is the very boundary the cleanup is trying to restore). Adopt A later, once db-into-models has pushed real query coverage behind models/, to add integration depth where it actually pays off. A real test DB now adds setup cost the refactor doesn't need yet — if B proves too thin for a given endpoint, fall back to A for that slice only.
  • Why not skip the harness and just refactor carefully? 500+ DB call-site edits and multi-thousand-line file splits across paid/auth endpoints is precisely where "careful" silently fails. A smoke net is the cheapest insurance; tsc alone proves nothing about runtime behavior.
