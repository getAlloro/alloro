Why

The admin Email Logs page is hand-rolled plain Tailwind — a bare <h1>, unstyled <select>/<input> filters, a static table, and a static modal. It ignores the shared admin design system that every sibling page (its direct analog AppLogs.tsx) already uses, so it looks out of place and unfinished. This is a pure presentation upgrade: match the sibling dashboard, animate the filters, and make the detail modal feel like the rest of the app. No behavior, API, or data changes.

What

Rebuild frontend/src/pages/admin/EmailLogs.tsx on the existing design-system primitives, decomposed into a feature folder (mirroring pages/admin/PracticeRanking/):

  • Heading → AdminPageHeader (navy Mail icon tile, animated entrance, description) — identical treatment to AppLogs.
  • Filters → AnimatedSelect for Category & Status; focus-animated date + search inputs inside a card with a staggered entrance; a Reset control when any filter is active.
  • Table → white card shell, Framer Motion staggered row entrance, hover-highlight rows, skeleton-row loading state, EmptyState for empty/error.
  • Modal → glass backdrop (backdrop-blur), spring in/out via modalVariants, icon + badges header, X close button, Escape-to-close. The sandboxed iframe preview is preserved verbatim (§17.4).
  • Layout bug → drop the page's own p-6 (the AdminLayout already pads content) and use space-y-6 like AppLogs.

Done when: the page visually matches the admin design language, all filters + table + modal animate, tsc is clean, and list/detail behavior is byte-for-byte the same as before.

Constraints

Must

  • Reuse existing primitives (AdminPageHeader, AnimatedSelect, EmptyState, ActionButton, animation variants).
  • Preserve the iframe sandbox="" preview exactly (untrusted stored HTML, §17.4).
  • Keep every filter/param/pagination behavior and the query hooks identical.
  • Each new file under 200 lines, single responsibility; aria-label on icon-only buttons; visible focus rings.

Must Not

  • No new npm dependencies.
  • No changes to api/email-logs.ts, the hooks, or any backend route.
  • No .minds-theme dark styling; no dark: utilities.
  • No raw fetch/axios, no any, no console.*.

Out Of Scope

  • Backend/API/schema changes; new columns or filters; CSV export; open-tracking accuracy.
  • App-wide prefers-reduced-motion gating (see Assumptions — matches existing admin pages).

Risk

Level: 1 (Suggestion) — presentation-only, reusing proven primitives.

Risks Identified

  • File/folder name coexistence (EmailLogs.tsx + EmailLogs/). Mitigation: identical to the working PracticeRanking.tsx + PracticeRanking/ pair; route import ./admin/EmailLogs resolves to the file.
  • Losing the security sandbox during the modal rewrite. Mitigation: keep sandbox="" and the untrusted-HTML comment verbatim; called out in T5 Verify.
  • AnimatePresence exit animation needs the modal mounted conditionally under the presence wrapper. Mitigation: wrap the conditional in the page, not inside the modal.

Blast radius: EmailLogs.tsx is imported only by pages/Admin.tsx (one route). New sub-files are consumed only by the page. No shared primitive is modified, so no other page is affected.

Assumptions (-i self-resolved)

  • Match the light admin theme, not the skill's dark default — the user asked to match the sibling dashboard, and dark navy on one admin page would clash with all siblings.
  • prefers-reduced-motion is not individually gated — no existing admin page does, and matching the sibling is the explicit goal. Revisit app-wide if desired.
  • Added a Reset filters affordance (low-risk UX win); no other new behavior.
