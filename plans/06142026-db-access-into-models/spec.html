Why

The conventions audit named DB-access-outside-models/ the single largest contract breach in the backend (dim 7). 126 non-model files import the raw connection and run ~507 direct db("table") / db.raw / db.transaction call-sites — controllers 96, services 12, workers 8, utils 8, routes 1. Query logic that lives in controllers and services is unmemoizable, untestable in isolation, duplicated across domains, and — because the writes leaked out with it — frequently runs without the transactions the owning model would have wrapped them in (dim 16). See checks/10-models.html for the full inventory.

Now, because: the audit is complete and ruled. The models/ layer is already exemplary (parameterized everywhere, transactions correct internally), so this is consolidation into a known-good layer — not a rewrite. It should be done while the audit's per-file file:line evidence is fresh.

What

A behavior-preserving refactor that moves every db(…) call out of controllers/, services/, utils/, workers/, and routes/ and behind a model in src/models/, reusing or extending the existing models. Done domain by domain: each domain is one task, one commit, one verification pass (smoke harness + npm run build + eslint). One concern per commit.

How we'll know it's done: a repo-wide grep for database/connection across src/controllers src/services src/utils src/workers src/routes returns 0 files (today: 126); npm run build stays green; the smoke harness passes after every domain; query results and side-effects are identical (same WHERE clauses, same column lists, same transaction boundaries — just relocated). No endpoint behavior changes.

Context

The target shape (already exists, copy it)

The models/ layer is the reference, not a thing to invent. src/models/BaseModel.ts defines QueryContext = Knex | Knex.Transaction and a static surface (findById, findOne, findMany, create, createReturningId, updateById, deleteById) that every method threads a trailing trx?: QueryContext through via this.table(trx). The canonical concrete example is src/models/GbpWorkItemModel.ts — every query method takes trx?: QueryContext and routes through this.table(trx); gbp-automation controllers/services call only these methods and hold zero inline db() (audit: 0 must-fix, the reference domain). Every extracted method must match this shape so callers can pass a transaction when they need one. Inside models/, ReviewModel.replaceApifyReviewsForPlace is the reference for a transaction-owning write (wraps del + upsert-loop in db.transaction).

The leak, by file

  • The handle: import { db } from "../../database/connection" (a knex instance; src/database/connection.ts), used dominantly as the query-builder call db("table")…. This token is neither knex nor .raw(, which is why the audit's narrow grep undercounted — the accurate surface is found by grepping the connection import.
  • Worst offenders (call-site density, from 10-models.html): controllers/admin-websites/feature-services/service.ai-command.ts ~106 · controllers/admin-websites/AdminWebsitesController.ts ~40 · controllers/practice-ranking/PracticeRankingController.ts ~33 · service.project-manager.ts ~33 · workers/processors/websiteRestore.processor.ts ~32 · service.page-editor.ts ~31 · service.menu-manager.ts ~28 · controllers/pm/PmTasksController.ts ~27 · controllers/pm/PmAiSynthController.ts ~24 · routes/checkup.ts ~22.
  • Existing models to extend, not duplicate: the admin-websites domain already has 33 models under src/models/website-builder/* that the feature-services bypass (audit, check 01). Most domains already have the right model — the work is adding methods, not adding model files.

Patterns To Follow

  • Extend the model that already owns the table; only create a new *Model.ts when no model owns it. New model files match GbpWorkItemModel.ts structure, naming (*Model suffix, descriptive query-fn names), and trx?: QueryContext threading.
  • Move the query verbatim first (same builder chain, same columns, same .raw() string), then name the method for what it fetches/writes. Do not "improve" the query while moving it.
  • When relocating a multi-table write, the destination model method owns the db.transaction (or accepts an injected trx); the caller passes one if it is composing several writes. Mirror ReviewModel.replaceApifyReviewsForPlace.
  • Leave existing console.*, response-shape, validation, and naming issues alone — those are separate audit dimensions with their own remediation plans. This plan touches dim 7 (and the dim-16 transactions that fall out of it) only.

Dependency — sequence after the smoke harness

  • This plan should run after plans/06142026-backend-smoke-test-harness exists. Each domain's verification step assumes that harness as the behavior-preserving safety net. It does not exist on disk yet — without it, every domain falls back to slow, error-prone manual endpoint verification, which is not acceptable at this volume. Do not start this plan until the harness is in place. (Build the harness first, or this plan's Verify steps degrade to manual.)

Reference file: src/models/GbpWorkItemModel.ts — closest existing analog for the method shape every extraction must match. Reference domain: gbp-automation (controllers route 100% through models, 0 inline db()) is the end-state every domain in this plan is converging toward.

Constraints

Must

  • Be behavior-preserving — same query semantics, same transaction boundaries, same return values. Relocation, not redesign.
  • Reuse/extend the existing model that owns the table; match GbpWorkItemModel.ts + BaseModel trx?: QueryContext shape.
  • Commit per domain (one concern per commit) and run the smoke harness + npm run build + eslint after each domain before moving on.
  • Where a leaked multi-table write had no transaction, the destination model method adds one (dim 16) — this is the one behavior the move is allowed to tighten, because it only prevents partial-write corruption.

Must Not

  • Change query logic, column selection, ordering, or filters while moving a query.
  • Fix unrelated audit dimensions in the same commit (no console.log→Pino, no response-shape, no validation, no renames, no file-size splits, no auth).
  • Add new dependencies or new patterns; the model layer's pattern already exists.
  • Touch frontend/, database/migrations, or seeds.
  • Big-bang the whole repo — domains are isolated and verified independently.

Out Of Scope

  • God-controller decomposition / file-size remediation (separate plan — admin-websites, user-website).
  • Misplaced-services relocation (utils/ → services/) — that is the utils plan; here we only pull their db() into models, in place.
  • Pino adoption, {success,data,error} response shape, input validation, auth-mounting, secret rotation — all separate audit-driven plans.
  • Naming drift (service.kebab.ts / missing Service suffix).

Risk

Level: 3 (Structural Risk) — high-volume, cross-cutting refactor (~126 files / ~507 sites). The risk is not the destination (the model layer is sound) but the volume: a subtly altered builder chain, a dropped .where, or a moved write that changes a transaction boundary can silently corrupt behavior, and the blast radius spans most of the backend.

Risks Identified

  • Subtle query-behavior drift during relocation (the headline risk). A copied query that loses a clause or changes return shape passes tsc but breaks at runtime. Mitigation: move queries verbatim before renaming; do not refactor the chain; rely on the smoke harness after every domain; keep each commit to one domain so a regression is bisectable to a single, reviewable diff.
  • No test net yet. The smoke harness (06142026-backend-smoke-test-harness) does not exist. Mitigation: hard prerequisite — do not begin until it lands. Until then, each domain would need full manual endpoint verification (slow, unreliable at this scale). This is the primary pushback below.
  • Transaction boundaries move with the writes. AdminWebsitesController.ts has 4 correct multi-table db.transaction blocks (addProjectLocation L5105, setPrimaryLocation L5212, removeProjectLocation L5296, + L597); relocating them must preserve atomicity, and the model method must own the trx. Mitigation: extract the whole transaction as one model method; pass an injected trx only where the caller composes further writes.
  • Layer-inversion landmine. models/BehavioralEventModel.ts imports services/behavioralIntelligence.ts (audit check 11) — a model calling up into a service. Pulling that service's db() into a model risks a circular import. Mitigation: when the services domain is reached, invert the dependency (service → model) as part of that domain's task; do not deepen the inversion.
  • Type-only false positives. AdminOrgCreationService.ts, TierManagementService.ts, OrganizationBootstrapService.ts import Knex only to type an injected trx (correct pattern), and routes/billing.ts matches on express.raw() not DB. Mitigation: these are NOT leaks — skip them; the connection-import grep already excludes them (they import the type, not db).

Blast radius: every domain whose controllers/services currently hold queries — admin-websites (28 controller files), pm (9), practice-ranking (9), agents (8), user-website (3), leadgen-tracking (4); top-level services/ (9 files), utils/ (8), workers/ (8), and routes/checkup.ts. Consumers are the route handlers that call these; behavior-preservation keeps them unaffected, but they are the verification surface. The models/ layer itself gains methods (additive) — existing model consumers are not changed.

Pushback

  • Do not attempt this without the test net. A behavior-preserving refactor at 507 sites is exactly the case where "looks the same" is a trap. Recommended path: (A) land 06142026-backend-smoke-test-harness first, then run this plan domain-by-domain against it — recommended. Alternatives if the harness slips: (B) scope this plan to read-only queries first (lower risk, still valuable) and defer write/transaction moves until the harness exists; (C) manual endpoint verification per domain — rejected at this volume, too slow and miss-prone. Recommendation: A.
  • This is the largest sweep in the cleanup. Sequence it after the harness and after (or alongside) the smaller dimension fixes so the diffs stay legible; do not interleave it with the god-controller decomposition, or the two refactors will collide in the same files and neither will be reviewable.
