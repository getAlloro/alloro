Why

PR 150 adds a useful review-volume comparison to the Summary agent, but it is not safe to merge as written. The customer metrics endpoint trusts tenant IDs from the query string, unfinished competitor sets can be treated as curated, median standing is mislabeled as market leadership, generator prose lacks deterministic evidence validation, operational failures look identical to “no competitors,” and the generated comparison is not rendered to the customer.

What

Harden PR 150 so a newly generated Summary can show one compact, read-only review comparison on the Practice Hub using only a finalized competitor set and validated evidence. Secure the metrics endpoint with server-derived tenant context, preserve prior Summary outputs without migration or backfill, keep the existing top-action selection behavior unchanged, and hide the comparison whenever the newest Summary lacks grounded Choosable data.

Constraints

  • Derive organization and accessible locations from authenticated request context; accept only a scoped locationId from the customer request (§5.5, §11.7).
  • Use only a finalized active competitor set; pending, curating, empty, and failed sources must be distinguishable.
  • Replace the ambiguous leadership flag with separate question-shaped booleans for “at or above median” and “has the most reviews” (§1.3).
  • Keep old Summary outputs valid by making new domain-summary evidence optional at the schema boundary.
  • Require grounded Choosable evidence before the new customer strip renders.
  • Select the strip from the newest Summary task by created_at with a deterministic ID tie-break; never fall back to an older comparison.
  • Keep the existing highest-priority hero action behavior unchanged.
  • Use Pino with organization/location context and no any in new code.
  • Update Alloro Docs in its separate working tree.

Must Not

  • Do not change ranking agent inputs, ranking formulas, rank_position, or SerpApi search_position.
  • Do not write directly to the database outside models (§7.4).
  • Do not collapse an operational failure into the same payload as an unconfigured competitor set (§3.2).
  • Do not render a comparison from an older Summary when the newest Summary lacks evidence.
  • Do not auto-backfill tasks, rerun agents, touch production data, deploy, merge, or push during execution without the later gate.
  • Do not modify the unrelated dirty main checkout.

Compatibility Contract

  • No database migration or data rewrite.
  • Prior Summary metadata without domain-summary evidence continues to parse.
  • Prior generations show no comparison strip.
  • The comparison becomes visible only after a new grounded Summary generation.

Risk

Level: 3 — the plan closes a tenant-isolation vulnerability and changes an AI-generated customer-facing read. The mitigations are explicit and testable, so execution can proceed only after this spec is approved.

Risks Identified

  • R1 — Cross-tenant metrics read (Level 3). Query-provided IDs can select another organization/location. Mitigation: authenticated RBAC + location scope, server-derived organization, required scoped location, canonical access-denied response, and a negative tenant test.
  • R2 — Uncurated comparisons (Level 2). Active discovery rows may exist before the owner finalizes the set. Mitigation: copy the finalized gate used by the ranking resolver and return a typed not-ready status otherwise.
  • R3 — False “leader” wording (Level 2). Being above the median does not mean leading the strongest competitor. Mitigation: separate median standing from true maximum standing. Test practice 550 versus competitors 100 and 1,000: at/above median is true; market leader is false.
  • R4 — Fabricated prose (Level 3). Domain-summary names and numbers are not currently walked by the evidence validator. Mitigation: optional typed evidence plus deterministic name/value validation against dashboard_metrics.choosable; invalid output triggers the existing Summary retry path.
  • R5 — Stale comparison (Level 2). Task selection can retain an older high-priority Summary. Mitigation: preserve the hero selector, but independently select comparison copy only from the newest Summary generation and hide on missing evidence.
  • R6 — Total Summary outage from optional competitor data (Level 2). Throwing on a competitor read could remove otherwise valid monthly output. Mitigation: log the operational error and return an explicit unavailable status, which the prompt and UI must not present as a real comparison.
  • R7 — Docs drift (Level 2). A new Practice Hub section without replica/copy parity leaves support guidance wrong. Mitigation: update both Practice Hub docs files and verify the docs build separately.

Blast radius: monthly Summary input/output for all future runs; the read-only dashboard metrics endpoint used by ProductionPanel, StatCardRow, and RankingsHubSurface; the task feed used by OneThingBanner; the Practice Hub layout; onboarding demo behavior; and Practice Hub documentation. No worker cadence, ranking calculation, database schema, or existing task row is changed.

Pushback

  • Recommended path A: hide the strip unless the newest Summary carries validated Choosable evidence. This is the selected path because it prevents stale or legacy copy from looking current.
  • Rejected path B: fall back to any older Summary with comparison data. This would make the page look complete while silently showing stale competitor counts.
  • Rejected path C: render deterministic metrics directly and bypass the Summary agent. That would change the product decision that the customer sees the Summary agent's contextual read.
