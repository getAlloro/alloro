Decision Required — Read First

This plan resolves a contradiction inside the contract itself. The /alloro-conventions contract (dimension 10) mandates Pino with pino-http and forbids console.log. But the codebase does the opposite: Pino is not installed (absent from package.json; express ^5.1.0 is there, no pino, no pino-http), there are zero pino imports in src/, and 277 of 922 .ts files use console.* as the de-facto logger. The contract and the code cannot both be right. The owner must pick a direction before the codemod runs — the spec cannot pick for you.

Recommendation: Option A. The deciding factor is not log aesthetics — it is the verified security finding. AuthPasswordController.ts:317 logs a live OTP and the user's email in plaintext (console.log(`[AUTH] Verification code for ${email}: ${code}`)), and OAuthFlowService.ts:50,62,92 log OAuth tokens. Pino's redact is a real, configured, app-wide mitigation that also covers future careless log lines; Option B's hand-rolled redaction does not, and only delays adopting a real logger. Picking A also makes the code match the contract instead of weakening the contract to match the code. Both options carry the same ~277-file codemod, so A's marginal cost over B is just two pinned deps and the pino-http wiring — cheap for the upside. The tasks below are written for Option A; if the owner chooses B, drop T1/T3-wiring and swap the logger module body for the console wrapper (the codemod task T4 and verification T6 are identical).

Why

The backend conventions audit (plans/06142026-backend-conventions-audit) flagged this as one of 5 systemic, repo-wide breaches, and check 17 (auth-core-shared) recorded it as a must-fix against dimension 10. The contract mandates Pino; Pino is not installed and console.* is everywhere. Left unresolved, every future backend PR either silently violates dimension 10 or gets blocked by it for no reason. Resolve the contradiction once, repo-wide, and stop the bleeding — including the plaintext-PII logging that rides on top of it.

What

A single owner decision (A or B) plus its mechanical follow-through: one logger module, request-logging wired into the bootstrap, and a repo-wide codemod that replaces console.* with the logger across the ~277 offending files — isolated in its own commit, separate from the logic/wiring changes. Done when: the decision is recorded in the convention doc, the logger module exists with redaction configured, request logging is live, and a verification grep finds zero raw console.* in src/ outside the logger module itself (excluding database/migrations and seeds, which are out of the audit's scope).

  • The codemod must respect the audit's exclusions: do not rewrite src/database/migrations/** or seeds/** (different rules; out of scope), and do not rewrite the frontend (frontend/ — separate /alloro-frontend contract).

Reference file: src/controllers/agents/feature-utils/agentLogger.ts — closest existing analog for a leveled logger wrapper; match its export/usage shape for the new src/lib/logger.ts.

Constraints

Must

  • Land the decision in the convention doc (dimension 10) before/with execution — spec-code-contract parity.
  • Isolate the ~277-file codemod in its own commit, separate from the logger-module + wiring + redaction commit (see Pushback).
  • Pin exact dependency versions — no ^ / ~ ranges for pino / pino-http (Operating Rules).
  • Configure redaction for secrets/PII (OTP code, password, token/accessToken/refreshToken, authorization, email/to/recipient) in the logger module.
  • Run the verification grep and confirm zero raw console.* outside the logger module.

Must Not

  • Mix logic/feature changes into the codemod commit — codemod = pure mechanical replacement.
  • Rewrite console.* in src/database/migrations/**, seeds/**, or frontend/**.
  • Refactor, rename, or relocate the existing domain loggers, or touch the index.ts CORS block — out of scope.
  • Add a second logging abstraction — one shared logger only (no parallel systems).
  • Change log destinations (files in src/logs/, Sentry) — that is a follow-up, not this plan.

Out Of Scope

  • Deleting the OTP-in-log line itself — owned by the security-hotfix (coordinate, don't duplicate).
  • Log shipping / aggregation / Sentry transport, log rotation, per-env log levels beyond a basic env switch.
  • The other 4 systemic breaches (validation, DB-outside-models, response shape, transactions) — separate plans.

Risk

Level: 2 (Concern) — the codemod is mechanical with low logic risk (a token-for-token call swap), but it is wide (~277 files) and changes a contract-level standard, so it is not zero-risk.

Risks Identified

  • A 277-file codemod is noisy in git and will bury any real change reviewed alongside it. Mitigation: isolate the codemod in its own commit (refactor: codemod console.* -> logger) with nothing else in it; put the logger module, wiring, redaction, and doc edit in a separate prior commit. Reviewers diff the small commit; the big one is a rubber-stamp.
  • Mechanical replacement can mangle edge cases — console.table, console.group, console.assert, multi-arg calls, or console referenced as a value. Mitigation: use an AST codemod (jscodeshift / ts-morph), not a regex; map only log/info/warn/error/debug; leave anything unmapped for a manual sweep and list it in the commit body.
  • Pino redaction misconfigured could still leak, or over-redact useful context. Mitigation: unit-assert the redact paths against a sample payload containing code/token/email before the codemod lands; coordinate the exact field list with the security-hotfix owner.
  • pino-http on Express 5 — confirm version compatibility at install (Express 5 changed router internals). Mitigation: pin a pino-http version with documented Express 5 support; smoke-test one request path locally before the codemod.

Blast radius: ~277 .ts files in src/ (every console.* consumer), src/index.ts (one middleware registration), package.json + lockfile, the 5 domain-logger wrappers (internals re-pointed), and the /alloro-conventions SKILL doc (dimension 10 text). No runtime behavior changes beyond log format/redaction.

Pushback

  • Do not commit the codemod and the logic together. If the logger module, the index.ts wiring, and the 277-file sweep all land in one commit, the meaningful 30 lines are invisible inside a 277-file diff and the change becomes unreviewable and unbisectable. Two commits, always: logic first, codemod second.
  • If the owner leans Option B to "avoid new deps," push back: the dep count (2) is trivial next to re-litigating PII redaction by hand forever and documenting drift away from a prod-grade standard. B is the worse long-term trade.

Change: Executed Option A (adopt Pino). Installed pino@10.3.1 + pino-http@11.0.0 (prod, exact-pinned) and pino-pretty@13.1.3 + ts-morph@27.0.0 (dev, exact-pinned). Created src/lib/logger.ts (configured pino instance, default + named levels, env-driven level, dev-only pino-pretty, exported REDACT_PATHS / REDACT_OPTIONS) with redaction over authorization/cookie, token/accessToken/refreshToken/secret, password, OTP code/otp, and email/to/recipient (top-level + *.key + req/res header shapes). Wired pino-http as the FIRST middleware in src/app.ts (the importable app factory — the bootstrap moved out of src/index.ts since the spec was written), gated off under VITEST. Ran an AST codemod (ts-morph) mapping console.{log→info,info,warn,error,debug} → shared logger across src/, with console.error(strMsg, err) → logger.error({ err }, strMsg). Added a redaction unit test (src/__tests__/logger.redaction.test.ts) and re-pointed the validate-middleware test from console spies to logger spies.

Reason — Execution deviations (vs spec assumptions):

  • Scope was per-call larger than stated. The spec's "~277 files" counts files; the actual mechanical surface was 1,373 console.* call-sites across 243 files (codemod), with only log/warn/error present (no console.info/console.debug existed). 604 of those were the console.error(msg, err) → { err } rewrite.
  • The "5 domain loggers" were 4. controllers/pm/pmActivityLogger.ts writes to a DB model and contains no console.*; the real console-wrapping loggers (agentLogger, ranking-logger, scraper-logger, rag-logger) had their internals re-pointed by the codemod as intended.
  • False-positive guard held. A naive regex flags searchconsole.sites / searchconsole.searchanalytics (Google Search Console SDK). The AST codemod matched only the global console identifier, so those 4 lines were correctly left untouched.
  • 4 call sites needed manual handling (codemod-excluded): three practice-ranking services bind a logger PARAM / use = console.log as a default-param value (service.identifier.ts, service.ranking-pipeline.ts, service.competitor-source-resolver.ts) — fixed by importing the shared logger as appLogger and wrapping (msg) => appLogger.info(msg) to avoid shadowing.
  • Pino's strict LogFn typing forced a follow-up arg codemod. Pino types only allow trailing args that match printf specifiers parsed out of a string-literal message, so the bulk's logger.X("msg", data) output (66 calls / 39 files) was a type error. A second AST pass rewrote string-first multi-arg calls to pino merge-object form: 1 trailing arg → { err: x } (error level / error-named) or { detail: x }; N trailing → { details: [...] }. Behavior-preserving (output form changes only).
  • validate.ts hand-corrected to idiomatic pino. Its warn/error calls were converted to merge-object-first (logger.warn({meta}, msg) / logger.error({ err, ...meta }, msg)) so the redaction-safe metadata is actually emitted rather than dropped as an unformatted printf extra.
  • Redaction test capture strategy. Pino writes via sonic-boom to fd 1, which a process.stdout.write spy cannot intercept; the unit test builds a logger from the exported REDACT_OPTIONS over an in-memory stream to assert the SHIPPED path list.
  • Residual. Exactly 1 console.* remains in app code — a browser-side console.error inside an HTML <script> string in controllers/googleauth/utils/templates/successTemplate.ts (runs in the user's browser, not Node) — correctly NOT repointed. Seeds + tests excluded per scope.
