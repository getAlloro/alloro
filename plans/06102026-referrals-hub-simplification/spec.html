Why

The Referrals Hub (/pmsStatistics) is dense: an AI executive-summary card, growth-opportunity cards, a doctor/self mix card, a velocity card, a full ranked-sources modal, and a chart carrying its own headline + "view pace" chrome — all orchestrated by the ~1500-line PMSVisualPillars.tsx. This is the second of four dashboard simplifications (after Practice Hub). It collapses the analytics view into: header + period toggle → one dual-line chart → four stat tiles → upload CTA → top-3 sources w/ trend → one action line.

What

Build a new slim surface PmsHubSurface.tsx as a drop-in for PmsDashboardSurface (identical props), rendered by PMSVisualPillars. Done when:

  • Header reads "REFERRALS HUB" with an interactive MONTH / QTR / YTD segmented toggle.
  • A dual-line (production + referrals) chart re-buckets per the selected period.
  • Four stat tiles: production (period-scoped, relabels THIS MO / THIS QTR / YTD), REFERRALS (period-scoped), SOURCES (active count), YTD (fixed annual reference).
  • Upload CTA preserved (reuse PmsIngestionCard).
  • Top-3 sources show rank · name · refs · % · trend arrow (trend from Referral-Engine trend_label, name-matched, — fallback).
  • One "1 ACTION" banner: "Protect your top N sources — X% of all referrals" (frontend heuristic).
  • Heavy sections no longer rendered; PMSVisualPillars plumbing (upload/automation/file-manager) untouched.
  • Onboarding wizard not broken. npx tsc --noEmit + lint clean.

No backend change — toggle aggregation + action are frontend-derived; trend reuses already-fetched RE data.

Constraints

Must

  • New surface accepts the identical PmsDashboardSurfaceProps.
  • Keep usePmsKeyData, the Referral-Engine fetch, automation polling, file-manager plumbing.
  • Period aggregation + the action heuristic are pure frontend (from monthlyData/topSources).
  • Reuse PmsIngestionCard for upload; preserve its pms-upload wizard target.
  • Leave retired component files on disk; only stop rendering them.

Must Not

  • No backend change. No new dependencies.
  • Do not gut or rewrite PMSVisualPillars — change only the render line + retire heavy child renders.
  • Do not touch other dashboard tabs or the Referral Engine page.
  • Do not remove the RE fetch (owner kept it for trends + action).

Out Of Scope

  • Backend per-source trend on sources[]; quarter/YTD backend params.
  • Rankings + Reviews&Posts redesigns (later passes).
  • Redesigning the file manager, automation timeline, or manual-entry modal.

Risk

Level: 3 (Structural — swapping the surface inside a 1500-line orchestrator on the primary analytics page).

Risks Identified

  • R1 — Orchestrator swap. Mitigation: drop-in with identical props; only PMSVisualPillars.tsx:1377 changes; plumbing untouched.
  • R2 — Fuzzy trend matching. RE source_label/referrer_name may not equal keyData sources[].name. Mitigation: normalized (trim/lowercase) match; — fallback; accept partial coverage.
  • R3 — Toggle "both" semantics are a best-guess contract (see Tasks T2). Mitigation: documented + tunable.
  • R4 — Minimal network reduction. RE fetch kept by decision → win is render weight, not fewer calls. Not a regression; stated for honesty.
  • R5 — Wizard re-point. pms-insights/pms-velocity targets lose their elements. Mitigation: re-point pms-insights → action banner, trim pms-velocity, keep pms-vitals (stat tiles) + pms-upload.

Blast radius: PMSVisualPillars.tsx (1 render line), new surface + helpers, wizardConfig.ts. Retired components (PmsReferralsMeaningCard, PmsGrowthOpportunities, PmsReferralMixCard, PmsVelocityCard, PmsTopSourcesCard, PmsProductionChart, PmsDashboardSurface) retained on disk; no other consumers verified before retiring.

Pushback

  • Reusing PmsIngestionCard keeps its 12-month availability calendar — heavier than the mockup's bare dashed CTA. Recommended (lower risk, preserves function); flag if you want the lean CTA instead.
  • If real fetch reduction matters more than keeping AI-derived trends, reconsider dropping the RE fetch (would move trend to backend or drop arrows).
