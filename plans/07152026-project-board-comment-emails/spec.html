Why

The PM board already supports task comments, mentions, task attachments, and in-app notifications, but the comment composer is not obvious enough and important collaboration events do not leave email evidence. Admin users need a clearer comment flow, image comments, and email delivery when someone is mentioned or a related card is moved.

What

Improve the PM task comment experience, add image attachments directly to comments, and send PM notification emails for comment mentions and meaningful card movement. Email is best-effort and recorded through the existing email service/logs; local and dev sends remain protected by the email interceptor.

Constraints

Must

  • Use a migration for any schema change.
  • Keep S3 delivery behind existing presigned URL endpoints.
  • Send mention emails to the actual mentioned internal users, not a broad admin blast.
  • Send movement emails only for cross-column moves and cross-project moves; same-column reorders are noise.
  • Use existing email interception so local/dev does not send live email.
  • Keep image-comment validation server-side.

Must Not

  • Do not redesign the whole PM board in this pass.
  • Do not introduce a new email transport or storage bucket.
  • Do not send email inside a DB transaction.
  • Do not render uploaded images or markdown through raw HTML.

Out Of Scope

  • Realtime push, notification preferences, Slack-style threads, watchers/subscribers, and customer-facing docs updates.

Risk

Level: 3 - schema change plus email side effects on a production-facing admin workflow.

Risks Identified

  • Email noise. Movement can happen often. Mitigation: email only cross-column and cross-project moves, skip same-column reorder, skip actor self-email, de-duplicate recipients.
  • Email inside transaction. Sending before commit could notify about rolled-back work. Mitigation: commit first, then call best-effort email service.
  • S3 orphan images. A comment may be created but image upload may fail. Mitigation: upload images after comment creation, show inline upload error, and keep the text comment intact.
  • Image abuse. Client MIME hints are not enough. Mitigation: if comment_id is present, backend accepts image MIME types only and verifies the comment belongs to the task.
  • Component sprawl. Existing comment components are already large. Mitigation: split composer/menu/image strip/markdown/item helpers into PM feature components instead of adding more bulk.

Blast radius: PM task comments, PM task attachments, PM task movement, PM notification emails, PM detail panel comment tab, and email logs.

Pushback

  • Do not email every drag event. Future-us will hate that inbox. Only movement that changes task state or project context gets email.

Done

  • Comment composer shows mention affordance, selected mention chips, image picker, paste/drop image support, previews, and disabled/loading send state.
  • Comment images persist as S3-backed attachments tied to the comment and render under that comment.
  • Mention emails are sent to mentioned internal users after comment create and newly-added mentions after edit, with no self-email and no duplicates.
  • Movement emails are sent for cross-column and cross-project moves, with no same-column reorder email and no self-email.
  • Local/dev email sends are intercepted by the existing email interceptor and logged through email_logs.
  • The current unified task conversation feed remains intact: standalone task attachments stay chronological in the feed while comment-owned images render only under their comment.
  • TypeScript, conventions, and targeted tests pass or only pre-existing advisory debt remains.
