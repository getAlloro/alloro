Why

Client websites served by the shared renderer are being flooded by datacenter bots (headless Chrome from AWS us-west-2, plus internet-wide scanners). For One Endodontics this turned ~1,300 real June visitors into ~21,000 reported, wrecking the dashboard and conversion math. A one-time data scrub already cleaned June, but the bots keep arriving daily and re-inflate it. The only durable fix is to stop the bots at the edge so they never load the page, never reach Rybbit, and never cost renderer CPU.

Hard constraints from the owner: the fix must never block legitimate SEO crawlers (Googlebot/Bing — an SEO outage is unacceptable) and must never block real visitors / potential clients.

What

Add an IP-origin block to the renderer's Caddy (alloro-renderer, 54.159.199.33) that denies requests originating from cloud/hosting networks (AWS + GCP + Azure + DigitalOcean + Oracle + Hetzner + OVH), while explicitly allowing our own infrastructure and verified search crawlers. Real visitors come from ISP/mobile networks and are structurally never in these ranges; Googlebot/Bing run on Google/Microsoft networks (not the blocked clouds) and are additionally allowlisted.

Done = a 7-day dry-run (log-only, nothing blocked; checked automatically each day by a cron that emails the verdict) proves the would-block set is all bots — zero Googlebot/Bing, zero engaged/converting visitors — then enforcement is switched on; all client sites keep loading; Googlebot still crawls; new-day Rybbit bot volume drops to baseline. Coverage = Approach B (multi-cloud CIDR), all client sites, via native Caddy remote_ip matchers (no custom binary).

Context

Target system (verified by recon)

  • alloro-renderer (54.159.199.33, AWS us-east-1) — serves every client domain. DNS points straight here (GoDaddy), no Cloudflare in front, so remote_ip = the real client IP.
  • /etc/caddy/Caddyfile — Caddy v2.11.1 (systemd service). Single catch-all https:// block → reverse_proxy localhost:7777 (pm2 website-builder). On-demand TLS via ask http://localhost:7777/verify-domain.
  • Built-in matchers http.matchers.remote_ip / client_ip are present. No MaxMind/ratelimit plugins → ASN-by-number blocking would need a custom binary; CIDR blocking works today.
  • Caddy currently writes no access log (only TLS/reverse-proxy warnings to journald) — the dry-run requires enabling one.

Blocklist source-of-truth (published ranges)

  • AWS ip-ranges.amazonaws.com/ip-ranges.json; GCP gstatic.com/ipranges/cloud.json (compute only — NOT goog.json, which contains Google services); DigitalOcean CSV; Oracle public_ip_ranges.json; Azure ServiceTags JSON; Hetzner/OVH via ASN→prefix derivation (RADb/BGP).

Reference analog: this is an infrastructure change on the renderer box, outside the app code trees (src/, frontend/), so the Code Constitution (backend §6 / frontend §12) does not govern it. It follows the renderer/ops + AGENTS.md deploy-safety conventions instead. Closest prior analog = the alloro-site nginx vhost handling (and its .bak-in-config landmine lesson: never leave backups where the server parses them).

Constraints

Must

  • Allowlist evaluated before the block; ACME path /.well-known/acme-challenge/* always exempt so on-demand cert issuance for new client domains never breaks.
  • Block only at the HTTP layer (never the TLS layer) so TLS-ALPN cert challenges are unaffected.
  • Run a 7-day dry-run (log-only), checked automatically each day by a renderer cron that emails a CLEAN/NOT-CLEAN verdict, and pass its acceptance gate before any enforcement.
  • caddy validate must pass before every reload; reloads are graceful (zero-downtime); keep a timestamped backup of the working Caddyfile outside /etc/caddy.
  • CIDR refresh script must validate the generated config before reloading; on failure keep the last-good list and alert — never reload an unvalidated config.

Must Not

  • Block Googlebot/Bing or any verified search crawler. Verify Google by published ranges + reverse-DNS, never by user-agent (UA is spoofable).
  • Block ISP/mobile (real visitor) traffic.
  • Block Google's crawler network — only GCP compute ranges (cloud.json), never Google services (goog.json).
  • Rebuild/replace the Caddy binary (no MaxMind plugin this round — config-only change).
  • Leave .bak/backup files inside /etc/caddy where Caddy would parse them.

Out Of Scope

  • MaxMind ASN-by-number blocking (custom binary) — future graduation if CIDR upkeep gets heavy.
  • Secret-header allowlisting of our own scrapers (needs app-code change) — phase 1 allowlists our tools by Elastic IP instead.
  • Proof-of-work / JS challenge (e.g., Anubis) — future backstop if bots ever move to residential IPs.
  • Rate-limiting (needs a plugin).
  • Cleaning historical Rybbit data for the other client sites (separate task).

Risk

Level: 3 — Structural/availability + SEO. The change is on the single box that keeps every client site online, and a crawler mistake risks deindexing.

Risks Identified

  • SEO deindexing if a crawler is blocked. Mitigation: AWS/GCP-compute/etc. ranges structurally exclude Googlebot; additionally allowlist Googlebot/Bing published ranges; dry-run proves zero crawlers in the would-block set; post-enforce monitor Search Console + alert on any 403 to an IP that reverse-DNS-verifies as Google/Bing.
  • Real visitor blocked (e.g., a patient on a datacenter-exit VPN). Mitigation: dry-run flags engaged/converting sessions (form submits, multi-page) inside the would-block set; if material, soften to a challenge or allow for that range. Residual low for a local dental practice.
  • All-sites outage from a bad config. Mitigation: caddy validate gate + graceful reload + working backup outside /etc/caddy + instant rollback.
  • Cert issuance breaks for new domains. Mitigation: exempt the ACME path; block is HTTP-layer only.
  • Stale/broken CIDR list under-blocks or breaks a reload. Mitigation: refresh script validates before reload; keeps last-good on failure; alerts.
  • Provider list sourcing — AWS/GCP/DO/Oracle have clean JSON/CSV; Azure's URL rotates; Hetzner/OVH need ASN→prefix derivation. Mitigation: ship AWS/GCP/DO/Oracle first; add Azure/Hetzner/OVH as derivable, defer any that can't be fetched reliably and log the gap.

Blast radius: every domain served by alloro-renderer — all client sites (One Endo, Caswell, Garrison, Surf City, Tri-City, …), all *.sites.getalloro.com previews, the landing page, and the on-demand TLS issuance flow. Our own AWS-hosted tools that fetch the sites (audit scraper on 52.203.199.155) are protected by the EIP allowlist.

Pushback

  • CIDR lists are reactive (bots can hop to an un-listed network). Accepted for phase 1 because coverage is a dial we can widen anytime and the dry-run + monitoring catch drift. The durable end-state is MaxMind ASN blocking (out of scope now, by owner choice, to avoid a binary swap on the shared box).
  • Allowlisting our scrapers by Elastic IP (not a secret header) is slightly less robust if an EIP changes, but avoids an app-code change this round. Revisit with the secret-header approach if EIPs churn.

Install as a weekly cron. Last-good retained on failure + alert.

/opt/alloro/edge-block/refresh-cidrs.sh, generated /etc/caddy/snippets/datacenter-cidrs.caddy, cron entry (version-controlled copy in the website-builder-rebuild repo)
Depends on
none

Script produces a non-empty, caddy validate-clean snippet; spot-check known bot IPs (44.252.x, 35.93.x) fall inside; known Googlebot IP falls outside.

T3: Allowlist snippet (our infra + verified crawlers + ACME)

Depends on
T2 (shares the refresh/validate harness)

Each EIP + a sample Googlebot range present; caddy validate clean.

T4: Dry-run analysis — automated daily, 7-day window

Do
Script that reads the access log + the T2/T3 CIDR lists and reports the would-block set = requests in datacenter ranges AND not allowlisted. Breaks down by network/ASN and UA; reverse-DNS-verifies and flags any Googlebot/Bing in the set; flags any IP showing engagement (form submit / multiple distinct paths); emits a one-line CLEAN/NOT-CLEAN verdict. A daily cron on the renderer runs it and emails the summary via the existing n8n emailService webhook (the same path the worker-watchdog uses to email dave@getalloro.com). Run 7 days to cover a full weekly crawl/traffic cycle.
Files
/opt/alloro/edge-block/analyze-dryrun.sh, daily cron entry, emailService webhook env
Depends on
T1, T2, T3
Verify
Acceptance gate: 7 consecutive daily reports show the would-block set is ~the bot volume with zero verified Googlebot/Bing and zero engaged/converting visitors. Any non-clean day resets the count after the allowlist is fixed.

T5: Enforce

Do
Add the block handler: @block = datacenter AND not-allowed AND not-ACME → respond 403 (or abort); everything else proxies as today. Back up the working Caddyfile outside /etc/caddy, caddy validate, graceful reload.
Files
/etc/caddy/Caddyfile
Depends on
T4 (gate passed)
Verify
Several client domains return 200 from a normal browser; a simulated AWS-range IP gets 403; an EIP and a Googlebot-range request pass; ACME path reachable.

T6: Monitoring, alerting, rollback

Do
Alert if any 403 hits an IP that reverse-DNS-verifies as Google/Bing (allowlist gap). Check Google Search Console for crawl-error spikes. Confirm Rybbit new-day bot volume drops to baseline across sites. Document the one-line rollback (restore backup Caddyfile + reload).
Files
/opt/alloro/edge-block/verify-crawlers.sh, runbook note in the plan folder
Depends on
T5
Verify
48h post-enforce: no verified-crawler 403s, no Search Console crawl errors, Rybbit baseline holds, all sites up.
