Why

Four renderer-level gaps confirmed live this session, each affecting every client site: (1) archived projects still render publicly — two confirmed ghost sites (garrison-orthodontics-3307.sites.getalloro.com, an orphaned full duplicate of the real Garrison site, and swift-medical-7604/DentalEMR), an active duplicate-content risk against sites whose SEO was just repaired; (2) any unknown path returns the homepage with HTTP 200 (soft-404) — stale URLs never leave Google's index, broken links are invisible in GSC, and it repeatedly confused this session's own debugging; (3) no sitemap.xml or robots.txt — both fall through to the homepage fallback, so Google discovers the freshly fixed metadata only via nav links; (4) the renderer trusts stored canonical/og:url verbatim and emits nothing when missing — the data is fixed for the 4 audited sites, but every other client site still has posts with missing canonicals, and nothing prevents future junk from rendering.

What

  • Archived projects stop resolving: hostname/domain lookups exclude archived_at IS NOT NULL; matched-but-archived requests get a dedicated "site no longer available" response (410), distinct from site-not-found.
  • Unknown paths return the existing pageNotFoundPage template with HTTP 404 instead of the homepage render.
  • Per-project robots.txt (allow all + Sitemap line) and sitemap.xml (published pages + published posts at /{post_type.slug}/{post.slug}, absolute URLs on the project's domain), Redis-cached and included in the ?nocache=1 flush list.
  • Canonical/og:url derived self-referentially from the actual serving context; stored canonical_url honored only when same-domain and path-matching; emitted even when the stored key is absent.
