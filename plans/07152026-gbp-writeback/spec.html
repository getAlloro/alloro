Alloro Funnel Engine · Stage A (Get Found) · Feature A6

GBP Write-Back

Let the owner correct and complete their own Google Business Profile — categories, hours, phone, website, description — with a one-approval patch to Google, and a captured snapshot so any write is reversible. The first lever that writes to a customer's live Google presence, built to be incapable of firing without the owner's explicit yes.

In Progress Branch: claude/a6-gbp-writeback → dev/dave Reference analog: §6.1 gbp-automation/ Rev 4 · 2026-07-16

WHY The service, then the feature

Who this lifts, and why it belongs in the funnel.

The owner is the hero; Alloro is the guide. A local practice owner is buried — running the business, not curating a Google listing. Their profile drifts: an old phone number, a missing category, hours that changed, no description. That drift quietly costs them the exact patients who were looking. They know the right answer; they don't have the time or the path to fix it across Google's clunky surface.

A6 gives them the path. Alloro already sees what's missing or wrong (A2 completeness scoring, live-read from Google). A6 closes the loop: the owner reviews Alloro's proposed correction, says yes once, and the fix lands on Google — done-for-them on execution, in-command on the decision. It turns "your profile is incomplete" from a nag into a fixed listing, in one click, with an undo.

Funnel role — Driver 2 (Get Found). Per library/canon/mvv.md, findability is the second driver in service of the freedom mission. A6 improves the accuracy and completeness of the owner's own profile. It is not a rank play. See Constraints → Value #6.

WHAT The behavior, precisely

Dave's bar: name the behavior and the edge cases so there is zero ambiguity.

The one flow

  1. Draft. A business_info work item is created for a location with a set of proposed field values (e.g. { phoneNumbers, categories, regularHours, websiteUri, profile.description }) and the updateMask naming exactly which fields change. Status starts draft. Nothing is sent.
  2. Approve. The owner (or Alloro operating done-for-them, with the owner's confirmation) approves the work item. Status → approved, stamping approved_by_user_id + approved_at. Still nothing is sent.
  3. Deploy — capture, then write. On explicit deploy, a worker job (a) reads the current live field values from Google and stores them as the rollback snapshot (previous_values), then (b) issues a single PATCH to Google's mybusinessbusinessinformation.googleapis.com/v1/{location} with the updateMask, then (c) records Google's response and marks the item published. Status guard: the write is only reachable from approved → deploying → published.
  4. Revert (rollback). A published business_info item can be reverted: a second PATCH restores the captured previous_values for the same updateMask. The snapshot is the rollback — it lives on Google's side, not just in a DB down-migration.

Out of this slice (adjacent / backlog)

  • Auto-suggestion of values from A2 gaps. Slice 1 accepts proposed values as input; wiring A2's gap output straight into a pre-filled draft is a follow-up slice.
  • GBP photo / gallery upload. A separate net-new write (tracker B6's GBP half) — not A6.
  • Recurring schedule. Business-info changes are episodic; A6 is on-demand, owner-initiated. No cron is added.

CONSTRAINTS The non-negotiables for a live-write lever

1 · Owner-approval is a STRUCTURAL gate, not a step (Option B canon). The write path must be incapable of firing without an explicit owner approval + explicit deploy. Enforced by the status machine: the Google PATCH is only reachable from approved → deploying, each transition guarded by .where({ id, status }) and stamping a user id. There is no code path that patches Google from a draft item, and no auto-send / no cron. Mirrors the shipped post/reply gate verbatim.
2 · No live writes in sandbox / build / tests / adversary. The write fn takes an injected http dependency (A5 pattern). Every test and the adversary pass feed a fake Google client that records the call and returns a canned response. A real axios.patch to Google fires only post-merge, owner-approved, on a real account — never during this work.
3 · Rollback = capture-before-write. Before the patch, the deploy job reads and stores the live field values (previous_values) for exactly the fields in the updateMask. If the read fails, the write does not proceed. Revert = a second PATCH restoring the snapshot. The rollback lives on Google's side, not only as a DB down-migration.
4 · Value #6 — no rank/visibility/guarantee framing. All labels/copy say "correct / complete your profile information on Google" — accuracy & completeness only. Never "improve your ranking," "get found higher," or any visibility promise. The honesty cap is on the claim, not just the output.
5 · Ships DISABLED, per account. business_info_writeback_enabled defaults false. Draft creation and deploy both refuse when off. Flagged for Dave in the PR: enable per account only after a real owner-walk. Additive + reversible migration; no data rewrite; existing review_reply/local_post rows untouched.
