Why

The website AI Command bulk editor reports edits as "executed" but silently drops them. Confirmed on production (project acc7cbfc…, getalloro.com, batch d698f83b…): a run that asked to "replace all cal.com links with /book-a-demo" reported 38 executed / 0 failed, yet every published page still serves cal.com and zero /book-a-demo.

The cause is not publishing, caching, or DNS — those were ruled out (the project has a verified custom_domain, the renderer serves status='published' rows, and the batch did auto-publish fresh versions). The cause is that multiple recommendations targeting the same page section overwrite each other, so only the last write per section survives. The status is computed from "the LLM returned HTML," never from "the change reached the published page," so it lies.

Evidence (live published content, same pages)

Page"Free Audit" (label edit)"/book-a-demo" (link edit)"cal.com" (old)
/2 — landed0 — lost1
/benefits2 — landed0 — lost2

Both edit types are page-section edits in the same batch. The label rename landed; the link change on the same sections was reverted. Sections with a single recommendation survived; sections with multiple recommendations kept only the last write.

What

Two changes to the AI Command execution path, plus tests. Scope decided with owner: core fix + verify net only (no deterministic find/replace, no per-batch concurrency lock — those are deferred).

  • Core fix — make the executor read from and write to one pinned draft per page for the whole batch, so edits to the same section stack instead of each being computed from a stale snapshot and clobbering the previous one.
  • Verify net — after publishing, re-read the published content and confirm each edit's HTML is actually present; if not, mark that recommendation failed (not executed) with a reason, and roll it into the batch stats. "Executed" becomes a truthful claim.

Done = a re-run of "replace all cal.com links with /book-a-demo" on a page whose hero has both a cal.com button and a label edit results in both the link and the label landing in the published page; and any edit that genuinely can't land is reported as failed, never as executed.

Constraints

Must

  • Read and write must resolve the same pinned draft row per path for the whole batch (the core fix).
  • A recommendation is only executed if its edited HTML is verifiably present in the published page; otherwise failed with a reason.
  • Verification re-reads from PageModel (§7.4) and runs after the existing publish loop.
  • Batch stats/summary reflect downgraded statuses (refreshStats after the verify pass).
  • Preserve existing behavior for layout and post edits, and for single-recommendation sections.

Must Not

  • No new dependencies.
  • Do not change service.page-editor.ts (createDraft/publishPage), the renderer, or the normal page editor.
  • Do not add a per-batch lock or deterministic find/replace (deferred — out of scope).
  • No inline db() outside models/; no console.*.
  • Do not modify the 715-line handlers file for size reasons.
  • No production data surgery in this plan (owner chose fix-forward).

Out Of Scope

  • Repairing the current getalloro.com content — owner will re-run the bulk edit after deploy (the re-run is the live validation).
  • Header/footer booking links — owner is handling those manually.
  • Concurrency lock against simultaneous manual editing during a batch (residual risk, deferred).
  • Deterministic (non-LLM) bulk find/replace for exact link/label swaps (deferred enhancement).

Risk

Level: 3 (Structural Risk) — changes the core write path of a live admin feature that mutates customer website content.

Risks Identified

  • Regression on the single-recommendation path. Mitigation: pinning derives the draft from the same published row analysis targeted; first read returns published content (unchanged behavior), later reads return the accumulating draft. Covered by a single-rec test.
  • Residual section-index drift — if a reused draft's section order differs from analysis, an edit can still target the wrong section. Mitigation: the verify net (T2) catches it — the edit won't be found in the published page, so the rec is marked failed and surfaced for re-run rather than silently lost.
  • Verify false-negatives — the agentic pipeline may reformat HTML so an exact substring match misses. Mitigation: compare on a normalized form (collapse whitespace) and match on a stable token from the edit (e.g. the changed href/text) rather than the entire blob; on ambiguity, prefer leaving status executed and logging a warn over a false failed.
  • External concurrent editing during a batch can still clobber (out of scope). Mitigation: documented as residual; the verify net at least makes the loss visible instead of silent.

Blast radius: only the AI Command execution path (admin website AI editor → executeBatch). Consumers: AiCommandController.executeAiCommandBatch (fire-and-forget background task) and the frontend AiCommandTab polling that renders stats. No change to routes, the publish transaction, the renderer, or the normal editor. The verify pass only downgrades statuses and re-reads content; it never writes page content.

Pushback

  • The deeper correct design is to make each page's edits one coherent, content-anchored, transactional operation (resolve once, apply all sections in memory, write once, verify, publish). The chosen "pin one draft for read+write + verify" is the minimal change that removes the clobbering; if same-section bugs recur, escalate to the full re-anchored/transactional design rather than adding more patches.
  • For exact bulk swaps ("replace X with Y"), an LLM per section is slow, costly, and drift-prone. A deterministic string/DOM replace would be strictly more reliable for that case. Deferred by owner decision — recommend revisiting as the next enhancement.
