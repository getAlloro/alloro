Why

Four follow-ups from the grade-scale work's live verification, bundled by request. One is a real correctness bug; the others harden trust and analytics.

#ProblemEvidence
T1GBP analysis silently skipped when the website scan is slow/fails. The independent website branch is awaited on the critical path before the GBP analysis; its throw aborts the whole job.Reported by user; root cause confirmed in auditLeadgen.processor.ts:396–399 + catch at :596.
T2Competitor cohort can be too thin to grade against (n=2).One Endodontics audit found 2 competitors; rank_score is a cohort percentile, so n=2 is fragile.
T3Website-scan "result" tags are fake and can contradict the real report.WebsiteScanStage.tsx:55 floats hardcoded green badges (Speed OK, CTA Found…) unconditionally.
T4Leadgen tracking events 400 (funnel analytics may be dropping).POST /api/leadgen/event → 400 observed locally (placeholder key). Prod parity unconfirmed.

What

  • T1: A slow/failed website analysis never aborts the GBP analysis. Website failure degrades to an empty website card; GBP + competitor + ranking still complete. realtime_status is monotonic (no backward UI flicker).
  • T2: When the same-specialty competitor search returns fewer than a floor (e.g. <4), broaden once (wider radius and/or adjacent category) and merge/dedupe; surface the true cohort size so a thin comparison is never presented as authoritative.
  • T3: Scan tags describe activity, not fake results — neutral process verbs ("Checking SSL…", "Scanning CTAs…") instead of asserted positives the report may later contradict.
  • T4: Confirm whether prod actually has a key mismatch; if so, align the backend LEADGEN_TRACKING_KEY with the deployed bundle's build-time VITE_LEADGEN_TRACKING_KEY and verify a tracking event returns 200.

Constraints

Must

  • T1: GBP analysis runs whenever its inputs (self GBP + competitors) succeeded, independent of the website branch.
  • T1: realtime_status never decreases.
  • T2: never present a thin cohort as authoritative — show the real count.
  • T4: back up prod .env before any edit; change a key only if a genuine mismatch is proven.

Must Not

  • Don't rewrite a working prod key (the local 400 was a dev placeholder — prod may already be correct).
  • Don't change website/GBP scoring logic or the grade scale (just-shipped).
  • T3: don't fabricate real telemetry — at scan time the analysis isn't done, so tags stay process-descriptive.
  • No new dependencies; no schema migration.

Out Of Scope

  • Adding an LLM-output retry/repair for the website analysis (separate hardening; T1 only needs graceful degradation).
  • Re-running historical failed audits.

Risk

Level 3 — Structural. T1 edits the core audit orchestration (blast radius: every audit). T4 writes prod config. Both demand care.

Risks & Mitigations

  • T1 reorder changes completion semantics. Decoupling means an audit can now "complete" with a null website card where it previously failed. Mitigation: that is the intended, better behavior (partial result > total failure); the frontend already greys out a null website card. Assert it explicitly in tests.
  • Monotonic guard could mask a legit reset. Mitigation: the pipeline only ever moves forward (0→5); a guard that ignores lower writes is safe. Scope the guard to realtime_status only.
  • T2 cohort broadening changes grade meaning. Comparing an endodontist against general dentists shifts the cohort. Mitigation / decision: prefer same-specialty; only broaden when below the floor, and label the cohort. Keep broadening conservative.
  • T4 prod write. Wrong key value breaks tracking further. Mitigation: read both sides first, prove the mismatch, back up .env, change the minimum, restart, verify a 200, keep the old value to roll back.

Pushback

  • Bundling a prod-ops change with a Level-3 pipeline refactor in one plan is broad. Recommended execution order: T1 → T4 → T2 → T3, committing T1 on its own (it's the bug) before the rest, so the reliability fix can ship and be watched independently of the softer changes.
