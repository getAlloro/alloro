Why

Google Search Console data is already harvested daily into website_builder.gsc_data and fully visualized in the admin dashboard. Clients can connect GSC in /settings/integrations but have nowhere to see their search performance — the data they own is locked behind admin-only routes. This surfaces it: practice owners see what people search to find them, whether they're showing up, and whether it's trending up. Now, because the entire pipeline already exists; this is surfacing, not building.

What

Two client-facing surfaces in the DFY website dashboard, both fed by one new client-scoped read endpoint:

  • A "Keywords" tab in the website dashboard pill nav (alongside Overview / Pages / Submissions / Posts / Menus): totals, a daily clicks/impressions trend chart, a range selector, and top-queries + top-pages tables.
  • A "Search Performance" card on the Overview tab: 3 headline metrics (clicks, impressions, avg position) + a trend sparkline + top 3 queries, with a "View all →" that opens the Keywords tab.
  • New endpoint GET /api/user/website/gsc/performance?rangeDays=90 that resolves the project from the auth-token org and reuses the existing service.gsc-performance.getDashboard() untouched.

Metrics shown (trimmed for clients): totals (clicks, impressions, avg position), daily trend, top queries, top pages. Countries / devices / per-dimension CTR are intentionally dropped as analyst noise.

Done when: a logged-in client whose project has an active GSC integration sees their real search data in both surfaces; the not-connected state links to settings; the connected-but-no-data-yet state renders cleanly; npx tsc --noEmit is clean for both frontend/ and root.

Patterns To Follow

  • Org-scoped read endpoint: mirror getWebsiteAnalytics exactly — orgId = req.organizationId → 400; ProjectModel.findByOrganizationId(orgId) → 404; WebsiteIntegrationModel.findByProjectAndPlatform(project.id, "gsc") → { success: true, hasIntegration: false, … } when absent; else { success: true, hasIntegration: true, …dashboard }.
  • Route guard: authenticateToken, rbacMiddleware, requireRole("admin", "manager") — identical to /analytics.
  • Frontend API client: mirror fetchWebsiteAnalytics — apiGet, no orgId in URL, throw on !success.
  • Card: mirror the Rybbit analyticsQuery + OverviewCard block in WebsiteOverview.tsx.
  • Charting: Recharts is already a dependency (the admin GSC chart uses it). Build a fresh client chart; do not import the admin component.

Reference file: frontend/src/api/websiteAnalytics.ts + getWebsiteAnalytics in UserWebsiteController.ts:651 — the closest existing analog (same harvest→admin-service→client-surface shape, shipped for Rybbit).

Data Flow

Keywords tab / Overview card → useWebsiteGscPerformance(rangeDays) [new hook] → fetchWebsiteGscPerformance() [new api client] → GET /api/user/website/gsc/performance [new route] → getGscPerformance() [new controller] orgId → ProjectModel.findByOrganizationId → WebsiteIntegrationModel.findByProjectAndPlatform(project.id,"gsc") → service.gsc-performance.getDashboard(integration, rangeDays) [REUSED] → reads website_builder.gsc_data (already harvested daily)

Constraints

Must

  • Resolve project strictly from req.organizationId (auth token). Never accept a caller-supplied projectId.
  • Reuse service.gsc-performance.getDashboard() with zero changes.
  • Return the full dashboard payload from the endpoint; trim (drop countries/devices) only in the client UI.
  • Handle three states in both surfaces: not-connected (link to /settings/integrations), connected-but-no-data-yet, and has-data.
  • Match the Rybbit analytics precedent for naming, error handling, and response envelope (success / hasIntegration).

Must Not

  • Edit or import any components/Admin/ GSC component (build fresh client-only — confirmed decision).
  • Edit service.gsc-performance.ts or any harvest / model code.
  • Add new dependencies (Recharts already present).
  • Add countries / devices / per-dimension CTR to the client view.
  • Expose tokens, connection IDs, or another org's data.

Out Of Scope

  • Org-level multi-website aggregation (one project per org today).
  • Countries / devices breakdowns, CSV export, period-over-period comparison.
  • Any change to GSC connect/auth flow or the harvest cron.
  • Extracting admin GSC components into a shared library (noted as follow-up debt below).

Risk

Level: 2 (Concern)

Risks Identified

  • Cross-org data exposure (security). A client must only ever see their own org's GSC data. Mitigation: resolve project solely from req.organizationId via ProjectModel.findByOrganizationId; the endpoint takes no projectId param. Identical to the shipped getWebsiteAnalytics guard.
  • Parallel-copy tech debt. "Build fresh client-only" means the client keyword table/chart duplicates the admin's. Mitigation: accepted deliberately to avoid touching admin code; logged as a future refactor (extract MetricCard/DimensionTable/chart to components/gsc/ consumed by both). Not done here.
  • Cross-module import. User controller imports a service under controllers/admin-websites/feature-services/. Mitigation: this is the established pattern — getWebsiteAnalytics already imports the Rybbit dashboard the same way (UserWebsiteController.ts:23). No new drift.
  • Empty / not-yet-harvested data. A freshly connected property has no rows; legacy payloads may be incomplete. Mitigation: explicit empty states; trust the service's limitations[] and dataDays fields.
  • Route ordering. /gsc/performance must not collide with /gsc. Mitigation: exact-path routes (not params) — Express already coexists /gsc, /gsc/sites, /gsc/connections. Place near them.

Blast radius: Additive. Backend: +1 route, +1 controller export (no existing handler touched). Frontend: edits to WebsiteDashboardTabs.tsx (union + options array — all consumers must handle "keywords"), DFYWebsite.tsx (WEBSITE_TABS + render switch), WebsiteOverview.tsx (new card). New files: api client, hook, KeywordsTab + chart/table parts. No DB migration (tables exist).

Pushback

  • Building fresh duplicates UI you already have. The honest best practice is extracting the admin GSC pieces into a shared component both sides consume. We chose fresh to keep the admin view zero-risk and ship faster — that is a reasonable trade, but it is debt. Recommended follow-up: a separate refactor plan to converge on components/gsc/ once both views are stable. Do not let the two copies silently diverge.
