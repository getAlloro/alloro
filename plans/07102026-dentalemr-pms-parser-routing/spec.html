Why

All organizations currently enter the same mapping-based PMS parser. That default path is useful for unknown formats, but its global procedure-log mapping calculates production with the wrong formula for DentalEMR and its copy/paste UI finalizes independent 50-row batches, which can count one patient more than once when their rows cross a boundary.

DentalEMR also exposes two assumptions the current UI loses: total referrals are distinct patients for the month, while source counts are distinct patient-source relationships and therefore are not additive; and a real XLSX export can contain summary tabs before multiple raw tabs, while the current converter reads only the first worksheet.

What

Add a server-owned pms_type to organizations, an admin assignment control, and a parser router with default and dentalemr implementations. File upload, file preview, paste preview, and paste persistence must all route through the same organization-selected parser. Organizations left on default retain current behavior.

DentalEMR parsing is deterministic and uses no LLM. It reproduces the supplied April/May summaries, maps the resulting rollup directly into the existing editable table, preserves the authoritative distinct-patient total, and uses a truthful phase progress indicator without semantic row batching.

Rev 2: Store an unassigned/configurable parser as null and normalize that value to the default parser on the server. A non-null custom parser such as dentalemr owns its calculation formula, so the client must hide both the right-side formula/mapping toggle and the sliding drawer. A null/default organization keeps the configurable formula control after uploaded or pasted columns are available.

DentalEMR Formula Contract

  1. Recognize raw worksheets/rows by the required headers Treatment Date, Status, Ins. Adj. Fee., Patient, and Referring Practice. Allow and ignore extra columns.
  2. Use Treatment Date for YYYY-MM bucketing. Do not trust a filename, worksheet name, or summary heading for the month.
  3. Preserve the existing procedure-log contract by including rows whose status is case-insensitively Done. Zero-production rows still count as referrals.
  4. Total production: sum every qualifying row's Ins. Adj. Fee.. The same field is summed per normalized source. Do not use Gross Revenue or the existing global formula.
  5. Total referrals: count distinct non-blank patients per month.
  6. Per-source referrals: count distinct patient + normalized-source pairs per month.
  7. Source normalization: trim whitespace, strip all leading/trailing asterisks, trim again, then map blank or case-insensitive exact 1endo to Self. Merge exact normalized names. No DentalEMR fuzzy-AI merge.
  8. Types: Self is self; every other normalized practice is doctor.
  9. Store total_referrals independently from source totals. A patient appearing under two sources counts once in the monthly total and once under each source.
  10. Persist parser metadata identifying the non-additive count semantics so downstream aggregation does not raise a false source-sum mismatch.

Truthful Progress UX

Remove the frontend's 50-row request batching. A 600-700-row DentalEMR export is about 132-184 KB, while the app accepts 50 MB JSON and 10 MB files. One request is faster than 12-15 sequential requests and lets the server deduplicate globally.

  • Before confirmation, show actual detected row count and payload size. Remove batch count/copy.
  • Use milestone segments, not a fabricated time percentage: Parse and, only when the default parser requires it, Clean sources.
  • The active segment is indeterminate/animated. A segment fills only when its request actually completes. Ready is the only 100% state.
  • DentalEMR returns requiresSanitization: false; its deterministic parser finishes after Parse. The default parser retains the existing sanitizer phase.
  • Expose accessible status with role="progressbar" and aria-valuetext. Omit aria-valuenow while a phase is indeterminate.
  • Do not add polling, WebSockets, SSE, Redis progress state, or artificial delay. Row iteration finishes too quickly for polling to observe; queue overhead would make the operation slower.

Constraints

Must

  • Use a follow-up migration to make organizations.pms_type nullable with no database default, convert stored default assignments to null, and preserve non-null custom parser keys. Do not rewrite an applied migration.
  • Normalize both null and the legacy explicit default value to the runtime default parser on the server. Keep stored assignment state separate from resolved runtime parser type.
  • Present null in the organization selector as “Default (configurable formula)” and keep custom parser choices such as DentalEMR explicit.
  • For any resolved custom parser, hide both the formula/mapping toggle and the sliding drawer, and clear stale mapping/drawer state. For the null/default parser, show the right-side formula control only after source columns and a mapping exist.
  • Support registry values default and dentalemr; unknown stored values safely fall back to default with a non-PII warning.
  • Let super admins change future parsing behavior from the organization overview. Historical jobs remain unchanged.
  • Use the same parser result contract for paste, file preview, and persisted file/paste uploads.
  • Keep the default route behaviorally identical, including mapping cache and optional sanitizer.
  • Reject paste payloads above the existing 10 MB PMS file limit with HTTP 413 and a clear upload-file instruction.
  • Preserve parsed totals until a user edits a referral/source row. Such an edit explicitly invalidates the authoritative override and returns that month to derived totals, with visible notice.
  • Record parser/count-semantics metadata in response_log and original_response_log.

Must not

  • Do not let form data or JSON pmsType choose the parser.
  • Do not treat hiding the frontend drawer as parser enforcement; the backend organization assignment remains authoritative.
  • Do not show an empty formula builder before an upload or paste provides column headers.
  • Do not modify the global default mapping or apply DentalEMR rules to unassigned organizations.
  • Do not run AI mapping or fuzzy source sanitation for the DentalEMR parser.
  • Do not compute total referrals by summing source rows.
  • Do not commit the supplied client files or any patient-level fixture.
  • Do not silently choose a summary/pivot sheet in a DentalEMR workbook.
  • Do not add a queue, job table, Redis state, WebSocket, or SSE solely for the progress bar.

Out of scope

  • Automatic production assignment of DentalEMR to a live organization; deployment and assignment are separate gated actions.
  • Reprocessing historical pms_jobs.
  • General cleanup of quotes, parenthetical scheduling instructions, misspellings, or provider duplicates beyond exact edge-asterisk normalization.
  • True streaming/resumable parsing for files or pastes above 10 MB.
  • Additional PMS vendors or an admin UI for authoring parser definitions.
  • A standalone formula editor that can be configured before any source columns are known.

Risk

Level 3 This changes the critical financial/referral ingestion path, adds organization configuration, and corrects assumptions shared by preview, persistence, and downstream agents.

Risks and mitigations

  • Default-client regression. The router sits in every ingestion path. Mitigation: default adapter delegates to the existing resolver/applier unchanged; add byte-equivalence tests for the Alloro template and current default procedure-log fixture.
  • Financial formula error. Existing mapping code and the confirmed DentalEMR formula disagree. Mitigation: isolate DentalEMR from the global mapping and reconcile four real location-month totals before acceptance.
  • Workbook sheet ambiguity. A summary sheet appears first in the supplied XLSX. Mitigation: inspect all DentalEMR sheets by required raw headers, report selected sheet names, and fail clearly when none match.
  • False referral mismatch. Source-patient relationships legitimately exceed global unique patients. Mitigation: persist count semantics, preserve top-level totals, and skip only the incompatible sum-reconciliation rule for that declared semantic.
  • Stale total after editing. The table cannot reconstruct patient-level uniqueness after source counts are manually changed. Mitigation: invalidate the authoritative total on any referral/source mutation and visibly return to derived mode.
  • Patient data exposure. Raw exports contain PII. Mitigation: synthetic committed tests, local-only acceptance files, no row-level logging, existing tenant-scoped raw-data persistence only.
  • Progress complexity. A queued job/poller would introduce retry, cleanup, tenant, and PII-state problems for sub-second work. Mitigation: milestone/indeterminate phases backed by existing requests.
  • Nullable assignment ambiguity. Persisted null and runtime default could be mistaken for two parsers. Mitigation: keep a separate nullable assignment type, normalize only at the server router boundary, and cover null/legacy-default/custom cases in tests.
  • Stale formula UI. A mapping resolved before an organization switches to a custom parser could leave the drawer visible. Mitigation: gate both controls on resolved parser metadata and clear mapping/drawer state whenever the active result is custom.

Blast radius

  • Database/admin: organizations schema/model, admin list/detail responses, admin organization overview, super-admin mutation route.
  • PMS ingestion: manual paste preview and submit, mapping preview/submit guard, direct upload, file-manager preview/overwrite, CSV/XLS/XLSX conversion.
  • Data contract: monthly rollup totals, parser metadata, raw-input persistence, source-sum reconciliation, monthly-agent input.
  • Frontend: paste hook/dialog, upload request types, manual-entry table state/transforms, mapping drawer routing.
  • Docs: Referrals Hub upload guidance and replica parity check.

Pushback

  • Do not build a numeric job-progress system for this. It would be slower and less reliable than the parser. The recommended phase bar is truthful and proportionate.
  • Do not “fix” the global Open-Dental mapping. That would silently change every default organization. DentalEMR must be an explicitly assigned custom parser.
  • Do not force additive totals. The supplied summaries prove the total and source breakdown answer different valid questions.
