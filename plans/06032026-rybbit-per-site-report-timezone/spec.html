Why

Every Rybbit analytics call hardcodes America/New_York as the bucketing timezone for all practices. Day/month boundaries are therefore computed in Eastern time even for practices in Central, Mountain, or Pacific zones, so their daily/monthly buckets do not match what their own staff see in Rybbit's dashboard. Measured impact is small (~0.3% of sessions shift near boundaries, Garrison probe) but it is a real correctness gap for non-Eastern practices, and it now also affects the live unique-visitor queries added 2026-06-03. Deferred from the 2026-06-03 website-overview data-accuracy investigation (plans/05312026-no-ticket-websites-tab-cards-overview/spec.md → "Data Accuracy Finding"); the larger ~10% summing-uniques bug was fixed in commit 24ffedba.

What

Introduce a per-site reporting timezone sourced from a new nullable website_builder.projects.rybbit_time_zone (IANA string), admin-settable through the existing Rybbit integration flow, and thread it through all four hardcoded call sites. When unset, every path falls back to America/New_York via one shared default constant — so existing Eastern practices are byte-for-byte unchanged. Done = no Rybbit request sends a hardcoded zone; each site's daily harvest, historic backfill upper-bound, live overview, live monthly-uniques, and the Proofline/Summary agent comparisons all bucket in that site's configured zone.

Pre-decided (Phase 2/3): Rybbit exposes no per-site timezone (verified against its open-source schema and site-settings docs — the sites table has no timezone column; the dashboard buckets off the time_zone query param, defaulting to the viewer's browser zone). So "match what the practice sees" resolves to the practice's local timezone, and the source must be ours. We also have no existing timezone column anywhere, so the chosen source is a new projects column set by an admin (user-selected over metadata-only, GBP auto-derivation, and WONTFIX).

Patterns To Follow

  • IANA-zone formatting already exists: getDateInTimeZone in service.rybbit-history.ts — reuse the Intl.DateTimeFormat approach for validation; no new dependency.

  • At execution, invoke the alloro-conventions skill (backend) and alloro-frontend skill (T9) before writing code.

Reference files: migration analog src/database/migrations/20260312000001_add_rybbit_site_id_to_projects.ts; save-flow analog service.rybbit-integration.ts + ProjectModel.updateRybbitSiteId.

Constraints

Must

  • Single source of truth: projects.rybbit_time_zone (nullable IANA string).
  • One shared default constant RYBBIT_DEFAULT_TIME_ZONE = "America/New_York"; every path falls back to it when the column is null.
  • Additive, nullable migration — no backfill, no default value at the DB level.
  • Validate the admin-supplied IANA zone via Intl before persisting; reject invalid.
  • Update every enumerated caller in the same task that changes a signature (no broken builds mid-plan).

Must Not

  • No new npm dependencies — use the platform Intl API, never a geo/tz library.
  • No automatic re-backfill on TZ change; no geo/GBP auto-derivation.
  • Do not store the zone in website_integrations.metadata (avoids a second source of truth).
  • Do not touch clarity / gsc / hubspot integrations, harvest payload shapes, or unrelated code.
  • No drive-by refactors.

Out Of Scope

  • Auto-deriving timezone from GBP business_data / address (possible future prefill, not now).
  • Automatic historic re-harvest when a site's zone changes (manual backfill is the documented op step).
  • Capturing timezone in the onboarding wizard; per-viewer/browser-zone display.

Risk

Level: 2 (Concern). Additive nullable migration + ET fallback = zero regression for existing Eastern practices; the only real edges are a few signature changes and a data-continuity caveat.

Risks Identified

  • Stored-data discontinuity. Changing a site's zone after history is harvested re-buckets new daily rows differently from existing ET rows. Mitigation: after setting a non-Eastern zone, run the existing historic backfill (POST /admin/websites/:projectId/integrations/:id/rybbit/backfill), which clears + re-harvests under the new zone. The live overview / monthly-uniques queries are computed on the fly and need no backfill.
  • Signature changes. getLatestCompleteRybbitReportDate(now, timeZone), fetchRybbitOverview/fetchRybbitMonthlyUniques gain a timeZone param, and getRybbitSiteId changes return shape to { siteId, timeZone }. Mitigation: all callers are enumerated below and updated in the same task.
  • Extra read in harvest adapter. The adapter must look up the project zone by project_id (the generic interface gives it only the integration). Mitigation: one indexed PK read per daily harvest job — negligible; not a hot path.
  • Inert until set. Non-Eastern practices stay on ET until an admin enters their zone. Mitigation: accepted (status quo until then); surface the current zone in getStatus so it is visible.

Blast radius (consumers of changed signatures): workers/processors/dataHarvest.processor.ts, WebsiteIntegrationsController.ts, UserWebsiteController.ts, controllers/agents/feature-services/service.agent-orchestrator.ts (unchanged — resolves internally), and frontend RybbitTab.tsx. No other consumers found.

Pushback

  • The measured impact is ~0.3%. This spec is deliberately the lightest correct fix; a geo→TZ library, a mandatory backfill, or auto-derivation would be gold-plating disproportionate to the gap. Recommended alternative: ship light, revisit auto-prefill only if non-Eastern volume grows.
  • "Admin set" introduces a human dependency (someone must know and enter each non-Eastern practice's zone). The cleaner-feeling auto-derive from GBP was rejected as unreliable (business_data is nullable) and heavier. If that becomes painful, a future task can prefill the field from GBP when present — without changing this source of truth.
