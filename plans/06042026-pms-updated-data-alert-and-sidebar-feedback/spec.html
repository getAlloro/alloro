Why

The PMS File Manager drawer (shipped 06/02/2026) mutates data but its UI waits on the server with no optimistic feedback: delete shows no in-flight state, edit-save has no clear confirmation, and every mutation fires a full 5-key surface refetch that reads as a page reload or "redirect to the dashboard."

Worse, edit and delete currently auto-rerun the monthly agent. That is the wrong default: a manager correcting one cell silently kicks off a multi-minute analysis run, and the dashboard behind the drawer flips into the processing state unexpectedly. We want edit/delete to be cheap and reversible, and the expensive rerun to be an explicit, user-initiated choice — surfaced through a prioritized "Updated data detected" alert with a single "Get updated insights" action.

What

Three coordinated changes on /pmsStatistics and the main dashboard Focus tab:

  • Stop auto-rerun on edit/delete. Uploads (new month and overwrite) keep auto-running. Edit and delete only record their event and mark insights stale.
  • Detect staleness and surface it. When an active month's data was edited/deleted after the most recent completed insights run for that location, show one location-level "Updated data detected" alert whose CTA, "Get updated insights," triggers the rerun and immediately flips the dashboard into the existing animated processing card.
  • Make the sidebar feel instant. Optimistic edit/delete with rollback, in-flight states on the delete confirm and the editor Save button, and removal of the now-unnecessary agentData/tasks invalidations.

The alert is rendered by a new shared DashboardAlert + DashboardAlertStack component used on both the main dashboard and the PMS surface. The stack cascades multiple alerts: the highest-priority alert is full-size on top, lower ones are scaled down behind it, with left/right arrows to cycle. The existing "PMS data ready?" calendar nudge and the setup-progress banner are refactored onto this same component.

Scope Ledger

Objective
Replace silent auto-rerun on PMS edit/delete with an explicit, prioritized "Updated data detected" alert, on a shared cascaded-alert component reused across the main dashboard and the Referrals Hub, plus optimistic sidebar feedback.
Intended user
Customer admin and manager users managing PMS data on the Referrals Hub.
Surfaces
/pmsStatistics (PMS dashboard surface + File Manager drawer) and the main dashboard Focus tab.
Staleness definition
Per location. Stale = the latest data_edited/file_deleted event across the location's active jobs is newer than the most recent completed monthly-agent run, and no run is currently in progress. Derived, auto-clearing — no stored/dismiss state, no migration.
System of record
Existing pms_jobs + pms_job_events + automation_status_detail JSONB. No schema change.
Out of scope
Changing the 12-month cap or aggregation, per-month stale alerts, dismiss/snooze persistence, queueing reruns, new DB tables/columns, touching ranking agents, and changing upload auto-run behavior.

Risk

Level: 3 - reverses behavior shipped two days ago, refactors a main-dashboard surface, and adds a correctness-sensitive staleness comparison. No migration, no auth-model change, no destructive data operations, so it is not Level 4.

Risks Identified

  • Reversing just-shipped auto-rerun. Edit/delete stop triggering runs. Mitigation: remove only the two rerunMonthlyAgents calls, keep the event writes and the active-run guard, and change the toasts in the same change so the UI stops lying.
  • Main-dashboard blast radius. The shared component replaces the duplicated nudge on the Focus tab. Mitigation: extract a purely presentational component; keep alert-data derivation in each surface's existing owner; snapshot both surfaces before/after.
  • Staleness correctness. Comparing event time vs per-job JSONB completedAt is easy to get subtly wrong (which job, soft-deleted rows, timezone, "never run yet"). Mitigation: one server-side method with explicit rules — ignore deleted/unapproved jobs, treat "no completed run" as not-stale, compare in UTC — covered by unit tests on the comparison.
  • Stale-insight viewing window. Between an edit and clicking the CTA, the cards show data that no longer matches. Mitigation: the alert is the explicit signal; copy makes clear insights reflect prior data until refreshed.
  • Optimistic rollback drift. A failed edit/delete must restore the exact prior cache. Mitigation: snapshot the file-manager query in onMutate, restore in onError, reconcile in onSettled; never optimistically mutate while a run is active (guard returns 409).
  • Cascade UX regression. A carousel can hide important alerts behind arrows. Mitigation: always render the highest-priority alert on top; show a count/indicator; collapse to a single card when only one alert is active.

Blast radius: PMS File Manager drawer, PMS key-data response contract (stats), main dashboard Focus tab, PMS dashboard surface, the existing calendar nudge (both copies), the setup-progress banner, the monthly-agent rerun trigger path, automation polling/processing state, and the Alloro Docs Referrals Hub page + replica.

Pushback

  • Auto-rerunning on a single-cell edit was never the right default — it makes a cheap correction expensive and surprising. Making the rerun explicit is the correct model; this spec treats that as settled, not optional.
  • Do not solve "show on both surfaces" by copying the alert markup a third time. The duplication that exists today is the bug; the shared component is the point.
  • Resist storing alert/dismiss state. A derived, auto-clearing signal cannot drift out of sync with the data; a stored flag will.
  • Keep the staleness math on the backend. The frontend should receive a boolean and timestamps, not reconstruct the comparison from per-month rows.
