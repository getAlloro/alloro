  • §20.2 requires failure-path and tenant-scope coverage, not tests that pass because they swallow errors.
  • §12.1 requires frontend API types to mirror backend API contracts through the existing typed API/hook path.

Constraints

Must

  • Start from current dev/dave truth and merge PR 153 intentionally.
  • Preserve ChoosableMetricsService.build(orgId, locationId, reviews) or an equivalent org-guarded service.
  • Keep source_status, source_reason, is_at_or_above_review_median, and has_most_reviews in the Choosable contract unless every downstream validator is intentionally updated.
  • Retain PR 153's intended owner-facing Ch3-7 copy and freshness fixes when they do not conflict with safer base behavior.
  • Repair the not-ready review-reply test so it proves the service path, not a swallowed mock crash.

Must Not

  • Do not accept the PR-side unguarded buildChoosableMetrics(locationId, reviews) path.
  • Do not downgrade current Summary validator safety checks.
  • Do not broaden the PR into unrelated dashboard, ranking, or agent prompt rewrites.
  • Do not merge or deploy as part of this fix unless explicitly requested later.

Out Of Scope

  • New product behavior beyond resolving PR 153's conflicts and review findings.
  • Production or dev deployment.
  • Database migrations.

Risk

Level: 3 - merge resolution crosses backend contracts, Summary validation, dashboard API shape, and tenant-isolated data.

Risks Identified

  • Tenant leakage through unguarded competitor lookup. Mitigation: keep the org-validated ChoosableMetricsService path and add/keep tests for mismatched org/location behavior.
  • Summary validation drift if the older PR-side Choosable shape wins the conflict. Mitigation: preserve the richer base contract and update prompt/frontend mirrors only if required.
  • False test confidence in the review-reply not-ready path. Mitigation: mock the full failure path and assert behavior after the service returns normally.
  • Ranking copy regression from sanitizer/prompt conflict. Mitigation: preserve top-three, unknown-position, and no-fabricated-rank guardrail tests.

Blast radius: GET /api/dashboard/metrics, monthly Summary agent inputs and validators, Practice Hub top-action display, Local Rankings narrative output, Patient Journey review/booking cards, and GBP review-reply deployment tests.

Pushback

  • Do not "just resolve conflicts" by picking one side. Future-us will hate that because the base branch already contains safety work the PR branch does not know about.
  • The recommended path is to treat dev/dave as canonical for guarded Choosable metrics and ranking guardrails, then layer in only the PR 153 additions that still fit those contracts.
