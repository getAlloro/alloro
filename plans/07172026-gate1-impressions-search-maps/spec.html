Why

The patient-journey funnel's first gate ("Get Found" = impressions) under-counts: readImpressions summed only GSC organic web-search impressions and ignored Google Maps/local impressions — even though the Maps data is already stored daily. The map pack is where people actually look for a local specialist, so the very first number an owner sees about their own business was confidently wrong (too low). This is the staked gate-1 prerequisite (Jidoka: fix the foundational number before building any get-found lever on top of it — see plans/07142026-alloro-funnel-engine/build-checklist.md § STAKED).

What

Gate 1's impressions value = whole-website GSC organic impressions (counted once) + whole-practice GBP Maps impressions (summed across ALL the org's locations) for the date window. This is the honest whole-practice aggregate — the impressions/visits/leads gates are all shared: true (org-wide, identical on every location tab), so folding one location's Maps into an org-wide organic number would be incoherent. An org has ONE website (one project → one GSC organic total) but MANY locations; per-location impressions is impossible today, so the number is deliberately practice-wide. Additive and best-effort: when the org has stored daily GBP data it is folded in; otherwise the Maps term is simply absent (organic-only), never a fabricated add, and the shared website is never multiplied across locations. Done = the number includes whole-practice Maps, the honesty edge-cases hold, the owner-facing label honestly describes the combined number, tests + tsc pass.

Grounded data path (verified, not assumed)

  • GSC organic → GscDataModel (keyed by project — one org has one project/website). GBP Maps → google_data_store, run_type='daily' (keyed by org, summed across all locations). Separate APIs, separate stores → no double-count.
  • A daily row stores two consecutive days (gbp_data.yesterday↔date_end, gbp_data.dayBefore↔date_start); a location's consecutive/retried runs overlap by one day → de-dup per (location_id, calendar-day), then sum across all locations. A (location, day) only counts when its stored side carries a visibility payload — an empty/missing payload is absent, never a fabricated zero, so metadata.maps.days reflects real coverage.
  • Whole-practice keying: the reader needs only organizationId (already resolved by the caller); locationId is no longer part of the Maps term. The model query is scoped to real locations (location_id NOT NULL) so an org-level aggregate row can never double-count per-location rows, and ordered by (date_start, id) so duplicate rows resolve deterministically.

Reference analog: the existing per-location readers in stageReaders.ts (readRank, readReviews) and the certified-clean src/controllers/gbp-automation/ (§6.1).

Risk

Level: 2 (Concern) — owner-facing number changes (goes up); mis-labeling would mislead.

Risks + mitigations

  • Double-count of organic vs Maps. Mitigation: separate APIs/stores confirmed; per-(location, calendar-day) dedup on overlap.
  • Fabricated/partial data reads as complete. Mitigation: a missing visibility side is absent; days exposes coverage. Honesty caveat (Rev 3): the daily writer ALWAYS emits a visibility object (zero-filled when Google returned nothing, service.agent-input-builder.ts flattenSingleDayGbp), so metadata.maps.days is stored-day coverage, NOT "days Google returned Maps data". A Google-empty day still counts as a stored day. Docstrings corrected to say so.
  • Misleading label (number now includes Maps, label still says "search"). Mitigation: backend label done (T3) AND the SPA labels aligned (Rev 3, T4) — subtext, detail-deck description + footer, and the shared multi-location note (which SHADOWS the source in the SPA tooltip) all now honestly describe search + Maps.
  • Blob-fallback double-count (Rev 3, C1 — flagged, not fixed). The daily processor loops every non-cancelled location (service.proofline-executor.ts:98); a location with no google_properties mapping hits the JSON-blob fallback (service.daily-agent-processor.ts:57-75) and the extractors read gbpData.locations[0] (service.agent-input-builder.ts:20-21) — the FIRST listing — storing it under that unmapped location's own location_id. So N unmapped siblings store the SAME first listing's Maps under N distinct location_ids, and the reader's (location, day) dedup then sums it N×. Why not fixed by identity-dedup: the stored gbp_data carries NO stable listing identity (no place_id / external_id) — only profile.{title,address,phone,website}, which are nullable AND themselves read from the same buggy locations[0]. Deduping on them would trade a detectable over-count for a silent UNDER-count (two profile-less locations, or a legit title collision, collapse). Mitigation: keep (location, day) dedup; flag for Dave with a data-check (see Flags). The gate-1 number CAN inflate for an org that has GBP data but some locations unmapped, until that check clears.
  • CTR blast radius (Rev 3, C3 — staked here, not owner-discovered). Adding whole-practice Maps to impressions RAISES the impressions denominator, so the impressions→visits ratio (the gauge-implied CTR) DROPS. That can (a) flip which step buildConversions marks as the leak and change buildHeadline, and (b) disagree with the detail deck's separate organic-only "Google click-through rate" (gsc.ctr, which stays Search-Console-only). The deck will show a higher organic CTR than the gauge's blended ratio implies. Mitigation: funnel math deliberately unchanged here; flagged for Dave to reconcile the deck CTR and re-check the "Clicked through" caption (patientJourney.utils.ts:164, a known mislabel — it captions the impressions→visits step as clicks).
  • Retroactive shrink on location delete (Rev 3, C4). google_data_store.location_id is ON DELETE SET NULL. Deleting a location NULLs its historical daily rows; the model's whereNotNull("location_id") then excludes them, so a past month's gate-1 impressions shrinks retroactively after a location is deleted. Acceptable (the location is gone), but noted so a "why did last month drop?" question has an answer.
  • Org-with-GBP-but-no-project gap (Rev 3, N2). readImpressions only runs when the org has a project (PatientJourneyService.ts:191 guards on projectId); otherwise the stage is the empty "not connected" read and the whole-practice Maps-only availability branch is never reached. An org with real GBP Maps data but no website project shows no impressions at all. Edge case, noted.

Blast radius: readImpressions consumers = PatientJourneyService.assemblePatientJourney (only caller). GoogleDataStoreModel new method has one consumer (the reader). Frontend copy touched: patientJourney.utils.ts (impressions subtext + type doc), patientJourneyDetailDeck.utils.ts (description + footer) — pure label strings, no data-flow change. Owner-facing surface: the "Get Found" gauge + its detail deck on the patient-journey dashboard → Alloro Docs parity check required before finalize.
