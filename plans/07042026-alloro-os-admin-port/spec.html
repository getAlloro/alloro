Execution Model (read first)

This plan is split into 7 phase files, each a self-contained executable spec with its own tasks, automated checks, and status pill. Execute them in dependency order with -x, one phase per session if desired. Each phase is done only when its Phase Gate (automated commands) passes. The user checks the result once, at the end, via test.html / test-results.json in this folder.

  • Per-phase (autonomous): execute tasks → run the Phase Gate (tsc backend + frontend, npm run check:conventions --strict, targeted vitest incl. supertest smokes) → set that phase file's status pill to Completed → update the Phase Index table below.
  • Deviations: recorded as Rev N entries in the affected phase file AND mirrored in this master's Revision Log.
  • Final (user, once): when all 7 phases are Completed, run the acceptance items in test-results.json (open test.html), then finalize with -d.
  • Commits: holistic per phase (author LagDave), on dev/dave per the branching workflow. Push deploys dev and runs the migration.

Why

Alloro OS (~/Desktop/alloro-os) is a working internal AI-indexed knowledge base — versioned documents, embeddings + semantic search, and a grounded RAG chat — but it lives as a separate app with its own auth, servers, docker stack, and deploy story. Folding it into the Alloro admin removes that entire second operational surface and puts internal knowledge where the team already works (admin tabs: Process · Projects · Support · OS).

What

A new top-level OS tab in the admin top bar (route /admin/os, full-width, super-admin gated) containing the ported OS features as sub-tabs: Library (documents, folders, categories, versions, drafts, locks, trash, imports, related docs, ⌘K search) and Chat (grounded RAG chat with citations, SSE streaming). All UI is recreated against the Alloro app light-mode design system (Spectral serif + Plus Jakarta Sans, warm paper #FAF8F3, terracotta #D66853) — no dark mode, no OS chrome. Done = all 7 phase gates green and the final acceptance checklist passes in one user session.

Explicitly excluded: Tasks/My-Tasks (pmtool owns tasks — comments port without task fields), notifications + Socket.IO (no WS infra port), Google OAuth + OS users/profiles (admin JWT + users table instead), dark mode, mobile tab bar, OS docker/deploy stack, data migration (source DB verified empty 2026-07-04).

Architecture Decisions (locked for all phases)

D1 — New top-level admin tab OS beside Process/Projects/Support in AdminTopBar.tsx; route prefix /admin/os; rendered full-width (no admin sidebar) like PM/Support via isFullWidth in AdminLayout.tsx:14.
D2 — Sub-navigation inside the OS surface: Library (default, /admin/os) and Chat (/admin/os/chat), plus quiet Trash access (/admin/os/trash). Document routes: /admin/os/doc/:id and /admin/os/doc/:id/edit.
D3 — Auth = existing authenticateToken + superAdminMiddleware on every /api/admin/os/* route (§11.1). No OAuth port, no OS users/refresh_tokens tables. People-pickers (owner, author) list internal admin users.
D4 (Rev 2) — Data: 16 new tables in a dedicated Postgres schema os (owner ask; mirrors the existing minds schema precedent) — os.documents, os.document_chunks, os.chat_messages, … (see migrations/). Table names are unprefixed inside the schema; models reference dot-qualified names exactly like MindBrainChunkModel's conn("minds.mind_brain_chunks") — i.e. db("os.documents"). User FKs are integer → public.users(id) (Alloro users; OS used uuid users). Purely additive migration confined to the new schema; production-safe; down() drops only os.* tables then the empty schema. knex_migrations bookkeeping stays in public.
D5 — pgvector: verified installed + active on dev AND prod (v0.8.0, PG 17.9, checked 2026-07-04). Chunks use vector(1536) + HNSW cosine, mirroring the existing minds.mind_brain_chunks migration (20260227000002).
D6 — Embeddings: OpenAI text-embedding-3-small @ 1536 via the house axios pattern (analog: src/controllers/minds/feature-services/service.minds-embedding.ts). Deliberate deviation from OS's 3-large — same dimension, cheaper, matches both existing Alloro embedding call sites. Env-overridable (OS_EMBEDDING_MODEL). A deterministic fake provider (ported from OS) keeps tests hermetic.
D7 — Chat + AI metadata LLM: Gemini gemini-3.5-flash via @google/genai — already a backend dependency (^1.43.0) and GEMINI_API_KEY already provisioned on dev + prod env. Faithful to the OS's deliberate Claude→Gemini move for first-token latency. Isolated behind one feature-service seam (service.os-llm.ts) so an Anthropic swap is a one-file change. Note: OS used @google/genai v2.x — adapt calls to the installed 1.43 API (or bump within the phase if trivially compatible).
D8 — Chat transport: SSE over HTTP (fetch + reader), exactly like src/controllers/minds/MindsChatController.ts:23-59. No Socket.IO anywhere. Edit-locks work via HTTP heartbeat; stale locks reaped by a repeatable BullMQ job.
D9 — Storage: existing S3 helper src/utils/core/s3.ts (bucket AWS_S3_IMPORTS_BUCKET) with an os/ key namespace: os/assets/{docId}/{assetId}, os/imports/{docId}/{filename}. No MinIO, no new bucket.
D10 — Background jobs: BullMQ processors registered in src/workers/worker.ts (runs under the existing minds-worker PM2 app), queue factory getOsQueue(name) in src/workers/queues.ts (prefix {os}). Jobs: os-ingest, os-convert, os-purge, repeatable os-lock-reaper. Idempotent, 3 attempts, backoff (§21).
D11 — Backend domain: src/controllers/admin-os/ (controllers + feature-services/ + feature-utils/), models src/models/Os*Model.ts, routes src/routes/admin/os.ts mounted in src/app.ts. Reference analog: §6.1 src/controllers/gbp-automation/.
D12 — Frontend: recreated components under frontend/src/components/Admin/os/, pages under frontend/src/pages/admin/os/, api modules frontend/src/api/admin-os*.ts, hooks frontend/src/hooks/queries/useAdminOs*.ts, keys in lib/queryClient.ts. Reference analog: §12.1 triad (api/admin-mission-control.ts + its hooks) and MindChatTab.tsx for streaming chat. OS source components are the design/behavior reference only — code is adapted, not copy-pasted wholesale.
D13 — Design: Alloro app light mode only. Reuse existing tokens — --color-alloro-bg #FAF8F3 (identical to OS warm paper), --color-alloro-orange #D66853, --font-display Spectral / --font-sans Plus Jakarta Sans / --font-mono JetBrains Mono (all already loaded in frontend/index.html:11). Spectral for document titles/reading body only; Jakarta for UI chrome; mono for meta/status. Hairline-divided rows over card grids; Lucide icons; status dots (indexed=green · indexing=amber pulse · failed=red · archived=muted).
D14 — Fresh start: no content migration (local OS Postgres verified 0 rows in all 21 tables, 2026-07-04). Seeding happens organically through the Imports feature (P6).

Constraints

  • All new backend queries in src/models/Os*Model.ts (§7.4); vector/tsvector raw SQL parameterized (§10.2), inside models only.
  • { success, data, error } envelope + domain OsError + one handleOsError (§8.1–§8.3); SSE endpoint exempt from envelope but not from auth.
  • Frontend data flow through api/admin-os*.ts + React Query hooks + QUERY_KEYS (§12.1, §14.2, §15.1); errors via react-hot-toast (§16.3).
  • Fonts/colors only via existing tokens (D13); Spectral reserved for document titles/body.
  • Each phase ships vitest coverage for its new models/services (§20.1–§20.2) with fake AI providers (§20.4).
  • Migration runs on dev first via dev/dave push (AGENTS.md deployment path).

Must Not

  • No Socket.IO / WebSocket infra; no new state library (§15.4).
  • No edits to client-dashboard pages, PM tool, or Support surfaces beyond the three shell anchor files (D1).
  • No NODE_ENV-gated behavior (repo rule: uniform runtime).
  • No new dependencies beyond the per-phase lists in P3/P6; nothing else without a Rev entry.
  • No task fields/UI on comments; no notifications tables.
  • No destructive migration statements outside the os schema; never drop the shared vector extension.

Out Of Scope

  • Tasks/My-Tasks (pmtool), notifications + presence, OS profile/settings pages, dark mode, mobile bottom-nav, Google OAuth, OS data migration, PDF vision-fallback transcription (P6 lists it as a deferred option), client-facing exposure of any OS feature.
  • Alloro Docs parity: N/A — internal super-admin tool, not client-dashboard UI (state again at -d).

Risk

Level: 2 — all schema is additive and namespaced; UI edits touch three small shell files; the heavy lifting is new, isolated code. No Level 3+ items remain after recon (pgvector and all API keys verified present on both environments).

Risks Identified

  • R1 — @google/genai major-version drift (installed 1.43 vs OS code written on 2.x). Mitigation: D7 seam file; P5 gate includes a live one-shot generation test; bump the package only if the 1.43 API lacks streaming parity, recorded as a Rev.
  • R2 — Embedding/LLM spend on ingest (per-save re-embed). Mitigation: re-ingest only on publish (not autosave), chunk-level batching, tiny corpus (team KB), 3-small pricing; fake providers in tests.
  • R3 — HNSW index build cost on writes. Mitigation: corpus is small (hundreds of chunks); same pattern already live for minds brains.
  • R4 — P6 parser deps (pdfjs-dist, turndown, jszip) enlarge the backend. Mitigation: confined to P6 (skippable phase); mammoth/xlsx/multer already present; @napi-rs/canvas vision fallback explicitly deferred.
  • R5 — Worker deploy: new processors must land together with queue producers. Mitigation: producers no-op safely when queue is empty; PM2 minds-worker reload is part of the existing deploy workflow.
  • R6 — Admin bundle growth (TipTap extensions). Mitigation: lazy-load the editor route exactly like OS did (React.lazy).

Blast radius (all additive edits): AdminTopBar.tsx, AdminLayout.tsx, frontend/src/pages/Admin.tsx, frontend/src/lib/queryClient.ts, src/app.ts, src/workers/queues.ts, src/workers/worker.ts, both package.jsons, one new migration. Everything else is net-new files.

Pushback

  • Chat provider: house pattern is Anthropic (minds). Kept Gemini (D7) because the OS made that switch deliberately for streaming latency and the dep+key already exist — but if a single LLM vendor matters more than first-token speed, swap the seam file to @anthropic-ai/sdk streaming (model claude-sonnet-4-6) during P5; nothing else changes.
  • Two knowledge systems now exist in-repo (minds brains for client agents, OS for internal docs). They serve different audiences and share only the pgvector pattern — do not merge them; do keep the embedding call shape identical so a future shared EmbeddingService extraction is mechanical.
