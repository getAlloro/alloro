Why

The "AI-indexed" half of the knowledge base: without ingest + retrieval there is no semantic search, no related docs, and no grounding for P5 chat. This phase turns the P1 stub processor into the real pipeline.

What

On create/publish/reindex the os-ingest job runs: heading-aware chunking → batch OpenAI embeddings (1536) → atomic chunk replacement → Gemini metadata (summary/category/tags, honoring meta_locked; title-fallback on model failure) → top-5 link suggestions (cosine ≥ OS_LINK_SUGGEST_FLOOR) → weighted tsv rebuild → status indexed. Three attempts with backoff, then processing_failed + a visible Reindex path. Retrieval service exposes parameterized cosine search (floor OS_SIMILARITY_FLOOR, k=OS_RETRIEVAL_K, indexed+non-archived only). UI: live status dots (poll while processing), Reindex button, ⌘K command palette scoped to OS routes (FTS + semantic + recent + nav), and a real Related rail (accept/reject suggestions, manual add).

Risk (phase-local)

  • Provider outages mid-ingest. Mitigation (ported): retry/backoff; metadata failure degrades to title-fallback (doc still becomes indexed); embedding failure → processing_failed + Reindex button.
  • Chunk replace vs concurrent retrieval. Mitigation: replace-all inside one transaction keyed by version_no; retrieval reads committed rows only.
  • Cost: re-embed on publish only (never autosave); batch embedding calls.
