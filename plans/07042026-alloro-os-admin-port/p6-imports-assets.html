P6 — File Imports (docx · xlsx · pdf · md) & Editor Assets

Why

The KB starts empty (D14). Imports are how the team seeds it — drag in the existing Notion exports, Word docs, and spreadsheets and get indexed markdown documents back. Editor image upload makes documents self-contained.

What

Batch upload (≤OS_IMPORT_BATCH_MAX_FILES files, ≤OS_IMPORT_MAX_FILE_MB each; multer memory → S3 os/imports/) creating documents in processing with an os.document_imports row each, then the os-convert job converts to markdown v1 — docx via mammoth→HTML→turndown(gfm), xlsx via sheet→markdown tables, pdf via pdfjs text extraction (low-text pages append a warning; vision transcription deferred), md passthrough — extracting embedded images to S3 os/assets/ + os.assets rows with rewritten markdown refs, recording warnings, setting converted/failed, then enqueueing ingest (P4). Grouped import can assign a category to the whole batch (OS grouped-import feature). Editor: paste/drop image upload → asset endpoint → markdown image insert. Asset delivery via redirect to a presigned S3 URL.

Constitution

§4.4 (dep list is closed), §5.2 (filename/path sanitize, mime allowlist), §11.2, §21.1–§21.4 (convert idempotent by import id), §10.5 (import row + doc creation transactional), §20.4 (fixtures synthetic). Analog: S3 usage in src/utils/core/s3.ts.

Risk (phase-local)

  • Malformed uploads (zip bombs, wrong extension). Mitigation: mime + extension allowlist, size caps, converter try/catch → failed + warning, never crash the worker (§3.1).
  • pdfjs-dist in Node needs the legacy build — pin the import path; text-only extraction keeps it dependency-light (canvas deferred with a visible warning instead).
  • S3 key collisions. Mitigation: keys include document id + uuid asset id (D9 namespace).
