Why

The generic org-type work is not complete in the PMS surface. Generic accounts still see health-specific copy such as PMS, practice management, Referral Source, Doctor referrals, Patient ID, and Production.

For generic organizations, this surface should read as a revenue-data workflow. It should ask users to upload revenue data, group revenue by neutral sources or channels, and avoid referral-source framing because generic businesses do not directly manage healthcare referrals.

What

Add PMS-specific label coverage on top of the existing org-type label system. Health organizations keep the current PMS/referral wording. Generic organizations see Revenue data, Revenue Hub, Revenue source or Source / Channel, records, customers, and revenue across upload, mapping, review, dashboard, processing, file-manager, shared alert, setup, and AI action states.

This is primarily a display-layer change. It must not rename PMS routes, API payload fields, database columns, monthly rollup keys, or agent data contracts. Rev 1 expands the scope only where Pilot QA proved it is necessary: shared dashboard alerts, setup copy, generated monthly-agent prompts, and a narrow generic display fallback for stale freeform AI copy already stored before prompts are clean.

Constraints

Must

  • For health orgs, preserve current wording: PMS, Practice Management System, Referral Source, Patient ID, Doctor referrals, Self referrals, and Production.
  • For generic orgs, visible PMS-surface copy must use revenue-data language and must not show PMS, Practice Management System, Referral Source, Doctor referrals, Patient ID, or Production.
  • Keep the source concept where the workflow needs it, but name it neutrally for generic orgs: Revenue source or Source / Channel, not Referral Source.
  • Use label helpers so components consume resolved strings, not explicit org-type checks sprinkled through JSX.
  • For generic orgs, shared dashboard alerts and setup states must use revenue-data language. The failed strings are May 2026 PMS data ready?, Upload PMS data, and referral and production insights.
  • For generic orgs, freeform AI/action text rendered in Practice Hub and Revenue Hub must not show healthcare words such as patients, doctor referral source, referrals, or production.
  • Existing stored agent output must be handled without rewriting historical database rows. Either translate only freeform display text for generic orgs or suppress stale healthcare copy until a clean generic rerun exists.
  • Update Alloro Docs parity because this changes dashboard labels, copy, walkthrough wording, and upload guidance.

Must Not

  • Do not rename PMS* component names, route paths, API functions, database tables, or response keys.
  • Do not change ingestion behavior, monthly rollup math, source classification, file parsing, or agent trigger logic.
  • Do not broaden this into N8N workflow edits. In-repo monthly-agent prompt clean-up is now in scope only because Pilot QA proved AI action text is visible in generic accounts.
  • Do not run the monthly agent pipeline during planning or execution unless explicitly needed for verification and confirmed email-safe. Prompt tests and stored-output display checks come first.
  • Do not use a global blind find-replace. Some terms are internal code/data names and must stay stable.
  • Do not touch unrelated dirty work in Patient Journey, Market Intelligence, embedded pilot, billing, or admin shell files.

Assumptions

  • The generic PMS surface should say Revenue data, not Lead data. This is stricter than the earlier broad generic map because the user specifically said generic organizations do not directly manage referrals in this context.
  • The generic mapping flow may still ask for a source/channel column because revenue attribution still needs grouping. The banned wording is referral-source framing, not the source grouping itself.
  • Direct/partner split values may remain in the data model as self/doctor. Generic UI labels should render them as neutral source categories or hide the healthcare-specific explanation.

Out Of Scope

  • Backend schema changes, org-type setup changes, Stripe pricing, N8N workflow prompt edits, and generic data-pipeline redesign.
  • Rewriting historical stored agent result rows in the database. The UI may transform or suppress stale freeform generated prose at render time, but the stored audit trail remains unchanged.

Risk

Level: 3. The behavior is display-layer only, but the blast radius is large across PMS upload, dashboards, file management, and docs.

Risks Identified

  • Data-model drift. Renaming internal PMS/referral keys would break parsers, rollups, and existing API consumers. Mitigation: copy changes stop at render labels and helper functions; data keys stay unchanged.
  • Inconsistent generic language. Some components currently use global Lead labels while PMS needs Revenue data. Mitigation: add PMS-specific labels instead of overloading the app-wide referralsShort keys.
  • File size creep. PMS components are already broad. Mitigation: compose labels in a typed helper or nested label object, then keep JSX edits small.
  • Docs drift. The docs app has hardcoded Referrals Hub and PMS-upload copy. Mitigation: execute docs parity in the same plan and verify the docs build separately.
  • False product promise. Calling it revenue data does not make the ingestion engine truly generic. Mitigation: avoid claims that Alloro can ingest arbitrary revenue systems; copy should say upload revenue data in the supported report shape.
  • Stale generated copy. Existing Summary / Referral Engine outputs were generated under older healthcare-heavy prompts, so generic org-type switching can still render old prose. Mitigation: clean in-repo prompts for future runs and add a narrow generic display fallback or suppression rule for freeform action text only.
  • Over-sanitizing real data. Source names, organization names, and location names can legitimately contain words that look medical or dental. Mitigation: do not rewrite raw entity names; only translate Alloro-authored labels and freeform generated prose fields such as action title/rationale/description.

Blast radius: frontend/src/constants/orgLabels.ts, PMS upload/modals/dashboard/file-manager components, shared dashboard alert utilities, setup/upload banners, Summary/Referral Engine action rendering, monthly-agent prompt files, Alloro Docs Referrals Hub and Integrations docs, and tests added for label/prompt resolution.

Pushback

  • This should not be treated as "finish all generic verticalization." It is a PMS-surface vocabulary correction. N8N prompts, external workflow payloads, and real generic ingestion semantics still need their own audit.
  • The better product name may eventually be Revenue Hub with no internal "PMS" concept at all. Future-us will hate a half-rename. For this plan, keep code identifiers stable and fix the user-visible layer only.
