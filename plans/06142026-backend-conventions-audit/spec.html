Executive Summary — Audit Complete

All 17 units audited (read-only, via subagents). Roughly 61 must-fix · 100 concerns · 78 nitpicks across the backend, plus 5 systemic breaches that recur on top of the per-unit counts. The audit was scoped to conventions but surfaced a cluster of live, verified security exposures that outrank everything else — called out first.

⚠ Security — verified, urgent (beyond the original conventions scope)

  • Unauthenticated admin / destructive endpoints. No global auth guard in index.ts; auth is per-router and many omit it. Verified bare-mounted with zero auth middleware: /api/admin/settings, /api/admin/app-logs, /api/admin/agent-insights, /api/admin/agent-outputs, /api/admin/pms-jobs, the /api/pms admin block (incl. DELETE /jobs/:id, marked "// No auth" in source), agentsV2 (IDOR — read any org's outputs), and the minds public skill API (proprietary-prompt theft + LLM cost). routes/admin/websites.ts: 176/210 registrations unguarded. The middleware guards themselves are sound → fix = mount them.
  • JWT secret fails open. auth.ts:13 || "dev-secret-key-change-in-prod" and billingGate.ts:69 || "secret" — forged tokens validate if JWT_SECRET is unset, and the two different literals desync sign/verify. Fix: one config source, throw at startup if unset.
  • Secrets committed in source. utils/core/domainMappings.ts holds 16 hardcoded JWT-shaped Clarity tokens. Rotate + move to env regardless of refactor.
  • Auth hygiene. OTP + temp-passwords use Math.random() (not crypto); one-time codes logged in plaintext (and the log endpoint is unauthenticated); a tester@google.com bypass util exists (verify it's disabled in prod); zero rate-limiting on login/OTP/reset (limiters exist but unwired); the public websiteContact upload form has its anti-abuse stack commented out + reCAPTCHA fails open.

Recommendation: a small security hotfix track, separate from and ahead of the slow conventions remediation.

Systemic breaches (repo-wide — count once, not per file)

  1. No request validation. 0 / 39 route files validate input; the lone validation/ schema validates LLM output. No validation middleware exists (dim 12).
  2. Pino not installed. The contract assumes Pino; it's absent from package.json. console.* is the de-facto logger across ~277 files. Owner decision: adopt Pino or amend the contract (dim 10).
  3. DB access outside models/. ~125 files / ~507 sites call db("table") directly in controllers/services/utils/workers/routes (dim 7).
  4. Response shape ignored. The nested {success,data,error:{code,message,details}} contract is essentially unused; a flat shape dominates (dim 9).
  5. No transactions on multi-table writes. Recurs across services + workers (websiteRestore wipes ~13 tables + S3 with zero transaction) — partial-write corruption risk (dim 16).

Rulings & reference

Nesting: feature-services/ + feature-utils/ (and pms-services/, user-website-services/ variants) is ACCEPTED — keep repo-wide; carve-out: must not be used to skip models/. Reference domain: gbp-automation is the model to copy (thin controllers, PascalCase *Service, the repo's only real {success,data,error} + sanitizer standard, mature partial-failure handling, 0 must-fix). Correction: user-website is admin-gated, not public — the real public renderer is a separate app.

Why

The backend (src/) has grown to ~743 files / ~142K lines, and an initial recon found 128 files (~17%) over the 300-line cap alone — before checking any other rule. We need a structured, per-unit map of where the codebase drifts from the /alloro-conventions contract so remediation can be planned and prioritized instead of guessed at.

This plan maps the audit. It does not run it. Map now, check later.

What

One master index (this file) plus 17 per-unit audit worksheets under checks/. Each worksheet scopes one slice of the backend (a feature domain or a cross-cutting layer), lists its real files with line counts, pre-flags the size offenders and unit-specific red flags found during recon, and carries the 16-point checklist plus an empty Findings section to fill during the check-later phase.

Done (this plan): all 17 worksheets exist and map real files. Done (execution, later): every worksheet audited, findings recorded, a ranked remediation backlog produced.

Context

Pattern note (decide once, repo-wide)

  • The repo has evolved domain-nested sub-folders: feature-services/, feature-utils/ (admin-websites, practice-ranking, agents), plus variants pms-services/ / pms-utils/ and user-website-services/. The skill's example uses plain services/ / utils/ nesting. This is either accepted evolution or naming drift — rule on it during the first scan and apply the ruling everywhere.

Reference unit: start with checks/13-routes.html (checkup.ts = 2,140 lines) — the single clearest, fastest-to-confirm violation. It validates the worksheet format before the big units.

Constraints

Must

  • Audit read-only — no application code changes during any scan.
  • Check every file in a unit against the 16-point checklist, not just file size.
  • Record findings inside the unit's own worksheet (spec-code parity).

Must Not

  • Modify, refactor, or "drive-by clean" any source during mapping or scanning.
  • Scan the frontend — that is the /alloro-frontend contract, a separate effort.
  • Scan database/migrations or seeds — different rules apply.

Out Of Scope

  • Fixing the violations. Remediation is a separate plan, written after findings exist.
  • Frontend (frontend/) and migration/seed files.

Risk

Level: 2 (Concern) — the audit itself is read-only and safe; the findings are expected to surface Level 3+ architectural debt (god-controllers, DB access outside models, business logic in routes).

Risks Identified

  • File-size is a weak proxy for real violations. Mitigation: every worksheet carries the full 16-point checklist; size offenders are only the starting point, not the verdict.
  • The 09-smaller-controller-domains bucket spans 247 files / 38 domains and could hide security-critical drift. Mitigation: sub-prioritize auth, billing, and websiteContact first within that worksheet.

Blast radius: none — no source files are touched by this plan or by the scans.

Pushback

  • Auditing all 741 files exhaustively is expensive. Recommended alternative: run the worksheets in the priority order below, stop at diminishing returns, and let the P1 findings seed the remediation plan rather than waiting for 100% coverage.

Audit Dimensions (the 16-point contract)

Every file in every unit is checked against these. Worksheets carry the same list as a runnable checklist.

  1. File ≤ 300 lines
  2. Functions ≤ 50 lines
  3. Nesting ≤ 4 levels
  4. Correct folder placement
  5. Data flow: Routes → Controllers → Services → Models
  6. No business logic in routes
  7. No DB queries outside models/
  8. Error handling on every async call
  9. { success, data, error } response shape
  10. No console.log — Pino logger only
  11. No hardcoded secrets or magic values
  12. Input validated/sanitized at route/middleware level
  13. Auth middleware on protected routes
  14. Naming (PascalCase controllers/services, descriptive fns, boolean-as-question)
  15. No commented-out code
  16. Transactions for multi-table writes
