Findings

Summary

Verdict: needs changes — decomposition-first, not a security panic. All 3 files exceed the 300-line cap and the controller is 5.6× over (1,690). The dominant violations are structural: oversized files/functions and DB access outside models/. Security-wise the brief's premise is wrong for this layer. Every route in src/routes/user/website.ts is gated by authenticateToken + rbacMiddleware + requireRole("admin"|"manager") — there is no anonymous traffic here. The genuinely public renderer is a separate app (website-builder-rebuild, per the resolver's own header). So the attacker model is an authenticated admin/manager editing their own org's site, which downgrades XSS/injection from "must-fix" to "concern." No SQL injection exists — every db.raw is a static literal and all variable inputs go through Knex's parameterized builder. Counts: 0 critical · ~4 must-fix · ~6 concerns · ~5 nitpicks.

Must-fix

  • UserWebsiteController.ts:1 — [dim 1] 1,690 lines, the worst density in the backend. Fix: split by domain into separate controllers (forms/submissions, posts, post-types+taxonomy, menus, domain, GSC, analytics, pages/versions). Each is an independent route group already — decomposition is mechanical and the primary remediation for this unit.
  • shortcodeResolver.service.ts:1 (1,003) & userWebsite.service.ts:1 (576) — [dim 1] both over cap. Fix: resolver splits cleanly into post-block / review-block / menu / conditional-engine modules; service splits into media, page-edit, and version-history services.
  • UserWebsiteController.ts:798-817, 1223-1296, 1459-1462, 1478-1480, 1495, 1514-1516, 1534-1537 & userWebsite.service.ts:324, 376, 410-411, 450-541 & shortcodeResolver.service.ts:187, 224, 284, 293, 629, 785, 797, 814 — [dim 7] DB access outside models/. 26 inline db(...) calls across the unit (10 controller / 6 service / 10 resolver), incl. raw db.raw aggregation, page-save writes, taxonomy CRUD, and the entire shortcode read layer. The controller writing rows directly (e.g. category/tag insert, post SEO update) is also [dim 6] business/DB logic in the controller layer. [systemic] repo-wide, but this is one of the densest offenders. Fix: move all queries into the relevant Models (FormSubmissionModel, PageModel, PostModel, MenuModel, PostBlockModel, etc.); controllers/services call models only.
  • UserWebsiteController.ts:1205 savePageSections (115), :773 getFormSubmissionsTimeseries (111), :687 getWebsiteAnalytics (87), :206 editPageComponent (82), :910 listFormSubmissions (78) and userWebsite.service.ts:427 restorePageVersion (132), :245 editPageComponent (114) and shortcodeResolver.service.ts:219 fetchFilteredPosts (122), :488 renderPostBlock (113) — [dim 2] functions far over the 50-line cap (9 functions, worst is 132). Fix: extract named helpers (e.g. concurrency-check, snapshot, version-bump in savePageSections; query-build vs token-replace in the resolver). Decomposition into models (above) shrinks most of these automatically.

Concerns

  • UserWebsiteController.ts:206 editPageComponent, :1320 resolvePreview, :1205 savePageSections — [dim 12] raw currentHtml / html / sections are accepted, then stored and re-served. resolveShortcodes escapes most tokens (escapeHtml) but intentionally injects three raw HTML sinks: {{post.content}} (shortcodeResolver.service.ts:537), the marker wrapper at :944, and buildVideoEmbed at :568/:951. This is a stored-XSS surface, but it is auth-gated to an admin/manager mutating their own org's site, and the public renderer re-resolves independently — so it is a real concern, not the must-fix the brief implied. Fix: sanitize stored HTML (e.g. DOMPurify server-side) on the edit/save path, or document the trust boundary explicitly. Track as systemic [dim 12] — 0/route input-validation schemas in this unit.
  • shortcodeResolver.service.ts:64-66, 102-110 — [dim 12] shortcode attribute parsing is hand-rolled regex with no length/cardinality bounds; a pathological template could enqueue many sub-queries (one extra db round-trip per post block, per menu, per review block — see N+1 below). Inputs feed Knex .where so they are parameterized (no SQLi / no path traversal — slugs never hit the filesystem). Fix: cap shortcode count + attribute list length per resolve call.
  • shortcodeResolver.service.ts:200-214 (post blocks) & :784-837 (menus) — [dim perf] N+1 query pattern: a per-shortcode loop issues fetchFilteredPosts (itself 1-3 queries) / per-menu menu_items + menu_templates lookups. Bounded by template authoring today, but unbounded in principle. Fix: batch by slug where feasible (post-block metadata is already batched; the post fetch is not).
  • UserWebsiteController.ts:50-91 — [dim 9] dual ad-hoc error handlers (handleError/handleGscError) emit a non-standard {success,error,message,...} shape, and success responses across the file are flat ({project,pages,...}, {success,allCount,...}) rather than the contract's {success,data,error}. [systemic] flat shape dominates repo-wide. Fix: none required unless standardizing globally; note for the systemic rollup.
  • userWebsite.service.ts:69-103, 217-239 — [dim 5] data-flow inversion: the service performs the tier/auth gate (getOrgAndValidateTier) which is closer to a middleware/authorization concern, while the controller performs raw DB writes. Layering is muddled in both directions. Fix: push the DFY-tier check into middleware; pull DB writes down into models.
  • UserWebsiteController.ts:1138-1199 exportFormSubmissions — [dim 8/11] hard-caps export at limit: 10000 (magic number, no pagination) and on error returns a bare res.status(500) bypassing the shared handler. Fix: stream/paginate the export; extract the cap to a named constant in config/.

Nitpicks

  • userWebsite.service.ts:305, 338, 548, UserWebsiteController.ts:66, 276, shortcodeResolver.service.ts:374, 428, 1000 — [dim 10] 8 console.* calls (log/warn/error). [systemic] Pino not installed repo-wide. Fix: swap to Pino when the logger lands.
  • userWebsite.service.ts:32-34 — [dim 11] in-file constants (DAILY_EDIT_LIMIT=50, USER_STORAGE_LIMIT=1GB) are named but live in the service rather than config/. Minor; they are at least not bare literals.
  • UserWebsiteController.ts:1477, 1513 — [dim 11] slug-generation regex (name.toLowerCase().replace(...)) is duplicated inline in createUserCategory and createUserTag. Fix: extract a slugify util.
  • shortcodeResolver.service.ts:328-333, 422 — [dim 15-adjacent] the conditional engine is flagged (in comments) as triplicated across this file, website-builder-rebuild, and a frontend component, with a "keep in sync" note. Not commented-out code, but a known DRY/drift hazard worth a shared module.
  • UserWebsiteController.ts:151, 154-161 etc. — [dim 14-adjacent] heavy (project as any) casting throughout fetchUserWebsiteData indicates the Project model type is incomplete. Fix: extend the type instead of casting.
  • No commented-out code found [dim 15] — clean. No hardcoded secrets [dim 11] — clean.
