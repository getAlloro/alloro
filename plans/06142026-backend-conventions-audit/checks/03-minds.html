Audit Dimensions

Run every file in scope against the 16-point contract (full detail in spec.html).

  • File ≤ 300 lines
  • Functions ≤ 50 lines
  • Nesting ≤ 4 levels
  • Correct folder placement
  • Data flow: Routes → Controllers → Services → Models
  • No business logic in routes
  • No DB queries outside models/
  • Error handling on every async call
  • { success, data, error } response shape
  • No console.log (Pino only)
  • No hardcoded secrets / magic values
  • Input validated at route/middleware

  • Naming conventions
  • No commented-out code

Findings

Summary

Confirmed the healthiest big domain. Routes are exemplary (thin, blanket-authed); models own DB access; transactions are used correctly; magic values are all named env-backed constants; zero commented-out code. The real risk is concentrated and unit-specific: a fully unauthenticated public skill-neuron endpoint that leaks proprietary system prompts and triggers LLM cost with no rate limit, and inconsistent error-handling on external LLM/embedding calls — the slowest, most failure-prone dependency is the least guarded, while the web-scraper calls in the same domain are guarded correctly. Severity: 3 must-fix · 5 concerns · 5 nitpicks (systemic items folded in, tagged).

Must-fix

  • ../../routes/mindsPublicApi.ts:15-17 + MindsSkillApiController.ts:7-49 — [dim 13 / security] GET/POST /api/minds/:agentSlug/:skillSlug is explicitly "no auth required" and returns the skill's entire neuron_markdown (the proprietary generated system prompt) as plain text, plus logs a call. Any caller can exfiltrate every agent's skill IP and drive load. No portal-key, no rate limit. Fix: require x-portal-key/x-internal-key like the portal endpoints do (mind/skill portal already verify keys), or gate behind auth middleware; add rate limiting at mount (index.ts:206-207).
  • feature-services/service.minds-embedding.ts:28-62, 265-286 and the Anthropic SDK calls in service.minds-chat.ts:200,257, service.minds-skills.ts:427,483,590, MindsPortalController.ts:86 — [dim 8] external OpenAI axios.post and Anthropic messages.create/stream have no timeout and no try/catch. A hung socket blocks the request indefinitely; an API error propagates raw. The same domain's scraper/discovery do set FETCH_TIMEOUT (service.minds-scraper.ts:91, service.minds-discovery.ts:193) — so the pattern exists and is simply not applied to the LLM/embedding path. Fix: add request timeouts (axios timeout; Anthropic client timeout/AbortSignal) and wrap each external call in try/catch with a typed error.
  • feature-services/service.minds-embedding.ts:292-356 (regenerateEmbeddings) — [dim 16] deletes all existing chunks (MindBrainChunkModel.deleteByMind, line 322) then generates embeddings + summary + bulkInsert (349) with no transaction. If embedding generation or the insert fails after the delete commits, the mind is left with zero RAG chunks (silent data loss); callers label this "non-blocking" and swallow the error (service.minds-crud.ts:107). Fix: wrap delete+insert in db.transaction, or generate embeddings first and only delete inside the same trx as the insert.

Concerns

  • MindsSkillsController.ts:7-33, 35-124, 147-160, 246-288 — [dim 5/6] controllers bypass the service layer and call models directly (MindSkillModel, MindModel, MindSkillNeuronModel), and embed business logic in the controller: listSkills computes stale-neuron enrichment inline, and regenerateStaleNeurons runs the whole stale-detection + regeneration loop in the controller. Contrast MindsController.ts, which is a clean thin controller delegating to service.minds-crud. Fix: move enrichment + stale-regeneration orchestration into service.minds-skills.
  • MindsPlatformCredentialsController.ts (whole file) — [dim 5] controller calls PlatformCredentialModel directly with no service layer; CRUD orchestration lives in the controller. Lower stakes than skills (thin CRUD), but same layering drift. Fix: introduce a credentials service or accept as a documented thin-CRUD exception.
  • Response shape [systemic dim 9] — across all minds controllers, success returns {success:true, data} (missing error:null) and errors return bare {error:"msg"} (no success:false, no machine-readable code). Worse: MindsPlatformCredentialsController.ts:15,51 return raw model objects (res.json(credentials) / res.json(safe)) with no envelope at all, and MindsSkillApiController.ts:44 returns text/plain. Fix: standardize on the {success,data,error} contract (defer to repo-wide remediation).
  • Input validation [systemic dim 12] — 0 route-level validation schemas; every minds route validates ad-hoc in the controller (if (!name), if (!message?.trim()), type guards in createCredential). Functional but inconsistent and unenforced. Fix: move to validation/ middleware on the route (defer to repo-wide remediation).
  • Console logging [systemic dim 10] — 135 console.* calls across all 30 .ts files in scope (Pino not installed). Heaviest: MindsSkillsController, MindsSkillUpgradeController, MindsParentingController (12 each). Fix: swap to Pino logger when adopted repo-wide.

Nitpicks

  • [dim 2] Oversized functions (often inflated by large inline prompt literals, but still over the 50-line cap): service.minds-skills.ts skillBuilderChatStream (~117), generateNeuron (~95), executeSkill (~86); service.minds-parenting.ts triggerReadingStream (~150), triggerReading (~101); service.skill-upgrade.ts triggerReadingStream (~154). Fix: extract prompt builders / stream-parse helpers (the SSE JSON-field streamer in skillBuilderChatStream is a self-contained helper).
  • [dim 1] 8 files over 300 lines (worst 647). All are cohesive single-responsibility units; size is from prompt strings + LLM orchestration, not tangled concerns. Split only opportunistically (service.minds-skills.ts could shed the skill-builder chat into its own file).
  • [dim 14] Type-safety drift: 89 Promise<any> controller return types and ~99 :any/as any occurrences (e.g. mind: any in service.minds-chat.ts:118, (event as any) SSE casts at service.minds-chat.ts:276-281, error: any everywhere). Hides real shapes. Fix: type controller handlers as Promise<void> and narrow error/event types.
  • MindsSkillApiController.ts:30,39 — duration metric is Date.now() - startTime measured immediately after startTime (no work between), so it always logs ~0ms. [dim 11] meaningless value. Fix: capture startTime before the model lookups, or drop the field on this delivery-only path.
  • utils/portalKey.ts:24-27 — verifyPortalKey calls crypto.timingSafeEqual on the two hex strings without a length guard; throws if lengths differ. Safe in practice (both are 64-char sha256 hex), but a malformed stored hash would crash instead of returning false. Fix: length-check before compare. (File is otherwise the cleanest in scope — proper hashing, constant-time compare; utils/ nesting is the accepted ruling.)
