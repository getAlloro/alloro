Priority Signals (start here)

  • AdminWebsitesController.ts = 5,567 lines (18× the cap). Near-certain god-controller mixing orchestration + business logic + likely inline DB access. The top single remediation target in the entire backend.
  • 27 of 50 files over cap; eight services sit in the 800–2,184 range. Decomposition + data-flow audit, not just trimming.
  • This domain defines the feature-services/ + feature-utils/ nesting pattern. Rule on accepted-vs-drift here — the verdict propagates to T02/T04 and beyond.
  • Verify business logic and DB access have not leaked out of models/ into the controller and feature-services (dimensions 5–7).

Audit Dimensions

Run every file in scope against the 16-point contract (full detail in spec.html).

  • File ≤ 300 lines
  • Functions ≤ 50 lines
  • Nesting ≤ 4 levels
  • Correct folder placement
  • Data flow: Routes → Controllers → Services → Models
  • No business logic in routes
  • No DB queries outside models/
  • Error handling on every async call
  • { success, data, error } response shape
  • No console.log (Pino only)
  • No hardcoded secrets / magic values
  • Input validated at route/middleware
  • Auth middleware on protected routes
  • Naming conventions
  • No commented-out code
  • Transactions for multi-table writes

Findings

Summary

This is the backend's rot epicenter and the verdict matches. AdminWebsitesController.ts is a genuine god-controller: 5,567 lines, 166 exported handlers, ~10 distinct domains, with business logic, domain models, and 28 raw db("website_builder.…") queries (including 4 multi-table db.transaction blocks) inlined directly in the controller. The feature-services/ + feature-utils/ split is sound and worth keeping — but it is undermined by ubiquitous direct DB access (805+ query-builder call-sites across 26 files, bypassing 33 existing models) and a handful of 200–450-line orchestrator functions. The most serious finding is orthogonal to size: 176 of 210 route registrations carry no auth middleware, and the router is mounted bare with no app-level guard, leaving most admin website endpoints (including destructive writes) publicly reachable. Severity counts: 5 must-fix · 6 concerns · 4 nitpicks.

feature-services nesting verdict

Accepted evolution — keep it, with one carve-out. Co-locating domain-specific services and utils beside the controller is explicitly blessed by the contract (Folder Structure: "Controllers can contain domain-nested services/ and utils/"). The service.*.ts / util.*.ts filename prefixes are a clear, self-documenting local convention and the internal decomposition of these files (named helpers, single-responsibility headers) is genuinely good. The pattern is fine to apply repo-wide. The carve-out: the convention is being used as a dumping ground that excuses skipping models/. Domain-nested services are allowed; domain-nested services running raw Knex are not. Adopt the folder pattern, but pair it with a hard rule that every db(…) call moves into models/website-builder/* (which already exists with 33 models). Nesting = accepted; DB-in-services = drift.

Must-fix

  • routes/admin/websites.ts (mounted bare at index.ts:183) — [dim 13] 176 of 210 route registrations have NO auth middleware; only ~34 (GSC/integrations + a few recent routes) spread ...adminWebsiteAuth/...adminGscAuth. Unguarded routes include POST / (createProject), POST /start-pipeline, DELETE /…/post-types/:postTypeId, DELETE /…/review-blocks/:id, PATCH /pages/:pageId/generation-status, etc. No app-level guard exists on /api/admin/websites. This is an unauthenticated admin write surface (Level 4 blast radius). Fix: apply auth at the router level — router.use(authenticateToken, superAdminMiddleware) at the top of the file — then remove the now-redundant per-route arrays, keeping the stricter adminGscAuth (adds rbacMiddleware) where it differs. Verify nothing relies on a route being public.
  • AdminWebsitesController.ts (whole file) — [dim 1] 5,567 lines, 18.5× the 300-line cap; [dim 6] business logic in the controller. Fix: decompose by the 10 domains already named in its own header doc-comment (Projects, Templates, Pages, HFCM, Posts, Menus, SEO, Reviews, AI-Command, Locations) into per-domain controller files; push all orchestration/business logic into the existing feature-services.
  • AdminWebsitesController.ts — [dim 7] 28 raw db("website_builder.…") queries inline in the controller (e.g. lines 327, 585, 597, 727, 1069, 5014, 5147, 5257, 5337). [dim 16] Four of these are multi-table db.transaction blocks (addProjectLocation 5105, setPrimaryLocation 5212, removeProjectLocation 5296, plus L597) writing ProjectIdentityModel + raw projects — transaction handling is correct but lives at the wrong layer. Fix: move every query into models/website-builder/* (ProjectModel/PageModel/etc. already exist); expose model methods that own the transaction, call them from a service.
  • feature-services/service.ai-command.ts — [dim 2] analyzeBatch (lines 87–540, ≈453 lines) and [dim 1] the file at 2,184 lines (7.3×). Also executeCreatePage (≈200), analyzeSpecializedBatch (≈200), executeCreatePost (≈160), executeUpdateMenu (≈130). [dim 7] 106 db(…) call-sites here. Fix: split the analyze pipeline and each execute-handler into their own modules; route DB access through Post/Page/Menu/Redirect models.
  • feature-services/service.identity-warmup.ts — [dim 2] runIdentityWarmup (lines 167–551, ≈384 lines, 7.7× the function cap) and [dim 1] file at 1,484 lines. Fix: the 8 pipeline stages are already documented in the header; extract each stage (GBP scrape, multi-URL scrape, image analysis, logo host, archetype, distillation, assembly) into its own named function/module ≤50 lines and have runIdentityWarmup be a thin sequencer.

Concerns

  • [dim 1] 27 of 50 files exceed 300 lines; 8 services sit in the 407–2,184 range (generation-pipeline 890, post-importer 874, page-editor 841, project-manager 721, seo-generation 680, url-scrape-strategies 612, gsc-integration 582, clarity-integration 567). Fix: systematic extraction; not all urgent, but all over cap.
  • [dim 7] DB access outside models/ is domain-wide, not just the controller: 26 files import db from database/connection and total 805+ query-builder call-sites, while 33 purpose-built models already exist and are bypassed. WebsiteIntegrationsController.ts (0 direct db(), delegates fully to WebsiteIntegrationModel) proves the correct pattern is achievable in this domain. Fix: treat models/ as the only DB layer; migrate services file-by-file.
  • [dim 9] Response error shape is non-contract everywhere: every error uses flat { success:false, error:"CODE", message:"…" } (42 sites) instead of the contract's nested { success:false, data:null, error:{ code, message, details } } — zero files use the nested shape. Also pagination is a sibling of data (3 sites) rather than nested in data. Internally consistent but uniformly wrong vs the contract. Fix: introduce a shared response helper and migrate; coordinate with frontend consumers (Level 3 blast radius — clients parse the flat shape today).
  • [dim 12] No input validation via validation/ schemas/middleware. Controllers hand-roll checks (e.g. req.body?.place_id … .toString().trim(), manual if (!placeId) return 400). The contract requires schema validation at the route/middleware level. Fix: add Zod schemas in validation/ applied as middleware (z is already imported in the controller).
  • [dim 5] Domain types/models are defined inside the controller: interface LocationsIdentityLocation (L4930), buildLocationEntryFromGbpLocal (L4952), buildBusinessFromGbpLocal (L4979), plus a mid-file import at L4928. Modeling logic belongs in a service/model, not the controller. Fix: relocate to the locations service/model.
  • [DRY/dim 14] parseIdentityJson is duplicated verbatim in AdminWebsitesController.ts:1326 and service.post-importer.ts:121 (and overlaps util.project-identity.parseProjectIdentity). Fix: consolidate into the existing feature-utils/util.project-identity.ts and import.

Nitpicks

  • [dim 10] console.* is endemic: 410 occurrences across the scope (180 in the controller alone) instead of Pino. Three services (identity-warmup, generation-pipeline, post-importer) wrap it in a private function log(){ console.log(...) }, so the grep under-counts the real channel. Fix: swap to the Pino logger with request/entity context.
  • [dim 11] Magic values: the "20 locations per project" cap is a bare literal repeated in the controller (L5046); MAX_SOURCE_CHARS = 100_000 and similar live as in-file constants rather than in config/. Fix: move domain limits to config/; in-file consts are a minor offense.
  • [dim 14] feature-services/ filenames use a service.kebab-case.ts prefix and export bare functions rather than the contract's PascalCaseService.ts with a Service class/object. Consistent within the domain and self-documenting, so low-priority, but it diverges from the stated naming rule — worth a one-line ruling when the nesting verdict propagates.
  • [dim 15] Commented-out code is effectively absent (grep heuristic returned only 2 hits, both genuine explanatory comments). Clean on this dimension.
