Findings

Summary

Verdict: this is the reference domain. The positive baseline the audit expected holds up — every file routes DB through models/, every controller handler is a thin try/catch → context → service → ok()/handleGbpError(), and the shared feature-utils/ layer (response shape, Google-error classification, input sanitizer) is the {success,data,error} + dim-12 gold standard the rest of the repo lacks. No unique console.* beyond 1 [systemic], no inline DB beyond the trx pattern + 1 raw join, zero hardcoded secrets, zero commented-out code. Violations are genuine but bounded: oversized files/functions, one dead-branch bug, a duplicated model-id constant, and one partial-failure gap. Severity: 0 must-fix · 5 concerns · 6 nitpicks (must-fix reserved for correctness/architecture breaks; the dead branch is the closest call, filed as a concern because it is currently behavior-neutral).

Must-fix

None. No correctness-breaking, security, or hard-architecture violations found.

Concerns

  • feature-services/GbpWorkItemActionService.ts:63-68 — [dim 15/correctness] reject() has a dead conditional: both the content_type === "local_post" branch and the else branch call GbpReviewReplyService.reject(params). It is a behavior-neutral landmine — reads as if local posts have a distinct reject path, but they do not (GbpLocalPostDeploymentService has no reject). Fix: collapse to a single unconditional call, or add the real local-post reject the branch implies. Don't leave a fake fork.
  • feature-services/GbpLocalPostDeploymentService.ts:266-288 & GbpReviewReplyDeploymentService.ts:83-100 — [dim 16] post-Google bookkeeping (markPublished + markSucceeded + WorkEvent.create + Notification.create / ReviewModel.updateReplyFields) runs as sequential awaits with no db.transaction wrapper. Deliberate (can't hold a trx open across the slow Google call), and the catch handles the Google-succeeded/local-failed case via LOCAL_PUBLISH_SYNC_FAILED — but a mid-sequence failure after markPublished can still half-write the local bookkeeping. Fix: wrap the post-Google local writes (everything after the API returns) in a single short transaction so the success path is atomic; keep the existing partial-failure fallback.
  • feature-services/GbpPublishedReplyService.ts:82-83,101-102 — [dim 16/dim 8] updateReply/deleteReply call Google then a single local write, but unlike the deploy services they record NO deployment attempt and have NO Google-succeeded/local-failed guard. If ReviewModel.updateReplyFields/clearReplyFields fails after Google succeeds, Google is live and the local reviews row is silently stale. Fix: mirror the if (googleResult) reconciliation used in the deploy services, or at minimum log + surface the divergence.
  • feature-services/GbpDraftGenerationService.ts:44-160 vs GbpLocalPostDraftService.ts:98-216 — [dim 2/DRY] the distinct-draft + safety-repair LLM loop is duplicated near-verbatim across both services (normalizedContent, isSameAsPreviousDraft, pickVariationInstruction, safetyRepairInstruction, generateDistinct*, generateSafe*). This is the same concept, not coincidental similarity — the convention's DRY carve-out doesn't apply. Fix: extract a shared feature-utils helper parameterized by variation text + safety validator; keep review-vs-post wording as inputs.
  • Long functions [dim 2] — GbpLocalPostDeploymentService.deployNow 139 lines (194-332), GbpReviewReplyService.generateDraft 134 lines (46-179), GbpReviewReplyDeploymentService.deployNow 123 lines (22-144), GbpLocalPostDraftService.completeQueuedGeneration 120 lines (569-688). All 2.4-2.8× the 50-line cap. Fix: extract the readiness/safety/property preflight and the success-vs-failure bookkeeping into named helpers (the codebase already favors this style — see the file-level helpers).

Nitpicks

  • Files over 300 [dim 1] — GbpLocalPostDraftService.ts 707, GbpPublishedLocalPostService.ts 640, AdminGbpAutomationController.ts 442, GbpAutomationController.ts 425, GbpReviewReplyService.ts 392, GbpLocalPostDeploymentService.ts 364. The two controllers are over purely from many thin handlers (split into client/admin sub-routers by resource if desired); the services are real split candidates, not alarms.
  • feature-services/GbpDraftGenerationService.ts:21 & GbpLocalPostDraftService.ts:31 — [dim 11] default LLM model string "claude-haiku-4-5-20251001" is hardcoded in both services as the env fallback. Fix: lift to a single named constant in config/ and import it in both (also kills a drift risk).
  • feature-services/GbpLocalPostDeploymentService.ts:160-166 & GbpReviewReplyService.ts:338-344 — [dim 11] BullMQ options (attempts: 3, delay: 30000, removeOnComplete age 86400, removeOnFail age 604800) are inline magic numbers duplicated across both deploy enqueues. Fix: extract a shared deploy-queue options constant.
  • feature-services/GbpPostMediaService.ts:13 vs routes/gbpAutomation.ts:9 — [dim 11] image size ceiling is 5 MB in the service but the multer route limit is 25 MB; the route limit is effectively dead (looser than the service). Fix: align the two on one named constant so the rejection boundary is single-sourced.
  • feature-services/GbpLocalPostScheduleService.ts:172-217 — [dim 15-adjacent/stale logic] processDueSettings only ever increments skipped (always writes lastLocalPostSkipReason: "per_post_image_required"); it never generates, so result.created is dead in the scheduled path. Likely intentional after per-post images became mandatory, but the unused counter + "process due" name now mislead. Fix: drop the dead branch/counter or add a comment stating the scheduled path is intentionally skip-only.
  • feature-services/GbpPublishedReplyService.ts:43-46 — [dim 14/clarity] getEditableReview validates property.google_connection_id then discards it; getAuthForLocation re-fetches the same property a line later. Harmless double-fetch. Fix: pass the resolved property/connection through instead of re-querying.

What this domain does right (positive baseline)

  • Data flow & response shape [dim 5/6/9] — both controllers are pure orchestration; feature-utils/controllerResponses.ts centralizes both {success,data,error} branches and maps error codes → HTTP status. This is the contract-correct shape the rest of the repo skips.
  • Google failure modes [dim 8] — feature-utils/googleApiErrors.ts classifies 401→reconnect / 403→permission / 429→rate-limit / 5xx→transient with a transient-retry flag; deploy paths distinguish transient (rethrow for BullMQ retry) from terminal, and deployNow correctly handles the Google-succeeded/local-failed partial failure instead of orphaning a live post.
  • Input handling & AI safety [dim 12] — GbpInputSanitizer (control-char strip, length caps, URL protocol allowlist) is applied at settingsPayload; GbpContentSafetyService/GbpLocalPostSafetyService gate PHI/medical-claim content; both generation services inject an untrustedInputNotice into the LLM prompt and run a regenerate-on-unsafe repair loop. GbpPostMediaService enforces MIME allowlist + size bounds + project ownership on uploads.
  • Transactions [dim 16] — every multi-table draft/approve/reject write threads trx from db.transaction into trx-aware model methods (19 call sites). The trx orchestration in services + queries-in-models split is a defensible read of the contract, not a dim-7 break.
  • Naming [dim 14] — uniform XxxController/XxxService PascalCase, descriptive function names, boolean-as-question (isSafe, isGoogleReady, hasBusinessManageScope). Auth [dim 13] applied at the router via router.use(...) on both client and admin route files.
