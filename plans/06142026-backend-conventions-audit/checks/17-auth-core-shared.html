Findings

Summary

This unit is the security/core spine, and the headline of the whole audit lives here. The in-scope files are individually well-engineered (env-only secrets, try/catch on every external call, no credentials logged), but two repo-wide breaches surface from this scope: (1) input validation via validation/ schemas is effectively non-existent, and (2) the mandated Pino logger is not installed or used anywhere — console.* is the de-facto standard. Severity counts for this unit: 2 must-fix (both repo-wide), 5 concerns, 4 nitpicks. No inline secrets found anywhere in scope; no DB access in config/, index.ts, scripts (correctly via models), or types.

Validation coverage (dim 12, repo-wide)

The contract is violated repo-wide. src/validation/ contains exactly one file, minds.schemas.ts (25 lines), and it does not validate HTTP input — it validates LLM output (a zod ProposalsSchema consumed by service.minds-comparison.ts via safeParse). There is no validation middleware in src/middleware/ (only auth/rbac/cors/rate-limit/billing). Across all 39 route files, exactly 0 apply a validation schema (the single validation/ import in routes+controllers is the LLM-output one). All 9 zod importers in src/ use it for parsing AI/LLM responses, never request bodies. No joi/express-validator/celebrate/yup is present at all. How validation actually happens: ad-hoc inline guards inside controllers (~8 controllers reference a VALIDATION_ERROR-style code; ~77 touch req.body/query/params with hand-rolled if (!x) return 400 checks of inconsistent shape). Coverage estimate: 1 schema file (LLM-output only); 0 of 39 route files apply request validation; request validation is inline, ad-hoc, and inconsistent where present. This reframes dim-12 remediation for every other unit: there is no shared schema layer to extend — one must be created from scratch (recommend a single validate(schema) middleware + per-domain schema files under validation/), and the work is repo-wide, not per-controller cleanup.

Must-fix

  • src/validation/ (whole layer) — [dim 12] No request-input validation layer exists. 0/39 route files validate input via schemas; the lone schema file validates LLM output. Client data reaches controllers untrusted, guarded only by scattered inline if checks. Fix: add a generic validateBody/validateQuery(schema) middleware (zod is already a dependency) and per-domain schema files under validation/; apply at the route level starting with auth/billing/public endpoints. Track as a dedicated repo-wide remediation plan.
  • (repo-wide logging) — [dim 10] Pino is not installed (absent from package.json, zero imports in src/) despite the contract mandating it. 277 of 922 .ts files use console.*; in-scope alone: llm-runner 15, index 5, oauth2Helper 3, stripe 1, plus the two scripts. Fix: either adopt Pino for real (install, wire pino-http, replace request-path console.*) or formally amend the contract to bless console.*. Until decided, treat dim-10 as a known repo-wide deviation, not a per-file defect — flag the decision to the owner.

Concerns

  • src/auth/oauth2Helper.ts:50,96,149,197 — [dim 7] Direct db("google_connections") queries (select + update) live in auth/, not models/. DB access outside models/ violates the data-flow contract. Fix: extract a GoogleConnectionModel (find-by-id, find-by-org, update-tokens) and call it from the helper.
  • src/agents/service.llm-runner.ts:1-697 — [dim 1] 697 lines, the only size offender in scope (cap 300). Cohesive but oversized. Fix: split into runAgent.ts, runWithTools.ts, and a shared extractJson.ts / system-block builder.
  • src/agents/service.llm-runner.ts:109,486 — [dim 2] runAgent (~296 lines) and runWithTools (~131 lines) far exceed the 50-line function cap. Fix: extract the zod-retry block, the cost-logging block, and the system-param builder into named helpers.
  • src/agents/service.llm-runner.ts:190,301,544 — [dim 8/failure-modes] Anthropic calls have try/catch + error context, but no explicit timeout or abort; they rely on SDK defaults, so a hung upstream can block a worker turn indefinitely. Fix: pass an explicit request timeout / AbortSignal (and a retry/backoff policy) on the messages.create calls.
  • src/index.ts:83-139 — [dim 6/4] A ~57-line inline CORS middleware with a hardcoded allowed-origins list lives in the entry point. That is real logic, and a corsCustomDomains.ts middleware already exists alongside it. Fix: move the CORS handler into middleware/ and the origin list into config/; index.ts should only register it.
