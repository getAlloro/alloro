Priority Signals (start here)

  • Misplaced services in utils/. aiCommandService.ts (884), service.dashboard-metrics.ts (876), pmsAggregator.ts (615), pmsAutomationStatus.ts (593) read like business services, not pure utils — the names even say "service". They likely belong in services/ or a domain (dimension 4, folder placement).
  • Double-violation risk: check these for DB access. Business logic + DB outside models/ = two breaches in one file.
  • website-utils/ cluster (aiCommandService, htmlValidator, uiChecker, linkChecker, builtinAnalyzer) — heavy logic; confirm purity vs hidden state / I/O.
  • Largest: aiCommandService 884, service.dashboard-metrics 876.

Audit Dimensions

Run every file in scope against the 16-point contract (full detail in spec.html). Dimension 4 (correct placement — pure util vs misplaced service) is primary here.

  • File ≤ 300 lines
  • Functions ≤ 50 lines
  • Nesting ≤ 4 levels
  • Correct folder placement (pure util, not a service)
  • Data flow: Routes → Controllers → Services → Models
  • No business logic in routes
  • No DB queries outside models/
  • Error handling on every async call
  • { success, data, error } response shape (callers)
  • No console.log (Pino only)
  • No hardcoded secrets / magic values
  • Input validated at route/middleware
  • Auth middleware on protected routes (callers)
  • Naming conventions
  • No commented-out code
  • Transactions for multi-table writes

Findings

Summary

Verdict: needs changes. The headline is confirmed: four files named/shaped as business services live in utils/ (dim 4), and three of them also hit the DB directly (dim 7) — double violations. Separately, the most severe single finding is unit-specific and NOT systemic: core/domainMappings.ts commits 9 live Microsoft Clarity JWT API tokens in plaintext source (dim 11). The website-utils/ analyzer cluster (htmlValidator, uiChecker, linkChecker, builtinAnalyzer) is the bright spot — genuinely pure (0 imports, 0 I/O, 0 DB), only guilty of the size cap. Severity counts: must-fix 4 · concerns 7 · nitpicks 4 (systemic dims 7/10/12 counted once each, not re-litigated).

Must-fix

  • core/domainMappings.ts:20,32,49,61,74,86,98,110 — [dim 11] 9 hardcoded Clarity JWT API tokens (clarity_apiToken, scope Data.Export, exp 2125) committed in source. Live credentials in version control — anyone with repo access can export every client's Clarity analytics. Fix: pull every token out to env/secret store (or a DB website_integrations row, which the Clarity architecture already models — see memory), reference via config/; rotate all 9 tokens since they are already leaked into git history.
  • pms/pmsAutomationStatus.ts:227,325,406,487,508,549,566,586 — [dim 4 + dim 7] A status service (8 direct db("pms_jobs") read+write calls across 6 exported fns) living in utils/. Double violation: business logic + all DB access outside models/. Fix: move to services/pms/ and route every db() call through a PmsJobModel method (e.g. PmsJobModel.updateAutomationStatusDetail()).
  • dashboard-metrics/service.dashboard-metrics.ts:334,460-483,540-556 — [dim 4 + dim 7] Self-described "service" (the filename literally says so) doing direct db("practice_rankings"), db("website_builder.form_submissions"), db("pms_jobs") — 8 inline queries. Double violation. Fix: relocate to services/dashboard-metrics/; push the 3 query blocks into PracticeRankingModel / FormSubmissionModel / PmsJobModel. (Also dim 2: computeDashboardMetrics ≈149 lines — extract the GBP-fetch + prior-rating blocks.)
  • website-utils/aiCommandService.ts:586-631, 722-767 — [dim 15] Two large dead blocks: __DEAD_SECTION_GEN (46-line unused template literal, even labeled "DEAD") and VISUAL_ANALYSIS_PROMPT (45-line const, superseded by getVisualAnalysisPrompt() which loads the prompt from a .md file). ~91 lines of confirmed-unused code, zero references. Fix: delete both. (Same file is also dim 4 — an Anthropic LLM service with module-level client state and network I/O, not a pure util; belongs in services/ or agents/.)

Concerns

  • pms/pmsAggregator.ts:284-615 — [dim 2] aggregatePmsData is a single ~331-line function — longer than the entire 300-line file cap, ~6.6× the 50-line fn cap. It does extraction, dedup, capping, source aggregation, trend computation, and Levenshtein dedup-candidate detection in one body. Fix: extract computeSourceTrends(), findDedupCandidates(), buildMonthMap(). (dim 4: it's a business service, but at least it goes through models — no direct db().)
  • website-utils/aiCommandService.ts, website-utils/pageEditorService.ts, rybbit/service.rybbit-data.ts, driveTimeMarket.ts, core/s3.ts, website-utils/screenshotService.ts — [dim 4] Network/external-I/O modules misfiled as "utils": Anthropic SDK (×2), axios + Google Places, AWS S3 SDK, Playwright browser. A util is stateless and side-effect-free; these hold clients and make external calls. Fix: move to services/ (or lib/ for the thin S3/screenshot wrappers).
  • website-utils/pageSnapshots.ts:76-143, startup/zombieJobCleanup.ts:7-28, website-utils/agenticHtmlPipeline.ts:126, website-utils/pageEditorPrompt.ts:27, core/notificationHelper.ts:68-72, rybbit/service.rybbit-data.ts:38 — [dim 7][systemic] Six more files with direct db() outside models/ (pageSnapshots even does .insert(); zombieJobCleanup uses db.raw()). Part of the repo-wide ~125-file pattern, but flagged here because "utils" should be the last place DB access hides. Fix: route through models.
  • dashboard-metrics/service.dashboard-metrics.ts:727-876 & pms/pmsAutomationStatus.ts — [dim 16] Multi-row writes without transactions. dashboard-metrics fires 5 parallel form-submission counts (read-only, fine) but the automation-status helpers do read-modify-write on automation_status_detail JSONB with no transaction/locking — concurrent agent steps can clobber each other's status. Fix: wrap RMW in a transaction with row lock, or use a JSONB-merge update.
  • website-utils/{htmlValidator,uiChecker,linkChecker,builtinAnalyzer}.ts (479/399/377/351) & core/notificationHelper.ts (322) — [dim 1] Over the 300-line cap. The four analyzers are otherwise exemplary pure utils (0 imports, pure input→recommendations[]) — correctly placed, just split each into rule-group files. notificationHelper is a notification service (DB + email I/O), so its fix is relocation, not just splitting.
  • website-utils/aiCommandService.ts — [dim 9][systemic] LLM helpers throw raw Errors on parse/empty failures rather than returning the {success,data,error} shape; callers must wrap. Acceptable for an internal helper, but the boundary controller must enforce the shape. Flag as systemic.
  • core/domainMappings.ts — [dim 4/14] Beyond the token leak: a hardcoded array of client GBP account/location IDs as a "util" is config data, not a utility. Fix: this is reference/config — move to config/ or a DB table; the per-org data already partly lives in projects/website_integrations.

Nitpicks

  • website-utils/aiCommandService.ts — [dim 14] Filename aiCommandService.ts sits in website-utils/ but lacks PascalCase service convention (AiCommandService.ts) — moot once it moves out of utils, but the "Service" suffix on a file in a utils folder is itself the smell.
  • dashboard-metrics/service.dashboard-metrics.ts & rybbit/service.rybbit-data.ts — [dim 14] service.kebab-name.ts naming diverges from the documented PascalCase XxxService.ts convention; pick one repo-wide.
  • core/notificationHelper.ts:20-24 — [dim 11] No-op NODE_ENV ternary (both branches return the same prod URL). Per repo memory NODE_ENV is uniform everywhere, so the conditional is dead. Fix: collapse to the single constant.
  • core/*, website-utils/*, pms/* — [dim 10][systemic] 110 console.* calls across 18 files (aiCommandService alone has 30). Matches the repo-wide Pino-not-installed finding; counted once.
