Priority Signals (start here)

  • Three giants: PracticeRankingController 1,948 + service.ranking-pipeline 1,933 + service.location-competitor-onboarding 1,612. Controller likely holds pipeline logic that belongs in services.
  • Heavy external-service surface (Apify, SerpAPI, Google Places, LLM). Audit failure-mode handling hard: retries, timeouts, partial failure, malformed responses (skill's Failure Mode Thinking).
  • Ranking math — check for magic numbers / thresholds that should be named constants in config/.
  • Same feature-services/ + feature-utils/ nesting as admin-websites — apply the T01 ruling.

Audit Dimensions

Run every file in scope against the 16-point contract (full detail in spec.html).

  • File ≤ 300 lines
  • Functions ≤ 50 lines
  • Nesting ≤ 4 levels
  • Correct folder placement
  • Data flow: Routes → Controllers → Services → Models
  • No business logic in routes
  • No DB queries outside models/
  • Error handling on every async call
  • { success, data, error } response shape
  • No console.log (Pino only)
  • No hardcoded secrets / magic values
  • Input validated at route/middleware
  • Auth middleware on protected routes
  • Naming conventions
  • No commented-out code
  • Transactions for multi-table writes

Findings

Summary

Verdict: needs changes — structurally heavy but the external-service plumbing is genuinely well-built. The headline problems are size/placement, not failure-handling. Nine files breach the 300-line cap (≈8,400 of 11,768 lines sit in the 3 giants), one function (processLocationRanking) is ~1,159 lines, and the controller does raw multi-join DB work and runs the whole background pipeline itself — a clear Routes→Controllers→Services→Models violation. Credit where due: the retry/resilience layer (service.ranking-resilience.ts), Apify poll-timeout, and the onboarding transaction are exemplary; ranking weights are already named constants, so the "magic ranking math" worry from Priority Signals is mostly a non-issue. Counts: 4 must-fix, 5 concerns, 4 nitpicks (systemic items folded in, tagged).

Must-fix

  • ../../routes/practiceRanking.ts:40-62 — [dim 13] missing auth on ~17 endpoints. Only the v2 /locations/:locationId/competitors* routes (and /photo) carry authenticateToken + rbacMiddleware + locationScopeMiddleware. Everything else — POST /trigger (starts paid Apify/SerpAPI/LLM runs), GET /list, /accounts, /latest, /results/:id, /history, DELETE /:id, DELETE /batch/:batchId, POST /retry/:id, /retry-batch/:batchId — is wide open: unauthenticated callers can trigger cost-incurring batch runs, enumerate/read every org's rankings, and delete records. Fix: mount authenticateToken (+ rbac/role/scope as appropriate) on the v1 block; confirm whether a parent router already wraps these before downgrading severity, but as written in this file there is no gate.
  • PracticeRankingController.ts (33 db( calls) — [dim 5/6/7] controller owns data-access + pipeline orchestration. Raw multi-join queries live inline: listAccounts (db("google_connections as gc").join("organizations as o"...) L464), getLatestRankings (4+ separate db("practice_rankings") / db("locations") reads, L1063-1173), triggerBatchAnalysis (L106 join), retryRanking/retryBatch read+update+re-update practice_rankings directly. The controller also runs the background job (setImmediate + inline status updates, L744-785) instead of delegating to a service. Fix: push every query into PracticeRankingModel / LocationModel / existing models; move retry/batch orchestration into a feature-service; controller should validate → call service → format. [systemic dim-7: db from database/connection used as db("table") in 8 scope files — controller 33, pipeline 10, competitor-cache 8, specialty-identifier 5, ranking-computation 4, onboarding 3, llm-webhook 2, source-resolver 1.]
  • feature-services/service.ranking-pipeline.ts:774-1933 — [dim 2] processLocationRanking is one ~1,159-line function (runs to EOF), the single hardest-to-maintain unit in scope. It sequences OAuth, GBP fetch, competitor resolution, Apify enrichment, SerpAPI, website audit, scoring, LLM, and persistence in one body. Fix: extract phase helpers (resolve-competitors, enrich-velocity, run-llm, persist-result) — several already exist as private helpers above it; the orchestrator should read as ~10 named steps. This file also breaches dim 1 at 1,933 lines.
  • feature-services/service.ranking-computation.ts:320-338 — [dim 15] committed commented-out code: a ~19-line dead createNotification(...) try/catch block left inline. Fix: delete it (git has history). Minor on its own but it's the only literal commented-out logic block found and it's exactly the hygiene rule.

Concerns

  • [dim 1] 9 files over 300 lines: PracticeRankingController 1,948 · ranking-pipeline 1,933 · location-competitor-onboarding 1,612 · places-competitor-discovery 1,016 · ranking-algorithm 770 · apify 709 · util.ranking-formatter 555 · ranking-llm 482 · ranking-computation 371. The 3 giants are the priority; onboarding (1,612) is dense but cohesive (discovery + curation + finalize) and is a reasonable 2-3 file split.
  • util.ranking-formatter.ts:555 — [dim 4/1] a "util" doing 11 response-shaping functions. It's the controller's whole presentation layer (formatTriggerResponse … formatTasksList). Not pure utilities; closer to a view/serializer module. Acceptable under the feature-utils ruling, but the size flag stands and the naming oversells it as generic.
  • [systemic dim-9] response shape. Controller has 0 nested error:{code,message,details} objects and 0 data:null — 56 flat error:"CODE", message:"..." literals + 18 ad-hoc success:true payloads. The contract's {success,data,error} shape is entirely absent here; flat shape dominates. Repo-wide, not unique to this unit.
  • [dim 8] partial-failure semantics. The retry/resilience layer is good, but processLocationRanking swallows several enrichment failures into "not measured" / empty defaults (e.g. velocity, photos) without surfacing a degraded-result flag to the caller — a run can "succeed" on thin data. Worth an explicit partial-success marker on the ranking record. (Not a swallowed-error-with-no-log case; failures are logged.)
  • service.apify.ts:99-142 — [dim 8] poll loop hard-codes 300000ms / 5000ms. Timeout + termination exist (good), but maxWaitMs=300000 and pollInterval=5000 are inline literals. Fix: name them (and ideally source from config) so the Apify SLA is tunable without code edits.

Nitpicks

  • [systemic dim-10] no Pino. Logging routes through util.ranking-logger.ts, a hand-rolled level gate over console.log/warn/error; CURRENT_LOG_LEVEL = DEBUG (L16) ships verbose. Plus 8 stray direct console.* across services (places-discovery, serpapi, apify, pipeline, competitor-cache, identifier, source-resolver, + 5 in the logger itself). Pino not installed repo-wide; noted, not per-file must-fix.
  • [dim 11] secrets read outside config/. process.env.SERPAPI_API_KEY, APIFY_TOKEN, GOOGLE_PLACES_API, COMPETITOR_CACHE_TTL_HOURS, IDENTIFIER_AGENT_WEBHOOK accessed inline in service files. Keys are not hardcoded (good) but bypass the contract's config/ indirection. Also note env-name inconsistency: APIFY_TOKEN vs const APIFY_API_TOKEN, GOOGLE_PLACES_API (not ..._KEY).
  • [dim 11] benchmark thresholds in-file. service.ranking-algorithm.ts normalization bands (avg:150, excellent:400, max:800, reviews-per-month, posts, photos) are a named object — fine — but live beside the algorithm rather than in config/. Tuning knobs that belong with the weights in config.
  • [systemic dim-12] no validation/ schemas. Input checks are hand-rolled in feature-utils/util.ranking-validator.ts + util.competitor-validator.ts (and parseInt/isNaN guards in the controller) rather than schema middleware from validation/. Functional, but off-pattern; repo-wide gap.
