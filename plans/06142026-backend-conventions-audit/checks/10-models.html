Audit Dimensions

Run every file in scope against the 16-point contract (full detail in spec.html). For this layer, dimensions 7 (no DB outside models — checked inversely), 11 (parameterization), and 16 (transactions) are primary.

  • File ≤ 300 lines
  • Functions ≤ 50 lines
  • Nesting ≤ 4 levels
  • Correct folder placement
  • Data flow: Routes → Controllers → Services → Models
  • No business logic in routes
  • No DB queries outside models/ (inverse grep)
  • Error handling on every async call
  • { success, data, error } response shape (callers)
  • No console.log (Pino only)
  • No hardcoded secrets / magic values
  • Parameterized queries — no string-concat SQL, minimal raw()
  • Auth middleware on protected routes (callers)
  • Naming conventions (Model suffix, descriptive query fns)
  • No commented-out code
  • Transactions for multi-table writes

Findings

Summary

Verdict: the models layer itself is exemplary; the architecture is breached from the outside. Internally, models/ is one of the cleanest layers audited — 97 of 109 files thread a QueryContext/trx param, every .raw() is parameterized or a static constant, there is zero string-interpolated SQL, and the largest models (ReviewModel, GbpWorkItemModel) are model-of-good-practice. But dimension 7 — all DB access lives in models/ and nowhere else — is violated at scale: 125 non-model files import the raw db connection and run ~507 direct query/raw call-sites outside models/. This is the single largest contract breach in the audit. Severity: 1 systemic must-fix (dim 7), ~3 concerns, ~3 nitpicks.

Note on the assigned grep: the prompt's knex|\.raw( grep matched only 22 files (and ~4 of those are type-only import { Knex } false positives). That grep massively undercounts the leak: the canonical handle is db from database/connection, and the dominant leak form is the query-builder call db("table")…, which contains neither the literal token knex nor .raw(. The true surface (125 files / 507 sites) was found by grepping the connection-import instead. Headline figures below use the accurate surface.

Must-fix

  • (dim 7, systemic) Pull all 507 inline queries behind models. 125 files reach the DB directly. This defeats the entire layering contract: query logic is unmemoizable, untestable in isolation, and duplicated across domains. Highest-leverage targets are the website-builder feature-services (one cluster, ~15 files, >400 sites) and routes/checkup.ts (queries in a route file). Recommend an incremental model-extraction pass per domain, not a big-bang.
  • (dim 6 + 7) routes/checkup.ts performs db("organizations").update() / db("weekly_ranking_snapshots").insert() inline. A route file doing writes is the worst single instance — move to a controller + model.

Concerns

  • (dim 16, downstream of the leak) Multi-table writes outside models run without transactions. Because writes leaked out of models/, many are unguarded: e.g. websiteRestore.processor.ts (~32 DB ops, 0 transactions), AgentsController.ts, service.competitor-cache.ts, behavioralIntelligence.ts, several pm/ and agents/ services. Partial-write corruption risk. Fixing dim 7 (move writes into models that own a db.transaction) resolves most of this at once. Inside models/, transaction discipline is correct — see ReviewModel.replaceApifyReviewsForPlace (wraps del + upsert-loop in db.transaction).
  • (dim 1) 11 models exceed the 300-line cap (ReviewModel 554, GbpWorkItemModel 512, AppUsageEventModel 424, PmsJobModel 404, + 7 more). All are cohesive single-entity models, so this is soft — but the two largest are split candidates if read-vs-write or metrics-vs-CRUD concerns can be separated.
  • (dim 10) console.log in a model: website-builder/SeoGenerationJobModel.ts:72 logs stale-job expiry via console.log instead of Pino.

Nitpicks

  • (dim 11, magic value) Hardcoded timezone in raw SQL. PmTaskModel.getAssignedVelocity embeds 'America/Los_Angeles' in its velocity queries (parameter-safe, but the TZ should be a named constant in config/, not inlined in 3 SQL branches).
  • (dim 4 false-positive, for the record) Four files flagged by the leak grep are type-only imports, NOT leaks: AdminOrgCreationService.ts, TierManagementService.ts, OrganizationBootstrapService.ts import Knex purely to type an injected trx param (correct transaction-passing pattern); routes/billing.ts matched on express.raw() (Stripe middleware), not DB. These should not be counted as violations.
  • (dim 11, advisory) Raw .raw() usage in models is fine but worth periodic review: 77 call-sites, all parameterized via ? + bound arrays or static constants. No injection risk found; flagged only so the count is visible.

Coverage

Deep-read: 5 models in full (ReviewModel 554, GbpWorkItemModel 512, PmTaskModel raw-query block, plus AppUsageEventModel & PmsJobModel write paths). Grep-sampled: the remaining ~104 models for .raw() parameterization, interpolation, console.*, commented code, deep nesting, and trx adoption (full-layer sweeps, not a sample cap). The dim-7 inverse leak was enumerated exhaustively across all of controllers/ services/ utils/ routes/ workers/ by both the assigned narrow grep and the corrected connection-import grep. Not examined line-by-line: the internal logic of the ~100 low-density leak files (their per-file dims 1–16 belong to the other audit units; this worksheet owns the dim-7 inventory only).
