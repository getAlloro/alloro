Audit Dimensions

Run every file in scope against the 16-point contract (full detail in spec.html).

  • File ≤ 300 lines
  • Functions ≤ 50 lines
  • Nesting ≤ 4 levels
  • Correct folder placement
  • Data flow: Routes → Controllers → Services → Models
  • No business logic in routes
  • No DB queries outside models/
  • Error handling on every async call
  • { success, data, error } response shape
  • No console.log (Pino only)
  • No hardcoded secrets / magic values
  • Input validated at route/middleware
  • Auth middleware on protected routes
  • Naming conventions
  • No commented-out code
  • Transactions for multi-table writes

Findings

Summary

Verdict: structurally sound, needs cleanup. Consistent PascalCase XxxController.ts naming confirmed — lowest-risk controller domain as flagged. Every handler is wrapped in try/catch with a per-file handleError, transactions are used correctly for multi-table writes, and the attachment security surface is genuinely well-built (MIME whitelist + hard blocklist + size cap + presigned-only downloads + a server-side text proxy that never hands the browser an executable URL). The real debt is the unit's heavy participation in two systemic problems (direct db() outside models/, console logging) plus a handful of unit-specific issues: dead placeholder code in PmController.ts, a dead no-op block in PmAiSynthController.ts, several functions well over the 50-line cap, 4 files over 300 lines, and an N+1 in listProjects. No SQL-injection or path-traversal exploit found — both raw-SQL and the unsanitized S3 taskId are mitigated (whitelisted branches / prior int-keyed findById 404). Severity counts: Must-fix 2 · Concerns 8 · Nitpicks 5 (systemic items folded into concerns, counted once).

Must-fix

  • PmController.ts:12-25,41 — [dim 15] Dead placeholder exports: extractTasks, batchCreateTasks, getLatestBrief, getBriefHistory, getGlobalActivity, getProjectActivity, getStats all return empty stubs and shadow real implementations that live in PmAiSynthController / PmActivityController / PmStatsController. Only listUsers is actually consumed (via routes/pm/users.ts); the routed activity/stats handlers correctly import the real controllers, so these stubs are pure dead code that invites a future mis-import returning silently-empty data. Fix: delete the seven placeholder exports and the two placeholder* helpers; keep only listUsers + its PmUserRow/displayNameForUser support.
  • PmAiSynthController.ts:134-137 — [dim 15] Dead no-op block: if (error?.batchId || (error as any)?.batch_id) { /* batch variable may not be in scope */ } has an empty body and an inert comment — it does nothing. Fix: delete the block; the real cleanup (mark orphaned "synthesizing" batches failed) already follows at lines 140-145.

Concerns

  • (scope-wide) — [systemic][dim 7] Direct db("table") / db.raw outside models/: ~50 call sites in 9 of 13 files. Counts: PmAiSynthController 19, PmActivityController 7, PmTasksController 6, PmProjectsController 5 (+5 raw), PmStatsController 3 (+3 raw), PmNotificationsController 3, PmCommentsController 3, PmAttachmentsController 3, PmController 1. Controllers query pm_* + users tables and even hand-roll raw aggregate subqueries directly. Fix (systemic): push queries behind the existing Pm*Model classes (notifications, activity, batch-task, stats aggregates have no model coverage yet) — out of scope for a one-file fix; flag for the architecture track.
  • (scope-wide) — [systemic][dim 9] Response shape is flat { success:false, error: "message-string" } in all ~64 error returns; the contract's nested error:{ code, message, details } object is never used, and success payloads vary between bare arrays, { items }-style objects, and top-level total siblings. Fix (systemic): adopt the { success, data, error } envelope with machine-readable error.code via a shared responder; track repo-wide.
  • (scope-wide) — [systemic][dim 12] No route-level input validation: none of the PM routes apply a validation/ schema; every check (title required, task_ids array, numeric userId) is hand-done inside controllers, and several IDs flow straight from req.params/req.body into queries untyped. Fix (systemic): add validation middleware per route; track repo-wide.
  • PmTasksController.ts:199-317 (moveTask ~118 lines), :429-559 (bulkMoveTasksToProject ~130 lines) — [dim 2] Functions far over the 50-line cap; both bury column-shift, completion-tracking, backlog-priority, notification fan-out, and position-compaction logic inline. Fix: extract helpers (shiftColumnPositions, applyCompletionTransition, compactColumns, notifyAssigneeCompleted) into a domain service alongside the controller.
  • PmAiSynthController.ts:23-148 (extractBatch ~125 lines) — [dim 2] Single function does input parsing, file extraction, prompt assembly, LLM call, response-format salvage, per-task validation/insert, and batch-counter updates. Fix: split into buildSynthPrompt, parseProposedTasks, and persistBatchTasks helpers.
  • PmCommentsController.ts:107-228 (createComment ~121 lines) — [dim 2] Comment insert + 3-tier recipient dedup + metadata build + batch notification insert + activity log in one body. Fix: extract buildCommentRecipients(task, mentions, authorId) and insertCommentNotifications.
  • PmProjectsController.ts:41-83 — [dim 2 / perf] N+1 in listProjects: Promise.all(projects.map(...)) fires 2 extra queries (statusCounts + dailyActivity) per project on top of 3 correlated subqueries already in the base select. Project-count growth multiplies round-trips. Fix: fold the per-project breakdowns into grouped joins or a single windowed query in a PmProjectModel method.
  • PmAttachmentsController.ts:87-97 + constants.ts:45 — [dim 12 / security] MIME type is taken from client-supplied file.mimetype with no magic-byte sniffing, and application/octet-stream sits on the allow-list as an escape hatch — so a renamed executable declaring octet-stream passes the gate (it is stored, never executed server-side, and downloads are presigned, so impact is bounded). Fix: sniff real content type (e.g. file-type on the buffer) and drop the octet-stream allowance, or store-but-force-download untrusted types.

Nitpicks

  • (scope-wide) — [systemic][dim 10] console.error in every file's handleError (13 sites). Pino is not installed repo-wide. pmActivityLogger.ts is clean — it does no logging, just delegates to PmActivityLogModel.create (the Pino-vs-console question for it resolves to "neither / N/A"). Fix: swap to the Pino instance once adopted.
  • s3-key.ts:14-17 — [dim 11 / security, defense-in-depth] taskId is interpolated into the S3 key (pm-attachments/${taskId}/...) without the same sanitization applied to the filename. Currently safe because every caller passes req.params.id only after PmTaskModel.findById(taskId) matched a real integer-keyed row (a traversal string would 404 first), but the util itself makes no guarantee. Fix: sanitize/coerce taskId to digits inside buildAttachmentS3Key so the helper is safe regardless of caller.
  • PmTasksController.ts, PmCommentsController.ts, PmProjectsController.ts et al. — [dim 14, minor] Pervasive any on query rows (tasks: any[], row: any, .map((r: any) =>)) erases type safety on DB shapes. Fix: type rows via model row interfaces.
  • PmAttachmentsController.ts:38-42 (and each file's handleError) — [dim 8, minor] On 500s the raw error.message is returned to the client (error: message), which can leak internal detail; PmAiSynthController already does the right thing by returning a generic string. Fix: return a generic message externally, keep error.message in the log only.
  • PmStatsController.ts:101,117,133 — [dim 11, minor] Timezone 'America/Los_Angeles' hardcoded in the velocity raw SQL. Not user-input (injection-safe) but a magic value. Fix: lift to a named config constant.
