Findings

Summary

Sound, well-documented unit with a genuinely correct send path (try/catch around the HTTP call, ERROR logged, failure result returned — no silent swallow, dim 8 passes). Two real unit-specific problems dominate: recipient PII (email addresses) is written to a disk log on every send, and the only credential (an authenticated webhook URL) is read straight from process.env instead of config/. Note: the sender is an n8n webhook, not Mailgun — the worksheet/memory Mailgun framing is stale; there are no Mailgun keys anywhere in scope, so the dim-11 secrets concern is reframed below, not dismissed. Verdict: needs changes. Severity: 3 must-fix · 4 concerns · 3 nitpicks.

Must-fix

  • emailService.ts:207 & :224 — [dim 13 / logging] success and failure logs include recipients: payload.recipients — full recipient email addresses (PII) persisted to logs/email.log on every send. :182 additionally logs originalRecipients (the pre-interception to/cc/bcc set). Fix: log recipientCount / a redacted or hashed identifier only; never raw addresses. The "Sending" log at :186 already does this correctly — match it.
  • emailService.ts:18 — [dim 11] WEBHOOK_URL = process.env.ALLORO_EMAIL_SERVICE_WEBHOOK read directly from env at module load, bypassing config/. This URL is the authenticated send endpoint — the unit's one secret. ADMIN_EMAILS (:19) and base.ts:23 APP_URL/NODE_ENV do the same. Fix: source all of these through a typed config/ accessor; this module should not call process.env or dotenv.config() directly.
  • emailService.ts:224-226 — [dim 13] error log embeds responseData: error.response?.data (raw upstream webhook response) alongside recipient PII. Upstream bodies can contain echoed payload/PII and internal detail. Fix: log error.response?.status + a bounded errorMessage only; drop raw responseData from the persisted entry.

Concerns

  • emailService.ts:40-45 (EmailResult) — [dim 9] introduces a third result shape ({ success, messageId?, error?, timestamp }) distinct from both the canonical { success, data, error } and the flat-dominant [systemic] pattern. Internal-only (never serialized to an HTTP response), so lower stakes, but it is its own contract. Fix: acceptable to keep as an internal type; document it as deliberate so it isn't mistaken for an endpoint shape.
  • templates/WelcomeCheckupEmail.ts:32,100 — [dim 9/14] sendWelcomeCheckupEmail returns bare Promise<boolean> (return result.success), diverging from every sibling send-helper that returns EmailResult. Callers lose the error string / messageId. Fix: return EmailResult like the others for a uniform helper contract.
  • emailService.ts:42-75 (logEmail) — [dim 10 systemic-adjacent] hand-rolled file logger using fs.appendFileSync (synchronous I/O on the send path) plus dev-only console. Reimplements logging instead of Pino. Fix: migrate to the project logger when Pino lands; at minimum avoid sync FS on the hot path. [systemic] (Pino not installed repo-wide).
  • emailInterceptor.ts:30-36,55-70 — [dim 8] fetchOwnPublicIp / isLiveSender fail closed into interception on any DNS/HTTP error (correct, safety-first) but the failure is swallowed via bare catch {} with no log — a misconfigured prod box silently rerouting all mail to dave@getalloro.com would be invisible. Fix: log a WARN on identity-check failure so a stuck-in-intercept prod box is detectable.

Nitpicks

  • templates/{UserSendInquiry,UserSendNotification,AdminSendErrorMessage,CheckupResultEmail}.ts — [dim 15/DRY] identical escapeHtml defined 4× verbatim. Fix: hoist one copy into base.ts (or utils/) and import.
  • base.ts:23-27 — [dim 11] the APP_URL ternary returns the same https://app.getalloro.com in both branches, making the NODE_ENV check dead code; the inline comment even says so. Fix: collapse to process.env.APP_URL || "https://app.getalloro.com" (sourced via config/).
  • templates/AdminSendErrorMessage.ts:73-88 — [dim 13] renders data.stackTrace into the admin email body (escaped). Internal admin-only recipient + interceptor guard make this low-risk, but stack traces can carry secrets/paths. Fix: note it's admin-only by design; reconsider if recipients ever broaden.
