Audit Dimensions

Run every file in scope against the 16-point contract (full detail in spec.html).

  • File ≤ 300 lines
  • Functions ≤ 50 lines
  • Nesting ≤ 4 levels
  • Correct folder placement
  • Data flow: Routes → Controllers → Services → Models
  • No business logic in routes
  • No DB queries outside models/
  • Error handling on every async call
  • { success, data, error } response shape
  • No console.log (Pino only)
  • No hardcoded secrets / magic values
  • Input validated at route/middleware
  • Auth middleware on protected routes
  • Naming conventions
  • No commented-out code
  • Transactions for multi-table writes

Findings

Summary

Verdict: needs changes. The domain is bifurcated. The file-manager pair (PmsFileManagerController.ts + PmsFileManagerService.ts) is the gold standard of the whole scope — genuinely thin controller, model-backed service, transactions wrapping multi-table writes, org/location scoping via findScopedJob. Everything else regressed from that bar: a 1,193-line god-controller, two ingestion paths with raw transactions in non-model layers, three services that each call the app over HTTP via hardcoded localhost, and — the headline — six fully unauthenticated admin routes including destructive delete/restart. The pure utils and the stateless paste/sanitize/json services are clean. Severity counts: 3 must-fix · 7 concerns · 6 nitpicks. Systemic-tagged items (console.*, no validation middleware, db() outside models, flat response shape) are noted briefly per wave-1 ruling and not re-litigated.

Must-fix

  • ../../routes/pms.ts:98-107 — [dim 13] Six PMS routes register with ZERO auth/RBAC middleware, self-described "ADMIN ENDPOINTS (No auth)": GET /jobs (lists every org's jobs, no scoping), PATCH /jobs/:id/approval, PATCH /jobs/:id/response (overwrites any job's data), DELETE /jobs/:id (permanent delete), POST /jobs/:id/retry, POST /jobs/:id/restart (deletes all agent_results/tasks/notifications then re-triggers). Any caller reaching the API can enumerate, mutate, and destroy PMS data across all organizations. The backing controllers (deleteJob, approveJob, updateResponseLog, restartJob) also perform no org-scoping — unlike the file-manager which scopes every call. The route file is outside this unit's scope but the controllers it exposes are in-scope, so flagging here. Fix: apply authenticateToken + rbacMiddleware (+ requireRole("admin")) to all six; add org/location scoping inside the controllers/services the same way PmsFileManagerService.findScopedJob does.
  • pms-retry.service.ts:457-483 (restartMonthlyAgents) / contrast :343-381 (retryMonthlyAgents) — [dim 8] Inconsistent partial-failure semantics with data-loss risk. Both call cleanupMonthlyRunData (hard-deletes agent_results, tasks, google_data_store, notifications) before re-triggering. On trigger failure, retryMonthlyAgents throws (500), but restartMonthlyAgents swallows it (restarted=false) and returns HTTP 200. So a failed restart leaves the org with deleted insights, no running job, and a success response — silent data loss. Fix: pick one contract; for restart, either re-trigger before destructive cleanup, wrap cleanup+trigger so a trigger failure surfaces as an error (and ideally is recoverable), or at minimum return a non-2xx + an explicit "data cleared but re-trigger failed, retry required" state.
  • pms-retry.service.ts:168, pms-finalize.service.ts:136, pms-approval.service.ts:199-209 — [dim 5 / dim 11] Three services POST to the app itself over HTTP (http://localhost:${process.env.PORT || 3000}/api/agents/monthly-agents-run). An in-process call shipped through the network loopback: adds latency, a failure mode (the agents route being down/unreachable while the same process is up), bypasses the type system, and hardcodes localhost + a magic 3000 fallback in three places. Breaks Routes→Controllers→Services→Models. Fix: extract the monthly-agents trigger into a shared service function and call it directly; if an HTTP boundary is genuinely required, centralize the base URL in config/ (no inline localhost/3000).

Concerns

  • PmsController.ts — [dim 1] 1,193 lines, ~20 endpoint handlers in one file — the god-controller for this domain. Fix: split by responsibility (upload, approval, automation/retry, column-mapping, paste) into separate controller files behind the existing route groups; the file-manager controller is the template.
  • PmsController.ts:932-971 — [dim 5/16] Controller opens an inline db.transaction and orchestrates a two-table write (PmsJobModel.create + PmsJobEventModel.create) directly. This is the only db(/transaction in either controller, and it duplicates pms-upload.service.processManualEntry:55-85 almost verbatim — business logic that leaked up into the controller. Fix: move into an upload/job-creation service method (reuse the upload service's pattern); keep the controller orchestration-only.
  • PmsController.ts:173-206 (getPmsSummary) — [dim 5] Live, auth-protected route (POST /pms/summary) that returns hardcoded all-zero placeholder data with a "Summary endpoint placeholder" message. Either dead-by-stub or a silent feature gap surfaced as real-looking data. Fix: implement, or remove the route + handler.
  • pms-upload.service.ts:132-344 (processFileUpload ~212 lines), PmsController.ts:841-1004 (uploadWithMapping ~163), pms-retry.service.ts:35-156 (cleanupMonthlyRunData ~121), pms-paste-analysis.service.ts:338-492 (sanitizeParsedData ~155) — [dim 2] Functions far over the 50-line cap. Fix: extract named helpers (mapping-resolve, persist-job, build-event payload, etc.).
  • file-upload.config.ts:13-24 — [dim 12/security] Multer fileFilter accepts on hasValidMimeType || hasValidExtension. The OR means a file with a benign .csv name but arbitrary/spoofed MIME passes (and vice-versa) — MIME validation is effectively bypassable by renaming. Size limit (10MB) and S3-key filename sanitization (pms-file-storage.util.ts:10) are correctly in place. Fix: require MIME and extension to agree, or validate parsed content shape after conversion before trusting the file.
  • pms-retry.service.ts:76,96 — [dim 11] When detail.startedAt is missing, deletion windows fall back to Date.now() - 86400_000 (a magic 24h window) and bound hard-deletes of tasks/notifications/agent_results by created_at BETWEEN. A too-wide fallback can delete unrelated rows created in that window. Fix: name the constant, and prefer deleting by explicit run/job linkage (IDs) rather than a time window where possible.
  • Whole scope — [dim 9, systemic] Response shape is non-contract everywhere: success is {success,data} (no error:null) and errors are {success:false, error:"<string>"} plus ad-hoc message/code/data/toastMessage fields (e.g. PmsController.ts:34,160-165,375-377). The nested {code,message,details} error object is never used. Note only — repo-wide ruling. Fix (repo-level): a shared response helper.

Nitpicks

  • [dim 14] Naming inconsistency inside pms-services/: lowercase-dotted pms-*.service.ts (10 files) next to PascalCase PmsFileManagerService.ts; same split in controllers (PmsController.ts vs the dotted services). Pick one convention for the folder. (Per wave-1, the pms-services//pms-utils/ folder layout itself is accepted.)
  • [dim 14, systemic-DB] db import style is itself inconsistent: default import db (retry/approval/data/automation) vs named import { db } (PmsController/upload/PmsFileManagerService) from the same database/connection.
  • [dim 7, systemic] db("table") used outside models/ in pms-data.service (4), pms-approval.service (6), pms-automation.service (2), pms-retry.service (5 — across agent_results/tasks/google_data_store/notifications/pms_jobs). Sharper than generic systemic: approval/data/automation all import PmsJobModel in the same file yet hand-roll raw db("pms_jobs") queries instead of using it. Fix: route these through PmsJobModel; the file-manager service shows it's already possible.
  • [dim 11] pms-paste-analysis.service.ts:17 — model id "claude-haiku-4-5-20251001" hardcoded inline; belongs in config/.
  • [dim 14] Duplicate toNumber: pms-paste-parse.service.ts:427 re-defines the helper already in pms-validator.util.ts:21; the file's own comment admits it "is not the canonical home." Import the shared one.
  • [dim 10, systemic] 59 console.* calls across 10 files (heaviest: PmsController 20, pms-paste-analysis 9, pms-retry 7). Pino not installed repo-wide — note only.
