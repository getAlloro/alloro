Audit Dimensions

Run every file in scope against the 16-point contract (full detail in spec.html).

  • File ≤ 300 lines
  • Functions ≤ 50 lines
  • Nesting ≤ 4 levels
  • Correct folder placement
  • Data flow: Routes → Controllers → Services → Models
  • No business logic in routes
  • No DB queries outside models/
  • Error handling on every async call
  • { success, data, error } response shape
  • No console.log (Pino only)
  • No hardcoded secrets / magic values
  • Input validated at route/middleware
  • Auth middleware on protected routes
  • Naming conventions
  • No commented-out code
  • Transactions for multi-table writes

Findings

Summary

Verdict: needs changes — multiple security must-fixes. This 39-domain bucket is structurally consistent with the rest of the backend (flat response shape, console.* logging, no validation middleware, DB-outside-models — all systemic) but carries the audit's heaviest concentration of auth-gap and disabled-protection findings. Severity counts (unit-specific, systemic excluded): 9 must-fix, 9 concerns, 5 nitpicks. Billing is the cleanest domain (auth + RBAC + Stripe signature verification + transactions + secrets via config/); the public form intake and several admin domains are the weakest.

Auth-gap check

There is no global authenticateToken in src/index.ts — the only global pre-mount middleware is billingGateMiddleware, which is explicitly non-blocking (it skips when no/invalid JWT is present). So auth is per-router, and several routers omit it entirely. Guarded correctly (authenticateToken + superAdminMiddleware via router.use or per-route): admin/organizations, admin/leadgenSubmissions, admin/media, admin/missionControl, admin/pmsPipeline, admin/schedules, admin/support, admin/aiSeoAudit, billing. UNGUARDED admin/internal domains (must-fix): adminAgentInsights (top-level route), admin/agentOutputs, admin/imports, admin/settings, appLogs. Their controllers also contain zero internal req.user/role checks (verified). Unauthenticated proxy/data domains (concern→must-fix): monday (full task CRUD), clarity (analytics fetch), audit (start/read/PATCH by id — only /retry is shared-secret gated). Public-by-design and acceptable: auth / auth-otp / auth-password (login flows), googleauth (OAuth), websiteContact (public form), billing/webhook (Stripe-signed), scraper (key), leadgenTracking (key + limiter), imports (public file serving), documentation.

Must-fix

  • routes/adminAgentInsights.ts, routes/admin/agentOutputs.ts, routes/admin/imports.ts, routes/admin/settings.ts — [dim 13] admin endpoints (incl. destructive DELETE, bulk-delete, file upload, settings writes) mounted with no auth middleware; controllers have no internal guard. Anyone can read/mutate/delete agent_results, agent recommendations, alloro_imports, and app settings. Fix: add router.use(authenticateToken, superAdminMiddleware) to each, matching admin/organizations.
  • routes/appLogs.ts:10,16 — [dim 13] unauthenticated GET / (read server log file) and DELETE / (clear log file). The logs contain leaked OTP/verification codes (see below), so this compounds. Fix: gate with authenticateToken + superAdminMiddleware.
  • routes/auth-password.ts, routes/auth-otp.ts, routes/auth.ts — [dim 12/sec] zero rate-limiting on /login, /register, OTP /request+/verify, /forgot-password, /reset-password, /resend-verification. Brute-force/credential-stuffing on login; email-bomb on the code senders; OTP /verify has no per-email attempt cap so the 6-digit space is brute-forceable in the 10-min window. The express-rate-limit infra exists (middleware/publicRateLimiter.ts) but is only wired to checkup/places/practiceRanking. Fix: apply per-IP + per-email limiters to all auth routes.
  • controllers/auth-otp/feature-services/service.jwt-management.ts:14 — [dim 11] JWT_SECRET falls back to a hardcoded literal "dev-secret-key-change-in-prod". NODE_ENV is uniformly "production" in this repo (per ecosystem config), so a missing env var would silently sign every token with a public string → forgeable tokens. Fix: throw on missing JWT_SECRET instead of defaulting.
  • controllers/auth-password/AuthPasswordController.ts:317 + OTP code path — [dim 11/sec] one-time auth codes are logged in plaintext (console.log(`[AUTH] Verification code for ${email}: ${code}`)). Anyone with log access (now also anyone, via unguarded appLogs) can read live login/reset codes. This is distinct from the systemic console issue — it's logging a secret value. Fix: never log code values; redact.
  • controllers/websiteContact/formSubmissionController.ts:212-388 — [dim 12/15/sec] on the public, unauthenticated, file-accepting /form-submission endpoint, nearly the entire anti-abuse stack is commented out "for debugging": honeypot, timing check, JS challenge, origin validation, content-pattern scoring, IP flood/429 throttle, and duplicate detection. Only post-hoc AI flagging runs. Fix: restore the protections (re-enable the IP throttle at minimum) and delete the dead blocks.
  • routes/websiteContact.ts:31-36 — [dim 12/sec] the route-level formSubmissionLimiter (50/15min) is commented out (// TODO: Re-enable), so the multipart endpoint (up to 10 files × 50MB) has no express rate limit; multer buffers uploads into memory before any app-level flood check runs. Fix: re-enable the limiter.
  • routes/monday.ts — [dim 13] all 7 endpoints (create/update/archive task, comments, board diag) are unauthenticated; controller does no internal auth. Fix: add authenticateToken (+ org scope) to the router.
  • controllers/websiteContact/formSubmissionController.ts:177-196 — [dim 12/sec] S3 file uploads sanitize the filename but enforce no MIME-type allowlist; any file type up to 50MB×10 is accepted and stored. Fix: validate file.mimetype against an allowlist per the file-upload convention.

Concerns

  • controllers/auth-otp/feature-utils/util.test-account.ts:7 — [dim 11/sec] hardcoded tester@google.com bypasses OTP and email-verification in verifyOtp, shipped in production code with no env gate. If that account exists with any role/org, it is a credential-free login. Fix: env-gate or remove.
  • controllers/auth-otp/feature-services/service.otp-generation.ts:12 & admin-organizations/AdminOrganizationsController.ts:898-920 — [dim 11/sec] security tokens use non-cryptographic Math.random(): 6-digit OTP codes and the admin-set temporary password (whose .sort(() => Math.random()-0.5) shuffle is also statistically biased). Fix: crypto.randomInt.
  • controllers/websiteContact/websiteContact-services/recaptchaService.ts:8-10 — [dim 12/sec] reCAPTCHA fails open: if RECAPTCHA_SECRET_KEY is unset it returns true (treats every submission as verified), silently disabling bot protection on env misconfig. Fix: fail-closed (or at least warn) in production.
  • routes/clarity.ts — [dim 13] all 4 endpoints (incl. POST fetch/getKeyData/getAIReadyData) unauthenticated; returns per-project analytics. Repo treats inbound Clarity token as "optional," but project data should not be world-readable. Fix: require auth or a shared key.
  • routes/audit.ts:30-32 — [dim 13] only /retry is shared-secret gated; GET /:auditId and PATCH /:auditId are open → IDOR read/mutate of any audit by id. The partial gate signals an inconsistent model. Fix: gate read-by-id and PATCH with the same tracking key.
  • controllers/googleauth/GoogleAuthController.ts:55-56,92 — [dim 11/sec] OAuth callback returns the Google refresh token (long-lived credential) in the JSON body and embeds tokens into an HTML success page; all googleauth routes are unauthenticated. It is an app-level connector-setup flow (single env refresh token), but echoing the refresh token to a browser page is risky. Fix: auth-gate the routes and stop returning the refresh token to the client.
  • controllers/admin-organizations/AdminOrganizationsController.ts — [dim 1] 1,091-line controller with 22 exported handlers spanning org lifecycle, billing ops, projects, users, and business-data. It does delegate to feature-services/, but the file should be split by sub-domain. Largest single offender in scope. (Auth is correct — super-admin gated.)
  • [dim 9 — systemic] Response shape: only ~6 in-scope files use the canonical {success, data, error} wrapper; ~40 use flat {success: true, ...} and the auth controllers use bare {error} / {token, user}. Tag [systemic]; auth domain is the most divergent.
  • [dim 7 — systemic] DB access outside models/: 24 in-scope files import database/connection directly. Top offenders: leadgen-tracking/LeadgenTrackingController.ts (12 db(), admin-leadgen/AdminLeadgenController.ts (9), billing/BillingService.ts (7, but transaction-wrapped), leadgen-tracking/feature-services/service.email-notification-queue.ts (7). Tag [systemic].

Nitpicks

  • websiteContact-services/recaptchaService.ts:17 — [dim 12] reCAPTCHA secret + attacker-supplied token are string-interpolated into the POST body without URL-encoding; use URLSearchParams.
  • controllers/googleauth/GoogleAuthController.ts:13-18,45-50,89 — [dim 10] verbose OAuth console.log of token metadata (presence booleans, scope, expiry). Stops short of leaking the secret value, but noisy. Tag [systemic] dim-10.
  • [dim 10 — systemic] Pino not installed; console.* is de-facto across 82 in-scope files / 1,008 occurrences. Top: admin-agent-outputs (33), admin-agent-insights (27), auth-password (19), gbp (17), auth/OAuthFlowService (17). Tag [systemic].
  • [dim 12 — systemic] No validation/ schema middleware on any in-scope route; validation is inline in controllers (the auth controllers do reasonable inline checks). Tag [systemic]; flagged specifically for the public/auth/billing surfaces above.
  • Several NODE_ENV === "production" branches (e.g. AdminOrganizationsController.ts:953, BillingService.ts:74, the commented origin check) are effectively dead — NODE_ENV is forced to "production" everywhere in this repo, so the non-prod fallbacks never execute.
