Priority Signals (start here)

  • checkup.ts = 2,140 lines. A route file this size is the single clearest data-flow violation in the backend — business logic living in routes (dimension 6). Scan this first: it's the cheapest confirmation and validates the worksheet format before the big controller domains.
  • admin/websites.ts 907, locations.ts 526, user/website.ts 430 — same smell, smaller. Logic that belongs in controllers/services.
  • For the 58 thin files: spot-check that input validation is applied via middleware from validation/. It likely isn't — see Check 17, where only one validation schema exists for the whole app.

Audit Dimensions

Run every file in scope against the 16-point contract (full detail in spec.html). Dimensions 6 (no business logic in routes) and 12 (validation via middleware) are primary.

  • File ≤ 300 lines
  • Functions ≤ 50 lines
  • Nesting ≤ 4 levels
  • Correct folder placement
  • Data flow: Routes → Controllers → Services → Models
  • No business logic in routes
  • No DB queries outside models/
  • Error handling on every async call
  • { success, data, error } response shape
  • No console.log (Pino only)
  • No hardcoded secrets / magic values
  • Input validated at route/middleware
  • Auth middleware on protected routes
  • Naming conventions
  • No commented-out code
  • Transactions for multi-table writes

Findings

Summary

The unit is sharply bimodal. Most of the ~58 thin files are exemplary: minds.ts and practiceRanking.ts show the codebase already knows the right pattern (router-level .use(authenticateToken, superAdminMiddleware) or per-route middleware chains, then pure path → controller one-liners with no try/catch because the controller owns errors). But two structural failures dominate: (1) checkup.ts (2,140 lines) is a wholesale data-flow inversion — scoring algorithms, economics tables, AI orchestration, multi-table transactions, and ~25 direct db("…") Knex calls all live inside route handlers; and (2) a cluster of admin route files mounted with zero authentication (settings, agent-outputs, app-logs, imports) — confirmed unprotected at the route, mount, and controller layers, and explicitly exempted from the only global gate (billingGate, which is a billing check, not auth). Validation is systemically hand-rolled: the entire app ships ONE schema file (validation/minds.schemas.ts) and not a single route in scope imports a validation middleware (dimension 12 effectively unmet repo-wide). console.* is used instead of Pino in checkup.ts (40×), locations.ts (11×), and auth.ts (1×). Severity counts: must-fix 6 · concern 7 · nitpick 5.

Must-fix

  • admin/settings.ts:6-8 — [dim 13] GET / · GET /:category/:key · PUT /:category/:key have NO auth at any layer. Mounted at /api/admin/settings (index.ts:185); billingGate exempts /api/admin; AdminSettingsController has no auth check. Anyone can read and overwrite arbitrary app settings unauthenticated. Fix: add router.use(authenticateToken, superAdminMiddleware) at the top, mirroring minds.ts:21.
  • appLogs.ts:10,16 — [dim 13] mounted at /api/admin/app-logs (index.ts:178) with no auth: GET / returns raw server log file contents (log/PII exposure) and DELETE / clears logs (destructive). Unauthenticated. Fix: router-level authenticateToken + superAdminMiddleware.
  • admin/agentOutputs.ts:14-24 & admin/imports.ts:18+ — [dim 13] full CRUD (incl. DELETE /:id and authenticated-looking file upload) with no auth middleware and no controller-side guard. Fix: add router-level auth to both, same pattern as minds.ts.
  • admin/websites.ts (whole file) — [dim 13] ~162 of ~196 routes delegate straight to a controller with NO auth; only ~34 spread ...adminWebsiteAuth/...adminGscAuth. AdminWebsitesController performs no auth check and there is no router.use(auth). So list/create/delete projects, all templates, posts, menus, SEO, redirects, AI-command, and integration CRUD are unauthenticated. Fix: hoist auth to router.use(...adminWebsiteAuth) once at the top and drop the per-route spreads (the partial application is the smell that proves the gap).
  • checkup.ts (entire file, 2,140 lines) — [dims 1, 2, 6, 7, 16] the prime offender. The POST /analyze handler alone is ~1,090 lines (153→1245), containing the Business-Clarity scoring engine, a hardcoded specialtyEconomics table (179-214), findings/gap generation, and AI orchestration. ~25 inline Knex calls hit organizations, users, vocabulary_configs, weekly_ranking_snapshots, vendors, checkup_shares, checkup_invitations directly (e.g. 1509-1618 transaction, 1645, 1734, 1895, 1996, 2036, 2112). Fix: extract a controllers/checkup/ domain — CheckupScoringService, CheckupAccountService, and models for every table touched. Routes should shrink to method + path + limiter + controller call. (NOTE: the file's default export checkupRoutes has no consumer in src/ — it appears currently unwired; the violations stand regardless, but verify before investing in the refactor.)
  • checkup.ts:2130-2138 — [dim 8] GET /invite/:token is an async handler with a bare top-level await db(...).first() and NO try/catch (unlike every sibling route). A DB error here is an unhandled rejection. Fix: wrap in try/catch returning a safe redirect, or move to a controller with the standard wrapper.

Concerns

  • checkup.ts (40×), locations.ts (11×: lines 88,138,170,198,246,304,339,399,434,479,517), auth.ts:25 — [dim 10] console.log/error/warn used throughout instead of the Pino logger. Fix: replace with the request-scoped logger; include route + org/user id context.
  • Validation systemically absent — [dim 12] only validation/minds.schemas.ts exists for the whole app and NO in-scope route imports a validation middleware. checkup.ts and locations.ts hand-roll checks (if (!name) … 400, inline email regex at 1307 & 1474, parseInt(req.params.id) NaN guards). Fix: introduce schema-based validation middleware (Zod is already a dependency — AdminWebsitesController imports it) and apply per route.
  • checkup.ts:2050,2120 — [dim 11] hardcoded URL fallbacks process.env.APP_URL || "https://app.getalloro.com" and || "https://getalloro.com" (two different defaults for the same var). Fix: resolve via config/ with one canonical default.
  • checkup.ts:179-214, 1587-1595 — [dim 11] large magic-value maps (specialtyEconomics case values/conversion rates; CATEGORY_TO_VERTICAL) embedded in a route. Fix: move to config/ or a service constant alongside the extracted scoring logic.
  • Response-shape drift — [dim 9] almost no in-scope route returns the contract { success, data, error }. locations.ts returns { success, location, total } / { success, error, message }; checkup.ts returns bespoke top-level keys (score, findings, gaps, token, etc.) and on error { success:false, error:"string" } (error as a bare string, not the {code,message,details} object). Fix: normalize via a response helper; this is a broad but low-risk cleanup.
  • checkup.ts:1890-1985 — [dim 13] PATCH /first-login, /ttfv, /billing-prompt-shown, GET /ttfv-status read req.organizationId but apply NO auth middleware to populate/verify it; they only null-check it. If wired, they depend on an upstream gate that isn't present on this router. Fix: attach authenticateToken (these are per-account, not public like /analyze).
  • checkup.ts:153-1245 — [dim 3] the /analyze handler nests well beyond 4 levels (enrichment if (placeId){ try{ if(details){ if(field){…} } } }, scoring branches, findings loops). Inseparable from the dim-6 refactor but called out for completeness.

Nitpicks

  • agentsV2.ts:29,32 and websiteContact.ts:34 — [dim 15] commented-out route registrations / middleware. Fix: delete; git has history.
  • checkup.ts:37-76 — [dim 6] deriveSpecialtyFromName() (a pure business helper) is defined inside the route file. Belongs in a service/util.
  • checkup.ts:87-93 — [dim 11] an inline rateLimit (emailLimiter) is constructed in the route file while sibling limiters are imported from middleware/publicRateLimiter. Move it there for consistency.
  • locations.ts (526) and admin/websites.ts (907) — [dim 1] over the 300-line cap. admin/websites.ts is otherwise clean routing; splitting into domain sub-routers (templates, posts, integrations, SEO, pages) is the natural fix. locations.ts shrinks naturally once console→Pino and validation move out.
  • checkup.ts:322,325,442,1089 etc. — [dim 14] heavy use of any (enrichedReviews: any[], placeDetails: any, (r: any)). Typing tightens once the scoring logic moves into a typed service.
