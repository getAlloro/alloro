Findings

Summary

Verdict: needs changes. The unit splits cleanly into thin delegating handlers/pure builders (good) and a few god-functions carrying the real orchestration. The headline issue is not "controller holds business logic that belongs in services" — that's mostly already extracted. The real damage is concentrated in three places: (1) four enormous, deeply-nested orchestration functions, (2) multi-table writes with zero transactions across five files (orphan-row risk on partial failure), and (3) the agent endpoints are mounted with no auth and no validation middleware, including a tenant-scoped GET that is an IDOR. types/agent-output-schemas.ts (431 lines) is acceptable — pure type + Zod declarations, no logic. Counts: 4 must-fix · 6 concerns · 6 nitpicks. Coverage: all 13 files read in full.

Must-fix

  • routes/agentsV2.ts (whole file) + AgentsController.ts:1212,1284 — [dim 13 + dim 5/IDOR] The router applies no middleware and there is no global auth gate (verified in src/index.ts:160-175 — only CORS shim, billing-webhook raw parser, json/urlencoded, billingGate run before mounts). Client-facing GETs /latest/:googleAccountId and /getLatestReferralEngineOutput/:googleAccountId read req.organizationId (set only by middleware/rbac.ts, applied per-router) and silently fall back to parseInt(googleAccountId) from the URL when it's undefined — so any unauthenticated caller can read any org's agent outputs by guessing the numeric id. The POST run-triggers (proofline/monthly/ranking/process-all) are also fully open and kick off expensive multi-account LLM + DB-write jobs. Fix: apply rbacMiddleware (+ locationScopeMiddleware for the scoped GETs), matching routes/dashboard.ts; gate the run-trigger POSTs behind admin/role (requireRole) or an internal-scheduler guard. Remove the param fallback once org scoping is guaranteed.
  • feature-services/service.governance-validator.ts:358-419, service.proofline-executor.ts:110-125, AgentsController.ts (processClient via orchestrator) service.agent-orchestrator.ts:1118-1148, service.ranking-executor.ts:159-183 — [dim 16] Multi-table writes with no transaction. Examples: governance inserts agent_results (guardian) → agent_results (governance) → agent_recommendations (3 tables, lines 358/382 + saveRecommendationsFromAgents); proofline & orchestrator processClient insert google_data_store then agent_results back-to-back. If the second write throws, the first is committed → orphaned raw-data rows / half-written governance state. Fix: wrap each related write-set in db.transaction(async (trx) => {...}) and thread trx through. (NB: DB-in-service is itself [systemic] dim-7 — see Concerns — but the missing transaction is unit-specific and corrupting.)
  • feature-services/service.agent-orchestrator.ts:480-897 (processMonthlyAgents ≈417 lines) · :1003-1209 (processClient ≈206) · service.governance-validator.ts:39-439 (runGuardianGovernanceAgents ≈400) · service.ranking-executor.ts:206-353 (processRankingWork ≈147) — [dim 2 + dim 3] Single functions 3-8× over the 50-line cap, each nesting 5+ levels (try → for → try → if-else → for). These are the actual "too much logic in one place" violations, not the controller. Fix: extract per-step named helpers (run-one-agent-with-retry, persist-results, build-RE-input, identify-then-update-ranking) so each stays under cap and ≤4 deep; the retry loop in particular is duplicated verbatim and should be one helper.
  • feature-services/service.webhook-orchestrator.ts:46-71 (callAgentWebhook) — [dim 8] The generic outbound webhook caller has no retry, no idempotency/correlation key, and no failure-mode handling — it does one axios.post (10-min timeout) and re-throws. Idempotency for these external calls lives only as check-then-act dedup in callers (e.g. AgentsController.ts:832, governance-validator.ts:139, orchestrator:1065) with no unique constraint or transaction, so concurrent/duplicate triggers (two schedulers, a retry, a late n8n callback) race and double-insert agent_results/tasks (TOCTOU). Contrast identifyLocationMeta (same file, line 107) which does use runWithRetry. Fix: route callAgentWebhook through runWithRetry with isRetryableExternalError; enforce idempotency with a DB unique index on the dedup tuple (organization_id, agent_type, date_start, date_end) + upsert/transaction instead of read-then-insert.

Concerns

  • [systemic] [dim 7] DB access outside models/: 38 db("table") calls across 8 of 13 files, all importing db from database/connection. Per-file: AgentsController.ts 9, service.agent-orchestrator.ts 7, service.ranking-executor.ts 6, service.governance-validator.ts 6, service.task-creator.ts 5, service.proofline-executor.ts 3, service.recommendation-parser.ts 1, service.ranking-recommendations.ts 1. Tables hit raw: google_connections, organizations, agent_results, agent_recommendations, tasks, google_data_store, practice_rankings, pms_jobs. Models do exist and are used selectively (AgentResultModel, GooglePropertyModel, LocationModel) which makes the raw queries inconsistent drift, not absence of a pattern. Fix (long-term): move queries into the matching models.
  • [dim 5/6] AgentsController.ts handlers are uneven. runProoflineAgent/runRankingAgent/runGuardianGovernance are correctly thin (delegate to executors). But runMonthlyAgents (≈284 lines, 126-409), runMonthlyAgentsTest (≈300, 415-718) and the disabled runGbpOptimizer (≈245, 724-970) still do real work in the controller: account fetch+join, GBP-config filtering, sequential client loop, dedup query, agent_results insert, task creation, notification + admin-email orchestration, post-run task counting. Fix: push each into a service.*-executor mirroring the proofline/ranking split already done; controller should reduce to validate → call executor → shape response.
  • [systemic] [dim 9] Response shape: not one endpoint uses the contract {success, data, error}. All use ad-hoc flat bodies, and on error error is a bare string code ("MONTHLY_AGENTS_ERROR") with a sibling message, not the {code, message, details} object. Payload keys vary per endpoint (processed, summaryId, batches, agents, data, pending…). Fix: standardize on the envelope; nest payloads under data.
  • [dim 11] Repeated magic numbers, no named constants: delay(30000) appears 5× (orchestrator 673/794/1027, governance 236/291), timeout: 600000 (webhook 59), maxTokens literals 16384/65536/4096/1024 (orchestrator 168/256/694, webhook 112), .limit(50) (governance 197/211), inline attempt <= 3 / MAX_*ATTEMPTS = 3 redeclared in 4 spots. Fix: hoist to config/ (e.g. AGENT_RETRY_DELAY_MS, WEBHOOK_TIMEOUT_MS, per-agent MAX_TOKENS, MAX_AGENT_ATTEMPTS).
  • [dim 8] agentLogger.ts:24 uses synchronous fs.appendFileSync on every log() call, invoked hundreds of times per run inside request/await paths — blocks the event loop under concurrency. Fix: async/stream writer, or fold into the proper logger (see dim-10 systemic).
  • [dim 8/4 — minor duplication] The five createTasksFrom* functions in service.task-creator.ts (63/133/202/289/366) are structurally near-identical (build taskData → per-item try/catch db("tasks").insert), as are the Guardian vs Governance retry blocks (governance-validator.ts 230-264 vs 285-321) and the Guardian vs Governance parse blocks (recommendation-parser.ts 32-91 vs 96-168). This is genuine concept-duplication, not coincidental — but per the contract's "premature abstraction is worse" caution, only consolidate the clearly-identical retry/insert mechanics, keep per-agent field mapping separate. Fix: extract a shared insertTask(trx, fields) + a single retry helper.

Nitpicks

  • [dim 1 — acceptable, no action] types/agent-output-schemas.ts (431) is over the 300 cap but is pure declarations (TS interfaces + Zod schemas + z.infer types), zero runtime logic/IO. Per "cap is a code rule, not a type-manifest rule," this is fine. Optional: split into daily.ts/monthly.ts/governance.ts if it keeps growing.
  • [systemic] [dim 10] No Pino (not installed repo-wide). In-scope raw console: agentLogger.ts:25,27 (inside the log/error helper) and service.ranking-recommendations.ts:71 console.warn (the one direct console outside the logger). Everything else funnels through log().
  • [systemic] [dim 12] No input validation on any agent route (0 schemas, no validation/ middleware). Controllers hand-roll a few guards (!googleAccountId, isNaN(organizationId)) but request bodies (referenceDate, month, pmsJobId, locationId) are otherwise trusted untyped. Fix: add route-level schemas.
  • [dim 3] service.agent-input-builder.ts:352-405 buildCopyCompanionPayload nests for → if/if/if → forEach → if (borderline 4-5 deep); also its file header claims "no logging" yet it calls log() 9×. Minor: flatten + fix the header claim.
  • [dim 14 — minor] File naming uses the service.kebab-name.ts convention (e.g. service.agent-orchestrator.ts) rather than the contract's PascalCaseService.ts. It's internally consistent across the unit, so treat as the established local pattern, not a per-file defect — flag only for repo-wide naming reconciliation.
  • [dim 15-adjacent / dead code] feature-utils/dateHelpers.ts:55-65 shouldRunMonthlyAgents guard if (dayOfMonth < 1) return false is dead (a date's day is always ≥1), and MONTH_PMS_DATA_AVAILABLE is a hardcoded true placeholder making the whole function effectively return true. Also buildCopyCompanionPayload/buildOpportunityPayload/buildCroOptimizerPayload and the Opportunity/CRO task creators are retained-but-disabled (kept intentionally per Plan-1 revival path; noted, not flagged).
