Findings

Summary

Verdict: needs changes. Failure-mode handling is the right lens for this unit, and it is uneven. Most processors are well-structured thin dispatchers with sane error propagation and genuinely thoughtful idempotency (scheduler, gbpAutomation, crmPush). But three concrete, data-integrity / reliability defects stand out beyond the systemic noise: (1) websiteRestore performs a destructive wipe-then-restore of ~13 tables + all S3 media with zero transaction wrapping — a mid-job failure or BullMQ retry irrecoverably destroys a project's website; (2) compilePublish's top-level catch marks the run failed but never re-throws, so BullMQ records the job as succeeded and never retries / dead-letters; (3) dataHarvestWorker is omitted from the shutdown() close sequence, so it cannot drain on SIGTERM. Severity counts: 3 must-fix · 5 concerns · 4 nitpicks. Systemic items (console-over-Pino, no payload validation, db() outside models/, flat result shape) are present and tagged but not re-litigated.

Must-fix

  • processors/websiteRestore.processor.ts:566-657 + :163-546 — [dim 16 / dim 8] Destructive multi-table restore with no transaction. wipeProjectData() deletes rows across ~13 tables (posts, assignments, attachments, categories, tags, pages, media, menus, menu_items, header_footer_code, form_submissions, newsletter_signups) and all S3 media, then the body re-inserts across the same ~13 tables (29 write ops). None of it is wrapped in a transaction. Any failure after the wipe — a bad zip entry, an S3 timeout, an unparseable JSON file, or a BullMQ lock-renewal kill mid-run — leaves the project permanently in a wiped/partial state. Because the job re-throws, BullMQ will retry it, re-running the wipe against already-wiped data. Fix: wrap the entire DB restore in db.transaction(async (trx) => …) and route every db("…") write through trx so a failure rolls back to the pre-restore state. The S3 wipe cannot be transactional, so reorder to upload-new-keys-first / delete-old-S3-last (or defer S3 deletion until after DB commit) to keep media recoverable. Set attempts: 1 for the restore worker (currently inherits default) — a non-idempotent destructive job must not auto-retry.
  • processors/compilePublish.processor.ts:180-183 — [dim 8] Catch swallows the failure without re-throwing. Every other processor re-throws after marking failed (see websiteBackup:263, websiteRestore:558, seoBulkGenerate:154, and even this file's own inner catch at :45). This top-level catch calls MindSyncRunModel.markFailed(runId, …) and returns normally, so BullMQ marks the job completed/succeeded — no retry, no failed event, no dead-letter. Compounded by the fact that CREATE_NEW_VERSION → PUBLISH_VERSION → FINALIZE_PROPOSALS is a multi-table write with no transaction (dim 16): a failure between publish and finalize leaves a published version with un-finalized proposals, and that partial state is now silently "successful". Fix: add throw err; as the last line of the catch; ideally wrap steps 5-8 in a transaction so a partial publish rolls back.
  • worker.ts:478-505 — [dim 8] dataHarvestWorker missing from graceful-shutdown sequence. It is declared (:432) and wired into the event-handler loop (:463) but is the only worker absent from shutdown()'s explicit .close() list. On SIGINT/SIGTERM (PM2 restart / deploy) an in-flight daily harvest is killed mid-iteration rather than drained, risking a partial harvest write and a stale retry-count row. Fix: add await dataHarvestWorker.close(); to shutdown(). Root cause is the 24-hand-maintained-.close()-calls pattern — consider iterating the same array used at :463 for both event wiring and shutdown so the lists can't drift again.

Concerns

  • processors/websiteRestore.processor.ts:52-560 — [dim 2] processWebsiteRestore is ~508 lines in a single function; [dim 1] file is 686 lines. The wipe is already extracted; the per-entity restore blocks (media / pages / taxonomy / posts / assignments / menus / code / submissions) should each become a named restoreX(trx, data, urlMap, …) helper. This is also the natural seam for threading the transaction handle. [dim 1] [systemic-adjacent]
  • processors/auditLeadgen.processor.ts:102-619 — [dim 2] processAuditLeadgen is ~515 lines; [dim 1] file is 619. The Branch-A/B/C fan-out closures and the 5-pillar message-builder block (:417-557) are extractable into named helpers. Logic itself is sound (good staged try/catch, failed-state persistence, no db()). [dim 1]
  • processors/websiteRestore.processor.ts & websiteBackup.processor.ts & seoBulkGenerate.processor.ts & websiteGeneration.processor.ts & websiteLayouts.processor.ts & reviewSync.processor.ts — [dim 7] direct db("website_builder.…") access (32 / 8 / 7 / 4 / 1 / 1 occurrences) instead of routing through models/. websiteBackup even comments that it bypasses PageModel/HeaderFooterCodeModel because those models assume a sort_order column the tables lack (:16,:20) — a real model/schema mismatch worth fixing so the model layer is usable. [systemic]
  • All processors — [dim 12] job payloads (job.data) are consumed unvalidated; most do ad-hoc null guards (e.g. gbpAutomation checks workItemId) but there is no schema validation of inbound job data. A malformed enqueue surfaces as a deep runtime error. [systemic]
  • All processors — [dim 9] results are flat / void returns and DB rows, not the { success, data, error } contract. Defensible for workers (no HTTP boundary), but flagged for consistency. [systemic]

Nitpicks

  • processors/compilePublish.processor.ts:12 — [dim 15] import { db } from "../../database/connection" is never used (dead import). Remove it.
  • Every file in scope — [dim 10] console.log/error/warn used throughout (123 calls; worker.ts alone has 30). No Pino logger exists in the repo. Per-job structured logging (jobId, queue, entity IDs) would replace the hand-rolled [TAG] [id] prefix helpers. [systemic]
  • processors/seoBulkGenerate.processor.ts:33-149 — [dim 15] the top-level try { body is indented at the outer level (not nested under the try), making the block boundary hard to read. Cosmetic, but re-indent.
  • worker.ts:500-501 — [nit] closeWbQueues() is awaited before gbpAutomationWorker.close(), so the gbp worker closes after the WB queues rather than alongside the other workers; harmless but inconsistent ordering. Fold all worker closes (via the array) before queue/connection teardown.
