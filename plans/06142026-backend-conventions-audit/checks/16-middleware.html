Findings

Summary

Verdict: the auth guards are sound — the wave-1 route auth-hole fix is validly "apply them at the route level." Each guard authenticates / authorizes correctly server-side: authenticateToken verifies the JWT and rejects 401/403; rbacMiddleware reads the role from organization_users (DB, not client) and requireRole enforces an allowlist with 401/403; superAdminMiddleware checks an env email allowlist; validateScraperKey checks an env key on an isolated route namespace and cannot bypass normal auth; CORS is a strict allowlist with origin reflection (no wildcard). They are correctly exported and already consumed by many routes, so remediation is purely "mount the existing guards on the unprotected admin routes." Severity counts: 2 must-fix · 4 concerns · 4 nitpicks. The two must-fixes are genuine security gaps, not style: a hardcoded JWT-secret fallback that fails OPEN, and the absence of rate limiting on the actual auth attack surface (login / OTP / register / password reset) despite the package being wired elsewhere.

Must-fix

  • auth.ts:13 & billingGate.ts:69 — [dim 11] hardcoded JWT-secret fallback (process.env.JWT_SECRET || "dev-secret-key-change-in-prod", and || "secret" in billingGate). If prod ever boots without JWT_SECRET set, every forged token validates — this is a fail-open auth bypass, not a dev convenience. Same literal repeated in 4 other files (AuthController.ts:15, PilotSessionService.ts:12, security-utils.ts:4, service.jwt-management.ts:14) so the values also drift (billingGate uses "secret" while everyone else uses "dev-secret-key-change-in-prod" — a token signed by one is rejected by the other). Fix: read the secret once from config/, throw at startup if unset (fail-closed), delete every inline fallback; align billingGate to the same source so the verify secret matches the sign secret.
  • publicRateLimiter.ts (whole file) — [dim 12 / security] the limiters are defined but the real auth attack surface has zero rate limiting: routes/auth-otp.ts (login / OTP issue+verify), routes/auth-password.ts (password reset), and routes/auth.ts (register/login) apply no limiter at all. express-rate-limit is wired only to Places/checkup/leadgen/telemetry. This directly violates the SKILL security rule "Failed auth attempts are rate-limited" and "Rate limiting on all public-facing endpoints (login, signup, password reset)." Fix: add a strict auth limiter (e.g. per-IP, low max, with the standard { success:false, error } body already used here) and mount it on the OTP-issue, OTP-verify, register, and password-reset routes.

Concerns

  • publicRateLimiter.ts:49,57,41 — [dim 15-adjacent / dead code] three exported limiters have zero consumers: referralFormLimiter, gbpAuthLimiter, checkupCompareLimiter. Either the referral form / GBP-OAuth / compare endpoints are silently unprotected (the file's own header claims it guards "Referral form abuse" and "GBP OAuth enumeration"), or these are dead exports. Fix: wire them to their intended routes, or delete them; resolve the gap the header advertises.
  • publicRateLimiter.ts:108-116 — [dim 5 / failure-mode] scraper detection uses a module-level in-memory Map + setInterval. Per-instance state: counts don't aggregate across PM2/cluster workers, so the "5 in 5 min" threshold is effectively N× looser under multiple instances, and the setInterval isn't cleared on shutdown. Also mixes a second responsibility (behavioral logging) into a rate-limit file. Fix: note as a known single-instance limitation; if multi-instance, back it with Redis (consistent with the shared-Redis note in repo memory).
  • rbac.ts:42-48 — [dim 13] users with no organization_users row fall through to req.userRole = "viewer" and next() rather than 403. Combined with admin routes that are missing requireRole (the wave-1 hole), an orgless authenticated user reaches handlers as a viewer. The guard itself is correct, but the default-allow posture means "apply the guards" must also include requireRole(...), not just rbacMiddleware. Fix: confirm every protected admin route layers requireRole; consider 403 for orgless on org-scoped routes.
  • scraperAuth.ts:30 — [dim 13] API-key comparison uses plain !== (not constant-time). Low practical risk over the network but it is a secret comparison. Fix: use crypto.timingSafeEqual on equal-length buffers.

Nitpicks

  • [systemic] [dim 9] error responses across this layer are flat ({ error } / { error, message, timestamp } / { success, error } / { success, errorCode, message }) — none use the contract { success, data, error:{ code, message, details } }. Four different ad-hoc shapes within eight files (auth.ts, rbac.ts, tokenRefresh.ts, billingGate.ts, scraperAuth.ts). Tagged systemic; worth one consistent rejection shape.
  • [systemic] [dim 10] 12 console.* calls across 6 files (Pino not installed). tokenRefresh.ts:77,97,109 log on every Google-API request (noisy info-level). Tagged systemic.
  • [systemic] [dim 7] DB access outside models/: raw db() in rbac.ts:37, tokenRefresh.ts:50,91, billingGate.ts:82,92, corsCustomDomains.ts:25, publicRateLimiter.ts:147. Middleware legitimately needs lookups, but these inline queries (org-role lookup, google_connections read/update) belong behind model functions. Tagged systemic.
  • billingGate.ts:101 & rbac.ts:37 — [dim 11] magic string literals "inactive" (subscription status) and the inline role-priority CASE belong in named constants in config/. billingGate.ts:29-43 exempt-prefix list is also a maintenance-sensitive security allowlist that would read better as a named, documented constant.
