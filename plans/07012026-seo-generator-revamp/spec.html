Why

The current SEO meta generator instructs the model to invent unverifiable trust signals ("5-star rated", "[X]+ happy patients" in SeoGeneration.high-impact.md) and superlative claims with no compliance guardrail — both a dental-board advertising risk and a GEO (AI search citability) liability, since AI engines favor specific, sourced facts over superlatives. There is also no answer-first / AI-citability layer at all, and schema generation defaults to generic LocalBusiness instead of the most specific applicable type.

What

Rebuild the SEO generation pipeline so every claim is source-traceable: extract practice_facts from existing business_data/page/post content (never invented), ban fabricated trust signals and superlatives, upgrade schema-type selection to the most specific applicable type, add a GEO/answer-first content layer (direct-answer opening line + sourced FAQ candidates) for pages and posts, and auto-apply generated content through recoverable write paths (new page version row; pre-write content snapshot for posts) so generated changes are never an unrecoverable overwrite. Done when: the 5+ prompt files no longer contain fabrication-inducing instructions, practice_facts extraction runs as a background job with literal source-excerpt provenance on every row, the generation service consumes only sourced facts, and the SEO panel UI surfaces what was extracted/used/auto-applied per page and post.

Constraints

Must

  • Every fact used in generation traces to a literal source_excerpt stored in practice_facts — structurally enforced (extraction worker discards any fact without a verbatim excerpt), not just a prompt instruction.
  • No fabricated superlatives or trust signals in any tier prompt, ever — including the existing high-impact.md CTA copy.
  • Auto-applied page content creates a new page version row (existing createPageVersionQuery path); auto-applied post content snapshots prior body into posts.previous_content before overwrite.
  • SEO panel UI shows, per generation: which facts were used, their source excerpt, and (for auto-applied content) a link/diff to the prior version or snapshot.
  • Schema recommendation selects the most specific applicable type, never a generic fallback when a more specific one is determinable from sourced facts.

Must Not

  • No new state-management library or DB beyond the additive Postgres migration.
  • No change to the existing bulk-generate flow's external contract (generateAllPageSeo/generateAllPostSeo) beyond consuming the new fact-sourcing layer.
  • No per-client content-style rule engine (e.g. Caswell's clinical-format rules) — out of scope, flagged as a follow-on risk below.
  • Do not refactor PageModel/PostModel beyond the methods this spec needs.

Out Of Scope

  • Per-client content-style/voice rule encoding (Caswell-style clinical format) — posts auto-apply uses the generic GEO instructions only this round.
  • A full post version-history system — the snapshot column is a single-step recovery, not a history log.
  • Manual fact-entry UI for Corey/Jo to add facts not present in any existing source — extraction is sourced-only this round.

Risk

Level: 4 — cross-cutting (new DB table, new worker, 5+ prompt files, 2 model files, controller/routes, frontend), auto-writes to live page/post content.

Risks Identified

  • Auto-applying generated content to live pages/posts with no pre-publish human gate. Mitigation: pages use the existing version-row system (recoverable, never overwrites the live row in place); posts get a previous_content snapshot written immediately before overwrite. Confirmed by owner: source-traceable extraction is the trust mechanism, with UI visibility after the fact — not a pre-publish approval queue.
  • Extraction itself could "launder" a stale or wrong fact already sitting in old page copy into a confidently-restated one. Mitigation: every extracted fact must carry a literal, verbatim source_excerpt — the extraction worker discards anything it can't quote back to source. This doesn't catch a source that's wrong at the root, only a source that's invented.
  • Posts have no version-history system, unlike pages — the snapshot column is a single-step undo, not a history log. Mitigation: documented as an accepted limitation this round; flagged as Out of Scope above rather than silently building full post versioning into this spec.
  • GEO/answer-first auto-apply on posts has no per-client content-style awareness (e.g. Caswell's clinical-format rules aren't machine-readable anywhere in the codebase — confirmed, zero hits for any content-style config). Mitigation: flagged as Out of Scope; auto-applied content on posts may not match an individual client's established voice until a follow-on spec encodes those rules.
  • New LLM call (fact extraction) adds AI cost beyond existing SEO generation. Mitigation: uses the existing model-rate pricing in src/services/ai-cost/pricing.ts — no new registration needed, but cost volume should be monitored post-launch since it runs per page/post, not just on-demand.

Blast radius: SeoController, seo.routes.ts, service.seo-generation.ts (touches both the bulk-generate and analyze flows), seoBulkGenerate.processor.ts, worker.ts registration, PageModel, PostModel, all SEO prompt files, frontend SeoPanel + PostTypeSeoButton + api/websites/seo.ts. Consumers: admin website editor users, DFY internal staff generating SEO in bulk, any client site whose pages/posts get auto-applied GEO content.

Pushback

  • Auto-extract + auto-apply (no pre-publish review) was explicitly chosen by the owner over the safer advisory-only / human-confirm alternative. The mitigations above (source-excerpt provenance, recoverable writes) reduce the blast radius of a bad write, but they do not prevent one from going live — if a source itself is wrong, the fact will be marked "sourced" and used anyway. This is an accepted tradeoff, not a closed risk.
  • Combining both Spec A (prompt fixes) and Spec B (data pipeline + auto-apply) into one plan, per owner's choice, makes this a Large/Level-4 spec rather than the smaller, faster-to-ship Spec A alone. Recommend the parallel sub-agent task breakdown below to keep wall-clock reasonable, and treat T1–T3 as the critical path — nothing else can safely execute until source-traceable facts exist.
