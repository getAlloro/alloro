Why

Adding a location today silently bumps the Stripe subscription quantity after the fact — the client never sees or confirms the new amount. Jo is onboarding Garrison's South Orange location (+$2,000/month) on the July 8 call and needs a flow that walks the client through the new total and charges with consent. Garrison (org 5) has an active Stripe subscription, default $2,000 price, quantity billing, 1 location — this is exactly the quantity 1→2 path.

What

The client add-location flow in Settings → Properties becomes: enter name → pick GBP (required; already-linked GBPs blocked) → see a billing summary (location name, per-location cost, current total, new total, prorated charge today) → confirm → Stripe charges the prorated delta on the card on file → only then is the location created. The Billing tab's plan card shows "N locations × $X/mo = $Z/mo". Done when a dev test-mode run completes the full flow on org 41 and the change is live on production before July 8.

Owner decisions (locked 07-02/07-03)

  • GBP stays required (overrides Jo's ticket item "make GBP optional") — and the same GBP cannot be linked twice for an org.
  • Location is created after payment succeeds — no pending-location records.
  • Proration: charge the prorated remainder of the current cycle immediately; full new total from next cycle.
  • Billing tab shows count + per-location price + total; cancel/reopen UI (Phase B) lives in Properties tab.

Risk

Level: 3 — this code moves real client money once merged to main.

Risks Identified

  • (L3) Double-charge or charge-without-location. Mitigation: the Must-column ordering + Stripe idempotency key + compensating revert; vitest on the ordering; dev test-card E2E before merge; charge-success/create-failure logs at error with org/GBP ids (§9.3).
  • (L3) POST /api/locations has four consumers — three are admin surfaces. If the payment gate lands on the shared endpoint, Mission Control adds would demand payment. Mitigation: new client endpoint POST /api/locations/purchase for PropertiesTab; legacy endpoint restricted to platform-admin/impersonation callers (T4 verifies the existing admin/impersonation marker before relying on it — if none exists, restrict via the /api/admin route family instead). Org admins must not be able to bypass payment via the legacy route.
  • (L2) SCA/3DS on the prorated invoice — card-on-file charges can require authentication and fail. Mitigation: treat as PAYMENT_FAILED + portal CTA; acceptable for July 8 (US cards, existing default payment method); note in acceptance run.
  • (L2) Quote drift — locations/price changing between quote and purchase. Mitigation: server recompute at purchase; if totals differ from the client-echoed quote, return QUOTE_STALE and refresh.
  • (L1) One Endo dup-GBP silent-500 becomes a friendly GBP_ALREADY_LINKED 409 — fixed as part of T5.

Blast radius: createLocation() FE callers (PropertiesTab + 3 admin surfaces — T4 keeps admin paths on the legacy endpoint); LocationService.createLocation (gains an option, existing callers unchanged); syncSubscriptionQuantity (untouched, but bypassed by the purchase path); BillingTab (additive section); no schema, no webhook contract changes.

Pushback

  • The ticket's "routed to Stripe to confirm" is implemented as an in-app confirm charging the card on file — Stripe Checkout cannot host a quantity-update confirmation for an existing subscription. The consent requirement (see the new amount before being charged) is met in-app; this is the standard Stripe pattern.
  • Charging the prorated delta immediately is the honest reading of "create after paid," but it means the client's card is hit mid-call on July 8 — Jo should say so out loud during the walkthrough.
