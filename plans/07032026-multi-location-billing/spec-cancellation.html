Why

Clients cannot cancel a single location today — the only paths are hard-deleting the location (destroys the org's data trail; currently exposed to clients in PropertiesTab) or cancelling the whole subscription. The owner wants end-of-period cancellation per location, a reopen window, and a hard rule: nothing is ever deleted — a cancelled location keeps all its data and can always be reopened.

What

Locations gain a status lifecycle: active → pending_cancellation → cancelled, with reopen from either non-active state. Cancelling stops the next invoice from including the location (the current, already-paid period is untouched); the location stays fully usable until the period ends; reopening before then simply restores it. Once cancelled: greyed out in Settings → Properties, gone from the sidebar switcher, all data retained. Cancelling the last active location schedules the whole subscription to cancel at period end (owner decision: all locations are cancellable, no floor guard). Reopening a fully-cancelled location goes through the Phase A payment flow. Client hard-delete is removed.

Owner decisions (locked 07-03)

  • All locations cancellable, including primary and last. Rows never disappear from the Properties list — only marked.
  • Background workflows (rankings, GBP automation, agents, PMS) keep running for cancelled locations for now; gating them is a separate later plan.
  • Cancel/reopen UI lives in Settings → Properties; the Billing tab only carries the count/total summary (Phase A).

Billing mechanics (per direction)

  • Cancel (not last): set pending_cancellation + cancel_effective_at = subscription current_period_end; decrement quantity immediately with proration none — the paid current period is untouched, the next invoice is lower.
  • Cancel (last active): set pending_cancellation; set subscription cancel_at_period_end = true (Stripe quantities cannot go to 0). The existing webhook flow handles the org status when it lapses.
  • Reopen while pending: restore active; increment quantity back with proration none (already paid this period) or clear cancel_at_period_end in the last-location case.
  • Reopen after cancelled: a paid re-add — reuse Phase A's LocationBillingService quote + prorated charge, then restore active.
  • No-sub / flat-override orgs: no Stripe mutation; no period end exists for no-sub orgs → cancellation is immediate (cancelled directly, cancelled_at = now). Flat-override orgs keep their price; status-only change with period-end timing from the sub if one exists, else immediate.
  • Effective quantity rule (single definition): quantity = count of status = 'active' locations. pending_cancellation is excluded because the decrement already happened at cancel time. The Math.max(count, 1) floor is removed from the sync path; the checkout path keeps a ≥1 rule.

Risk

Level: 3 — subscription lifecycle mutations + a production schema change.

Risks Identified

  • (L3) Quantity drift between DB status and Stripe. A cancel that decrements Stripe but fails the DB write (or vice versa) leaves money wrong. Mitigation: DB status transition commits first, Stripe call second, compensating status revert on Stripe failure; sync recount derives from status so a later sync self-heals; every mismatch logged at error.
  • (L3) Removing the Math.max(count, 1) floor changes sync semantics for all orgs. Mitigation: floor removal applies only to the status-aware recount path; checkout keeps ≥1; unit tests cover 0-active (last-location) branching into sub-cancel instead of quantity 0.
  • (L2) Cancelled primary location. is_primary may sit on a cancelled row; the provider falls back (primary → first returned location) so the switcher is safe, but any backend path resolving "the primary location" may pick a cancelled one. Mitigation: accepted per owner ("just mark as cancelled"); audit direct is_primary consumers during execution and list them in the summary; full handling belongs to the workflow-gating plan.
  • (L2) Webhook interplay. Last-location cancel sets cancel_at_period_end; the existing customer.subscription.updated/deleted handlers will move org status — BillingTab then shows the org-level cancelled banner. Mitigation: T8 aligns copy so "your subscription ends on X because your last location was cancelled" reads correctly; reopen-while-pending clears the flag.
  • (L2) Finalizer double-run / missed run. Mitigation: §21.1 idempotent predicate (status = 'pending_cancellation' AND cancel_effective_at <= now()), hourly repeatable job, bounded retries + dead-letter, and the flip is also safe to run manually.
  • (L1) localStorage selected location becomes cancelled — provider fallback already handles a missing id (verified in -b).

Blast radius: every consumer of GET /api/locations (switcher, PropertiesTab, RBAC grant filtering) sees the new default filter; LocationModel.findByOrganizationId/countByOrganizationId callers must be audited for status-awareness (billing sync, RBAC middleware :118–145, admin surfaces); DELETE /api/locations/:id removal affects PropertiesTab only (admin project-identity locations are a separate system, untouched). Downstream location_id tables are untouched — data stays.

Pushback

  • Allowing the last location to cancel the whole subscription via a Properties-tab action is a big lever next to small ones. The confirm dialog must say explicitly: "This is your only active location — cancelling it ends your Alloro subscription on <date>."
  • Keeping workflows running for cancelled locations means Alloro keeps paying compute/API costs for non-paying locations. Deferral accepted, but the follow-up gating plan should come soon after — noted as the explicit next step.
